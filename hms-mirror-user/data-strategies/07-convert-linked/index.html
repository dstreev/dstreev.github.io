    CONVERT_LINKED Strategy¶

    Purpose¶

    Converts tables previously created by the LINKED strategy (pointing to source data) into SCHEMA_ONLY-style tables (pointing to the target namespace). This is the second phase of a two-step migration: LINKED first for immediate access, then CONVERT_LINKED after data has been copied externally (e.g., via distcp).

    The core operation is an ALTER TABLE SET LOCATION that swings the table's storage pointer from the source cluster's namespace to the target cluster's namespace. After conversion, the RIGHT cluster owns its local copy of the data rather than reading from the LEFT cluster remotely.

    How It Works¶

    1. Reads the existing linked table definition from the RIGHT cluster (the table must already exist on RIGHT, having been created by a prior LINKED run)
    2. Validates the table is eligible (not ACID, AVRO schema check passes)
    3. Translates the table location from the source namespace to the target namespace using TranslatorService.translateTableLocation()
    4. Generates ALTER TABLE SET LOCATION SQL pointing to the new target path
    5. If the table was a legacy managed table (detected via HMS_MIRROR_LEGACY_MANAGED_FLAG), sets external.table.purge to true so the target takes data ownership

    Partitioned Table Handling¶

    When CONVERT_LINKED encounters a partitioned table, it takes a different path: it drops the existing RIGHT table and delegates to the SCHEMA_ONLY strategy to rebuild it from scratch. Before dropping, it removes EXTERNAL_TABLE_PURGE if set (to prevent data deletion on drop). This is necessary because partition locations cannot be individually updated via a single ALTER TABLE SET LOCATION.

    Schema Translation Details¶

    Aspect Behavior
    Location Translated from source namespace to target namespace (this is the core operation)
    EXTERNAL_TABLE_PURGE Set to true when converting legacy managed tables (detected via hms-mirror_LegacyManaged flag)
    Table structure Preserved from the existing linked table (non-partitioned) or rebuilt via SCHEMA_ONLY (partitioned)
    AVRO schema URLs Checked via AVROCheck() before proceeding
    Statistics Not explicitly refreshed for non-partitioned conversion

    Data Mapping¶

    • NO data movement -- assumes data was copied externally (e.g., distcp) before this strategy runs
    • The translated location must match where the external copy placed the data
    • After conversion, the RIGHT cluster owns its copy of the data and no longer depends on LEFT cluster storage access
    • If data has not been copied to the target namespace, the table will point to a nonexistent location

    Configuration Parameters¶

    Parameter Effect
    warehouse.external / warehouse.managed Target directories used for location translation
    Translator location maps Custom path mappings for non-standard layouts

    Minimal configuration is required beyond what was set for the original LINKED run. The translator must be configured so that the source locations map correctly to the target namespace.

    Where This Strategy Is NOT Applicable¶

    • ACID tables -- LINKED does not support ACID, so there is nothing to convert. The strategy explicitly checks TableUtils.isACID() and rejects ACID tables.
    • Tables not previously created by LINKED -- If the RIGHT table does not exist, the strategy issues an error: "Table doesn't exist. To transfer, run 'SCHEMA_ONLY'".
    • When data has not been copied -- This strategy does not move data. If the target location is empty, queries against the converted table will return no results.
    • Views and non-native tables -- Only Hive-native external tables created by LINKED are eligible.

    Use Cases¶

    1. Phase 2 of gradual migration -- After LINKED provided immediate query access and data was independently copied via distcp, CONVERT_LINKED updates the table metadata to point to the local copy. This is the designed two-phase migration workflow.
    2. Migration cutover -- Switch linked tables from remote storage to local storage during a planned maintenance window. Run LINKED for all tables, validate access, copy data at your own pace, then run CONVERT_LINKED to finalize.
    3. Hybrid migration schedules -- Different tables may be ready for cutover at different times. CONVERT_LINKED can be run iteratively against subsets of tables as their data copies complete.

    Generated SQL Examples¶

    (a) Non-Partitioned Table: ALTER TABLE SET LOCATION¶

    USE target_db;
ALTER TABLE orders SET LOCATION "s3a://target-bucket/warehouse/external/target_db.db/orders";

    (b) Legacy Managed Table: SET LOCATION + EXTERNAL_TABLE_PURGE¶

    When the table was originally a legacy managed table (detected by the hms-mirror_LegacyManaged property):

    USE target_db;
ALTER TABLE orders SET LOCATION "s3a://target-bucket/warehouse/external/target_db.db/orders";
ALTER TABLE orders SET TBLPROPERTIES ("external.table.purge"="true");

    (c) Partitioned Table: DROP and RECREATE via SCHEMA_ONLY¶

    For partitioned tables, the strategy removes purge protection, drops the existing linked table, and delegates to SCHEMA_ONLY:

    USE target_db;
ALTER TABLE orders_partitioned UNSET TBLPROPERTIES ("external.table.purge");
DROP TABLE IF EXISTS orders_partitioned;
-- (followed by SCHEMA_ONLY generated DDL with translated locations)

    Execution Flow¶

    build()
  │
  ├── Set strategy = CONVERT_LINKED
  ├── Check RIGHT table exists ──► if not: error "run SCHEMA_ONLY"
  ├── Check not ACID ──► if ACID: error "not eligible"
  │
  ├── [Partitioned?]
  │     YES ──► Remove EXTERNAL_TABLE_PURGE
  │             DROP TABLE
  │             Delegate to SchemaOnlyDataStrategy.build()
  │
  │     NO ──► AVROCheck()
  │            Translate location via TranslatorService
  │            ALTER TABLE SET LOCATION
  │            Check HMS_MIRROR_LEGACY_MANAGED_FLAG
  │              └── if set: ADD EXTERNAL_TABLE_PURGE
  │
  └── return success/failure

execute()
  └── tableService.runTableSql(tableMirror, Environment.RIGHT)
