    STORAGE_MIGRATION Strategy¶

    Purpose¶

    Migrates table data within a single cluster (or across clusters) from one storage namespace to another. This handles scenarios like HDFS-to-S3 migration, namespace changes, warehouse directory reorganization, and Ozone migration. Unlike other strategies that copy data between clusters, STORAGE_MIGRATION transforms tables in-place by renaming, recreating, and moving data on the same cluster.

    How It Works¶

    SQL Mode (default)¶

    1. Add HMS_STORAGE_MIGRATION_FLAG property with timestamp to mark the table as being migrated.
    2. Remove the TRANSLATED_TO_EXTERNAL property if present (prevents rename conflicts).
    3. Rename original table to archive name (e.g., orders -> orders_storage_migration).
    4. Create new table with the original name pointing to the new (translated) storage location.
    5. INSERT OVERWRITE new table SELECT * from archive table (data movement via Hive SQL).
    6. Drop archive table after verification (generated as cleanup SQL).

    For partitioned tables, the INSERT uses dynamic partitioning: 

    FROM orders_storage_migration INSERT OVERWRITE TABLE orders PARTITION (region) SELECT *

    Distcp Mode (without archive)¶

    When storageMigration.distcp is enabled and createArchive is false:

    1. Generate ALTER TABLE SET LOCATION for the table pointing to the new storage path.
    2. For partitioned tables, generate ALTER TABLE PARTITION (...) SET LOCATION for each partition individually.
    3. Generate distcp commands for external execution (data movement happens outside of Hive).
    4. No intermediate/archive table is needed; the table metadata is updated in-place.

    Distcp Mode (with archive)¶

    When both storageMigration.distcp and createArchive are enabled:

    1. Rename the original table to the archive name.
    2. Create a new table at the target location.
    3. For partitioned tables, generate ALTER TABLE ADD PARTITION ... LOCATION for each partition with the new location.
    4. Generate distcp commands for data movement.
    5. The archive table preserves access to the original data at the source location.

    All SQL is generated against the LEFT environment only, since STORAGE_MIGRATION operates on a single cluster.

    Schema Translation Details¶

    • Location translation is driven by TranslationTypeEnum:
    • RELATIVE: Only the [protocol]://namespace prefix is changed. The relative path structure from the source is preserved. For example, hdfs://old-cluster/warehouse/db/table becomes s3a://bucket/warehouse/db/table.
    • ALIGNED: Full translation using the system-built location maps. Paths are restructured to align with warehouse directory conventions (e.g., <warehouse>/external/<db.db>/<table>).
    • Warehouse directories (external/managed) determine the target root path based on table type.
    • Partition locations are individually translated (each partition may have a different source path).
    • ACID tables: Location is stripped from the CREATE statement (Hive 3 manages ACID locations internally). If migrateACID.downgrade is set, the table is converted to EXTERNAL+PURGE.
    • HMS_STORAGE_MIGRATION_FLAG property is added with a timestamp to mark migrated tables.
    • Table type is preserved (external stays external, managed stays managed) unless explicitly downgraded.
    • If Iceberg conversion is enabled (icebergConversion.enable), Hive session variables are set to trigger the conversion during table creation.

    Data Mapping¶

    • Data moves within the cluster (or to a new namespace accessible by the same cluster).
    • SQL mode: Data is serialized/deserialized through HiveServer2. This may change the file layout (e.g., rewriting ORC files, adjusting compression). Supports format transformation.
    • Distcp mode: Bit-for-bit file copy that preserves the exact file structure. No format transformation. Requires that partition locations match their partition spec directory structure for correct path translation.
    • ALIGNED mode validates that translated table paths match the warehouse convention (<warehouse>/<db.db>/<table_name>). Mismatches produce warnings or errors.
    • RELATIVE mode is more permissive, accepting non-standard directory layouts as long as the namespace prefix can be replaced.

    Strict Mode Behavior¶

    When storageMigration.strict is true (the default): - Any location mapping issues (mismatched partition paths, paths not aligning with warehouse directories) cause the migration for that table to fail, and all generated SQL is cleared. - This prevents partial or incorrect migrations.

    When strict is false: - Issues are logged as warnings, but migration proceeds. - A STORAGE_MIGRATION_NOT_STRICT_ISSUE error is added to alert the user.

    Configuration Parameters¶

    Parameter Description Default
    transfer.storageMigration.translationType RELATIVE or ALIGNED. Controls how source paths map to target paths. RELATIVE
    transfer.storageMigration.dataMovementStrategy SQL or DISTCP. SQL uses Hive INSERT; DISTCP generates Hadoop distcp commands. SQL
    transfer.storageMigration.dataFlow PULL or PUSH. Direction of distcp execution (which cluster runs the job). PULL
    transfer.storageMigration.strict When true, any location issues abort the table migration. true
    transfer.storageMigration.createArchive When true with DISTCP, creates archive table preserving original data. false
    transfer.storageMigration.consolidateTablesForDistcp When true, consolidates distcp commands across tables in the same database. false
    transfer.storageMigration.skipDatabaseLocationAdjustments When true, database LOCATION/MANAGEDLOCATION properties are not updated. false
    transfer.storageMigrationPostfix Suffix appended to the archive table name. _storage_migration
    transfer.warehouse.external Target warehouse directory for external tables. (required)
    transfer.warehouse.managed Target warehouse directory for managed tables. (required)
    migrateACID.on Include ACID tables in the storage migration. false
    migrateACID.downgrade Convert ACID tables to EXTERNAL+PURGE during migration. false
    migrateACID.partitionLimit Maximum partitions for ACID tables (safety valve). (configurable)
    optimization.overrides Session-level Hive property overrides applied during data movement SQL. (none)
    optimization.sortDynamicPartitionInserts Use sorted dynamic partition inserts for better write performance. false
    icebergConversion.enable Convert target tables to Iceberg format during migration. false

    Where This Strategy Is NOT Applicable¶

    • Tables already in the target namespace: Validated at build time. If the table location already starts with the target namespace + warehouse directory, the migration is skipped with a "Table has already been migrated" message.
    • Cross-cluster migrations where both clusters need separate copies: Use SQL, EXPORT_IMPORT, or HYBRID instead.
    • Schema-only changes: Use SCHEMA_ONLY.
    • Views: Views are metadata-only objects with no storage to migrate.
    • Distcp with non-standard partition layouts: When partition directory names do not match partition spec values (e.g., /table/odd/ for partition=1), distcp mode will fail in strict mode or produce warnings.

    Use Cases¶

    1. HDFS to S3 migration -- Move data from on-premises HDFS to cloud object storage while keeping the same Hive metastore. Use ALIGNED translation to reorganize into standard warehouse layout, or RELATIVE to preserve the existing directory structure.

    2. Namespace consolidation -- Multiple HDFS namespaces merged into one after cluster consolidation. RELATIVE translation replaces only the hdfs://old-nameservice prefix with hdfs://new-nameservice.

    3. Warehouse restructuring -- Reorganize from a flat layout to the Hive 3 aligned warehouse structure with separate external (/warehouse/tablespace/external/hive) and managed (/warehouse/tablespace/managed/hive) directories. ALIGNED translation ensures correct placement.

    4. Ozone migration -- Move from HDFS to Apache Ozone storage within the same cluster. The namespace changes from hdfs:// to ofs:// or o3fs://.

    5. Cloud-to-cloud migration -- Move from one S3 bucket to another (region change, account change). Both SQL and distcp modes work; distcp is preferred for large datasets to avoid serialization overhead.

    6. Distcp-based large table migration -- For tables too large for Hive SQL INSERT (multi-terabyte), enable distcp mode to generate distcp plans. The generated ALTER TABLE SET LOCATION commands update the metadata after external distcp execution completes.

    Generated SQL Examples¶

    (a) SQL-Based Storage Migration (Non-Partitioned Table)¶

    Rename the original table, create a new table at the target location, transfer data via SQL, then drop the archive.

    USE sales_db;

-- Rename original table to archive name
ALTER TABLE orders RENAME TO orders_storage_migration;

-- Create new table at target location
CREATE EXTERNAL TABLE IF NOT EXISTS orders (
    order_id    BIGINT,
    customer_id BIGINT,
    order_date  STRING,
    amount      DECIMAL(10,2)
)
STORED AS ORC
LOCATION 's3a://data-lake/warehouse/external/sales_db.db/orders'
TBLPROPERTIES (
    'HMS_STORAGE_MIGRATION_FLAG'='2026-02-16 10:00:00',
    'EXTERNAL_TABLE_PURGE'='true'
);

-- Transfer data from archive to new table
FROM orders_storage_migration INSERT OVERWRITE TABLE orders SELECT *;

-- Cleanup: Drop archive table (generated as cleanup SQL)
DROP TABLE IF EXISTS orders_storage_migration;

    (b) Distcp-Based Storage Migration (ALTER TABLE SET LOCATION)¶

    When distcp mode is enabled without archive, the table and partition locations are updated in-place. Data is moved externally via distcp.

    USE sales_db;

-- Update table location to new storage namespace
ALTER TABLE orders SET LOCATION "s3a://data-lake/warehouse/external/sales_db.db/orders";

-- Distcp commands generated separately (executed outside Hive):
-- hadoop distcp hdfs://source-cluster/warehouse/sales_db.db/orders s3a://data-lake/warehouse/external/sales_db.db/orders

    (c) ACID Table Storage Migration (SQL Mode with Downgrade)¶

    ACID tables have their location stripped (Hive 3 manages ACID locations). With downgrade enabled, the table is converted to EXTERNAL+PURGE.

    USE finance_db;

-- Rename original ACID table
ALTER TABLE transactions RENAME TO transactions_storage_migration;

-- Create new table (downgraded from ACID to EXTERNAL+PURGE)
CREATE EXTERNAL TABLE IF NOT EXISTS transactions (
    txn_id      BIGINT,
    account_id  BIGINT,
    amount      DECIMAL(10,2),
    txn_date    STRING
)
STORED AS ORC
LOCATION 's3a://data-lake/warehouse/external/finance_db.db/transactions'
TBLPROPERTIES (
    'HMS_STORAGE_MIGRATION_FLAG'='2026-02-16 10:00:00',
    'EXTERNAL_TABLE_PURGE'='true',
    'DOWNGRADED_FROM_ACID'='true'
);

-- Transfer data
SET hive.exec.dynamic.partition.mode=nonstrict;
FROM transactions_storage_migration INSERT OVERWRITE TABLE transactions SELECT *;

-- Cleanup
DROP TABLE IF EXISTS transactions_storage_migration;

    (d) Partitioned Table with Individual Partition Location Updates (Distcp Mode)¶

    Each partition's location is individually translated and updated. This handles cases where partitions may have non-standard locations.

    USE analytics_db;

-- Update table-level location
ALTER TABLE events SET LOCATION "s3a://data-lake/warehouse/external/analytics_db.db/events";

-- Update each partition's location individually
ALTER TABLE events PARTITION (year='2023', month='01')
    SET LOCATION "s3a://data-lake/warehouse/external/analytics_db.db/events/year=2023/month=01";
ALTER TABLE events PARTITION (year='2023', month='02')
    SET LOCATION "s3a://data-lake/warehouse/external/analytics_db.db/events/year=2023/month=02";
ALTER TABLE events PARTITION (year='2024', month='01')
    SET LOCATION "s3a://data-lake/warehouse/external/analytics_db.db/events/year=2024/month=01";
ALTER TABLE events PARTITION (year='2024', month='02')
    SET LOCATION "s3a://data-lake/warehouse/external/analytics_db.db/events/year=2024/month=02";

-- Distcp commands generated separately (one per unique source directory level):
-- hadoop distcp hdfs://source-cluster/warehouse/analytics_db.db/events s3a://data-lake/warehouse/external/analytics_db.db/events

    (e) SQL-Based Storage Migration (Partitioned Table with Dynamic Partitions)¶

    For partitioned tables in SQL mode, dynamic partitioning is used to transfer data efficiently.

    USE analytics_db;

-- Rename original table
ALTER TABLE events RENAME TO events_storage_migration;

-- Create new table at target location
CREATE EXTERNAL TABLE IF NOT EXISTS events (
    event_id    BIGINT,
    user_id     BIGINT,
    event_type  STRING,
    payload     STRING
)
PARTITIONED BY (year STRING, month STRING)
STORED AS ORC
LOCATION 's3a://data-lake/warehouse/external/analytics_db.db/events'
TBLPROPERTIES (
    'HMS_STORAGE_MIGRATION_FLAG'='2026-02-16 10:00:00',
    'EXTERNAL_TABLE_PURGE'='true',
    'DISCOVER_PARTITIONS'='true'
);

-- Set dynamic partition mode
SET hive.optimize.sort.dynamic.partition=true;
SET hive.optimize.sort.dynamic.partition.threshold=0;

-- Transfer data with dynamic partitioning (declarative)
FROM events_storage_migration
INSERT OVERWRITE TABLE events PARTITION (year, month)
SELECT *;

-- Cleanup
DROP TABLE IF EXISTS events_storage_migration;
