    SQL Strategy¶

    Purpose¶

    Migrates table data from source to target using SQL INSERT...SELECT through HiveServer2. Creates an intermediate SHADOW table on the target cluster that points to the source's data, then uses a local INSERT INTO...SELECT to transfer data to the final target table. This keeps all SQL execution on the target cluster.

    How It Works¶

    1. Create SHADOW table on RIGHT -- an external table pointing to LEFT's data location (no translation)
    2. Create final target table on RIGHT -- with translated location in target namespace
    3. Execute INSERT INTO target SELECT * FROM shadow -- data flows through HiveServer2
    4. Drop SHADOW table (cleanup, unless saveWorkingTables=true)

    The SHADOW table is key: it is a lightweight external table on the RIGHT cluster that gives RIGHT's HiveServer2 access to LEFT's data. This means only ONE cluster (RIGHT) executes SQL -- there is no cross-cluster SQL coordination.

    LEFT Cluster                           RIGHT Cluster
+------------------+                   +----------------------------------+
| Source Table     |                   | SHADOW Table                     |
| (original data)  |                   | (external, points to LEFT data)  |
|                  |                   |                                  |
| hdfs://left/     | <-- reads from -- | LOCATION 'hdfs://left/...'       |
| warehouse/db/tbl |                   +----------------------------------+
+------------------+                              |
                                        INSERT INTO...SELECT *
                                                  |
                                                  v
                                       +----------------------------------+
                                       | Target Table                     |
                                       | (translated location)            |
                                       |                                  |
                                       | LOCATION 's3a://target/...'      |
                                       +----------------------------------+

    Routing to Other Strategies¶

    The SQL strategy delegates to other implementations based on configuration:

    • If intermediateStorage or targetNamespace is configured, delegates to IntermediateDataStrategy
    • If the source table is ACID and migrateACID.inplace is set, delegates to SQLAcidInPlaceDataStrategy
    • If the source table is ACID and migrateACID.on is set, delegates to IntermediateDataStrategy

    Schema Translation Details¶

    • SHADOW table: uses LEFT's original location (no translation), marked with HMS_MIRROR_SHADOW_TABLE property
    • Target table: full location translation to target namespace
    • Managed-to-External conversion: applies when migrating from legacy Hive (2.x) to non-legacy Hive (3.x). Managed non-transactional tables become EXTERNAL TABLE with EXTERNAL_TABLE_PURGE=true to preserve data ownership semantics
    • ACID handling: when migrateACID.downgrade is enabled, ACID tables are converted to external. When ACID tables are detected, the strategy routes to IntermediateDataStrategy for proper handling
    • Partition handling for INSERT:
    • Dynamic partitioning is enabled via SET hive.exec.dynamic.partition=true and SET hive.exec.dynamic.partition.mode=nonstrict
    • Three optimization modes controlled by optimization.skip and optimization.sortDynamicPartitionInserts:
      • SKIP (optimization.skip=true): Forces all partitions via declarative INSERT OVERWRITE TABLE ... PARTITION (cols) SELECT *. Sets hive.optimize.sort.dynamic.partition=false
      • SORT_DYNAMIC_PARTITION (optimization.sortDynamicPartitionInserts=true): Enables partition sorting via SET hive.optimize.sort.dynamic.partition=true and SET hive.optimize.sort.dynamic.partition.threshold=0. Uses declarative partition clause
      • PRESCRIPTIVE (default): Uses explicit DISTRIBUTE BY for partition columns via INSERT OVERWRITE TABLE ... PARTITION (cols) SELECT * DISTRIBUTE BY partCols. Sets hive.optimize.sort.dynamic.partition=false and hive.optimize.sort.dynamic.partition.threshold=-1

    Properties Managed¶

    Property Action Purpose
    HMS_MIRROR_METADATA_FLAG Added to RIGHT Timestamp marking migration
    HMS_MIRROR_SHADOW_TABLE Added to SHADOW Identifies shadow tables for cleanup
    EXTERNAL_TABLE_PURGE Added to RIGHT Data ownership on converted managed tables
    DISCOVER_PARTITIONS Added to RIGHT Enables auto partition discovery
    DOWNGRADED_FROM_ACID Added to RIGHT Marks ACID-to-external conversions
    COLUMN_STATS_ACCURATE Removed Stale stats from source
    numFiles, numRows, etc. Removed Stale metrics from source
    transient_lastDdlTime Removed Source timestamp, irrelevant on target

    Data Mapping¶

    • Data flows: LEFT storage -> (read by RIGHT's HiveServer2 via SHADOW) -> RIGHT storage
    • All data passes through RIGHT cluster's compute resources (Tez/MapReduce)
    • Data format may change (e.g., source ORC -> target Parquet if Iceberg conversion is enabled)
    • Non-partitioned tables use FROM shadow INSERT OVERWRITE TABLE target SELECT *
    • Partitioned tables include partition columns in SELECT and map them via INSERT INTO...PARTITION clause
    • MSCK REPAIR TABLE is run on the SHADOW table to register source partitions on RIGHT before the INSERT

    Configuration Parameters¶

    Parameter Default Effect
    hybrid.sqlPartitionLimit 500 Maximum number of partitions before the strategy fails with an error. Prevents OOM/timeout on very large partitioned tables
    optimization.sortDynamicPartitionInserts false Enables hive.optimize.sort.dynamic.partition=true for partition sort optimization
    optimization.skip false Forces declarative partition mode, disabling sort optimization
    optimization.overrides (none) Custom SET statements injected before INSERT. Keyed by environment (LEFT/RIGHT)
    transfer.shadowPrefix hms_mirror_shadow_ Prefix for shadow table name. Table orders becomes hms_mirror_shadow_orders
    saveWorkingTables false When true, keeps shadow table after migration for debugging/validation
    icebergConversion.enable false Converts target to Iceberg format during INSERT. Sets iceberg.mr.schema.auto.conversion=true
    icebergConversion.version 2 Iceberg format version (1 or 2)
    migrateACID.on false Enables ACID table migration (routes to Intermediate strategy)
    migrateACID.downgrade false Enables ACID-to-external conversion
    migrateACID.inplace false Enables in-place ACID downgrade on source cluster
    migrateACID.artificialBucketThreshold 2 Removes CLUSTERED BY if bucket count is at or below this threshold

    Where This Strategy Is NOT Applicable¶

    • Clusters without shared storage access -- RIGHT must be able to read LEFT's data to create the SHADOW table. If RIGHT cannot reach hdfs://left-cluster/..., the SHADOW table is useless
    • Very large tables -- HiveServer2 memory and time constraints make SQL INSERT impractical for multi-TB tables. The entire dataset flows through HiveServer2's JVM
    • ACID tables between incompatible Hive versions -- Legacy ACID (Hive 1/2) uses a different storage format than Hive 3 ACID. Use EXPORT_IMPORT or HYBRID instead
    • Tables exceeding sqlPartitionLimit -- The strategy will abort with an error. HYBRID auto-routes these to EXPORT_IMPORT
    • Network-isolated clusters -- If there is no network path from RIGHT compute nodes to LEFT storage, the INSERT via SHADOW will fail
    • When intermediateStorage is configured -- The strategy automatically delegates to IntermediateDataStrategy, which uses a different transfer mechanism

    Use Cases¶

    1. Small-to-medium table migration -- Tables under a few GB where SQL INSERT through HiveServer2 completes in a reasonable time. The overhead of EXPORT/IMPORT's file copy is unnecessary for small datasets
    2. Format conversion during migration -- Change from ORC to Parquet or convert to Iceberg in a single step. The INSERT...SELECT re-serializes data in the target table's format
    3. Partition consolidation -- Dynamic partitioning can reorganize partition layouts during INSERT. The PRESCRIPTIVE mode with DISTRIBUTE BY gives explicit control over partition distribution
    4. ACID downgrade with data -- Combine ACID-to-external conversion with data movement in one operation. The source ACID table's data is read through the SHADOW table and written as non-transactional data to the target
    5. Development/test environment seeding -- Copy production data via SQL. Use optimization.overrides to inject custom SET statements (e.g., resource limits, queue assignments)

    Generated SQL Examples¶

    (a) Non-Partitioned Table¶

    A simple external table sales.orders being migrated from an HDP cluster to CDP with S3 storage.

    -- Executed on RIGHT cluster
USE sales;

-- Drop any previous shadow table
DROP TABLE IF EXISTS hms_mirror_shadow_orders;

-- Create SHADOW table pointing to LEFT's data
CREATE EXTERNAL TABLE hms_mirror_shadow_orders (
  order_id    BIGINT,
  customer_id BIGINT,
  order_date  STRING,
  total       DECIMAL(10,2)
)
STORED AS ORC
LOCATION 'hdfs://HDP-source/warehouse/sales.db/orders'
TBLPROPERTIES (
  'HMS_MIRROR_SHADOW_TABLE'='true'
);

-- Create final target table with translated location
CREATE EXTERNAL TABLE orders (
  order_id    BIGINT,
  customer_id BIGINT,
  order_date  STRING,
  total       DECIMAL(10,2)
)
STORED AS ORC
LOCATION 's3a://data-lake/warehouse/external/sales.db/orders'
TBLPROPERTIES (
  'HMS_MIRROR_METADATA_FLAG'='2024-03-15 10:30:00',
  'external.table.purge'='true',
  'hmsMirror_LegacyManaged'='true',
  'hmsMirror_Converted'='true'
);

-- Transfer data via SQL INSERT
FROM hms_mirror_shadow_orders INSERT OVERWRITE TABLE orders SELECT *;

-- Cleanup (runs after successful migration)
USE sales;
DROP TABLE IF EXISTS hms_mirror_shadow_orders;

    (b) Partitioned Table with Dynamic Partitions and Sort Optimization¶

    A partitioned table analytics.web_events with 150 partitions, using sortDynamicPartitionInserts.

    -- Executed on RIGHT cluster
USE analytics;

-- Drop any previous shadow table
DROP TABLE IF EXISTS hms_mirror_shadow_web_events;

-- Create SHADOW table pointing to LEFT's data
CREATE EXTERNAL TABLE hms_mirror_shadow_web_events (
  event_id    BIGINT,
  user_id     BIGINT,
  event_type  STRING,
  payload     STRING
)
PARTITIONED BY (event_date STRING, region STRING)
STORED AS ORC
LOCATION 'hdfs://HDP-source/warehouse/analytics.db/web_events'
TBLPROPERTIES (
  'HMS_MIRROR_SHADOW_TABLE'='true'
);

-- Repair partitions on the shadow table so RIGHT can see LEFT's partitions
MSCK REPAIR TABLE hms_mirror_shadow_web_events;

-- Create final target table
CREATE EXTERNAL TABLE web_events (
  event_id    BIGINT,
  user_id     BIGINT,
  event_type  STRING,
  payload     STRING
)
PARTITIONED BY (event_date STRING, region STRING)
STORED AS ORC
LOCATION 's3a://data-lake/warehouse/external/analytics.db/web_events'
TBLPROPERTIES (
  'HMS_MIRROR_METADATA_FLAG'='2024-03-15 10:30:00',
  'external.table.purge'='true',
  'discover.partitions'='true'
);

-- Enable sort optimization for dynamic partitions
SET hive.optimize.sort.dynamic.partition=true;
SET hive.optimize.sort.dynamic.partition.threshold=0;

-- Transfer data with declarative dynamic partitions
FROM hms_mirror_shadow_web_events
INSERT OVERWRITE TABLE web_events PARTITION (event_date, region)
SELECT *;

-- Cleanup
USE analytics;
DROP TABLE IF EXISTS hms_mirror_shadow_web_events;

    (c) ACID Downgrade Scenario¶

    An ACID managed table finance.transactions being downgraded to an external table. Since this is an ACID table, the SQL strategy routes to the IntermediateDataStrategy, which uses a TRANSFER table on the LEFT and a SHADOW table on the RIGHT.

    -- Phase 1: Executed on LEFT cluster
USE finance;

-- Create transfer (external) table from ACID source on LEFT
CREATE EXTERNAL TABLE hms_mirror_transfer_transactions (
  txn_id      BIGINT,
  account_id  BIGINT,
  amount      DECIMAL(12,2),
  txn_type    STRING,
  txn_date    STRING
)
STORED AS ORC
LOCATION 'hdfs://HDP-source/apps/hive/warehouse/export_finance/transactions'
TBLPROPERTIES (
  'HMS_MIRROR_TRANSFER_TABLE'='true',
  'external.table.purge'='true'
);

-- Move ACID data into the external transfer table
SET hive.optimize.sort.dynamic.partition=false;
SET hive.optimize.sort.dynamic.partition.threshold=-1;
FROM transactions INSERT OVERWRITE TABLE hms_mirror_transfer_transactions SELECT *;

-- Phase 2: Executed on RIGHT cluster
USE finance;

-- Create shadow table on RIGHT pointing to the transfer location on LEFT
DROP TABLE IF EXISTS hms_mirror_shadow_transactions;
CREATE EXTERNAL TABLE hms_mirror_shadow_transactions (
  txn_id      BIGINT,
  account_id  BIGINT,
  amount      DECIMAL(12,2),
  txn_type    STRING,
  txn_date    STRING
)
STORED AS ORC
LOCATION 'hdfs://HDP-source/apps/hive/warehouse/export_finance/transactions'
TBLPROPERTIES (
  'HMS_MIRROR_SHADOW_TABLE'='true'
);

-- Create final target as EXTERNAL (downgraded from ACID)
CREATE EXTERNAL TABLE transactions (
  txn_id      BIGINT,
  account_id  BIGINT,
  amount      DECIMAL(12,2),
  txn_type    STRING,
  txn_date    STRING
)
STORED AS ORC
LOCATION 's3a://data-lake/warehouse/external/finance.db/transactions'
TBLPROPERTIES (
  'HMS_MIRROR_METADATA_FLAG'='2024-03-15 10:30:00',
  'external.table.purge'='true',
  'DOWNGRADED_FROM_ACID'='true'
);

-- Transfer data from shadow to target
FROM hms_mirror_shadow_transactions INSERT OVERWRITE TABLE transactions SELECT *;

-- Cleanup on LEFT
USE finance;
DROP TABLE IF EXISTS hms_mirror_transfer_transactions;

-- Cleanup on RIGHT
USE finance;
DROP TABLE IF EXISTS hms_mirror_shadow_transactions;
