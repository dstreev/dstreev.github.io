    HYBRID Strategy¶

    Purpose¶

    Automatically selects between SQL and EXPORT_IMPORT on a per-table basis, optimizing for each table's characteristics. Large partitioned tables route to SQL (which handles dynamic partitioning efficiently), while small/unpartitioned tables and ACID tables route to EXPORT_IMPORT.

    How It Works¶

    For each table, HYBRID evaluates:

    IF table is ACID AND legacy-to-non-legacy migration:
    -> Use INTERMEDIATE (forced, format incompatibility)
ELSE IF table is partitioned AND partition count > exportImportPartitionLimit:
    -> Use SQL (handles large partition counts better)
    -> OR Use INTERMEDIATE if intermediateStorage or target namespace override is set
ELSE:
    -> Use EXPORT_IMPORT (default, handles most cases well)

    The strategy delegates to the actual SQL, EXPORT_IMPORT, or INTERMEDIATE strategy implementation. HYBRID is a router, not its own execution engine. It does not override buildOutDefinition() or buildOutSql() directly; instead, its build() and execute() methods dispatch to the selected delegate strategy.

    Selection Criteria Details¶

    • Partition count is the primary routing metric for non-ACID tables.
    • exportImportPartitionLimit (default: 100) is the threshold. Tables with partition counts at or below this limit use EXPORT_IMPORT. Tables above this limit use SQL.
    • sqlPartitionLimit (default: 500) is the upper safety valve. Tables exceeding this limit are flagged for manual handling or skipped entirely.
    • ACID tables in a legacy-to-non-legacy migration always route to INTERMEDIATE (format incompatibility between Hive 1/2 ACID and Hive 3 ACID requires an intermediate step). If migrateACID is not enabled, ACID tables produce an error.
    • Non-partitioned tables always route to EXPORT_IMPORT (simpler, more reliable for small/medium datasets).
    • When intermediateStorage is configured or a target namespace override is present, tables that would normally route to SQL instead route to INTERMEDIATE (SQL_INTERMEDIATE), because direct cross-namespace SQL is not possible without a bridge.

    Schema Translation Details¶

    Same as the delegated strategy (SQL, EXPORT_IMPORT, or INTERMEDIATE). The HYBRID strategy itself does not modify any translation behavior. Location translation, table property management, and type conversions are all handled by the delegate.

    Data Mapping¶

    Same as the delegated strategy. Data flows depend on whether SQL, EXPORT_IMPORT, or INTERMEDIATE was selected:

    • SQL route: LEFT storage -> SHADOW table on RIGHT (points to LEFT's data) -> INSERT INTO RIGHT table -> RIGHT storage
    • EXPORT_IMPORT route: LEFT storage -> EXPORT to directory -> IMPORT on RIGHT -> RIGHT storage
    • INTERMEDIATE route (ACID or cross-namespace): LEFT storage -> TRANSFER staging table -> INSERT INTO RIGHT table -> RIGHT storage

    Configuration Parameters¶

    Parameter Description Default
    hybrid.exportImportPartitionLimit Partition count threshold. Tables at or below this use EXPORT_IMPORT; above this use SQL. 100
    hybrid.sqlPartitionLimit Maximum partitions for the SQL route. Tables exceeding this are skipped (safety valve). 500
    hybrid.sqlSizeLimit Maximum estimated table size (bytes) for the SQL route. 1073741824 (1 GB)
    intermediateStorage If set, ACID tables and large-partition tables route through INTERMEDIATE strategy. (none)
    migrateACID.on Enables ACID table migration. Without this, ACID tables in HYBRID produce an error. false
    migrateACID.downgrade Convert ACID to EXTERNAL+PURGE during migration. false

    All SQL strategy parameters apply when the SQL route is selected. All EXPORT_IMPORT strategy parameters apply when the EXPORT_IMPORT route is selected. All INTERMEDIATE strategy parameters apply when the INTERMEDIATE route is selected.

    Where This Strategy Is NOT Applicable¶

    • When you need predictable, uniform behavior across all tables (use SQL or EXPORT_IMPORT directly).
    • Single-cluster storage migration (use STORAGE_MIGRATION).
    • Schema-only operations (use SCHEMA_ONLY).
    • When clusters do not share storage access (neither SQL nor EXPORT_IMPORT will work without a shared filesystem or intermediate storage).
    • When you need to migrate views (HYBRID does not handle views).

    Use Cases¶

    1. Production database migration -- Mixed workloads with tables from 1 KB to 1 TB, some ACID, some partitioned. HYBRID routes each table to the optimal strategy automatically.

    2. HDP to CDP migration -- Handles the full spectrum: non-ACID tables with few partitions via EXPORT_IMPORT, non-ACID tables with many partitions via SQL, and legacy ACID tables via INTERMEDIATE.

    3. Hands-off migration -- When you do not want to manually categorize tables by strategy. HYBRID makes reasonable choices based on partition counts and table types.

    4. Mixed partition density -- Databases with both heavily-partitioned (1,000+ partitions) and simple tables benefit from per-table routing rather than a one-size-fits-all strategy.

    Generated SQL Examples¶

    (a) Table Routed to SQL -- Large Partitioned Table¶

    A partitioned table with 250 partitions exceeds the exportImportPartitionLimit of 100, so HYBRID routes it to the SQL strategy.

    -- Routed to SQL: partition count (250) > exportImportPartitionLimit (100)

-- On RIGHT cluster: Create shadow table pointing to LEFT's storage
USE target_db;
CREATE EXTERNAL TABLE IF NOT EXISTS hms_mirror_shadow_orders
LIKE source_db.orders
LOCATION 'hdfs://source-cluster/warehouse/source_db.db/orders';

-- On RIGHT cluster: Create the final target table
CREATE EXTERNAL TABLE IF NOT EXISTS orders (
    order_id    BIGINT,
    customer_id BIGINT,
    order_date  STRING,
    amount      DECIMAL(10,2)
)
PARTITIONED BY (region STRING)
STORED AS ORC
LOCATION 's3a://data-lake/warehouse/external/target_db.db/orders'
TBLPROPERTIES (
    'HMS_MIRROR_METADATA_FLAG'='2026-02-16 10:00:00',
    'EXTERNAL_TABLE_PURGE'='true'
);

-- On RIGHT cluster: Move data via SQL
FROM hms_mirror_shadow_orders
INSERT OVERWRITE TABLE orders PARTITION (region)
SELECT *;

-- Cleanup: Drop shadow table
DROP TABLE IF EXISTS hms_mirror_shadow_orders;

    (b) Table Routed to EXPORT_IMPORT -- Small/Unpartitioned Table¶

    An unpartitioned table (or a table with 50 partitions, which is below the 100 threshold) routes to EXPORT_IMPORT.

    -- Routed to EXPORT_IMPORT: unpartitioned table (always uses EXPORT_IMPORT)

-- On LEFT cluster: Export table data
USE source_db;
EXPORT TABLE customers TO '/apps/hive/warehouse/export_hms_mirror/customers';

-- On RIGHT cluster: Import table data
USE target_db;
IMPORT EXTERNAL TABLE customers
FROM '/apps/hive/warehouse/export_hms_mirror/customers'
LOCATION 's3a://data-lake/warehouse/external/target_db.db/customers';

ALTER TABLE customers SET TBLPROPERTIES (
    'HMS_MIRROR_METADATA_FLAG'='2026-02-16 10:00:00',
    'EXTERNAL_TABLE_PURGE'='true'
);

    (c) ACID Table Routed to INTERMEDIATE (Legacy Migration with Downgrade)¶

    An ACID table in a legacy-to-non-legacy (HDP 2 to CDP) migration is forced to the INTERMEDIATE strategy regardless of partition count. With migrateACID.downgrade enabled, the table is converted from ACID to EXTERNAL+PURGE.

    -- Routed to INTERMEDIATE: ACID table in legacy-to-non-legacy migration
-- Downgrade: ACID -> EXTERNAL + PURGE

-- On LEFT cluster: Create transfer table (external, pointing to intermediate storage)
USE source_db;
CREATE EXTERNAL TABLE IF NOT EXISTS hms_mirror_transfer_transactions (
    txn_id      BIGINT,
    account_id  BIGINT,
    amount      DECIMAL(10,2),
    txn_date    STRING
)
STORED AS ORC
LOCATION 's3a://intermediate-bucket/hms_mirror/source_db/transactions'
TBLPROPERTIES ('HMS_MIRROR_TRANSFER_TABLE'='true');

-- On LEFT cluster: Move data from ACID source to transfer table
FROM transactions
INSERT OVERWRITE TABLE hms_mirror_transfer_transactions
SELECT *;

-- On RIGHT cluster: Create target table (downgraded to external)
USE target_db;
CREATE EXTERNAL TABLE IF NOT EXISTS transactions (
    txn_id      BIGINT,
    account_id  BIGINT,
    amount      DECIMAL(10,2),
    txn_date    STRING
)
STORED AS ORC
LOCATION 's3a://data-lake/warehouse/external/target_db.db/transactions'
TBLPROPERTIES (
    'HMS_MIRROR_METADATA_FLAG'='2026-02-16 10:00:00',
    'EXTERNAL_TABLE_PURGE'='true',
    'DOWNGRADED_FROM_ACID'='true'
);

-- On RIGHT cluster: Load data from intermediate storage
FROM hms_mirror_transfer_transactions
INSERT OVERWRITE TABLE transactions
SELECT *;

-- Cleanup
DROP TABLE IF EXISTS hms_mirror_transfer_transactions;
