    EXPORT_IMPORT Strategy¶

    Purpose¶

    Uses Hive's native EXPORT TABLE / IMPORT TABLE commands to move data between clusters. The EXPORT command writes table data files and a _metadata file to an intermediate storage location, and IMPORT reads it on the target cluster to reconstruct the table. This is the only strategy that natively supports ACID table data movement between compatible Hive versions.

    How It Works¶

    1. Execute EXPORT TABLE on LEFT -- writes data files and _metadata file to export directory
    2. Execute IMPORT EXTERNAL TABLE on RIGHT -- reads from the export directory, creates table with translated location
    3. For ACID downgrade: exports to an archive-suffixed intermediate table first
    LEFT Cluster                        Shared Storage                     RIGHT Cluster
+------------------+               +---------------------+            +------------------+
| Source Table     |   EXPORT TO   | Export Directory     |  IMPORT   | Target Table     |
| (original data)  | -----------> | /{prefix}/{db}/{tbl} | --------> | (translated loc) |
|                  |               |   _metadata          |           |                  |
| hdfs://left/     |               |   data files...      |           | s3a://target/    |
| warehouse/db/tbl |               +---------------------+            | warehouse/db/tbl |
+------------------+                                                  +------------------+

    The export directory path is constructed as: {namespace}{exportBaseDirPrefix}{database}/{table}

    Where {namespace} is determined by: - If intermediateStorage is configured: the intermediate storage URI - If targetNamespace is configured: the target namespace URI - Otherwise: the LEFT cluster's namespace (from the source table's location)

    ACID Table Handling¶

    For ACID tables, Hive's EXPORT natively handles the transactional delta files. The critical constraint is version compatibility: both LEFT and RIGHT must be on the same major Hive version (both legacy or both non-legacy). The strategy explicitly checks this and fails with an error if there is a mismatch:

    Can't process ACID tables with EXPORT_IMPORT between 'legacy' and 'non-legacy'
hive environments. The processes aren't compatible.

    For ACID in-place downgrade, the strategy delegates to ExportImportAcidDowngradeInPlaceDataStrategy, which: 1. Renames the original ACID table to {tableName}_archive 2. Exports the archived table 3. Imports it back as an external table at the original name

    Schema Translation Details¶

    • Export location: determined by exportBaseDirPrefix, intermediateStorage, or targetNamespace configuration
    • Import location: translated to target namespace via the Translator. For non-ACID tables, the IMPORT EXTERNAL TABLE ... LOCATION form is used to explicitly set the target path
    • ACID tables: natively supported when both clusters run compatible Hive versions (both legacy Hive 1/2 or both non-legacy Hive 3+). Imported using IMPORT TABLE (not IMPORT EXTERNAL TABLE) to preserve transactional properties
    • ACID downgrade: exports the original ACID table, imports as EXTERNAL TABLE. The DOWNGRADED_FROM_ACID property is added and EXTERNAL_TABLE_PURGE is set for data ownership
    • External tables: always imported with IMPORT EXTERNAL TABLE ... LOCATION to ensure proper location mapping
    • Partition metadata: preserved through the export/import cycle. The _metadata file captures full partition specifications
    • Table properties: preserved through export, minus stripped statistics (COLUMN_STATS_ACCURATE, numFiles, numRows, rawDataSize, totalSize, transient_lastDdlTime)
    • Managed-to-External conversion: when migrating from legacy to non-legacy Hive, managed non-transactional tables are converted to EXTERNAL TABLE with EXTERNAL_TABLE_PURGE=true
    • Ownership transfer: ALTER TABLE ... SET OWNER USER is executed on the RIGHT if ownership transfer is enabled and the RIGHT cluster is non-legacy Hive

    Location Alignment Checking¶

    When using ALIGNED translation type, the strategy verifies that the target table location falls within the database's configured location. If the table location does not align with the database location, the migration fails with an error:

    Table location {location} is not aligned with the database location
using translation type: ALIGNED

    For RELATIVE translation type, a warning is issued but the migration proceeds.

    Data Mapping¶

    • Data flows: LEFT storage -> export directory -> RIGHT storage
    • Export directory must be accessible by BOTH clusters (shared HDFS namespace, S3 bucket, ADLS container, etc.)
    • File format is preserved exactly (ORC stays ORC, Parquet stays Parquet). No serialization/deserialization occurs
    • ACID delta files are handled natively by Hive's export mechanism -- compacted and written as standard data files in the export directory
    • intermediateStorage enables multi-hop: Source cluster -> cloud bucket -> Target cluster. Both clusters must be able to access the intermediate location independently
    • The remoteWorkingDirectory (default: hms_mirror_working) is appended to intermediate storage paths to isolate migration artifacts

    Configuration Parameters¶

    Parameter Default Effect
    hybrid.exportImportPartitionLimit 100 Maximum number of partitions before the strategy fails. Prevents runaway exports on heavily partitioned tables
    transfer.exportBaseDirPrefix /apps/hive/warehouse/export_ Base directory path for export output. Each table's export is written to {namespace}{prefix}{database}/{table}
    transfer.remoteWorkingDirectory hms_mirror_working Subdirectory name used under intermediate/target storage for temporary artifacts
    transfer.intermediateStorage (none) Fully-qualified URI for multi-hop migrations (e.g., s3a://migration-bucket/staging/). When set, the export directory is written here instead of on the LEFT cluster's filesystem
    migrateACID.on false Enables ACID table migration via EXPORT/IMPORT
    migrateACID.downgrade false Converts ACID tables to EXTERNAL+PURGE during import
    migrateACID.inplace false Performs ACID downgrade on the source cluster itself (renames original, exports, imports back as external)
    warehouse.external (none) Target warehouse directory for external tables. Used for location translation during IMPORT
    warehouse.managed (none) Target warehouse directory for managed tables
    ownershipTransfer.table false When true, runs ALTER TABLE SET OWNER after import to preserve ownership
    forceExternalLocation false Always specifies explicit LOCATION in the IMPORT statement
    createIfNotExists true Adds IF NOT EXISTS semantics to the schema check

    Where This Strategy Is NOT Applicable¶

    • ACID tables between legacy Hive (1/2) and non-legacy Hive (3+) -- The ACID storage formats are incompatible. Hive 2 ACID uses a different delta file structure than Hive 3 ACID. The EXPORT from one version cannot be IMPORT-ed by the other. Use SQL strategy with ACID downgrade instead
    • Clusters with no shared storage for the export directory -- Both clusters must be able to read the export directory. If cluster A writes to hdfs://clusterA/export/ and cluster B cannot reach that path, the IMPORT will fail. Use intermediateStorage (e.g., S3) to bridge the gap
    • Very large tables where export time exceeds acceptable window -- EXPORT copies all data files to the export directory. For multi-TB tables, this can be slow. Consider SCHEMA_ONLY with distcp for very large datasets
    • When format conversion is needed -- EXPORT preserves file format exactly. If you need to convert ORC to Parquet, use the SQL strategy instead
    • When Iceberg conversion is desired -- EXPORT/IMPORT does not support Iceberg table format. Use the SQL strategy with icebergConversion.enable=true
    • Views and non-native tables -- The strategy explicitly rejects views, HBase-backed tables, Kafka tables, and other non-native Hive tables with the error: Can't process ACID tables, VIEWs, or Non Native Hive Tables with this strategy

    Use Cases¶

    1. ACID table migration -- The only native approach for moving transactional table data between compatible Hive clusters. Hive's EXPORT mechanism handles delta files, compaction state, and transaction metadata natively. No other strategy preserves ACID semantics without downgrading
    2. Exact data fidelity -- When you need bit-for-bit identical data files on the target. EXPORT copies data files directly without serialization/deserialization through SQL, eliminating any risk of floating-point drift, encoding changes, or implicit type coercion
    3. Network-isolated clusters with shared cloud storage -- Export to S3 from on-prem cluster A, import from S3 on cluster B. Configure intermediateStorage to point to the shared cloud bucket. The two clusters never need direct network connectivity
    4. Large table migration -- More efficient than SQL for large tables because EXPORT performs a direct file copy rather than row-by-row INSERT...SELECT through HiveServer2. The data does not pass through the JVM heap
    5. Cross-version migration with intermediate storage -- HDP cluster exports to S3 using intermediateStorage, CDP cluster imports from S3. This is the standard pattern for HDP-to-CDP migrations where the clusters have no shared HDFS namespace

    Generated SQL Examples¶

    (a) Basic Non-ACID Table Export/Import¶

    An external table sales.orders being migrated from HDP to CDP. Both clusters can access the LEFT cluster's HDFS namespace.

    -- Executed on LEFT cluster
USE sales;
EXPORT TABLE orders TO "hdfs://HDP-source/apps/hive/warehouse/export_sales/orders";

-- Executed on RIGHT cluster
USE sales;
IMPORT EXTERNAL TABLE orders
  FROM "hdfs://HDP-source/apps/hive/warehouse/export_sales/orders"
  LOCATION "s3a://data-lake/warehouse/external/sales.db/orders";
ALTER TABLE orders SET OWNER USER data_engineering;

    (b) ACID Table Export/Import¶

    A transactional table finance.transactions being migrated between two CDP clusters running compatible Hive 3 versions. The ACID format is preserved.

    -- Executed on LEFT cluster (CDP)
USE finance;
EXPORT TABLE transactions
  TO "hdfs://CDP-source/apps/hive/warehouse/export_finance/transactions";

-- Executed on RIGHT cluster (CDP)
USE finance;
IMPORT TABLE transactions
  FROM "hdfs://CDP-source/apps/hive/warehouse/export_finance/transactions";

    Note: IMPORT TABLE (not IMPORT EXTERNAL TABLE) is used to preserve transactional properties. The ACID delta files in the export directory are read natively by the target cluster's Hive.

    (c) Multi-Hop with intermediateStorage¶

    An external table analytics.web_events being migrated from an on-prem HDP cluster to CDP in the cloud. The clusters cannot see each other's HDFS, but both can access an S3 bucket.

    -- Configuration:
--   intermediateStorage: s3a://migration-bucket/staging/

-- Executed on LEFT cluster (HDP, on-prem)
USE analytics;
EXPORT TABLE web_events
  TO "s3a://migration-bucket/staging/hms_mirror_working/job_20240315/analytics/web_events";

-- Executed on RIGHT cluster (CDP, cloud)
USE analytics;
IMPORT EXTERNAL TABLE web_events
  FROM "s3a://migration-bucket/staging/hms_mirror_working/job_20240315/analytics/web_events"
  LOCATION "s3a://data-lake/warehouse/external/analytics.db/web_events";

    The intermediate S3 path includes the remoteWorkingDirectory (hms_mirror_working), the job key, database name, and table name to prevent collisions between concurrent migration jobs.

    (d) ACID Downgrade via Export/Import¶

    A transactional (ACID) table finance.ledger on HDP (legacy Hive) being migrated to CDP (non-legacy Hive). Since the ACID formats are incompatible across Hive versions, the table is downgraded to EXTERNAL with in-place downgrade on the LEFT.

    -- Executed on LEFT cluster (HDP, legacy Hive)
USE finance;

-- Remove property that would prevent rename
ALTER TABLE ledger UNSET TBLPROPERTIES ("TRANSLATED_TO_EXTERNAL");

-- Rename original ACID table to archive name
ALTER TABLE ledger RENAME TO ledger_archive;

-- Export the archived ACID table
EXPORT TABLE ledger_archive
  TO "hdfs://HDP-source/apps/hive/warehouse/export_finance/ledger_archive";

-- Import back as EXTERNAL table (downgraded)
IMPORT EXTERNAL TABLE ledger
  FROM "hdfs://HDP-source/apps/hive/warehouse/export_finance/ledger_archive";

-- On RIGHT cluster (CDP, non-legacy Hive), if not in-place:
USE finance;

-- Drop existing table if schema sync is active
DROP TABLE IF EXISTS ledger;

IMPORT EXTERNAL TABLE ledger
  FROM "hdfs://HDP-source/apps/hive/warehouse/export_finance/ledger"
  LOCATION "s3a://data-lake/warehouse/external/finance.db/ledger";
ALTER TABLE ledger SET OWNER USER finance_team;

    When migrateACID.inplace=true, both the EXPORT and IMPORT happen on the LEFT cluster -- the original ACID table is renamed, exported, and re-imported as an external table. The RIGHT cluster is not involved.

    When migrateACID.inplace=false (cross-cluster downgrade), the EXPORT happens on LEFT and the IMPORT happens on RIGHT with IMPORT EXTERNAL TABLE to create a non-transactional copy.
