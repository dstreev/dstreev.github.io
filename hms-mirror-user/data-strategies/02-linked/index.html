    LINKED Strategy¶

    Purpose¶

    Creates external tables on the target cluster that point directly to the source cluster's data locations. No namespace translation is performed -- the original storage paths are preserved exactly so that both clusters access the same physical data files. This is a non-destructive way to give the target cluster read access to source data without any data movement or duplication.

    LINKED is primarily intended for testing and validation scenarios: it lets you stand up a new cluster (e.g., CDP) and immediately run queries against production data that lives on the original cluster (e.g., HDP), verifying Hive compatibility before committing to a full migration.

    How It Works¶

    1. Reads table definition from LEFT (source) -- The full CREATE TABLE definition is retrieved from the source cluster's Hive Metastore, including all columns, SerDe, storage format, LOCATION, and TBLPROPERTIES.

    2. Creates EXTERNAL TABLE on RIGHT using LEFT's exact storage locations (no translation) -- The CopySpec is configured with replaceLocation=false, meaning the source LOCATION is preserved as-is. The table on RIGHT will resolve data from the same HDFS/S3/ABFS paths that LEFT uses.

    3. Does NOT set EXTERNAL_TABLE_PURGE -- The CopySpec is configured with takeOwnership=false. Since RIGHT does not own the data, the PURGE flag is never set. This prevents accidental data deletion if the table is dropped on the target cluster.

    4. Generates partition discovery -- After the CREATE TABLE, either explicit ALTER TABLE ADD PARTITION ... LOCATION statements (when partition metadata is loaded) or MSCK REPAIR TABLE (for auto-discovery) is generated. Partition locations are also preserved from the source without translation.

    5. Reuses SchemaOnlyDataStrategy for SQL generation -- The buildOutSql() method delegates to SchemaOnlyDataStrategy.buildOutSql(), which handles the USE, CREATE, DROP, REPLACE, and partition SQL generation. The key difference is in the definition phase: LINKED does not translate locations.

    Schema Translation Details¶

    Locations Are NOT Translated¶

    This is the defining characteristic of the LINKED strategy. Source paths are preserved exactly:

    Source:  hdfs://HDP50/apps/hive/warehouse/sales.db/orders
Target:  hdfs://HDP50/apps/hive/warehouse/sales.db/orders  (same path)

    The RIGHT cluster must have network-level access to the LEFT cluster's storage for this to work. This typically means: - HDFS federation or cross-cluster HDFS access via dfs.nameservices - Cross-account S3 access via IAM role policies or bucket policies - Shared NAS/SAN storage visible to both clusters

    Table Type Forced to EXTERNAL¶

    Regardless of the source table type, the target table is created as EXTERNAL. This is because LINKED tables are read-only references -- they should never modify the underlying data.

    EXTERNAL_TABLE_PURGE Is NOT Set¶

    Since the data is owned by the source cluster, the PURGE flag is explicitly excluded. This prevents a DROP TABLE on the RIGHT cluster from deleting data on the LEFT cluster.

    ACID Tables Are NOT Supported¶

    The LinkedDataStrategy.build() method explicitly checks for ACID tables and rejects them:

    if (TableUtils.isACID(let)) {
    tableMirror.addIssue(Environment.LEFT, "You can't 'LINK' ACID tables.");
    rtn = Boolean.FALSE;
}

    ACID (transactional) tables use internal delta file structures managed by the Hive metastore. These files cannot be read as a simple external table because the ACID reader requires metastore transaction state that is not shared between clusters.

    Views Are NOT Supported¶

    Views reference other tables by name. If the underlying tables do not exist on the RIGHT cluster (or exist under different database names), the view will fail. LINKED does not attempt to resolve or rewrite view SQL.

    Non-Native Tables Are NOT Supported¶

    Tables backed by external systems (HBase, Kafka, JDBC) require those systems to be configured and accessible from the RIGHT cluster. LINKED does not validate or configure these dependencies.

    Statistics Properties Stripped¶

    The same stale properties removed by all strategies are stripped: - COLUMN_STATS_ACCURATE - numFiles, numRows, rawDataSize, totalSize - discover.partitions (removed then re-added based on target cluster config) - transient_lastDdlTime - external, last_modified_by, last_modified_time

    HMS_MIRROR_METADATA_FLAG Added¶

    The hmsMirror_Metadata_Stage1 property is added with the migration timestamp, marking the table as having been created by HMS-Mirror.

    Managed to External Conversion¶

    When migrating from a legacy (Hive 2) source, managed non-ACID tables are converted to EXTERNAL (via copySpec.setUpgrade(true) when convertManaged() returns true). However, unlike SCHEMA_ONLY, the PURGE flag is NOT set because the RIGHT cluster does not own the data.

    Data Mapping¶

    • NO data movement -- Both clusters read the same physical files at the same storage paths.
    • Requires network connectivity -- The RIGHT cluster must be able to access LEFT's storage. This means HDFS cross-cluster access, HDFS federation, shared S3 bucket access, or shared NAS storage.
    • Changes to data on LEFT are immediately visible from RIGHT -- Since both clusters point to the same files, any data modifications (inserts, updates, compactions) on the source are immediately reflected in queries on the target.
    • Write conflicts are possible -- If both clusters attempt to write to the same locations, data corruption can occur. LINKED tables should be treated as read-only on the target.

    Configuration Parameters¶

    Parameter Effect
    createIfNotExists Adds IF NOT EXISTS to CREATE TABLE DDL.
    enableAutoTableStats Generates ANALYZE TABLE after table creation.
    copyAvroSchemaUrls Preserves avro.schema.url and copies Avro schema files.
    ownershipTransfer.table Generates ALTER TABLE SET OWNER for ownership transfer.
    sync Enables synchronization mode: creates missing tables, drops orphaned tables, replaces tables with schema differences.
    readOnly Should be set to true for LINKED. Creates tables without PURGE flag (redundant for LINKED since PURGE is never set, but reinforces the safety intent).
    noPurge Should be set to true for LINKED. Forces no PURGE flag.
    skipLinkCheck Must be false for LINKED. Clusters MUST be linked (storage-accessible) for this strategy to work.
    dbPrefix Prepend a string to database names on the RIGHT cluster (useful for testing to avoid conflicts).
    dbRename Rename a single database on the target.

    Parameters NOT Used¶

    • transfer.warehouse.externalDirectory / transfer.warehouse.managedDirectory -- Not used because locations are not translated.
    • transfer.targetNamespace -- Not used because the source namespace is preserved.
    • transfer.storageMigration.translationType -- Not applicable.
    • migrateACID.* -- ACID tables are rejected by LINKED.
    • migrateVIEW.on -- Views are not supported.
    • migrateNonNative -- Non-native tables are not supported.

    Where This Strategy Is NOT Applicable¶

    • ACID/transactional tables -- ACID tables use internal delta file structures that cannot be read as external tables. The metastore transaction state is cluster-specific.
    • Views -- View SQL references tables by name. If those tables do not exist on the RIGHT cluster or exist under different names, the view will fail.
    • When clusters do not share storage access -- If the RIGHT cluster cannot reach LEFT's HDFS/S3/ABFS, the LINKED tables will fail at query time with file-not-found errors.
    • When data isolation is required -- Both clusters share the same physical data. If the RIGHT cluster could accidentally modify or delete data, LINKED is unsafe.
    • When you want RIGHT to own its own copy of the data -- Use SCHEMA_ONLY (with external distcp) or SQL/HYBRID (with engine-driven data movement) instead.
    • Long-term production use -- LINKED creates a dependency on the source cluster's storage. It is designed for testing and validation, not as a permanent architecture.

    Use Cases¶

    1. Hive 3 testing -- Set up a CDP test cluster that queries HDP production data without copying terabytes of data. Run your existing HiveQL workloads against the CDP engine and identify compatibility issues (deprecated UDFs, behavioral changes, performance regressions) before committing to a full migration.

    2. Gradual migration -- Start with LINKED for immediate access on the target cluster, then later use CONVERT_LINKED to take ownership after the data has been separately copied (via distcp, S3 replication, etc.). This two-phase approach provides immediate value while the data copy runs in the background.

    3. Cross-cluster analytics -- Allow a secondary analytics or reporting cluster to query primary cluster data without duplicating storage. Useful when a dedicated cluster is needed for heavy analytical workloads that should not impact production.

    4. Migration validation -- Create linked tables, run a suite of validation queries, and compare results with the source cluster. This confirms that the target Hive engine produces identical results before switching production workloads.

    Generated SQL Examples¶

    (a) Simple External Table Linked to HDFS Source¶

    Source (LEFT, legacy Hive on hdfs://HDP50): 

    -- Original table on source cluster (legacy managed, non-ACID)
CREATE TABLE `sales.db`.`customers`(
  `c_id` bigint,
  `c_name` varchar(100),
  `c_email` varchar(200),
  `c_signup_date` date)
ROW FORMAT SERDE
  'org.apache.hadoop.hive.ql.io.orc.OrcSerde'
STORED AS INPUTFORMAT
  'org.apache.hadoop.hive.ql.io.orc.OrcInputFormat'
OUTPUTFORMAT
  'org.apache.hadoop.hive.ql.io.orc.OrcOutputFormat'
LOCATION
  'hdfs://HDP50/apps/hive/warehouse/sales.db/customers';

    Generated RIGHT SQL (target CDP cluster, locations NOT translated): 

    USE sales;

CREATE EXTERNAL TABLE `customers`(
  `c_id` bigint,
  `c_name` varchar(100),
  `c_email` varchar(200),
  `c_signup_date` date)
ROW FORMAT SERDE
  'org.apache.hadoop.hive.ql.io.orc.OrcSerde'
STORED AS INPUTFORMAT
  'org.apache.hadoop.hive.ql.io.orc.OrcInputFormat'
OUTPUTFORMAT
  'org.apache.hadoop.hive.ql.io.orc.OrcOutputFormat'
LOCATION
  'hdfs://HDP50/apps/hive/warehouse/sales.db/customers'
TBLPROPERTIES (
  'hmsMirror_Metadata_Stage1'='2026-02-16 10:30:00',
  'hmsMirror_LegacyManaged'='true',
  'hmsMirror_Converted'='true',
  'discover.partitions'='true');

    Note: The LOCATION still points to hdfs://HDP50 (the source cluster). The CDP cluster must have HDFS access to this nameservice. EXTERNAL_TABLE_PURGE is NOT present because the target does not own the data.

    (b) Partitioned Table with MSCK REPAIR¶

    Source (LEFT): partitioned external table on the source cluster.

    Generated RIGHT SQL: 

    USE sales;

CREATE EXTERNAL TABLE `orders`(
  `o_id` bigint,
  `o_customer_id` bigint,
  `o_total` decimal(12,2),
  `o_status` varchar(20))
PARTITIONED BY (
  `o_order_date` string)
ROW FORMAT SERDE
  'org.apache.hadoop.hive.ql.io.orc.OrcSerde'
STORED AS INPUTFORMAT
  'org.apache.hadoop.hive.ql.io.orc.OrcInputFormat'
OUTPUTFORMAT
  'org.apache.hadoop.hive.ql.io.orc.OrcOutputFormat'
LOCATION
  'hdfs://HDP50/apps/hive/warehouse/sales.db/orders'
TBLPROPERTIES (
  'hmsMirror_Metadata_Stage1'='2026-02-16 10:30:00',
  'hmsMirror_LegacyManaged'='true',
  'hmsMirror_Converted'='true',
  'discover.partitions'='true');

MSCK REPAIR TABLE sales.orders;

    Note: The MSCK REPAIR TABLE command discovers all partitions under the source LOCATION by scanning the directory structure for o_order_date=<value>/ subdirectories. This requires the target cluster to have read access to hdfs://HDP50.

    Relationship to CONVERT_LINKED¶

    After using LINKED, the CONVERT_LINKED strategy converts these linked schemas to owned schemas by translating locations from the source namespace to the target namespace. This supports a two-phase migration workflow:

    Phase 1: LINKED                         Phase 2: CONVERT_LINKED
+-------------------------------+       +----------------------------------+
| Create tables on RIGHT        |       | Translate locations to RIGHT     |
| pointing to LEFT's data       |       | namespace                        |
| (immediate read access)       |       | Set PURGE flag where appropriate |
+-------------------------------+       +----------------------------------+
          |                                        |
          v                                        v
  RIGHT queries LEFT data             RIGHT queries its own data copy
  (data copy runs in background)      (source cluster can be retired)

    The CONVERT_LINKED process: - For non-partitioned tables: issues ALTER TABLE SET LOCATION to the translated target path, and adds EXTERNAL_TABLE_PURGE=true if the table was originally a legacy managed table. - For partitioned tables: drops and recreates the table using SCHEMA_ONLY (because each partition location must be individually translated, which cannot be done with ALTER TABLE alone).

    This two-phase approach provides immediate value (queries work on day one via LINKED) while allowing the potentially long-running data copy to proceed independently.

    Safety Warnings¶

    • LINKED is intended for testing and validation only. Tables on the RIGHT cluster point to LEFT cluster data. Dropping tables on RIGHT with EXTERNAL_TABLE_PURGE=true set would delete data on LEFT. LINKED explicitly prevents this by never setting the PURGE flag.
    • Always use readOnly=true and noPurge=true as defense-in-depth when running LINKED migrations.
    • DROP DATABASE CASCADE on RIGHT could still trigger data deletion if any table on RIGHT has PURGE set from a previous non-LINKED operation. Verify table properties before running destructive commands.
    • LINKED tables create a runtime dependency on the source cluster's storage. If the source cluster goes down or the storage becomes inaccessible, all LINKED tables on the target will fail at query time.
