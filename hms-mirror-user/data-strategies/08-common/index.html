    COMMON Strategy¶

    Purpose¶

    Creates table definitions on the target cluster that point to the same storage location as the source cluster. Both clusters share the same physical data. Unlike LINKED (which implies the target is borrowing temporary access to source data), COMMON implies a designed shared-storage architecture where both clusters legitimately access the same storage layer.

    This strategy registers existing tables in a second metastore without moving any data or translating any paths.

    How It Works¶

    1. Reads table definition from LEFT cluster
    2. Validates the table is Hive-native and not ACID
    3. Builds a CopySpec with replaceLocation = false (preserving source paths)
    4. If the target is CDP (non-legacy), upgrades legacy managed tables to EXTERNAL with EXTERNAL_TABLE_PURGE
    5. Creates the table on RIGHT using the same storage location as LEFT
    6. For partitioned tables, generates partition registration SQL (explicit ALTER TABLE ADD PARTITION or MSCK REPAIR TABLE)

    Sync Mode¶

    When sync is enabled, COMMON supports incremental schema management:

    • If LEFT exists and RIGHT does not: CREATE the table on RIGHT
    • If both exist and schemas match: no action (skip with message)
    • If both exist and schemas differ: REPLACE (DROP + CREATE) on RIGHT, unless RIGHT has EXTERNAL_TABLE_PURGE set (which blocks replacement to prevent accidental data deletion)
    • In sync mode, takeOwnership is forced to false to prevent either side from claiming purge rights over shared data

    Non-Sync Mode¶

    Without sync:

    • If RIGHT already exists: no action (skip)
    • If RIGHT does not exist: CREATE on RIGHT

    Schema Translation Details¶

    Aspect Behavior
    Locations NOT translated -- same paths used on both clusters (replaceLocation = false)
    Table type Forced to EXTERNAL (via upgrade path if source is legacy managed)
    EXTERNAL_TABLE_PURGE Set based on takeOwnership logic: true unless readOnly or noPurge is set. Forced false in sync mode.
    ACID properties NOT supported -- ACID tables are rejected before build
    Statistics properties Stripped (standard behavior via CopySpec)
    HMS_MIRROR_METADATA_FLAG Added to mark the table as migrated by HMS-Mirror
    Partition locations Preserved as-is from source (no translation)
    Owner Transferred via ALTER TABLE SET OWNER when ownershipTransfer.table is enabled and target is non-legacy

    Data Mapping¶

    • NO data movement -- both clusters read/write the same physical storage
    • BOTH clusters can potentially modify the data (no isolation)
    • Changes from either cluster are immediately visible to the other
    • The storage layer must support concurrent access from multiple compute engines
    • When takeOwnership is true, RIGHT can delete data on table drop (via EXTERNAL_TABLE_PURGE)

    Configuration Parameters¶

    Parameter Effect
    readOnly If true, disables takeOwnership (RIGHT cannot delete shared data)
    noPurge If true, disables takeOwnership regardless of other settings
    sync Enables incremental schema management (compare and update existing tables)
    createIfNotExists Adds IF NOT EXISTS to DDL
    ownershipTransfer.table If true, transfers table owner from LEFT to RIGHT
    loadMetadataDetails If true, generates explicit partition ADD statements; otherwise uses MSCK REPAIR
    Warehouse settings NOT used -- locations are not translated in COMMON

    Where This Strategy Is NOT Applicable¶

    • ACID/transactional tables -- Concurrent ACID access from multiple metastores is unsafe. The strategy explicitly rejects ACID tables with the message: "Can't use COMMON for ACID tables".
    • When data isolation is required -- Both clusters share the same physical data. Writes from one cluster are immediately visible to (and can conflict with) the other.
    • When clusters use different storage systems -- COMMON requires both clusters to access the same storage namespace (e.g., both reading from the same S3 bucket or HDFS cluster).
    • When ownership semantics must be clear -- With shared data, DROP TABLE on one cluster could delete data needed by the other (when EXTERNAL_TABLE_PURGE is set).
    • Views -- Only Hive-native tables are supported.
    • Non-native tables -- Tables backed by HBase, Kafka, etc. are not handled by this strategy.

    Use Cases¶

    1. Dual-compute on shared cloud storage -- Two CDP clusters both reading/writing the same S3/ABFS data lake. Metastore entries registered on both so that either cluster can query the data.
    2. Parallel Hive metastore instances -- Multiple HMS instances over the same HDFS warehouse (e.g., separate dev/prod metastores on shared storage, or a migration from one metastore version to another on the same storage).
    3. Register existing tables in a new metastore -- When standing up a new cluster with access to existing shared storage, use COMMON to populate the new metastore without copying any data.
    4. Read-only secondary cluster -- With readOnly = true, register table definitions on a secondary cluster for read access without granting purge/delete capabilities.

    Generated SQL Examples¶

    (a) External Table on Shared S3 Storage¶

    USE target_db;
CREATE EXTERNAL TABLE IF NOT EXISTS orders (
    order_id    BIGINT,
    customer_id BIGINT,
    order_date  STRING,
    total       DECIMAL(10,2)
)
STORED AS ORC
LOCATION 's3a://shared-bucket/warehouse/sales.db/orders'
TBLPROPERTIES (
    "external.table.purge"="true",
    "hms-mirror_Metadata_Stage1"="2026-02-16 12:00:00"
);

    (b) Shared Storage with readOnly (No Purge)¶

    USE target_db;
CREATE EXTERNAL TABLE IF NOT EXISTS orders (
    order_id    BIGINT,
    customer_id BIGINT,
    order_date  STRING,
    total       DECIMAL(10,2)
)
STORED AS ORC
LOCATION 's3a://shared-bucket/warehouse/sales.db/orders'
TBLPROPERTIES (
    "hms-mirror_Metadata_Stage1"="2026-02-16 12:00:00"
);

    (c) Partitioned Table with Explicit Partition Locations¶

    USE target_db;
CREATE EXTERNAL TABLE IF NOT EXISTS events (
    event_id  BIGINT,
    event_type STRING
)
PARTITIONED BY (event_date STRING)
STORED AS PARQUET
LOCATION 's3a://shared-bucket/warehouse/analytics.db/events'
TBLPROPERTIES (
    "external.table.purge"="true",
    "hms-mirror_Metadata_Stage1"="2026-02-16 12:00:00"
);

ALTER TABLE events ADD IF NOT EXISTS
  PARTITION (event_date='2025-01-01') LOCATION 's3a://shared-bucket/warehouse/analytics.db/events/event_date=2025-01-01'
  PARTITION (event_date='2025-02-01') LOCATION 's3a://shared-bucket/warehouse/analytics.db/events/event_date=2025-02-01';

    (d) Sync Mode: Schema Replacement¶

    When sync detects a schema mismatch:

    USE target_db;
DROP TABLE IF EXISTS orders;
CREATE EXTERNAL TABLE IF NOT EXISTS orders (
    order_id    BIGINT,
    customer_id BIGINT,
    order_date  STRING,
    total       DECIMAL(10,2),
    status      STRING           -- new column not in previous schema
)
STORED AS ORC
LOCATION 's3a://shared-bucket/warehouse/sales.db/orders'
TBLPROPERTIES (
    "hms-mirror_Metadata_Stage1"="2026-02-16 12:00:00"
);

    Execution Flow¶

    build()
  │
  ├── Set strategy = COMMON
  ├── Check not ACID ──► if ACID: error "Can't transfer SCHEMA reference on COMMON storage"
  │
  ├── buildOutDefinition()
  │     ├── Validate table is Hive-native and not ACID
  │     ├── Create CopySpec(LEFT → RIGHT, replaceLocation=false)
  │     ├── Set upgrade=true if convertManaged
  │     ├── Set takeOwnership based on readOnly/noPurge flags
  │     ├── [Sync mode?]
  │     │     ├── Both exist, schemas match → LEAVE (no action)
  │     │     ├── Both exist, schemas differ → REPLACE (unless PURGE set)
  │     │     ├── LEFT exists, RIGHT missing → CREATE
  │     │     └── Force takeOwnership=false
  │     ├── [Non-sync mode?]
  │     │     ├── RIGHT exists → LEAVE (no action)
  │     │     └── RIGHT missing → CREATE
  │     └── buildTableSchema(copySpec)
  │
  ├── buildOutSql()
  │     ├── Generate USE database
  │     ├── [CreateStrategy]
  │     │     ├── NOTHING/LEAVE → no SQL
  │     │     ├── DROP → DROP TABLE
  │     │     ├── REPLACE → DROP TABLE + CREATE TABLE
  │     │     └── CREATE → CREATE TABLE (+ SET OWNER if configured)
  │     └── [Partitioned?]
  │           ├── loadMetadataDetails → explicit ALTER TABLE ADD PARTITION
  │           └── partitionDiscoveryInitMSCK → MSCK REPAIR TABLE
  │
  └── return success/failure

execute()
  └── tableService.runTableSql(tableMirror, Environment.RIGHT)
