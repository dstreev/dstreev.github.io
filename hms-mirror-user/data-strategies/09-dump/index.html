    DUMP Strategy¶

    Purpose¶

    Extracts DDL (CREATE TABLE) statements from the source cluster without executing anything on any cluster. Produces SQL scripts for review, archival, or manual execution. This is a read-only, zero-risk strategy -- it only reads metadata and writes output files.

    DUMP is the simplest strategy in HMS-Mirror. It requires minimal configuration and makes no changes to either cluster.

    How It Works¶

    1. Reads table definitions from the LEFT cluster
    2. Standardizes the definition (strips database prefix from table names)
    3. For non-legacy ACID tables, strips the LOCATION property (since ACID location is managed by the metastore)
    4. Generates CREATE TABLE DDL and writes it to the LEFT environment's SQL output
    5. For partitioned non-ACID tables, generates partition registration SQL
    6. The execute() method is a no-op -- it returns true without executing any SQL

    Key Distinction¶

    Unlike all other strategies, DUMP writes SQL to the LEFT environment's output (not RIGHT). There is no target cluster -- the output is purely for documentation or manual use.

    Schema Translation Details¶

    Aspect Behavior
    Locations Output as-is from the source cluster (no translation)
    Table properties Included as-is
    ACID properties Included as-is; LOCATION stripped for non-legacy ACID tables
    Database prefix Stripped from table names in CREATE statements
    Partition metadata Included if loaded (explicit ALTER TABLE ADD PARTITION) or via MSCK REPAIR TABLE
    Views DDL included (when migrateVIEW is enabled)
    Non-native tables DDL included (when migrateNonNative is enabled)
    Owner Transferred via ALTER TABLE SET OWNER when ownershipTransfer.table is enabled and source is non-legacy

    Data Mapping¶

    • NO data movement
    • NO schema changes on any cluster
    • NO SQL execution -- the execute() method is a no-op
    • Read-only metadata extraction written to output report files

    Configuration Parameters¶

    Parameter Effect
    migrateVIEW Include view DDL in output
    migrateNonNative Include non-native table DDL (HBase, Kafka-backed tables)
    ownershipTransfer.table Include ALTER TABLE SET OWNER in output
    loadMetadataDetails Load partition metadata for explicit partition DDL
    transfer.storageMigration.distcp When true, partition repair SQL is placed in cleanup SQL section

    This is the lowest-configuration strategy -- it only needs a connection to the source cluster (LEFT). No RIGHT cluster, warehouse directories, or translator configuration is required.

    Where This Strategy Is NOT Applicable¶

    • When you need actual migration -- DUMP does not create tables or move data on any cluster.
    • When you need location translation -- DUMP preserves source paths exactly as they are. Use SCHEMA_ONLY or other strategies for path translation.
    • When you need automated execution -- DUMP generates scripts but does not run them. It is a reporting tool, not a migration tool.

    Use Cases¶

    1. Schema documentation -- Extract all table definitions for documentation, compliance audits, or governance inventories.
    2. Pre-migration analysis -- Review what tables exist, their types, locations, properties, and partition structures before planning a migration strategy.
    3. Disaster recovery planning -- Archive DDL scripts as part of a backup strategy. In case of metastore corruption, the DDL output can be used to recreate table definitions.
    4. Manual migration -- Generate scripts for teams that prefer to review, modify, and execute SQL manually rather than using automated migration.
    5. Table inventory -- Catalog all tables, their types, storage formats, locations, and properties across a cluster.
    6. Migration validation -- Compare DUMP output before and after a migration to verify that table definitions were correctly transferred.

    Generated SQL Examples¶

    (a) Simple External Table¶

    USE sales_db;
CREATE EXTERNAL TABLE IF NOT EXISTS orders (
    order_id    BIGINT,
    customer_id BIGINT,
    order_date  STRING,
    total       DECIMAL(10,2)
)
STORED AS ORC
LOCATION 'hdfs://source-cluster/warehouse/sales_db.db/orders';

    (b) ACID Table (Non-Legacy -- Location Stripped)¶

    For non-legacy (Hive 3) ACID tables, the LOCATION clause is removed since it is managed by the metastore:

    USE inventory_db;
CREATE TABLE IF NOT EXISTS stock_levels (
    product_id   BIGINT,
    warehouse_id BIGINT,
    quantity     INT,
    updated_at   TIMESTAMP
)
CLUSTERED BY (product_id) INTO 8 BUCKETS
STORED AS ORC
TBLPROPERTIES (
    "transactional"="true",
    "transactional_properties"="default"
);

    (c) View¶

    USE sales_db;
CREATE VIEW IF NOT EXISTS active_orders AS
SELECT order_id, customer_id, order_date, total
FROM orders
WHERE status = 'ACTIVE';

    (d) Partitioned Table with Explicit Partition Metadata¶

    When loadMetadataDetails is enabled:

    USE analytics_db;
CREATE EXTERNAL TABLE IF NOT EXISTS events (
    event_id   BIGINT,
    event_type STRING,
    payload    STRING
)
PARTITIONED BY (event_date STRING)
STORED AS PARQUET
LOCATION 'hdfs://source-cluster/warehouse/analytics_db.db/events';

ALTER TABLE events ADD IF NOT EXISTS
  PARTITION (event_date='2025-01-01') LOCATION 'hdfs://source-cluster/warehouse/analytics_db.db/events/event_date=2025-01-01'
  PARTITION (event_date='2025-02-01') LOCATION 'hdfs://source-cluster/warehouse/analytics_db.db/events/event_date=2025-02-01';

    (e) Partitioned Table with MSCK REPAIR¶

    When partitionDiscoveryInitMSCK is enabled instead of loading metadata details:

    USE analytics_db;
CREATE EXTERNAL TABLE IF NOT EXISTS events (
    event_id   BIGINT,
    event_type STRING,
    payload    STRING
)
PARTITIONED BY (event_date STRING)
STORED AS PARQUET
LOCATION 'hdfs://source-cluster/warehouse/analytics_db.db/events';

MSCK REPAIR TABLE events;

    (f) Table with Owner Transfer¶

    When ownershipTransfer.table is enabled:

    USE sales_db;
CREATE EXTERNAL TABLE IF NOT EXISTS orders (
    order_id    BIGINT,
    customer_id BIGINT,
    order_date  STRING,
    total       DECIMAL(10,2)
)
STORED AS ORC
LOCATION 'hdfs://source-cluster/warehouse/sales_db.db/orders';

ALTER TABLE orders SET OWNER USER data_engineering;

    Execution Flow¶

    build()
  │
  ├── Set strategy = DUMP
  │
  ├── buildOutDefinition()
  │     ├── Strip database prefix from table name
  │     └── [Non-legacy ACID?] Strip LOCATION property
  │
  └── buildOutSql()
        ├── Generate USE database
        ├── Generate CREATE TABLE statement (written to LEFT SQL)
        ├── [ownershipTransfer?] Generate ALTER TABLE SET OWNER
        └── [Partitioned and non-ACID?]
              ├── [partitionDiscoveryInitMSCK] → MSCK REPAIR TABLE
              └── [loadMetadataDetails] → ALTER TABLE ADD PARTITION statements

execute()
  └── return true (no-op)
