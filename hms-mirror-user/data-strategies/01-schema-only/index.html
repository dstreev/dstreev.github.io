    SCHEMA_ONLY Strategy¶

    Purpose¶

    Transfers table schema (DDL) from source to target cluster without moving data. Storage locations are translated from the source namespace to the target namespace so that table definitions on the target cluster point to the correct paths. Data is assumed to be moved externally via tools such as distcp, S3 sync, cloud replication, or similar mechanisms. HMS-Mirror also generates distcp plans alongside the schema migration to facilitate this external data copy.

    How It Works¶

    1. Reads table definition from LEFT (source) -- The full CREATE TABLE definition, including columns, SerDe, storage format, LOCATION, and TBLPROPERTIES, is retrieved from the source cluster's Hive Metastore.

    2. Translates storage locations from source namespace to target namespace -- The TranslatorService replaces the source HDFS/S3/ABFS namespace with the target namespace. Translation uses ordered location maps (auto-generated from warehouse configuration and/or user-provided custom mappings), with longest-path-match-wins semantics.

    3. Converts table types if needed (managed non-ACID to EXTERNAL + PURGE on Hive 3 upgrade) -- When migrating from a legacy Hive (1/2) environment to a non-legacy (Hive 3+) environment and convertManaged() returns true, managed non-transactional tables are converted to EXTERNAL TABLE with EXTERNAL_TABLE_PURGE=true, preserving data ownership semantics.

    4. Strips ACID properties if migrating transactional tables with downgrade -- When migrateACID.downgrade=true, transactional properties (transactional, transactional_properties, BUCKETING_VERSION) are removed, the table is made EXTERNAL with PURGE, and the DOWNGRADED_FROM_ACID property is set. The LOCATION is also stripped because Hive 3 manages ACID table locations internally.

    5. Generates CREATE TABLE DDL for RIGHT (target) -- The buildOutSql() method produces USE <database> and CREATE TABLE (or CREATE EXTERNAL TABLE) statements targeting the RIGHT cluster, incorporating all translated locations, converted types, and adjusted properties.

    6. Generates partition discovery statements -- If partition metadata was loaded (loadMetadataDetails=true), explicit ALTER TABLE ADD PARTITION ... LOCATION statements are generated with individually translated partition paths. Otherwise, if the RIGHT cluster supports auto-discovery via MSCK, a MSCK REPAIR TABLE statement is generated.

    7. Optionally generates ALTER TABLE SET OWNER -- When ownershipTransfer.table=true and the target cluster is non-legacy, an ALTER TABLE SET OWNER statement is generated to transfer the table owner from the source.

    Schema Translation Details¶

    Table LOCATION¶

    The source namespace is replaced with the target namespace. The translation mode determines how the path is constructed:

    • RELATIVE -- Simple namespace swap. The path suffix after the namespace is preserved as-is. 

      hdfs://hdp-source/apps/hive/warehouse/sales.db/orders
-->  s3a://data-lake-bucket/apps/hive/warehouse/sales.db/orders

    • ALIGNED -- Warehouse plans drive the target path structure, potentially reorganizing the directory layout. 

      hdfs://hdp-source/apps/hive/warehouse/sales.db/orders
-->  s3a://data-lake-bucket/warehouse/external/sales.db/orders

    Partition Locations¶

    Each partition path is individually translated through TranslatorService.translatePartitionLocations(). This is critical when partitions have non-standard locations that differ from the table's base path.

    Managed to External Conversion¶

    On Hive 2 to Hive 3 upgrade (legacy to non-legacy), non-ACID managed tables become EXTERNAL with EXTERNAL_TABLE_PURGE=true:

    Hive 2 (Legacy)                    Hive 3 (CDP)
+-------------------+              +-----------------------------+
| MANAGED TABLE     |   -------->  | EXTERNAL TABLE              |
| (non-ACID)        |              | + EXTERNAL_TABLE_PURGE=true |
+-------------------+              +-----------------------------+

    Additional properties set during conversion: - hmsMirror_LegacyManaged=true -- marks that this table was originally a legacy managed table - hmsMirror_Converted=true -- marks that the table type was converted

    ACID Handling¶

    When migrateACID.on=true and migrateACID.downgrade=true:

    • transactional property removed
    • transactional_properties property removed
    • BUCKETING_VERSION property removed
    • Table made EXTERNAL
    • EXTERNAL_TABLE_PURGE=true set (unless noPurge=true)
    • DOWNGRADED_FROM_ACID=true set
    • LOCATION stripped (Hive 3 manages ACID locations internally; the downgraded table uses the database default external location)
    • Artificial buckets removed if bucket count is at or below artificialBucketThreshold

    Properties Stripped¶

    The following stale or irrelevant properties are removed from the target definition:

    • COLUMN_STATS_ACCURATE
    • numFiles
    • numRows
    • rawDataSize
    • totalSize
    • discover.partitions (removed then re-added based on target cluster configuration)
    • transient_lastDdlTime
    • external (controlled by CREATE statement, not TBLPROPERTIES)
    • last_modified_by
    • last_modified_time

    Properties Added¶

    • hmsMirror_Metadata_Stage1 -- timestamp of when the schema migration was performed
    • discover.partitions=true -- set for external/converted partitioned tables when the target cluster has auto partition discovery enabled
    • EXTERNAL_TABLE_PURGE=true -- set when the target should own the data (upgrade conversion, or explicit ownership)

    Iceberg Conversion¶

    When icebergConversion.enable=true, the target table definition is converted to Iceberg format: - Non-Iceberg storage constructs are stripped from the definition - format-version property is set to the configured Iceberg version (1 or 2)

    Data Mapping¶

    • NO data movement -- This strategy only handles metadata (schema definitions and metastore entries).
    • Assumes data is moved externally and arrives at the translated location paths before queries are run against the target tables.
    • The generated distcp plans (always produced by SCHEMA_ONLY) provide a mapping of source paths to target paths that should be used for the external data copy.
    • The generated locations must match where the external data copy places the files. If the data lands at a different path than the translated LOCATION, the table will not resolve its data.

    Configuration Parameters That Affect This Strategy¶

    Parameter Effect
    transfer.warehouse.externalDirectory Target warehouse base path for external tables. Drives location translation when using ALIGNED mode.
    transfer.warehouse.managedDirectory Target warehouse base path for managed tables. Drives location translation for ACID tables (before downgrade strips LOCATION).
    transfer.targetNamespace Target storage namespace (e.g., hdfs://new-cluster or s3a://bucket).
    transfer.storageMigration.translationType RELATIVE (simple namespace swap) or ALIGNED (warehouse plan-driven layout).
    migrateACID.on Include ACID tables in the migration. Without this, ACID tables are skipped.
    migrateACID.downgrade Convert ACID tables to EXTERNAL+PURGE by stripping transactional properties.
    migrateACID.artificialBucketThreshold Remove CLUSTERED BY ... INTO n BUCKETS if n is at or below this value (default: 2). Cleans up legacy artificial bucketing.
    migrateACID.partitionLimit Skip ACID tables with more partitions than this limit (default: 500).
    migrateVIEW.on Include views in migration. Views always use SCHEMA_ONLY logic regardless of the top-level strategy.
    migrateNonNative Include non-native tables (HBase, Kafka, JDBC). These are passed through as-is with a warning about dependency availability.
    createIfNotExists Adds IF NOT EXISTS to CREATE TABLE DDL. When true and the table already exists, creation will fail softly and continue with partition statements.
    forceExternalLocation Always include the LOCATION clause in CREATE TABLE, even when the default database location would suffice.
    enableAutoTableStats / enableAutoColumnStats Generates ANALYZE TABLE statements after table creation to compute statistics on the target.
    copyAvroSchemaUrls Preserves the avro.schema.url table property and copies the referenced Avro schema file between clusters. Requires clusters to be linked.
    icebergConversion.enable Convert the target table to Iceberg format during schema creation.
    icebergConversion.version Iceberg format version (1 or 2). Default: 2.
    icebergConversion.fileTypeTranslation Target file format for Iceberg conversion (PARQUET, ORC, AVRO, STANDARD).
    ownershipTransfer.table Generate ALTER TABLE SET OWNER to transfer table ownership from source to target.
    readOnly Create tables without PURGE flag, suitable for DR or testing scenarios.
    noPurge Force removal of PURGE flag from all tables.
    sync Enable synchronization mode: drop and recreate tables if schemas differ, drop tables on RIGHT that no longer exist on LEFT.

    Where This Strategy Is NOT Applicable¶

    • When you need data to move as part of the migration process. Use SQL, EXPORT_IMPORT, or HYBRID instead -- these strategies generate INSERT...SELECT or EXPORT/IMPORT commands that move data through the Hive engine.
    • When you want RIGHT to query LEFT's data directly without copying it. Use LINKED instead -- it preserves source locations so both clusters read the same physical files.
    • When migrating within the same cluster to a new storage location. Use STORAGE_MIGRATION instead -- it handles in-place data movement with table renaming and archiving.
    • When both clusters share the same physical storage. Use COMMON instead -- it creates schemas with the same locations, no translation needed.

    Use Cases¶

    1. Pre-migration validation -- Create schemas on the target cluster, validate that they compile and are structurally correct, then run a separate data copy (distcp or cloud replication) as a second phase. The dry-run mode (execute=false) makes this completely non-destructive.

    2. HDP to CDP schema upgrade -- Migrate table definitions from an HDP 2.x cluster (legacy Hive) to a CDP cluster (Hive 3). Managed non-ACID tables are automatically converted to EXTERNAL+PURGE. ACID tables can be downgraded. Run distcp separately to copy data from HDFS source to the translated target paths.

    3. Disaster recovery -- Replicate schemas to a standby cluster. Data is replicated independently via HDFS snapshots, S3 cross-region replication, or similar mechanisms. Use readOnly=true to avoid setting PURGE flags on the DR cluster, preventing accidental data deletion.

    4. Cloud migration with distcp -- Create target schemas pointing to S3/ABFS/GCS locations, then run distcp from on-premises HDFS to the matching cloud paths. The generated distcp plans provide the exact source-to-target path mapping.

    5. Schema synchronization -- Use sync=true to keep two clusters' metastores aligned. Tables that exist on LEFT but not RIGHT are created. Tables that exist on RIGHT but not LEFT are dropped. Tables that exist on both but differ are replaced. Tables that match are left alone (or have partitions amended).

    Generated SQL Examples¶

    (a) Simple External Table -- HDP to CDP¶

    Source (LEFT, legacy Hive on hdfs://HDP50): 

    -- Original table on source cluster (legacy managed, non-ACID)
CREATE TABLE `sales.db`.`customers`(
  `c_id` bigint,
  `c_name` varchar(100),
  `c_email` varchar(200),
  `c_signup_date` date)
ROW FORMAT SERDE
  'org.apache.hadoop.hive.ql.io.orc.OrcSerde'
STORED AS INPUTFORMAT
  'org.apache.hadoop.hive.ql.io.orc.OrcInputFormat'
OUTPUTFORMAT
  'org.apache.hadoop.hive.ql.io.orc.OrcOutputFormat'
LOCATION
  'hdfs://HDP50/apps/hive/warehouse/sales.db/customers';

    Generated RIGHT SQL (target CDP on hdfs://CDP-PROD): 

    USE sales;

CREATE EXTERNAL TABLE `customers`(
  `c_id` bigint,
  `c_name` varchar(100),
  `c_email` varchar(200),
  `c_signup_date` date)
ROW FORMAT SERDE
  'org.apache.hadoop.hive.ql.io.orc.OrcSerde'
STORED AS INPUTFORMAT
  'org.apache.hadoop.hive.ql.io.orc.OrcInputFormat'
OUTPUTFORMAT
  'org.apache.hadoop.hive.ql.io.orc.OrcOutputFormat'
LOCATION
  'hdfs://CDP-PROD/warehouse/external/sales.db/customers'
TBLPROPERTIES (
  'hmsMirror_Metadata_Stage1'='2026-02-16 10:30:00',
  'hmsMirror_LegacyManaged'='true',
  'hmsMirror_Converted'='true',
  'external.table.purge'='true',
  'discover.partitions'='true');

    (b) Partitioned Table with Metadata Loaded¶

    Source (LEFT): partitioned external table with explicit partition locations.

    Generated RIGHT SQL: 

    USE sales;

CREATE EXTERNAL TABLE `orders`(
  `o_id` bigint,
  `o_customer_id` bigint,
  `o_total` decimal(12,2),
  `o_status` varchar(20))
PARTITIONED BY (
  `o_order_date` string)
ROW FORMAT SERDE
  'org.apache.hadoop.hive.ql.io.orc.OrcSerde'
STORED AS INPUTFORMAT
  'org.apache.hadoop.hive.ql.io.orc.OrcInputFormat'
OUTPUTFORMAT
  'org.apache.hadoop.hive.ql.io.orc.OrcOutputFormat'
LOCATION
  'hdfs://CDP-PROD/warehouse/external/sales.db/orders'
TBLPROPERTIES (
  'hmsMirror_Metadata_Stage1'='2026-02-16 10:30:00',
  'hmsMirror_Converted'='true',
  'external.table.purge'='true',
  'discover.partitions'='true');

ALTER TABLE orders ADD
  PARTITION (o_order_date='2025-01') LOCATION 'hdfs://CDP-PROD/warehouse/external/sales.db/orders/o_order_date=2025-01'
  PARTITION (o_order_date='2025-02') LOCATION 'hdfs://CDP-PROD/warehouse/external/sales.db/orders/o_order_date=2025-02'
  PARTITION (o_order_date='2025-03') LOCATION 'hdfs://CDP-PROD/warehouse/external/sales.db/orders/o_order_date=2025-03';

    When metadata is NOT loaded, the partition statements are replaced with: 

    MSCK REPAIR TABLE sales.orders;

    (c) ACID Downgrade¶

    Source (LEFT): a transactional (ACID) managed table on legacy Hive, bucketed into 2 buckets.

    -- Original on source
CREATE TABLE `sales.db`.`transactions`(
  `txn_id` bigint,
  `txn_amount` decimal(12,2),
  `txn_type` varchar(20))
CLUSTERED BY (txn_id) INTO 2 BUCKETS
ROW FORMAT SERDE
  'org.apache.hadoop.hive.ql.io.orc.OrcSerde'
STORED AS INPUTFORMAT
  'org.apache.hadoop.hive.ql.io.orc.OrcInputFormat'
OUTPUTFORMAT
  'org.apache.hadoop.hive.ql.io.orc.OrcOutputFormat'
TBLPROPERTIES (
  'transactional'='true',
  'transactional_properties'='default');

    Generated RIGHT SQL (with migrateACID.on=true, migrateACID.downgrade=true, artificialBucketThreshold=2): 

    USE sales;

CREATE EXTERNAL TABLE `transactions`(
  `txn_id` bigint,
  `txn_amount` decimal(12,2),
  `txn_type` varchar(20))
ROW FORMAT SERDE
  'org.apache.hadoop.hive.ql.io.orc.OrcSerde'
STORED AS INPUTFORMAT
  'org.apache.hadoop.hive.ql.io.orc.OrcInputFormat'
OUTPUTFORMAT
  'org.apache.hadoop.hive.ql.io.orc.OrcOutputFormat'
TBLPROPERTIES (
  'hmsMirror_Metadata_Stage1'='2026-02-16 10:30:00',
  'external.table.purge'='true',
  'DOWNGRADED_FROM_ACID'='true',
  'discover.partitions'='true');

    Note: LOCATION is stripped (Hive 3 will place the data at the database's default external location). The CLUSTERED BY clause is removed because the bucket count (2) is at or below the artificialBucketThreshold. Transactional properties are removed.

    (d) Iceberg Conversion¶

    Source (LEFT): standard ORC external table.

    Generated RIGHT SQL (with icebergConversion.enable=true, icebergConversion.version=2): 

    USE analytics;

CREATE EXTERNAL TABLE `events`(
  `event_id` bigint,
  `event_type` varchar(50),
  `event_payload` string,
  `event_ts` timestamp)
STORED BY 'org.apache.iceberg.mr.hive.HiveIcebergStorageHandler'
LOCATION
  's3a://data-lake-bucket/warehouse/external/analytics.db/events'
TBLPROPERTIES (
  'hmsMirror_Metadata_Stage1'='2026-02-16 10:30:00',
  'format-version'='2',
  'external.table.purge'='true',
  'discover.partitions'='true');

    Note: The original ROW FORMAT SERDE and STORED AS clauses are replaced by the Iceberg storage handler. The format-version property is set according to the configured Iceberg version.
