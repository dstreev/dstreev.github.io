    Data Strategy: The Core Concept¶

    Overview¶

    A Data Strategy is the central abstraction that drives everything in HMS-Mirror. It defines how a table migration is performed — what SQL is generated, whether data moves, how locations are translated, and what intermediate objects are created. Every migration job selects a data strategy, and that choice determines the entire execution flow.

    The strategy is not just a "mode switch." It is a self-contained engine that understands how to:

    1. Build a target schema from a source table definition (with location translation, property management, and type conversions)
    2. Generate SQL for schema creation, data movement, and cleanup
    3. Execute the generated SQL against source and/or target clusters
    4. Handle edge cases — ACID tables, partitions, views, non-native tables, format conversions

    Architecture¶

    Strategy Lifecycle¶

    Every strategy follows a two-phase lifecycle:

    Phase 1: Build                          Phase 2: Execute
┌────────────────────────────┐          ┌─────────────────────────┐
│ buildOutDefinition()       │          │ execute()               │
│   - Translate locations    │          │   - Run DDL on target   │
│   - Convert table types    │          │   - Run DML for data    │
│   - Set properties         │          │   - Clean up staging    │
│                            │          │                         │
│ buildOutSql()              │          │                         │
│   - Generate CREATE TABLE  │          │                         │
│   - Generate INSERT/SELECT │          │                         │
│   - Generate cleanup DDL   │          │                         │
└────────────────────────────┘          └─────────────────────────┘

    This separation is critical: the build phase is always safe (it only generates SQL), while the execute phase is where changes happen. Dry-run mode skips execution entirely, producing a report of what would happen.

    Four Environments¶

    Each table being migrated can have definitions in up to four environments:

    Environment Purpose Used By
    LEFT Source cluster (where data lives today) All strategies
    RIGHT Target cluster (where data should end up) All strategies
    SHADOW Intermediate table on RIGHT pointing to LEFT's data SQL, HYBRID
    TRANSFER Staging table for multi-hop migrations INTERMEDIATE, EXPORT_IMPORT

    Most strategies only use LEFT and RIGHT. The SQL strategy uses SHADOW as a bridge — it creates a temporary external table on the RIGHT cluster that points to LEFT's storage, allowing a local INSERT...SELECT. The TRANSFER environment is used for complex multi-hop scenarios with intermediate storage.

    The BuildWhat Decision¶

    Before building SQL, each strategy determines what needs to be built via the BuildWhat flags:

    • transferTable — create a TRANSFER environment staging table
    • shadowTable — create a SHADOW environment bridge table
    • transferSql — generate SQL for the TRANSFER environment
    • shadowSql — generate SQL for the SHADOW environment
    • rightSql — generate SQL for the RIGHT (final target) environment

    Simple strategies like SCHEMA_ONLY only set rightSql. Complex strategies like INTERMEDIATE set all flags.

    Location Translation¶

    The most important transformation in any strategy is location translation — converting storage paths from the source namespace to the target namespace.

    How Translation Works¶

    Source:  hdfs://source-cluster/warehouse/sales.db/orders
Target:  s3a://data-lake/warehouse/external/sales.db/orders

    Translation is driven by the Translator, which maintains ordered location maps:

    1. Auto-generated maps — derived from warehouse directory configuration
    2. User-provided maps — custom path mappings for non-standard layouts
    3. Merged maps — sorted by path length (longest match wins)

    Translation Modes¶

    Mode Behavior When Used
    ALIGNED Paths must align with warehouse directory structure Standard warehouse layouts
    RELATIVE Maintains relative path relationships with flexible base Non-standard or mixed layouts

    What Gets Translated¶

    • Table LOCATION in CREATE TABLE
    • Partition locations in ALTER TABLE ADD PARTITION
    • Database LOCATION and MANAGEDLOCATION properties
    • Export/import intermediate paths

    What Does NOT Get Translated¶

    • LINKED strategy intentionally preserves source locations
    • COMMON strategy preserves shared locations
    • DUMP strategy outputs source locations as-is

    Table Type Handling¶

    Managed vs External Conversion¶

    When migrating from Hive 2 (HDP) to Hive 3 (CDP), managed non-transactional tables must become external tables:

    Hive 2 (Legacy)                    Hive 3 (CDP)
┌─────────────────┐               ┌─────────────────────────┐
│ MANAGED TABLE   │   ────────►   │ EXTERNAL TABLE          │
│ (non-ACID)      │               │ + EXTERNAL_TABLE_PURGE  │
└─────────────────┘               └─────────────────────────┘

    The EXTERNAL_TABLE_PURGE property ensures that dropping the table on the target still deletes the underlying data — preserving the managed table's ownership semantics.

    ACID Table Handling¶

    ACID (transactional) tables require special handling because:

    • Legacy ACID (Hive 1/2) uses a different storage format than Hive 3 ACID
    • EXPORT/IMPORT of ACID between incompatible versions fails silently
    • Some strategies (LINKED, COMMON) cannot handle ACID tables at all

    When migrateACID is enabled, the strategy pipeline can:

    1. Downgrade — convert ACID to EXTERNAL+PURGE (strip transactional properties)
    2. Migrate via intermediate — use staging tables to handle format conversion
    3. In-place downgrade — modify the source cluster directly (single-cluster operation)

    Artificial Bucket Removal¶

    Hive 1/2 required ACID tables to be bucketed. Hive 3 removed this requirement. During migration, the strategy can detect "artificial" buckets (threshold: 2 or fewer) and remove the CLUSTERED BY clause, simplifying the target schema.

    Partition Handling¶

    Partitioned tables add complexity to every strategy. HMS-Mirror handles partitions through three mechanisms:

    1. Metadata-Driven (Explicit)¶

    When partition metadata is loaded from the source metastore, each partition gets an explicit ALTER TABLE ADD PARTITION with its translated location:

    ALTER TABLE orders ADD PARTITION (year=2024, month=1)
  LOCATION 's3a://bucket/warehouse/orders/year=2024/month=1';
ALTER TABLE orders ADD PARTITION (year=2024, month=2)
  LOCATION 's3a://bucket/warehouse/orders/year=2024/month=2';

    2. Auto-Discovery (MSCK REPAIR)¶

    When metadata is not loaded, the strategy generates:

    MSCK REPAIR TABLE orders;

    This relies on the storage directory structure following the standard key=value/ convention.

    3. Partition Limits¶

    Large tables (thousands of partitions) can cause resource exhaustion. Each strategy enforces configurable limits:

    • sqlPartitionLimit — max partitions for SQL strategy (routes to EXPORT_IMPORT above limit)
    • exportImportPartitionLimit — max partitions for EXPORT_IMPORT
    • HYBRID uses these limits to auto-select between SQL and EXPORT_IMPORT

    Configuration Parameters That Affect All Strategies¶

    Transfer Configuration¶

    Parameter Effect
    transferPrefix Prefix for intermediate transfer tables (default: hms_mirror_transfer_)
    shadowPrefix Prefix for shadow bridge tables (default: hms_mirror_shadow_)
    exportBaseDirPrefix Base directory for EXPORT operations
    intermediateStorage Multi-hop intermediate location (e.g., S3 bucket between clusters)
    warehouse.external Target warehouse directory for external tables
    warehouse.managed Target warehouse directory for managed tables

    Migration Flags¶

    Parameter Effect
    migrateACID.on Enable ACID table migration
    migrateACID.downgrade Convert ACID to EXTERNAL+PURGE
    migrateACID.inplace Modify source cluster directly
    migrateACID.artificialBucketThreshold Remove buckets if count <= threshold
    migrateVIEW Include views in migration
    migrateNonNative Include HBase/Kafka-backed tables

    Output Modifiers¶

    Parameter Effect
    createIfNotExists Add IF NOT EXISTS to DDL (default: true)
    forceExternalLocation Always specify LOCATION in CREATE TABLE
    enableAutoTableStats Run ANALYZE TABLE after migration
    enableAutoColumnStats Run ANALYZE TABLE COLUMNS after migration
    copyAvroSchemaUrls Preserve avro.schema.url property
    saveWorkingTables Keep intermediate tables after migration

    Iceberg Conversion (Overlay)¶

    Iceberg conversion is not a standalone strategy — it's a configuration overlay that can be applied during SQL or SCHEMA_ONLY strategies:

    Parameter Effect
    icebergConversion.enable Convert target tables to Iceberg format
    icebergConversion.version Iceberg format version (1 or 2)
    icebergConversion.fileTypeTranslation Target file format (PARQUET, ORC, AVRO, STANDARD)
    icebergConversion.inplace In-place conversion without data movement

    Table Properties Management¶

    During migration, strategies manage several table properties:

    Properties Removed (Legacy/Stats)¶

    • COLUMN_STATS_ACCURATE — stale statistics from source
    • numFiles, numRows, rawDataSize, totalSize — stale metrics
    • transient_lastDdlTime — source timestamp, irrelevant on target
    • BUCKETING_VERSION — Hive 3 internal property

    Properties Added¶

    • HMS_MIRROR_METADATA_FLAG — timestamp marking when schema was migrated
    • EXTERNAL_TABLE_PURGE — set when taking data ownership
    • DISCOVER_PARTITIONS — enable automatic partition discovery
    • HMS_STORAGE_MIGRATION_FLAG — timestamp for storage migration
    • DOWNGRADED_FROM_ACID — marks tables that were converted from ACID

    Properties on Intermediate Tables¶

    • HMS_MIRROR_TRANSFER_TABLE — marks transfer staging tables
    • HMS_MIRROR_SHADOW_TABLE — marks shadow bridge tables

    These markers ensure intermediate tables are identifiable for cleanup.

    Strategy Applicability Matrix¶

    Strategy Non-ACID External ACID/Managed Partitioned Views Non-Native Linked Clusters Single Cluster
    SCHEMA_ONLY Yes Limited Yes Yes Optional Yes Yes
    LINKED Yes No Yes No No Required No
    SQL Yes Downgrade Yes No No Required No
    EXPORT_IMPORT Yes Yes Yes No No Required No
    HYBRID Yes Auto Yes No No Required No
    STORAGE_MIGRATION Yes Limited Yes No No No Yes
    CONVERT_LINKED Yes No Limited No No Yes No
    COMMON Yes No Yes No No Yes Yes
    DUMP Yes Yes Yes Yes Yes Any Any

    Execution Flow¶

    User selects strategy in JobDto
         │
         ▼
DataStrategyService.getStrategy(enum)
         │
         ▼
strategy.build(dbMirror, tableMirror)
    │
    ├── whatToBuild() ──► determines which environments need tables/SQL
    ├── buildOutDefinition() ──► translates schema, locations, properties
    └── buildOutSql() ──► generates DDL and DML statements
         │
         ▼
strategy.execute(dbMirror, tableMirror)   [skipped in dry-run]
    │
    ├── Execute DDL on RIGHT (CREATE TABLE)
    ├── Execute DML (INSERT/SELECT, EXPORT/IMPORT)
    └── Execute cleanup (DROP intermediate tables)
         │
         ▼
Results written to ConversionResult
    │
    ├── Generated SQL (for review/audit)
    ├── Errors and warnings
    └── Migration status per table
