    Managing Jobs¶

    Jobs are where everything comes together - combining datasets, configurations, and connections to execute migrations. This guide covers creating, validating, and running jobs through the web UI.

    What is a Job?¶

    A job combines:

    • Dataset - What databases and tables to migrate
    • Configuration - How to perform the migration
    • Data Strategy - Which migration method to use
    • LEFT Connection - Source cluster connection
    • RIGHT Connection - Target cluster connection (if needed)
    • Options - Strategy-specific settings

    Jobs can be saved, validated, run multiple times, and their results tracked.

    Viewing Jobs¶

    Jobs Page¶

    Navigate to Jobs → Build from the main menu to see all saved jobs.

    Each job card displays: - Job name and description - Data strategy - Referenced components - Status (ready, running, completed) - Quick action buttons

    Job Actions¶

    Icon Action Description
    ✏️ Edit Modify job settings
    ▶️ Run Execute the job
    ✓ Validate Check for errors
    ✨ AI Analysis Get AI-powered recommendations
    📋 Summary View job summary
    🗑️ Delete Remove the job

    Creating a Job¶

    Using the Job Wizard¶

    The wizard guides you through job creation in 5 steps.

    1. Click New Job or Build Job
    2. Complete each wizard page:
    3. Job Details
    4. Data Strategy
    5. References
    6. Options
    7. Summary

    Step 1: Job Details¶

    Basic information about the job.

    Field Description Example
    Name Unique job identifier hdp-to-cdp-sales-migration
    Description Purpose and notes Migrate sales database from HDP to CDP

    Step 2: Data Strategy¶

    Select the migration method.

    Strategy Description When to Use
    SCHEMA_ONLY Schema without data Testing, schema sync
    LINKED Views to source data Quick access
    SQL Data via INSERT Small-medium tables
    EXPORT_IMPORT Hive export/import ACID tables
    HYBRID Automatic selection Mixed workloads
    STORAGE_MIGRATION Same-cluster move Storage changes
    COMMON Shared storage Linked clusters
    DUMP DDL extraction Documentation
    ICEBERG_CONVERSION Format conversion Modernization

    Tip

    Use the Strategy Selection Wizard if unsure which strategy to choose.

    Step 3: References¶

    Select the components for this job.

    Reference Description Required
    Dataset What to migrate Yes
    Configuration How to migrate Yes
    LEFT Connection Source cluster Yes
    RIGHT Connection Target cluster Depends on strategy

    Reference Selection¶

    1. Click the dropdown for each reference
    2. Select from saved items
    3. Or click Create New to build on the fly

    Connection Requirements by Strategy¶

    Strategy LEFT RIGHT
    SCHEMA_ONLY Required Required
    LINKED Required Required
    SQL Required Required
    EXPORT_IMPORT Required Required
    HYBRID Required Required
    STORAGE_MIGRATION Required Optional
    COMMON Required Required
    DUMP Required Not needed
    ICEBERG_CONVERSION Required Optional

    Step 4: Options¶

    Strategy-specific settings.

    Storage/Namespace Options¶

    For strategies that move data or change locations:

    Option Description
    Reset to Default Reset to default warehouse location
    Global Location Map Namespace translation rules

    Hybrid Strategy Options¶

    When using HYBRID strategy:

    Option Description Default
    SQL Partition Limit Max partitions for SQL 100
    SQL Size Limit Max table size for SQL 1GB
    Export Partition Limit Max partitions for EXPORT 500

    Step 5: Summary¶

    Review the complete job configuration before saving.

    Job Summary
───────────────────────────────
Name: hdp-to-cdp-sales-migration
Strategy: HYBRID

References:
  Dataset: sales-tables (3 databases)
  Configuration: hdp-migration-config
  LEFT: hdp-prod-cluster
  RIGHT: cdp-prod-cluster

Options:
  SQL Partition Limit: 100
  Export Partition Limit: 500

Warnings:
  ⚠ ACID tables will be downgraded
───────────────────────────────

    Click Save to create the job.

    Validating Jobs¶

    Before running, validate jobs to catch configuration issues.

    Run Validation¶

    1. Click Validate on the job card
    2. Wait for validation to complete
    3. Review results

    Validation Checks¶

    HMS-Mirror validates:

    • Connections - Can connect to clusters
    • Dataset - Databases exist, patterns valid
    • Configuration - Settings compatible with strategy
    • Strategy compatibility - All pieces work together
    • Permissions - Access to required resources

    Validation Results¶

    All Passed 

    ✓ Job validation passed

Connections:
  ✓ LEFT: Connected successfully
  ✓ RIGHT: Connected successfully

Dataset:
  ✓ sales_db: 45 tables matched
  ✓ inventory_db: 23 tables matched

Configuration:
  ✓ All settings valid

Strategy:
  ✓ HYBRID compatible with configuration

    With Warnings 

    ✓ Job validation passed with warnings

⚠ Warnings:
  - 3 ACID tables will be downgraded to external
  - Large table 'orders' (50GB) will use EXPORT_IMPORT
  - 2 views will be skipped (copyViews disabled)

    With Errors 

    ✗ Job validation failed

✗ Errors:
  - Cannot connect to RIGHT cluster
  - Database 'missing_db' not found
  - Invalid warehouse path format

    Addressing Validation Issues¶

    1. Review each error and warning
    2. Edit the job or referenced components
    3. Re-validate until all issues resolved

    Running Jobs¶

    Dry Run vs Execute¶

    Mode Description Effect
    Dry Run Generate scripts only No changes to clusters
    Execute Run the migration Creates tables, moves data

    Best Practice

    Always run a dry run first to review planned operations.

    Starting a Job¶

    1. Click Run on the job card
    2. Select mode:
    3. Dry Run (default and recommended first)
    4. Execute
    5. Click Start

    Monitoring Progress¶

    While the job runs, you can monitor:

    Progress Bar 

    [████████░░░░░░░░░░░░] 40%
Processing: sales_db.orders
Tables: 18/45

    Status Updates - Connecting to clusters - Discovering tables - Analyzing schemas - Generating DDL - Executing statements (Execute mode) - Generating reports

    Job States¶

    State Description
    Ready Job saved, not yet run
    Running Currently executing
    Completed Finished successfully
    Failed Encountered errors
    Cancelled Stopped by user

    Viewing Results¶

    Reports Page¶

    After completion, click View Report to see results.

    Report Contents¶

    Summary 

    Migration Summary
─────────────────
Total Tables: 68
Successful: 65
Failed: 2
Skipped: 1

Duration: 15m 32s
Strategy: HYBRID

    Per-Table Details

    Each table shows: - Status (SUCCESS, FAILED, SKIPPED) - Strategy used (for HYBRID) - Generated SQL - Execution time - Errors (if any)

    Generated Scripts

    View or download the SQL scripts: - DDL statements - Data movement queries - Cleanup scripts

    Error Analysis¶

    For failed tables: 

    Table: sales_db.large_orders
Status: FAILED
Error: Exceeded partition limit (1500 > 500)
Suggestion: Increase partitionLimit in configuration

    AI Report Analysis¶

    Click the sparkles icon to get AI-powered analysis: - Success rate assessment - Error categorization - Root cause analysis - Recommendations for failures - Best practices suggestions

    Job Management¶

    Editing Jobs¶

    1. Click Edit on the job card
    2. Modify settings
    3. Save changes

    Note

    Running jobs cannot be edited.

    Copying Jobs¶

    Create a new job based on an existing one:

    1. Click Copy on the job card
    2. Enter a new name
    3. Modify as needed
    4. Save

    Deleting Jobs¶

    1. Click Delete on the job card
    2. Confirm deletion

    Note

    Deleting a job does not undo migrations or delete reports.

    Re-running Jobs¶

    Jobs can be run multiple times:

    1. Click Run on the job card
    2. Select mode
    3. Start

    Each run creates a new report with a timestamp.

    Job Examples¶

    Schema Synchronization Job¶

    name: nightly-schema-sync
strategy: SCHEMA_ONLY
dataset: all-databases
configuration: schema-only-config
leftConnection: prod-cluster
rightConnection: dr-cluster

    Full Migration Job¶

    name: hdp-to-cdp-migration
strategy: HYBRID
dataset: production-tables
configuration: full-migration-optimized
leftConnection: hdp-cluster
rightConnection: cdp-cluster
options:
  hybrid:
    sqlPartitionLimit: 100
    exportPartitionLimit: 500

    Storage Migration Job¶

    name: hdfs-to-s3-migration
strategy: STORAGE_MIGRATION
dataset: analytics-database
configuration: s3-migration-config
leftConnection: cluster-hdfs
options:
  storage:
    targetNamespace: s3a://data-lake

    Strategy Selection Wizard¶

    If you're unsure which strategy to use, the wizard helps you decide.

    Using the Wizard¶

    1. On the Data Strategy page, click Help Me Choose
    2. Answer questions about your migration:
    3. Do you need to move data?
    4. Do you have ACID tables?
    5. What are your table sizes?
    6. Same cluster or different?
    7. Get a recommended strategy with explanation

    Wizard Questions¶

    1. Data Movement
    2. Do tables need data copied to target?

    3. Or just schema definitions?

    4. Table Types

    5. Do you have ACID (transactional) tables?

    6. Any views to migrate?

    7. Scale

    8. Approximate table sizes?

    9. Number of partitions?

    10. Infrastructure

    11. Same cluster or different?
    12. Shared storage available?

    Wizard Result¶

    Recommended Strategy: HYBRID

Reasoning:
- You have mixed table sizes → HYBRID handles this automatically
- ACID tables present → Will use EXPORT_IMPORT for those
- Small tables → Will use SQL for efficiency

Alternative: EXPORT_IMPORT
- If all tables are large
- If exact data fidelity is critical

    Best Practices¶

    Job Organization¶

    Naming Conventions - Include purpose: schema-sync-, full-migration- - Include source/target: hdp-to-cdp- - Include scope: -sales-db, -phase-1

    Example: hdp-to-cdp-full-migration-sales-db-phase-1

    Execution Strategy¶

    Start Small 1. Test with a single small database first 2. Verify results 3. Scale up gradually

    Use Dry Runs 1. Always dry run before execute 2. Review generated SQL 3. Check for warnings

    Phase Large Migrations 1. Split into multiple jobs by database or table group 2. Run sequentially or in parallel (different databases) 3. Verify each phase before proceeding

    Error Handling¶

    Before Running - Validate thoroughly - Address all errors - Understand warnings

    During Running - Monitor progress - Check for early failures - Be ready to cancel if needed

    After Running - Review reports immediately - Investigate failures - Document issues and solutions

    Documentation¶

    • Use job descriptions to document purpose
    • Keep notes on configuration choices
    • Track job history for auditing

    Troubleshooting¶

    Job Won't Start¶

    Connection Issues 

    Error: Cannot connect to LEFT cluster
    - Verify cluster is running - Check network connectivity - Validate credentials

    Validation Errors 

    Error: Job validation failed
    - Run validation to see specific issues - Fix errors before running

    Job Fails During Execution¶

    Permission Errors 

    Error: Permission denied creating table
    - Check user permissions on target - Verify service account access

    Resource Errors 

    Error: Out of memory
    - Reduce parallelism - Increase HMS-Mirror memory - Split into smaller jobs

    Slow Performance¶

    Large Tables - Consider EXPORT_IMPORT over SQL - Increase partition limits appropriately - Run during off-peak hours

    Many Tables - Split into multiple jobs - Process databases in parallel - Use appropriate strategy for scale
