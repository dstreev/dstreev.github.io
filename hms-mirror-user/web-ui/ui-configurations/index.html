    Managing Configurations¶

    Configurations define how HMS-Mirror performs migrations - what options to apply, how to handle special cases, and what optimizations to use. This guide covers creating and managing configurations through the web UI.

    What is a Configuration?¶

    A configuration is a reusable set of migration settings that controls:

    • Migration behavior - How to handle views, non-native tables, and properties
    • Optimizations - Query tuning and performance settings
    • Transfer settings - Warehouse directories and storage paths
    • ACID handling - Transactional table options
    • Conversions - Format transformations like Iceberg

    Configurations are separate from jobs, allowing you to reuse the same settings across multiple migrations.

    Viewing Configurations¶

    Configurations Page¶

    Navigate to Configurations from the main menu to see all saved configurations.

    Each configuration card displays: - Configuration name - Description (if provided) - Key settings overview - Quick action buttons (Edit, Copy, Delete, AI Analysis)

    AI Analysis¶

    Click the sparkles icon (✨) on any configuration card to get AI-powered analysis:

    • Configuration review and recommendations
    • Potential issues identified
    • Best practices suggestions
    • Risk assessment

    Creating a Configuration¶

    Using the Configuration Wizard¶

    The wizard walks you through each configuration section.

    1. Click New Configuration or the + button
    2. Complete each wizard page:
    3. Details
    4. Migration
    5. Optimizations
    6. Transfer
    7. ACID
    8. Conversions

    Page 1: Details¶

    Basic information about the configuration.

    Field Description Example
    Name Unique identifier hdp-to-cdp-standard
    Description Purpose and notes Standard settings for HDP to CDP migration

    Page 2: Migration¶

    Controls which tables are processed and how.

    Views¶

    Option Description
    Copy Views Migrate view definitions to target
    Views Only Only process views, skip tables

    Non-Native Tables¶

    Tables backed by external systems (HBase, Kudu, etc.)

    Option Description
    Skip Don't process non-native tables
    Include Attempt to migrate

    Partition Handling¶

    Option Description
    Evaluate Partition Location Check and handle partition-specific locations
    Partition Discovery How to find partitions

    Table Properties¶

    Option Description
    Property Filter Regex to exclude certain properties
    Reset Location Reset external table locations

    Page 3: Optimizations¶

    Query performance and tuning settings.

    Query Optimizations¶

    Option Description Default
    Sort Dynamic Partition Inserts Order data for better partitioning false
    Auto Tune Let HMS-Mirror optimize settings false

    Compression¶

    Option Description
    Text Compression Compression codec for text formats
    Codec GZIP, SNAPPY, LZ4, ZSTD

    Statistics¶

    Option Description
    Gather Statistics Collect table/column stats
    Shadow Statistics Use shadow approach for stats

    Property Overrides¶

    Custom Hive session properties applied during migration.

    Structure 

    Property Key → LEFT Value, RIGHT Value

    Common Overrides

    Property Description
    hive.exec.dynamic.partition.mode Partition mode (strict/nonstrict)
    hive.exec.max.dynamic.partitions Max partitions per query
    mapreduce.job.reduces Number of reducers

    Adding Overrides

    1. Enter the property key
    2. Set the value for LEFT cluster (optional)
    3. Set the value for RIGHT cluster (optional)
    4. Click Add to save the override

    Page 4: Transfer¶

    Where data goes and how paths are translated.

    Warehouse Directories¶

    Define target locations for migrated tables.

    Directory Description Example
    External Warehouse Location for external tables s3a://bucket/warehouse/external
    Managed Warehouse Location for managed tables s3a://bucket/warehouse/managed

    Storage Settings¶

    Option Description
    Common Storage Shared storage path for both clusters
    Intermediate Storage Temp location for EXPORT_IMPORT

    Namespace Translation¶

    For STORAGE_MIGRATION strategy:

    Option Description
    Translation Type RELATIVE or ALIGNED
    Source Namespace Original storage namespace
    Target Namespace New storage namespace

    Page 5: ACID¶

    Settings for transactional (ACID) tables.

    Basic Options¶

    Option Description
    Include ACID Tables Process transactional tables
    ACID Only Only process ACID tables

    Migration Options¶

    Option Description
    Downgrade ACID Convert to external tables
    In-Place Migrate within same cluster
    Migrate Full ACID migration

    Limits¶

    Option Description Default
    Partition Limit Max partitions for ACID migration 0 (no limit)
    Bucket Threshold Min buckets for SQL strategy 0

    Page 6: Conversions¶

    Table format transformations.

    Iceberg Conversion¶

    Convert Hive tables to Apache Iceberg format.

    Option Description
    Enable Iceberg Turn on Iceberg conversion
    Conversion Mode IN_PLACE or SHADOW
    Target Version Iceberg format version (1 or 2)
    File Format Output format (PARQUET, ORC, AVRO)

    Editing Configurations¶

    Modify an Existing Configuration¶

    1. Navigate to Configurations
    2. Click Edit on the configuration card
    3. The wizard opens with current values
    4. Modify settings as needed
    5. Click Save

    Copying Configurations¶

    Create a new configuration based on an existing one:

    1. Click Copy on the configuration card
    2. Enter a new name
    3. Modify settings as needed
    4. Save the new configuration

    Configuration Examples¶

    Basic Schema Migration¶

    name: schema-only-basic
migration:
  copyViews: false
  evaluatePartitionLocation: false
optimizations:
  sortDynamicPartitionInserts: false
acid:
  includeAcidTables: false

    Full Data Migration with Optimization¶

    name: full-migration-optimized
migration:
  copyViews: true
  evaluatePartitionLocation: true
optimizations:
  sortDynamicPartitionInserts: true
  autoTune: true
  textCompression: GZIP
transfer:
  warehouse:
    external: 's3a://data-lake/warehouse/external'
    managed: 's3a://data-lake/warehouse/managed'
acid:
  includeAcidTables: true
  downgradeAcid: true

    ACID Table Migration¶

    name: acid-migration
migration:
  copyViews: false
acid:
  includeAcidTables: true
  acidOnly: true
  downgradeAcid: false
  artificialBucketThreshold: 2
  partitionLimit: 500

    Iceberg Conversion¶

    name: iceberg-conversion
migration:
  copyViews: false
iceberg:
  enabled: true
  conversionMode: SHADOW
  targetVersion: 2
  fileFormat: PARQUET

    Storage Migration (HDFS to S3)¶

    name: hdfs-to-s3-migration
transfer:
  warehouse:
    external: 's3a://new-warehouse/external'
    managed: 's3a://new-warehouse/managed'
  storageMigration:
    translationType: ALIGNED
optimizations:
  sortDynamicPartitionInserts: true

    Configuration Validation¶

    HMS-Mirror validates configurations before they can be used.

    Validation Checks¶

    • Required fields are populated
    • Paths are properly formatted
    • Settings are compatible with selected strategy
    • No conflicting options

    Validation Results¶

    Configuration: hdp-migration-config

✓ Passed
  - All required fields present
  - Paths properly formatted
  - Settings compatible

⚠ Warnings
  - ACID tables will be downgraded to external
  - Consider enabling sortDynamicPartitionInserts for large partitioned tables

✗ Errors (if any)
  - Invalid warehouse path format

    Importing and Exporting¶

    Export a Configuration¶

    1. Click Export on the configuration card
    2. Save the YAML file

    Import a Configuration¶

    1. Click Import Configuration
    2. Select the YAML file
    3. Review and save

    Export Format¶

    name: example-config
description: Example configuration
migration:
  copyViews: true
  evaluatePartitionLocation: true
  nonNativeTableHandling: SKIP
optimizations:
  sortDynamicPartitionInserts: true
  autoTune: false
  overrides:
    - key: hive.exec.dynamic.partition.mode
      leftValue: nonstrict
      rightValue: nonstrict
transfer:
  warehouse:
    external: s3a://warehouse/external
    managed: s3a://warehouse/managed
acid:
  includeAcidTables: true
  downgradeAcid: true
  partitionLimit: 100
iceberg:
  enabled: false

    Best Practices¶

    Naming Conventions¶

    Create descriptive names that indicate purpose: - schema-only-basic - Basic schema migration - hdp-cdp-full-migration - Full migration from HDP to CDP - acid-downgrade-prod - Production ACID downgrade - iceberg-conversion-test - Iceberg conversion testing

    Configuration Management¶

    • Start simple: Begin with minimal settings, add complexity as needed
    • Test first: Use configurations in dry-run mode before executing
    • Document: Use descriptions to explain configuration purposes
    • Version: Create new configurations rather than modifying existing ones for major changes

    Common Patterns¶

    Development Environment - Disable optimization for faster testing - Smaller partition limits - Skip ACID tables initially

    Production Environment - Enable all optimizations - Proper warehouse paths - Include all table types - Appropriate partition limits

    Reusability¶

    Design configurations for reuse: - Keep environment-specific paths in separate configurations - Use consistent naming conventions - Document any assumptions in the description

    Troubleshooting¶

    Common Issues¶

    Path Format Errors 

    Error: Invalid warehouse path
    Ensure paths start with proper scheme (hdfs://, s3a://, etc.)

    Missing Required Fields 

    Error: External warehouse directory required for STORAGE_MIGRATION
    Fill in all required fields for your chosen strategy

    Incompatible Settings 

    Error: Cannot use ACID downgrade with in-place migration
    Review and resolve conflicting options

    Getting Help¶

    • Use AI Analysis for configuration recommendations
    • Check the troubleshooting guide for common errors
    • Review logs for detailed error messages
