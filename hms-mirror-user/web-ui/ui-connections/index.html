    Managing Connections¶

    Connections define how HMS-Mirror communicates with your DataLake environments powered by Hive and Hive Metastore. This guide covers creating, editing, and managing connections through the web UI.

    What is a Connection?¶

    A connection contains all the information needed to connect to a cluster:

    • HiveServer2 settings - JDBC URL, driver, and connection properties
    • Metastore Direct settings - Direct database access (optional)
    • Authentication - Kerberos, LDAP, or no authentication
    • Session properties - Hive session settings

    Connections are reusable - define them once and use them in multiple jobs.

    Viewing Connections¶

    Connections Page¶

    Navigate to Connections from the main menu to see all saved connections.

    Each connection card displays: - Connection name - Cluster type (Legacy or Non-Legacy) - HiveServer2 URI (if configured) - Metastore type (if configured) - Quick action buttons

    Connection Details¶

    Click on a connection card to view full details:

    Connection: prod-cluster
├── Type: Non-Legacy (CDP)
├── HiveServer2
│   ├── URI: jdbc:hive2://hs2.example.com:10000/default
│   └── Properties: transportMode=binary, ssl=true
├── Metastore Direct
│   ├── Type: MYSQL
│   └── URI: jdbc:mysql://metastore.example.com:3306/hive
└── Partition Discovery
    └── Auto: true

    Creating a New Connection¶

    Using the Connection Wizard¶

    The wizard guides you through connection setup step by step.

    1. Click New Connection or the + button
    2. Follow the wizard pages:
    3. Basic Information
    4. HiveServer2 Configuration
    5. Metastore Direct (Optional)
    6. Session Properties

    Basic Information¶

    Field Description Example
    Name Unique identifier for this connection prod-hdp-cluster
    Description Optional description Production HDP 3.1 cluster
    Legacy Check if HDP or CDH (pre-CDP) ☑ for HDP/CDH

    Note

    The Legacy flag affects how HMS-Mirror handles certain table types and properties.

    HiveServer2 Configuration¶

    Configure the primary connection method.

    URI Format¶

    The HiveServer2 JDBC URI follows this pattern:

    jdbc:hive2://<host>:<port>/<database>[;param1=value1;param2=value2]

    Common URI patterns:

    Standard Binary Transport 

    jdbc:hive2://hs2.example.com:10000/default

    HTTP Transport 

    jdbc:hive2://hs2.example.com:10001/default;transportMode=http;httpPath=cliservice

    SSL Enabled 

    jdbc:hive2://hs2.example.com:10000/default;ssl=true

    Knox Gateway 

    jdbc:hive2://knox.example.com:8443/;ssl=true;transportMode=http;httpPath=gateway/cdp-proxy-api/hive

    Zookeeper Discovery 

    jdbc:hive2://zk1:2181,zk2:2181,zk3:2181/;serviceDiscoveryMode=zooKeeper;zooKeeperNamespace=hiveserver2

    Driver Class¶

    Usually auto-detected, but can be specified:

    Driver Class
    Apache Hive org.apache.hive.jdbc.HiveDriver

    URI Parameters¶

    All connection properties should be included directly in the JDBC URI string. Common parameters include:

    Parameter Description Values
    transportMode Transport protocol binary, http
    httpPath HTTP endpoint path cliservice, gateway/...
    ssl Enable SSL/TLS true, false
    sslTrustStore Path to trust store /path/to/truststore.jks
    sslTrustStorePassword Trust store password password
    principal Kerberos service principal hive/_HOST@REALM
    user Username for LDAP auth username
    password Password for LDAP auth password

    Important

    Include all connection properties in the URI string rather than as separate connection properties. For example: jdbc:hive2://host:10000/default;ssl=true;transportMode=binary

    Metastore Direct Configuration¶

    Optional: Configure direct Metastore database access for advanced operations.

    Metastore Types¶

    Type Description
    MYSQL MySQL or MariaDB
    POSTGRESQL PostgreSQL
    ORACLE Oracle Database
    MSSQL Microsoft SQL Server

    URI Formats¶

    MySQL/MariaDB 

    jdbc:mysql://metastore.example.com:3306/hive?useSSL=false

    PostgreSQL 

    jdbc:postgresql://metastore.example.com:5432/hive

    Oracle 

    jdbc:oracle:thin:@metastore.example.com:1521:hive

    Credentials¶

    Field Description
    Username Database user with read access
    Password User's password

    Security Note

    Passwords are stored encrypted. Consider using a service account with minimal required permissions.

    Partition Discovery¶

    Configure how HMS-Mirror discovers table partitions.

    Option Description
    Auto Automatically discover partitions
    Manual Specify partition locations manually

    Authentication Methods¶

    No Authentication¶

    Use for development clusters without security enabled.

    No additional configuration required in the URI.

    Kerberos Authentication¶

    For clusters with Kerberos security, the service principal must be included in the URI:

    jdbc:hive2://hs2.example.com:10000/default;principal=hive/_HOST@YOUR.REALM

    Prerequisites - Kerberos credentials configured in Utilities > Preferences > Kerberos - Access to KDC - Properly configured /etc/krb5.conf on the HMS-Mirror server

    Important: The Kerberos service principal must be in the JDBC URI, not as a separate connection property. Your user credentials (keytab or password) are managed centrally through the Preferences > Kerberos tab.

    See Kerberos Authentication for detailed configuration.

    LDAP Authentication¶

    For clusters using username/password authentication, include credentials in the URI:

    jdbc:hive2://hs2.example.com:10000/default;user=your-username;password=your-password

    Knox Gateway Authentication¶

    For clusters accessed through Apache Knox, include all parameters in the URI:

    jdbc:hive2://knox.example.com:8443/;ssl=true;transportMode=http;httpPath=gateway/cdp-proxy-api/hive;user=knox-username;password=knox-password

    Testing Connections¶

    Always test connections before saving.

    Running a Test¶

    1. After configuring the connection, click Test Connection
    2. HMS-Mirror attempts to connect and run basic queries
    3. Review the results

    Successful Test¶

    ✓ Connection Successful

HiveServer2:
  ✓ Connected successfully
  ✓ Server version: 3.1.3000
  ✓ Databases found: 25

Metastore Direct (if configured):
  ✓ Connected successfully
  ✓ Metastore version: 3.1.0

    Failed Test¶

    ✗ Connection Failed

Error: Could not establish connection
Details: java.sql.SQLException: Cannot connect to jdbc:hive2://...
Cause: Connection refused

Suggestions:
- Verify the host and port are correct
- Check network connectivity
- Verify the service is running

    Troubleshooting Connection Failures¶

    Error Possible Cause Solution
    Connection refused Service not running or wrong port Verify HS2 is running, check port
    Host not found DNS or hostname issue Use IP address or fix DNS
    Authentication failed Wrong credentials Verify username/password or Kerberos ticket
    SSL handshake failed Certificate issue Check trust store configuration
    Timeout Network/firewall issue Check firewall rules and network path

    Editing Connections¶

    Modify an Existing Connection¶

    1. Navigate to Connections
    2. Click the Edit button on the connection card
    3. Make your changes
    4. Click Test Connection to verify
    5. Click Save

    What Can Be Changed¶

    • Connection name and description
    • HiveServer2 URI and properties
    • Metastore Direct settings
    • Authentication configuration
    • Session properties

    What Can't Be Changed¶

    Some operations require creating a new connection: - If jobs reference the connection and are running

    Deleting Connections¶

    Remove a Connection¶

    1. Navigate to Connections
    2. Click the Delete button on the connection card
    3. Confirm the deletion

    Deletion Restrictions¶

    You cannot delete a connection that is: - Currently in use by a running job - Referenced by saved jobs (you'll be warned)

    Handling Referenced Connections¶

    If a connection is referenced by jobs: 1. Update or delete the referencing jobs first 2. Then delete the connection

    Or force delete (will invalidate referencing jobs).

    Importing and Exporting¶

    Export a Connection¶

    Export connection configuration for backup or sharing:

    1. Click the Export button on a connection
    2. Save the YAML file
    name: prod-cluster
legacyHive: false
hiveServer2:
  uri: jdbc:hive2://hs2.example.com:10000/default;transportMode=binary;ssl=true
metastoreDirect:
  type: MYSQL
  uri: jdbc:mysql://metastore.example.com:3306/hive

    Import a Connection¶

    Import a previously exported connection:

    1. Click Import Connection
    2. Select or drag the YAML file
    3. Review the imported settings
    4. Click Save

    Note

    Sensitive data (passwords) are not included in exports. You'll need to re-enter them after import.

    Session Properties¶

    Session properties are Hive configuration settings applied when HMS-Mirror connects.

    Common Session Properties¶

    Property Description Example
    hive.exec.dynamic.partition Enable dynamic partitioning true
    hive.exec.dynamic.partition.mode Partitioning mode nonstrict
    hive.exec.compress.output Compress query output true
    hive.exec.parallel Enable parallel execution true

    Adding Session Properties¶

    Session properties are configured separately from connection properties and are applied after the connection is established.

    Best Practices¶

    Naming Conventions¶

    Use clear, descriptive names: - Include environment: prod-, dev-, test- - Include cluster identifier: -hdp-cluster, -cdp-east - Example: prod-cdp-east-cluster

    Security¶

    • Use service accounts with minimum required permissions
    • Rotate credentials regularly
    • Use keytab files instead of passwords where possible
    • Don't share connections with production credentials

    Organization¶

    • Create separate connections for each environment
    • Document connection purposes in descriptions
    • Group related connections by naming convention

    Testing¶

    • Always test connections after creation
    • Re-test after any infrastructure changes
    • Test periodically to catch configuration drift
