    Reports and Results¶

    After running a migration job, HMS-Mirror generates detailed reports. This guide explains how to view, interpret, and use these reports.

    Accessing Reports¶

    From Jobs¶

    After a job completes: 1. Navigate to Jobs → Build 2. Find the completed job 3. Click View Report or the report icon

    From Reports Page¶

    View all historical reports: 1. Navigate to Reports from the main menu 2. Browse or search for reports 3. Click to view details

    Generated SQL¶

    View all generated SQL statements.

    SQL Sections¶

    DDL Scripts 

    -- RIGHT Cluster DDL

USE sales_db;

CREATE EXTERNAL TABLE IF NOT EXISTS customers (
  customer_id INT,
  name STRING,
  email STRING
)
STORED AS PARQUET
LOCATION 's3a://data-lake/warehouse/sales_db/customers';

    DML Scripts (Data Movement) 

    -- Data Movement

INSERT INTO sales_db.customers
SELECT * FROM left_sales_db.customers;

    Cleanup Scripts 

    -- Cleanup (if needed)

DROP TABLE IF EXISTS temp_customers;

    Download Scripts¶

    Click Download SQL to get scripts as files: - left_cluster.sql - Scripts for source cluster - right_cluster.sql - Scripts for target cluster - cleanup.sql - Post-migration cleanup

    Report Formats¶

    Web UI¶

    Interactive viewing with: - Expandable sections - Sortable tables - Search and filter - Click-through details

    Markdown Export¶

    Human-readable text format: - Good for documentation - Version control friendly - Email attachments

    YAML Export¶

    Machine-readable format: - Automation integration - Custom processing - Archival

    JSON Export¶

    API-friendly format: - Integration with other tools - Programmatic access - Dashboard feeds

    Report Management¶

    Viewing History¶

    All reports are saved and accessible: 1. Navigate to Reports 2. Filter by date, job, or status 3. Click to view any historical report

    Comparing Reports¶

    Compare runs of the same job: - View changes between runs - Track progress over time - Identify regressions

    Exporting Reports¶

    1. Open a report
    2. Click Export
    3. Choose format (Markdown, YAML, JSON)
    4. Download the file

    Archiving Reports¶

    Reports are stored in RocksDB: - Automatically retained - No manual management needed - Backed up with RocksDB directory

    Understanding Results¶

    Success Criteria¶

    A table is successful when: - Schema created on target - Data moved (if applicable) - Partitions registered - No errors during execution

    Failure Reasons¶

    Common failure categories:

    Category Examples
    Permission Can't create table, can't write data
    Resource Timeout, out of memory
    Configuration Partition limit, size threshold
    Data Type mismatch, corrupt data
    Connectivity Network issues, service unavailable

    Skipped Tables¶

    Tables may be skipped due to: - Exclude patterns matching - Table type not supported - Configuration settings (e.g., skip ACID)

    Using Reports for Remediation¶

    Re-running Failed Tables¶

    1. Note failed tables from report
    2. Adjust configuration or limits
    3. Create new dataset with only failed tables
    4. Run new job with adjusted settings

    Verifying Migration¶

    Use reports to verify: - [ ] All expected tables migrated - [ ] Row counts match (if available) - [ ] No critical errors - [ ] Warnings are acceptable

    Documentation¶

    Export reports for: - Migration documentation - Audit trails - Stakeholder communication - Post-migration review
