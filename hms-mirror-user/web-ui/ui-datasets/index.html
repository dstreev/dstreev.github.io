    Managing Datasets¶

    Datasets define what databases and tables to migrate. This guide covers creating and managing datasets through the web UI.

    What is a Dataset?¶

    A dataset specifies:

    • Which databases to include in a migration
    • Which tables within those databases (using filters)
    • Database-specific settings like renaming or custom locations

    Datasets are reusable - define them once and use them in multiple jobs with different configurations or strategies.

    Viewing Datasets¶

    Datasets Page¶

    Navigate to Datasets from the main menu to see all saved datasets.

    Each dataset card displays: - Dataset name - Description (if provided) - Number of databases included - Quick action buttons

    Dataset Details¶

    Click on a dataset to see: - List of included databases - Table filter patterns for each database - Target database settings

    Creating a Dataset¶

    New Dataset Form¶

    1. Click New Dataset or the + button
    2. Fill in the basic information
    3. Add databases with their settings
    4. Save the dataset

    Basic Information¶

    Field Description Example
    Name Unique identifier sales-tables
    Description Purpose and scope All tables from sales databases

    Adding Databases¶

    Click Add Database to include a database in the dataset.

    Database Settings¶

    Field Description Required
    Database Name Source database name Yes
    Include Pattern Regex for tables to include No
    Exclude Pattern Regex for tables to exclude No
    Target Name Rename database on target No
    Target Warehouse Custom location on target No

    Table Filtering¶

    Use regex patterns to control which tables are included.

    Include Pattern¶

    Tables matching this pattern are included. Leave empty to include all.

    Pattern Description Matches
    .* All tables Everything
    fact_.* Tables starting with fact_ fact_sales, fact_orders
    .*_dim Tables ending with _dim customer_dim, product_dim
    fact_.*\|dim_.* Multiple patterns fact_, dim_
    customer Exact match customer (exact)
    customer.* Starting with customer, customer_info, customers

    Exclude Pattern¶

    Tables matching this pattern are excluded, even if they match the include pattern.

    Pattern Description Excludes
    test_.* Test tables test_orders, test_data
    .*_backup Backup tables orders_backup, customers_backup
    tmp_.*\|temp_.* Temporary tables tmp_, temp_
    .*_archive Archive tables sales_archive, logs_archive

    Filter Evaluation Order¶

    1. Include pattern is applied first (defaults to all if empty)
    2. Exclude pattern removes matches from the included set
    3. Final set is processed

    Example

    Database: sales_db
Include: .*
Exclude: test_.*|.*_backup

Result:
✓ customers
✓ orders
✓ products
✗ test_orders (excluded by test_.*)
✗ customers_backup (excluded by .*_backup)

    Common Filter Patterns¶

    All fact tables 

    Include: fact_.*
Exclude: (empty)

    All tables except test and temp 

    Include: .*
Exclude: test_.*|temp_.*|tmp_.*

    Specific tables only 

    Include: customers|orders|products
Exclude: (empty)

    Tables with specific suffix 

    Include: .*_current|.*_latest
Exclude: (empty)

    Database Renaming¶

    Change the database name on the target cluster.

    Use Cases¶

    • Environment prefixes: sales → prod_sales
    • Namespace changes: legacy_db → new_db
    • Version suffixes: analytics → analytics_v2

    Configuration¶

    Source Target Name Result
    sales sales_migrated Creates sales_migrated on target
    orders prod_orders Creates prod_orders on target

    Custom Warehouse Locations¶

    Override the default warehouse location for specific databases.

    When to Use¶

    • Different storage tiers for different databases
    • Regulatory requirements for data location
    • Performance optimization (different storage backends)

    Configuration¶

    Database Target Warehouse
    pii_data s3a://secure-bucket/warehouse/pii
    analytics s3a://analytics-bucket/warehouse
    temp_data hdfs:///tmp/warehouse

    Dataset Examples¶

    Simple Single Database¶

    name: sales-all-tables
description: All tables from sales database
databases:
  - name: sales_db
    includePattern: .*

    Multiple Databases with Filters¶

    name: production-data
description: Production fact and dimension tables
databases:
  - name: sales_db
    includePattern: fact_.*|dim_.*
    excludePattern: test_.*
  - name: inventory_db
    includePattern: .*
    excludePattern: .*_staging|.*_temp
  - name: customers_db
    includePattern: customer.*|account.*

    Database Renaming¶

    name: hdp-to-cdp-migration
description: Migrate with renamed databases
databases:
  - name: sales_db
    targetName: cdp_sales_db
    includePattern: .*
  - name: inventory_db
    targetName: cdp_inventory_db
    includePattern: .*

    Custom Warehouse Locations¶

    name: tiered-storage-migration
description: Different storage tiers per database
databases:
  - name: hot_data
    targetWarehouse: s3a://fast-storage/warehouse
    includePattern: .*
  - name: cold_data
    targetWarehouse: s3a://archive-storage/warehouse
    includePattern: .*
  - name: pii_data
    targetWarehouse: s3a://secure-storage/warehouse
    includePattern: .*

    Selective Table Migration¶

    name: incremental-migration-batch-1
description: First batch of tables for phased migration
databases:
  - name: enterprise_db
    includePattern: customers|orders|products|inventory
  - name: analytics_db
    includePattern: daily_.*|weekly_.*
    excludePattern: .*_test

    Editing Datasets¶

    Modify an Existing Dataset¶

    1. Navigate to Datasets
    2. Click Edit on the dataset card
    3. Modify databases, filters, or settings
    4. Save changes

    Adding/Removing Databases¶

    • Click Add Database to include new databases
    • Click the Remove button on a database to exclude it

    Updating Filters¶

    Modify include/exclude patterns to change which tables are processed.

    Note

    Filter changes take effect on the next job run.

    Copying Datasets¶

    Create a new dataset based on an existing one:

    1. Click Copy on the dataset card
    2. Enter a new name
    3. Modify as needed
    4. Save

    Importing and Exporting¶

    Export a Dataset¶

    1. Click Export on the dataset card
    2. Save the YAML file

    Import a Dataset¶

    1. Click Import Dataset
    2. Select the YAML file
    3. Review and save

    Export Format¶

    name: example-dataset
description: Example dataset for documentation
databases:
  - name: sales_db
    includePattern: ".*"
    excludePattern: "test_.*"
    targetName: null
    targetWarehouse: null
  - name: analytics_db
    includePattern: "fact_.*|dim_.*"
    excludePattern: ".*_staging"
    targetName: analytics_prod_db
    targetWarehouse: "s3a://analytics/warehouse"

    Dataset Validation¶

    When used in a job, HMS-Mirror validates datasets:

    Validation Checks¶

    • Database names are valid
    • Regex patterns are syntactically correct
    • Target names don't conflict
    • Databases exist on source cluster

    Validation Results¶

    Dataset: production-migration

✓ Passed
  - All databases found on source
  - Regex patterns valid

⚠ Warnings
  - Database 'legacy_db' has 0 tables matching filters

✗ Errors (if any)
  - Database 'missing_db' not found on source cluster
  - Invalid regex in exclude pattern: [unclosed

    Best Practices¶

    Naming Conventions¶

    Use descriptive names: - By purpose: financial-reports-tables - By environment: prod-to-dr-sync - By phase: migration-phase-1-batch-2 - By department: marketing-analytics

    Filter Design¶

    Start broad, then narrow 1. Begin with .* to include all 2. Add exclude patterns for unwanted tables 3. Refine as needed

    Test patterns first - Use a tool like regex101.com to test patterns - Verify matches before running migrations

    Be specific - Avoid overly broad patterns that might include future tables - Document pattern intentions in descriptions

    Dataset Organization¶

    By migration phase 

    migration-phase-1-critical
migration-phase-2-secondary
migration-phase-3-archive

    By data domain 

    customer-data-tables
financial-data-tables
operational-data-tables

    By frequency 

    daily-sync-dataset
weekly-sync-dataset
one-time-migration

    Performance Considerations¶

    • Smaller datasets run faster than large ones
    • Split large migrations into multiple datasets
    • Consider table sizes when grouping

    Troubleshooting¶

    No Tables Found¶

    Warning: 0 tables matched for database 'sales_db'

    Possible causes: - Include pattern too restrictive - Database name misspelled - Tables don't exist on source

    Solutions: - Verify database and table names on source - Test regex pattern - Use .* to include all, then refine

    Too Many Tables¶

    Warning: 10,000+ tables matched

    Considerations: - Migration will take longer - Consider splitting into multiple datasets - Use more restrictive filters

    Pattern Errors¶

    Error: Invalid regex pattern: [unclosed bracket

    Solution: - Fix regex syntax - Escape special characters properly - Test pattern before saving

    Conflicting Targets¶

    Error: Multiple databases targeting 'sales_db'

    Solution: - Use unique target names for each database - Remove duplicate database entries
