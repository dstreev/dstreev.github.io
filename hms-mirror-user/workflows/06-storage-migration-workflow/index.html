    STORAGE_MIGRATION Workflow¶

    Overview¶

    The STORAGE_MIGRATION strategy relocates table data within a single cluster from one storage location to another. This is used for cloud migrations (HDFS to S3/ADLS), namespace changes, warehouse reorganization, or moving to different storage tiers.

    Unlike other strategies that migrate between clusters, STORAGE_MIGRATION operates within the same metastore, changing only the physical data location.

    When to Use¶

    • Cloud adoption: Moving from HDFS to S3, ADLS, or GCS
    • Storage tiering: Moving data to different storage classes
    • Namespace changes: Reorganizing warehouse directory structure
    • Cluster consolidation: Moving data when changing HDFS namespace
    • Warehouse standardization: Aligning table locations with conventions

    Prerequisites¶

    1. Single cluster operation: Source and target are the same metastore
    2. Target storage: New storage location configured and accessible
    3. Permissions: Hive user can read source and write to target
    4. Storage capacity: Sufficient space in target location
    5. Connection: Only LEFT connection needed (same cluster)

    Key Concepts¶

    Translation Types¶

    Type Description Use Case
    RELATIVE Maintains relative path from source Moving warehouse root
    ALIGNED Aligns to target warehouse conventions Reorganizing structure

    Data Movement Strategies¶

    Strategy Description Performance
    SQL Uses INSERT to copy/transform data Slower, allows transformation
    DISTCP Direct file copy via Hadoop DistCp Faster, no transformation

    Step-by-Step Workflow¶

    Step 1: Plan Storage Layout¶

    Define your target storage structure:

    Current (HDFS):
hdfs://cluster/warehouse/
├── external/
│   └── db.db/
│       └── table/
└── managed/
    └── db.db/
        └── table/

Target (S3):
s3a://datalake-bucket/warehouse/
├── external/hive/
│   └── db.db/
│       └── table/
└── managed/hive/
    └── db.db/
        └── table/

    Step 2: Configure Connection¶

    Only LEFT connection is needed (no RIGHT for storage migration):

    1. LEFT Connection:
    2. HiveServer2 URI of the cluster
    3. HCFS Namespace: Current namespace (e.g., hdfs://cluster)
    4. Platform Type: Your cluster version

    Step 3: Define Dataset¶

    1. Create Dataset with databases/tables to migrate
    2. Consider filtering by database or table patterns
    3. Note: Large tables may need special handling

    Step 4: Create Configuration¶

    1. Transfer Settings: 

      transferConfig:
  storageMigration:
    translationType: ALIGNED  # or RELATIVE
    dataMovementStrategy: SQL  # or DISTCP
    dataFlow: PULL  # for DISTCP
    strict: true
    createArchive: false  # true to keep source copy
    skipDatabaseLocationAdjustments: false

  warehouse:
    externalDirectory: "s3a://bucket/warehouse/external/hive"
    managedDirectory: "s3a://bucket/warehouse/managed/hive"

    2. ACID Handling (if applicable):

    3. ACID tables can be migrated but may need downgrade
    4. Consider converting to EXTERNAL if moving to cloud

    Step 5: Create Job¶

    1. Data Strategy: Select STORAGE_MIGRATION

    2. Target Namespace:

    3. Set to new storage prefix (e.g., s3a://bucket)

    4. This overrides the connection's namespace

    5. References:

    6. Select LEFT connection only
    7. RIGHT connection not used

    Step 6: Validate Job¶

    1. Run validation
    2. Review translation preview: 
      Current: hdfs://cluster/warehouse/db.db/table
Target:  s3a://bucket/warehouse/external/hive/db.db/table
    3. Verify warehouse directory calculations

    Step 7: Execute (SQL Strategy)¶

    For SQL-based migration:

    1. HMS-Mirror creates transfer tables at target location
    2. Data is copied via INSERT statements
    3. Original tables are renamed (archived)
    4. Transfer tables are renamed to original names
    -- Generated SQL (simplified)
CREATE TABLE db.table_hms_mirror_transfer
LOCATION 's3a://bucket/warehouse/external/hive/db.db/table'
AS SELECT * FROM db.table;

ALTER TABLE db.table RENAME TO db.table_storage_migration_archive;
ALTER TABLE db.table_hms_mirror_transfer RENAME TO db.table;

    Step 7 (Alternative): Execute (DISTCP Strategy)¶

    For DISTCP-based migration:

    1. HMS-Mirror generates distcp commands
    2. Execute distcp commands to copy files
    3. HMS-Mirror updates table locations
    # Generated distcp command
hadoop distcp \
  -Dmapreduce.job.reduces=50 \
  -update -strategy dynamic \
  hdfs://cluster/warehouse/db.db/table \
  s3a://bucket/warehouse/external/hive/db.db/table

    Step 8: Update Database Locations¶

    Unless skipDatabaseLocationAdjustments is true:

    ALTER DATABASE db SET LOCATION 's3a://bucket/warehouse/external/hive/db.db';
ALTER DATABASE db SET MANAGEDLOCATION 's3a://bucket/warehouse/managed/hive/db.db';

    Step 9: Validate Migration¶

    1. Verify table locations: 

      DESCRIBE FORMATTED db.table;
-- Check Location points to new storage

    2. Verify data accessibility: 

      SELECT COUNT(*) FROM db.table;

    3. Verify storage: 

      # Old location should have archive or be empty
hdfs dfs -ls hdfs://cluster/warehouse/db.db/

# New location should have data
aws s3 ls s3://bucket/warehouse/external/hive/db.db/table/

    Step 10: Cleanup Archives¶

    After validation, remove archived tables:

    -- Review archives first
SHOW TABLES LIKE '*_storage_migration_archive';

-- Drop when ready
DROP TABLE db.table_storage_migration_archive;

    Or remove source data directly: 

    hdfs dfs -rm -r hdfs://cluster/warehouse/db.db/table_old

    Configuration Options¶

    For Cloud Migration (HDFS to S3)¶

    transferConfig:
  storageMigration:
    translationType: ALIGNED
    dataMovementStrategy: DISTCP
    dataFlow: PULL
    strict: true

  warehouse:
    externalDirectory: "s3a://datalake/warehouse/external"
    managedDirectory: "s3a://datalake/warehouse/managed"

    For Warehouse Reorganization¶

    transferConfig:
  storageMigration:
    translationType: ALIGNED
    dataMovementStrategy: SQL
    createArchive: true  # Keep original as backup

  warehouse:
    externalDirectory: "/new/warehouse/external"
    managedDirectory: "/new/warehouse/managed"

    Validation Checklist¶

    • Target storage locations created
    • Tables updated to new locations
    • Data accessible from new locations
    • Database LOCATION/MANAGEDLOCATION updated
    • Partitioned tables have all partitions updated
    • Archive tables created (if createArchive=true)
    • Source data ready for cleanup

    Troubleshooting¶

    Issue: "Target location already exists"¶

    Solutions: - Enable createIfNotExists - Clear target location before migration - Use different directory naming

    Issue: "Permission denied on target storage"¶

    Solutions: - Configure cloud storage credentials for Hive - Set up S3/ADLS IAM roles/service principals - Verify Hadoop credential provider configuration

    Issue: "ACID table migration failing"¶

    Solutions: - ACID tables require special handling - Consider downgrading to EXTERNAL first - Run compaction before migration

    Issue: "Large table causing timeouts"¶

    Solutions: - Use DISTCP instead of SQL - Migrate large tables separately - Increase query timeouts

    Issue: "Partitions not updating"¶

    Solutions: - Run MSCK REPAIR TABLE after migration - Check partition-level location handling - Verify all partition locations translated

    Best Practices¶

    1. Test with small dataset first: Validate configuration before full migration
    2. Use DISTCP for large tables: Much faster than SQL for big data
    3. Keep archives initially: Easier rollback if issues found
    4. Monitor storage metrics: Track space usage during migration
    5. Schedule during low activity: Migration can impact cluster performance
    6. Validate incrementally: Check tables as they complete
