    COMMON Storage Workflow¶

    Overview¶

    The COMMON strategy is used when both source and target clusters share the same underlying storage. Schema definitions are transferred between clusters while table locations remain unchanged, pointing to the shared storage.

    This is common in cloud environments where multiple clusters (dev, test, prod, or multi-tenant) access the same data lake storage.

    When to Use¶

    • Shared data lake: Multiple clusters access same S3/ADLS/GCS storage
    • Multi-cluster environments: Dev/Test/Prod clusters sharing datasets
    • Cluster upgrades: New cluster version accessing existing data
    • DR scenarios: Standby cluster shares storage with primary
    • Data sharing: Read-only access to shared datasets

    When NOT to Use¶

    • When clusters have separate storage namespaces
    • When data isolation is required between clusters
    • When you need to copy/transform data during migration

    Architecture¶

    Cluster A (Source)                    Shared Storage                Cluster B (Target)
+------------------+                +------------------+           +------------------+
| Hive Metastore A |                |                  |           | Hive Metastore B |
| - db.table       |--------------->| s3a://datalake/  |<----------| - db.table       |
|   LOCATION: s3a  |                | warehouse/       |           |   LOCATION: s3a  |
+------------------+                | db.db/table/     |           +------------------+
                                    +------------------+

    Prerequisites¶

    1. Shared storage: Both clusters can access the same storage location
    2. Storage permissions: Both clusters have appropriate read (and optionally write) permissions
    3. Compatible Hive versions: Schema compatibility between cluster versions
    4. Network access: Both clusters can reach the shared storage

    Step-by-Step Workflow¶

    Step 1: Verify Shared Storage Access¶

    Confirm both clusters can access the storage:

    # From source cluster
hdfs dfs -ls s3a://datalake/warehouse/

# From target cluster
hdfs dfs -ls s3a://datalake/warehouse/

    Step 2: Configure Connections¶

    1. LEFT (Source) Connection:
    2. HiveServer2 URI of source cluster

    3. HCFS Namespace: Shared storage (e.g., s3a://datalake)

    4. RIGHT (Target) Connection:

    5. HiveServer2 URI of target cluster
    6. HCFS Namespace: Same shared storage (e.g., s3a://datalake)

    Important: Both connections should have the same HCFS Namespace since storage is shared.

    Step 3: Define Dataset¶

    1. Create Dataset with databases/tables to share
    2. Tables will be registered on target without data movement

    Step 4: Create Configuration¶

    1. Basic Settings:

    2. createIfNotExists: Enable for idempotent operations

    3. Ownership Transfer (optional): 

      transferOwnership:
  database: false  # Usually keep original ownership
  table: false

    4. Views (if needed):

    5. Enable view migration if views should be available on target

    Step 5: Create Job¶

    1. Data Strategy: Select COMMON
    2. References: Select Dataset, Configuration, both Connections

    Step 6: Execute Migration¶

    1. Execute job
    2. HMS-Mirror creates tables on target cluster:
    -- Generated DDL (location unchanged)
CREATE EXTERNAL TABLE target_db.shared_table (
  col1 STRING,
  col2 INT
)
STORED AS PARQUET
LOCATION 's3a://datalake/warehouse/db.db/shared_table';

    Step 7: Validate¶

    1. Verify tables on target: 

      USE target_db;
SHOW TABLES;
DESCRIBE FORMATTED shared_table;
-- Verify LOCATION is same as source

    2. Verify data access: 

      SELECT COUNT(*) FROM shared_table;

    3. Verify both clusters see same data: 

      -- Run on both clusters, should match
SELECT COUNT(*) FROM db.table;

    Use Cases¶

    Use Case 1: Cluster Upgrade (Side-by-Side)¶

    Upgrade from HDP to CDP while sharing storage:

    HDP 3.1 Cluster          S3 Data Lake           CDP 7.2 Cluster
+---------------+       +---------------+       +---------------+
| Metastore     |       |               |       | Metastore     |
| db.fact_sales |------>| s3://bucket/  |<------| db.fact_sales |
|               |       | warehouse/    |       |               |
+---------------+       +---------------+       +---------------+

1. Set up CDP cluster
2. Run COMMON migration to register tables
3. Test applications on CDP
4. Cut over when ready
5. Decommission HDP cluster

    Use Case 2: Multi-Tenant Data Sharing¶

    Share datasets between team clusters:

    Team A Cluster                      Team B Cluster
+---------------+                   +---------------+
| db_team_a     |                   | shared_data   |
| (read/write)  |                   | (read only)   |
+---------------+                   +---------------+
       |                                   ^
       v                                   |
+------------------------------------------+
| s3://datalake/shared/                    |
| └── team_a.db/                           |
|     └── shared_dataset/                  |
+------------------------------------------+

    Use Case 3: Disaster Recovery¶

    Standby cluster with shared storage:

    Primary Cluster                     Standby Cluster
+---------------+                   +---------------+
| Active        |                   | Passive       |
| Metastore     |                   | Metastore     |
| (read/write)  |                   | (read only)   |
+---------------+                   +---------------+
       |                                   |
       v                                   v
+------------------------------------------+
| s3://datalake/warehouse/                 |
| (shared storage with replication)        |
+------------------------------------------+

Normal: Primary handles all workloads
Failover: Switch to standby, promote to primary

    Important Considerations¶

    Data Ownership¶

    With COMMON storage, only one cluster should "own" the data:

    • Owner cluster: Can CREATE, INSERT, UPDATE, DELETE
    • Consumer clusters: Should be read-only
    -- On consumer cluster, consider read-only views
CREATE VIEW readonly_sales AS
SELECT * FROM shared_db.sales;
-- Application uses view instead of base table

    Partition Management¶

    Partitions created on one cluster need to be visible on others:

    -- After new partitions added on source
-- Run on target cluster to discover them
MSCK REPAIR TABLE db.partitioned_table;

    Schema Changes¶

    Schema changes on source must be propagated:

    1. Option A: Re-run COMMON migration
    2. Option B: Manually sync schema changes
    3. Option C: Use schema registry for coordination

    Concurrency¶

    With shared storage, concurrent writes can cause issues:

    • Use table-level locking if both clusters write
    • Prefer single-writer, multiple-reader model
    • Consider ACID tables for transactional consistency

    Validation Checklist¶

    • Storage accessible from both clusters
    • Tables registered on target cluster
    • Locations match source tables
    • Data query results match between clusters
    • Partition visibility correct
    • Permission model defined (read-only vs read-write)

    Troubleshooting¶

    Issue: "Table data not visible on target"¶

    Solutions: - Verify storage credentials on target cluster - Check storage bucket policies/ACLs - Run MSCK REPAIR for partitioned tables

    Issue: "Schema mismatch between clusters"¶

    Solutions: - Hive versions may have different defaults - Re-run migration with updated configuration - Manually adjust schema on target

    Issue: "Partition not visible on target"¶

    Solutions: - Run MSCK REPAIR TABLE tablename - Partitions created after migration need sync

    Issue: "Permission denied on target writes"¶

    Solutions: - Verify IAM roles/policies allow target cluster writes - Check if table is meant to be read-only on target - Configure appropriate storage permissions
