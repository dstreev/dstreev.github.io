    ACID Table Migration Workflow¶

    Overview¶

    ACID (Atomicity, Consistency, Isolation, Durability) tables in Hive provide transactional semantics including INSERT, UPDATE, DELETE, and MERGE operations. Migrating ACID tables requires special consideration due to their internal file structure and transactional nature.

    This workflow covers strategies for migrating ACID tables between clusters.

    Understanding ACID Tables¶

    ACID Table Characteristics¶

    Property Value Migration Impact
    Table Type MANAGED_TABLE Must handle ownership
    transactional true Requires special handling
    Format ORC (typically) Format preserved or converted
    Bucketing Optional (required in older Hive) May need adjustment
    Delete Deltas Present Must be compacted or converted

    ACID File Structure¶

    table_name/
├── base_0000001/          # Base files
│   ├── bucket_00000
│   └── bucket_00001
├── delta_0000002_0000002/ # Insert deltas
│   ├── bucket_00000
│   └── bucket_00001
└── delete_delta_0000003/  # Delete deltas (if any)
    ├── bucket_00000
    └── bucket_00001

    Migration Options¶

    Option 1: ACID Downgrade (Recommended)¶

    Convert ACID tables to EXTERNAL tables with purge during migration.

    Benefits: - Simplest migration path - Works with all strategies (SQL, EXPORT_IMPORT, HYBRID) - No transactional overhead on target - Compatible with Iceberg conversion

    Configuration: 

    migrateACID:
  on: true
  only: false
  downgrade: true
  artificialBucketThreshold: 2
  partitionLimit: 500

    Option 2: ACID Preservation¶

    Migrate ACID tables as ACID on target.

    Benefits: - Preserves full transactional capabilities - No application changes needed

    Limitations: - Requires SQL strategy - Target must support full ACID - More complex, longer migration

    Configuration: 

    migrateACID:
  on: true
  only: false
  downgrade: false

    Option 3: In-Place Operations¶

    Perform ACID operations within the source cluster.

    Use Cases: - ACID compaction before migration - ACID downgrade in place - Storage migration with ACID

    Configuration: 

    migrateACID:
  on: true
  inplace: true

    Step-by-Step: ACID Downgrade Migration¶

    Step 1: Identify ACID Tables¶

    -- Find all ACID tables
SELECT d.NAME as db_name, t.TBL_NAME, t.TBL_TYPE
FROM metastore.TBLS t
JOIN metastore.DBS d ON t.DB_ID = d.DB_ID
JOIN metastore.TABLE_PARAMS p ON t.TBL_ID = p.TBL_ID
WHERE p.PARAM_KEY = 'transactional'
  AND p.PARAM_VALUE = 'true';

    Step 2: Check for Pending Transactions¶

    -- Verify no open transactions
SHOW TRANSACTIONS;

-- Check compaction status
SHOW COMPACTIONS;

    Step 3: Run Compaction (Recommended)¶

    Compact ACID tables before migration for cleaner data:

    -- Major compaction for each ACID table
ALTER TABLE db.acid_table COMPACT 'major';

-- Monitor compaction
SHOW COMPACTIONS;

    Step 4: Configure Migration¶

    # Configuration
migrateACID:
  on: true
  downgrade: true  # Convert to EXTERNAL
  artificialBucketThreshold: 2
  partitionLimit: 500

# For explicit tables-only migration
# only: true

    Step 5: Create Dataset¶

    Include ACID tables in your dataset:

    databases:
  - databaseName: your_database
    filter:
      # Optional: filter specific tables
      includeRegEx: ".*"

    Step 6: Create Job¶

    1. Data Strategy: SQL (required for ACID with downgrade)
    2. Configure hybrid settings for partition limits

    Step 7: Execute Migration¶

    The migration process:

    1. Read: HMS-Mirror reads ACID table via HiveServer2
    2. Transform: Data extracted, transactions materialized
    3. Write: Data written as standard ORC/Parquet files
    4. Create: EXTERNAL table created on target with purge flag

    Generated DDL Example: 

    -- Original ACID table becomes:
CREATE EXTERNAL TABLE target_db.former_acid_table (
  id BIGINT,
  name STRING,
  value DECIMAL(10,2)
)
STORED AS ORC
LOCATION 'hdfs://target/warehouse/external/db.db/former_acid_table'
TBLPROPERTIES (
  'external.table.purge'='true'  -- Data deleted on DROP
);

-- Data migrated via:
INSERT INTO target_db.former_acid_table
SELECT * FROM source_db.acid_table;

    Step 8: Validate¶

    -- On target cluster
DESCRIBE FORMATTED former_acid_table;
-- Verify: Table Type = EXTERNAL_TABLE
-- Verify: transactional = null or false
-- Verify: external.table.purge = true

-- Verify data
SELECT COUNT(*) FROM former_acid_table;

    Handling Artificial Buckets¶

    Older Hive versions required bucket clauses for ACID. These "artificial" buckets should be removed:

    migrateACID:
  artificialBucketThreshold: 2

    How it works: - Tables with CLUSTERED BY and <= threshold buckets have bucketing removed - Tables with > threshold buckets keep their bucketing (assumed intentional)

    Example: 

    -- Source (HDP with artificial bucket)
CREATE TABLE acid_table (id INT, name STRING)
CLUSTERED BY (id) INTO 2 BUCKETS
STORED AS ORC
TBLPROPERTIES ('transactional'='true');

-- Target (bucket removed, threshold=2)
CREATE EXTERNAL TABLE acid_table (id INT, name STRING)
STORED AS ORC
TBLPROPERTIES ('external.table.purge'='true');

    Partition Limits¶

    ACID tables with many partitions require careful handling:

    migrateACID:
  partitionLimit: 500  # Tables exceeding this are skipped

    For partition-heavy ACID tables: 1. Increase limit if cluster resources allow 2. Migrate in partition ranges 3. Use SCHEMA_ONLY + distcp with compaction

    In-Place ACID Operations¶

    For operations within the same cluster:

    In-Place Compaction¶

    migrateACID:
  on: true
  inplace: true
# Run compaction without migration

    In-Place Downgrade¶

    Convert ACID to EXTERNAL without moving between clusters:

    migrateACID:
  on: true
  inplace: true
  downgrade: true

    This creates a new EXTERNAL table from the ACID table data.

    ACID to Iceberg Conversion¶

    Convert ACID tables directly to Iceberg format:

    migrateACID:
  on: true
  downgrade: true  # First downgrade from ACID

icebergConversion:
  enable: true
  fileTypeTranslation: PARQUET
  version: 2

    Process: 1. ACID table read via SQL 2. Transactions materialized 3. Data written as Iceberg table 4. Full Iceberg capabilities available

    Troubleshooting¶

    Issue: "Timeout reading ACID table"¶

    Solutions: - Run major compaction first - Reduce partition scope - Increase query timeout in overrides

    Issue: "Bucket count mismatch"¶

    Solutions: - Set artificialBucketThreshold appropriately - Verify source table bucketing intent

    Issue: "Open transactions preventing read"¶

    Solutions: - Wait for transactions to complete - Abort orphaned transactions (carefully) - Run compaction to finalize

    Issue: "Out of memory during migration"¶

    Solutions: - Reduce concurrent table migrations - Increase Tez container size - Migrate large tables separately

    Issue: "Delete deltas not handled"¶

    Solutions: - Run major compaction before migration - SQL strategy materializes deletes automatically

    Best Practices¶

    1. Compact before migrating: Major compaction creates cleaner files
    2. Downgrade when possible: Simpler and faster migration
    3. Monitor transactions: Ensure no open transactions
    4. Test with sample first: Validate approach on subset
    5. Plan for large tables: ACID tables often contain more data

    Validation Checklist¶

    • All ACID tables identified
    • Compaction completed (if run)
    • Tables migrated to target
    • Data counts match source
    • External/purge flags set correctly
    • Bucketing handled appropriately
    • Applications tested with new tables
