    CONVERT_LINKED Workflow¶

    Overview¶

    The CONVERT_LINKED strategy converts previously created LINKED tables (tables pointing to source cluster storage) into regular tables pointing to target cluster storage. This is used after data has been copied to the target via external means (distcp, cloud sync, etc.).

    This completes the second phase of a two-phase migration: first LINKED for testing, then CONVERT_LINKED after data copy.

    When to Use¶

    • After LINKED migration: Tables were created using LINKED strategy
    • After data copy: Data has been copied to target storage externally
    • Testing complete: Validation on linked tables is finished
    • Ready for cutover: Ready to make target tables independent

    Prerequisites¶

    1. Existing LINKED tables: Tables on target pointing to source storage
    2. Data copied: Target storage contains the data files
    3. Both connections: Source and target clusters accessible
    4. Permissions: ALTER TABLE permissions on target

    Two-Phase Migration Overview¶

    Phase 1: LINKED
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│  Source Cluster │     │ Target Cluster  │     │                 │
│                 │     │                 │     │  Applications   │
│  db.table       │────>│  db.table       │<────│  (testing)      │
│  LOCATION: src  │     │  LOCATION: src  │     │                 │
└─────────────────┘     └─────────────────┘     └─────────────────┘
         │
         │ Tables point to source, apps test on target
         v

Phase 2: Data Copy (External - distcp/sync)
┌─────────────────┐                          ┌─────────────────┐
│  Source Storage │ ──────distcp───────────> │ Target Storage  │
│  hdfs://src/... │                          │ hdfs://tgt/...  │
└─────────────────┘                          └─────────────────┘

Phase 3: CONVERT_LINKED
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│  Source Cluster │     │ Target Cluster  │     │                 │
│                 │     │                 │     │  Applications   │
│  db.table       │     │  db.table       │<────│  (production)   │
│  LOCATION: src  │     │  LOCATION: tgt  │     │                 │
└─────────────────┘     └─────────────────┘     └─────────────────┘
                               │
                               │ Tables now point to target storage
                               v

    Step-by-Step Workflow¶

    Step 1: Verify LINKED Tables Exist¶

    1. Connect to target cluster
    2. Verify tables are LINKED (point to source): 
      DESCRIBE FORMATTED db.table;
-- Location should show source cluster path
-- e.g., hdfs://source-cluster/warehouse/db.db/table

    Step 2: Copy Data to Target Storage¶

    Use external tools to copy data:

    Option A: Hadoop DistCp 

    # Use distcp workbook from LINKED job
hadoop distcp \
  -update \
  hdfs://source-cluster/warehouse/db.db/table \
  hdfs://target-cluster/warehouse/db.db/table

    Option B: Cloud Provider Tools 

    # AWS
aws s3 sync s3://source-bucket/warehouse/ s3://target-bucket/warehouse/

# Azure
azcopy sync "source-container" "target-container"

    Step 3: Verify Data Copy¶

    # Compare file counts
hdfs dfs -count hdfs://source-cluster/warehouse/db.db/table
hdfs dfs -count hdfs://target-cluster/warehouse/db.db/table

# Compare sizes
hdfs dfs -du -s hdfs://source-cluster/warehouse/db.db/table
hdfs dfs -du -s hdfs://target-cluster/warehouse/db.db/table

    Step 4: Configure Connections¶

    Same connections used for LINKED migration:

    1. LEFT Connection: Source cluster
    2. RIGHT Connection: Target cluster with correct HCFS namespace

    Step 5: Use Same Dataset¶

    Use the same Dataset from LINKED migration (or create matching one).

    Step 6: Create Configuration¶

    # Configuration for CONVERT_LINKED
createIfNotExists: true

# Warehouse should match target storage layout
transferConfig:
  warehouse:
    externalDirectory: "hdfs://target-cluster/warehouse/external"
    managedDirectory: "hdfs://target-cluster/warehouse/managed"

    Step 7: Create Job¶

    1. Data Strategy: Select CONVERT_LINKED
    2. References: Same dataset, configuration, connections as LINKED job

    Step 8: Validate Job¶

    1. Run validation
    2. Review location translations: 
      Current (LINKED): hdfs://source-cluster/warehouse/db.db/table
Target (CONVERTED): hdfs://target-cluster/warehouse/db.db/table
    3. Verify all linked tables are identified

    Step 9: Execute Conversion¶

    1. Execute job
    2. HMS-Mirror runs ALTER TABLE commands: 
      ALTER TABLE db.table SET LOCATION 'hdfs://target-cluster/warehouse/db.db/table';

-- For partitioned tables, each partition is also updated
ALTER TABLE db.table PARTITION (year=2024, month=1)
SET LOCATION 'hdfs://target-cluster/warehouse/db.db/table/year=2024/month=1';

    Step 10: Validate Conversion¶

    1. Verify new locations: 

      DESCRIBE FORMATTED db.table;
-- Location should now show target cluster path

    2. Verify data access: 

      SELECT COUNT(*) FROM db.table;
-- Should return same count as before

    3. Verify partitions updated: 

      SHOW PARTITIONS db.table;
-- Each partition should point to target storage

    Step 11: Update Purge Flag (if applicable)¶

    Tables migrated from legacy managed to EXTERNAL may need purge flag:

    -- If table was legacy managed and should delete data on drop
ALTER TABLE db.table SET TBLPROPERTIES ('external.table.purge'='true');

    HMS-Mirror handles this automatically based on source table type.

    Handling Partitioned Tables¶

    Partitioned tables require updating each partition location:

    -- HMS-Mirror generates for each partition
ALTER TABLE db.partitioned_table
PARTITION (year=2024, month=1)
SET LOCATION 'hdfs://target/warehouse/db.db/table/year=2024/month=1';

ALTER TABLE db.partitioned_table
PARTITION (year=2024, month=2)
SET LOCATION 'hdfs://target/warehouse/db.db/table/year=2024/month=2';
-- ... for all partitions

    For tables with many partitions, this can take significant time.

    Legacy Managed Table Handling¶

    Tables that were "legacy managed" (pre-Hive 3 managed without ACID) are converted to EXTERNAL during LINKED. CONVERT_LINKED sets the appropriate purge behavior:

    Original Type After LINKED After CONVERT_LINKED
    Legacy Managed EXTERNAL (no purge) EXTERNAL with purge=true
    EXTERNAL EXTERNAL EXTERNAL (unchanged)
    ACID Handled separately Handled separately

    Validation Checklist¶

    • All linked tables identified
    • Data copy to target verified
    • Table locations updated to target
    • Partition locations updated (for partitioned tables)
    • Data accessible via target queries
    • Purge flag set correctly for legacy tables
    • Row counts match expected values

    Troubleshooting¶

    Issue: "Data not found at new location"¶

    Solutions: - Verify data copy completed successfully - Check target path matches expected location - Ensure distcp preserved directory structure

    Issue: "Partition locations not updating"¶

    Solutions: - Very large partition counts take time - Check for partition copy completeness - Run MSCK REPAIR TABLE to sync partitions

    Issue: "Table still shows source location"¶

    Solutions: - Verify ALTER TABLE executed successfully - Check for permission issues on target - Re-run conversion for failed tables

    Issue: "Permission denied on ALTER"¶

    Solutions: - Verify Hive user has ALTER permissions - Check table ownership on target - Ensure target metastore allows modifications

    Best Practices¶

    1. Complete data copy first: Ensure all data is on target before converting
    2. Verify file integrity: Compare checksums if critical
    3. Test before cutover: Run queries against both locations
    4. Keep source temporarily: Don't delete source until conversion verified
    5. Handle large tables separately: Partition-heavy tables may need extra time
