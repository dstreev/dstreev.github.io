    Phased Migration Workflow¶

    Overview¶

    Large-scale migrations benefit from a phased approach that reduces risk, allows validation at each stage, and provides rollback points. This workflow covers planning and executing multi-phase migrations.

    Why Phased Migration?¶

    Approach Risk Rollback Validation Duration
    Big Bang High Difficult Post-migration Short
    Phased Low Easy Continuous Longer

    Phased Migration Strategies¶

    Strategy 1: By Database Priority¶

    Phase 1: Non-critical databases (dev, test, sandbox)
Phase 2: Supporting databases (reference data, dimensions)
Phase 3: Core databases (transactions, facts)
Phase 4: Critical databases (real-time, customer-facing)

    Strategy 2: By Table Type¶

    Phase 1: External tables (simplest)
Phase 2: Non-ACID managed tables
Phase 3: ACID tables
Phase 4: Views (depend on tables)

    Strategy 3: By Application¶

    Phase 1: Batch analytics (overnight processing)
Phase 2: Reporting (business hours)
Phase 3: Ad-hoc queries (interactive)
Phase 4: Real-time (streaming)

    Strategy 4: By Data Freshness¶

    Phase 1: Historical/archive data (cold)
Phase 2: Recent data (warm)
Phase 3: Current data (hot)
Phase 4: Real-time sync (active)

    Step-by-Step Workflow¶

    Phase 0: Planning¶

    Step 0.1: Inventory and Categorize¶

    Create inventory of all databases and tables:

    -- Get database inventory
SELECT
  d.NAME as database_name,
  COUNT(t.TBL_ID) as table_count,
  SUM(CASE WHEN p.PARAM_VALUE = 'true' THEN 1 ELSE 0 END) as acid_count
FROM metastore.DBS d
LEFT JOIN metastore.TBLS t ON d.DB_ID = t.DB_ID
LEFT JOIN metastore.TABLE_PARAMS p ON t.TBL_ID = p.TBL_ID
  AND p.PARAM_KEY = 'transactional'
GROUP BY d.NAME;

    Step 0.2: Identify Dependencies¶

    Map dependencies between databases/tables:

    fact_sales → dim_product, dim_customer, dim_date
reporting_views → fact_sales, fact_inventory

    Step 0.3: Define Phases¶

    Create phase plan:

    Phase Databases Dependencies Duration Rollback Plan
    1 dev_, test_ None 1 week Delete and retry
    2 dim_* None 1 week Delete and retry
    3 fact_* dim_* 2 weeks Restore from source
    4 reporting_* fact_, dim_ 1 week Recreate views

    Step 0.4: Create Datasets for Each Phase¶

    Phase 1 Dataset: 

    name: "phase-1-dev-test"
databases:
  - databaseName: dev_analytics
  - databaseName: test_analytics
  - databaseName: sandbox

    Phase 2 Dataset: 

    name: "phase-2-dimensions"
databases:
  - databaseName: dimension_data
    filter:
      includeRegEx: "dim_.*"

    Continue for each phase...

    Phase 1: Non-Critical Migration¶

    Step 1.1: Create Phase 1 Job¶

    1. Dataset: phase-1-dev-test
    2. Strategy: HYBRID (or appropriate)
    3. Configuration: Standard settings

    Step 1.2: Validate Phase 1¶

    1. Run job validation
    2. Review table counts
    3. Check for issues

    Step 1.3: Execute Phase 1¶

    1. Execute migration
    2. Monitor progress
    3. Handle errors

    Step 1.4: Verify Phase 1¶

    1. Verify tables on target
    2. Run sample queries
    3. Connect test applications
    4. Validate for 1-2 weeks

    Step 1.5: Sign-off Phase 1¶

    • All tables migrated
    • Data verified
    • Applications tested
    • Performance acceptable
    • Stakeholder approval

    Phase 2-N: Repeat Pattern¶

    For each subsequent phase:

    1. Pre-requisites: Verify previous phase complete
    2. Create Job: For phase dataset
    3. Validate: Pre-execution checks
    4. Execute: Run migration
    5. Verify: Post-migration validation
    6. Sign-off: Stakeholder approval

    Final Phase: Views and Dependencies¶

    Views should be migrated last:

    # Views configuration
migrateVIEW:
  on: true

# Dataset for views
databases:
  - databaseName: reporting
    filter:
      tableTypes: ["VIRTUAL_VIEW"]

    Views are automatically created after all base tables.

    Managing Between Phases¶

    Handling Source Changes¶

    During extended migrations, source tables may change:

    Option A: Freeze and Migrate 1. Freeze source changes during phase 2. Migrate tables 3. Unfreeze after verification

    Option B: Delta Sync 1. Enable DR mode with sync 2. Run sync jobs to catch changes 3. Final sync before cutover

    Rollback Procedures¶

    For each phase, document rollback:

    Phase Rollback Procedure:
1. Identify affected tables
2. Drop target tables
3. Verify source still intact
4. Re-run migration when ready
5. Notify stakeholders

    Communication Plan¶

    Event Notify Method
    Phase start All stakeholders Email
    Phase complete All stakeholders Email
    Issues found Technical team Slack/Teams
    Rollback required All stakeholders Email + meeting

    Parallel Phase Execution¶

    For independent databases, phases can run in parallel:

    Week 1: Phase 1A (dev) + Phase 1B (test)
Week 2: Phase 2A (dim) + Phase 2B (reference)
Week 3: Phase 3 (facts - sequential, dependent)
Week 4: Phase 4 (views - sequential, dependent)

    Best Practices¶

    1. Start small: First phase should be low-risk
    2. Allow validation time: Don't rush between phases
    3. Document everything: Issues, decisions, sign-offs
    4. Communicate regularly: Status updates to stakeholders
    5. Have rollback ready: Test rollback procedures
    6. Learn and adapt: Apply learnings to later phases

    Validation Checklist¶

    Per Phase: - [ ] All tables migrated - [ ] Data counts verified - [ ] Sample queries successful - [ ] Application connectivity tested - [ ] Performance acceptable - [ ] Dependencies satisfied - [ ] Stakeholder sign-off

    Overall: - [ ] All phases complete - [ ] All applications migrated - [ ] Final validation complete - [ ] Source decommissioning planned
