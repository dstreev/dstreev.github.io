    ACID Downgrade Workflow¶

    Overview¶

    ACID downgrade converts transactional (ACID) tables to non-transactional EXTERNAL tables. This is often the simplest migration path when full ACID capabilities aren't needed on the target cluster, or when migrating to platforms with limited ACID support.

    When to Downgrade¶

    Scenario Recommendation
    Migration to cloud storage Downgrade (cloud may have ACID limitations)
    Iceberg conversion planned Downgrade (Iceberg provides better ACID)
    Read-heavy workloads Downgrade (no transactional overhead)
    Mixed platform targets Downgrade (broader compatibility)
    Full ACID needed Do NOT downgrade

    What Changes During Downgrade¶

    Attribute ACID Table Downgraded Table
    Table Type MANAGED_TABLE EXTERNAL_TABLE
    transactional true false/null
    Purge Behavior Always purge external.table.purge=true
    File Format ORC (typically) Same or converted
    Bucket Clause Often required Removed if artificial
    DELETE/UPDATE Supported Not supported

    Step-by-Step Workflow¶

    Step 1: Identify ACID Tables¶

    -- Find all ACID tables
SELECT
  d.NAME as db_name,
  t.TBL_NAME,
  t.TBL_TYPE
FROM metastore.TBLS t
JOIN metastore.DBS d ON t.DB_ID = d.DB_ID
JOIN metastore.TABLE_PARAMS p ON t.TBL_ID = p.TBL_ID
WHERE p.PARAM_KEY = 'transactional'
  AND p.PARAM_VALUE = 'true';

    Step 2: Verify Application Compatibility¶

    Before downgrading, confirm applications don't need: - UPDATE statements - DELETE statements - MERGE statements - Snapshot isolation

    If applications use these, either: - Modify applications to use INSERT-only patterns - Keep tables as ACID - Convert to Iceberg for modern ACID

    Step 3: Prepare for Downgrade¶

    Run Compaction (Recommended)¶

    Compact ACID tables for clean files:

    -- Major compaction for each ACID table
ALTER TABLE db.acid_table COMPACT 'major' AND WAIT;

-- Or initiate and check later
ALTER TABLE db.acid_table COMPACT 'major';

-- Monitor compaction
SHOW COMPACTIONS;

    Verify No Open Transactions¶

    -- Check for open transactions
SHOW TRANSACTIONS;

-- Verify no locks
SHOW LOCKS;

    Step 4: Configure Downgrade¶

    # Configuration for ACID downgrade
migrateACID:
  on: true
  downgrade: true
  only: false  # true to migrate ONLY ACID tables
  artificialBucketThreshold: 2  # Remove fake buckets
  partitionLimit: 500

    Step 5: Create Job¶

    1. Data Strategy: SQL (required for downgrade)
    2. Enable ACID migration with downgrade
    3. Select appropriate datasets

    Step 6: Execute Downgrade Migration¶

    HMS-Mirror executes:

    1. Reads ACID table via HiveServer2
    2. Materializes transactions: Applies all deltas
    3. Creates EXTERNAL table on target
    4. Writes clean data: Standard ORC/Parquet files
    5. Sets purge flag: For proper cleanup behavior

    Generated DDL: 

    -- Target table definition
CREATE EXTERNAL TABLE db.formerly_acid_table (
  id BIGINT,
  name STRING,
  value DECIMAL(10,2)
)
STORED AS ORC
LOCATION 'hdfs://target/warehouse/external/db.db/formerly_acid_table'
TBLPROPERTIES (
  'external.table.purge'='true',
  'hms_mirror_original_type'='MANAGED_TABLE'
);

-- Data migration
INSERT INTO db.formerly_acid_table
SELECT * FROM source_db.acid_table;

    Step 7: Validate Downgrade¶

    -- On target cluster
DESCRIBE FORMATTED db.formerly_acid_table;

-- Verify:
-- Table Type: EXTERNAL_TABLE
-- transactional: null or absent
-- external.table.purge: true

-- Verify data
SELECT COUNT(*) FROM db.formerly_acid_table;

    Handling Bucket Clauses¶

    Artificial Buckets (Remove)¶

    HDP ACID tables often have artificial bucket clauses:

    -- Source (HDP with artificial bucket)
CREATE TABLE orders (
  order_id INT,
  customer_id INT
)
CLUSTERED BY (order_id) INTO 2 BUCKETS
STORED AS ORC
TBLPROPERTIES ('transactional'='true');

    Configuration to remove: 

    migrateACID:
  artificialBucketThreshold: 2

    Result: 

    -- Target (bucket removed)
CREATE EXTERNAL TABLE orders (
  order_id INT,
  customer_id INT
)
STORED AS ORC
TBLPROPERTIES ('external.table.purge'='true');

    Intentional Buckets (Preserve)¶

    For tables with intentional bucketing:

    -- Source (intentional bucketing for joins)
CREATE TABLE customer_orders (
  order_id INT,
  customer_id INT
)
CLUSTERED BY (customer_id) INTO 32 BUCKETS
STORED AS ORC
TBLPROPERTIES ('transactional'='true');

    Set threshold appropriately: 

    migrateACID:
  artificialBucketThreshold: 2  # Only remove <= 2 buckets

    Result: 

    -- Target (bucketing preserved)
CREATE EXTERNAL TABLE customer_orders (
  order_id INT,
  customer_id INT
)
CLUSTERED BY (customer_id) INTO 32 BUCKETS
STORED AS ORC
TBLPROPERTIES ('external.table.purge'='true');

    Purge Flag Behavior¶

    With external.table.purge=true¶

    -- Dropping table DELETES the data
DROP TABLE db.table;
-- Data at location is removed

    Without Purge Flag (or false)¶

    -- Dropping table keeps the data
DROP TABLE db.table;
-- Data at location remains

    HMS-Mirror sets external.table.purge=true for downgraded tables to maintain the original MANAGED table's cleanup behavior.

    In-Place Downgrade¶

    For downgrading within the same cluster:

    migrateACID:
  on: true
  downgrade: true
  inplace: true

    Process: 1. Creates new EXTERNAL table from ACID data 2. Data rewritten to new location 3. Original ACID table optionally renamed/archived 4. New table renamed to original name

    ACID-Only Migration¶

    To migrate only ACID tables (skip non-ACID):

    migrateACID:
  on: true
  only: true  # Only process ACID tables
  downgrade: true

    Use case: Separate jobs for ACID and non-ACID tables.

    Validation Checklist¶

    • All ACID tables identified
    • Compaction completed (if run)
    • Tables converted to EXTERNAL
    • transactional property removed
    • external.table.purge set to true
    • Bucketing handled correctly
    • Data counts match source
    • Applications tested with new tables

    Troubleshooting¶

    Issue: "Compaction not completing"¶

    Solutions: - Check for blocking transactions - Increase compaction resources - Skip compaction (HMS-Mirror handles delta reads)

    Issue: "Bucket mismatch errors"¶

    Solutions: - Adjust artificialBucketThreshold - Verify bucketing intent in source tables

    Issue: "Data mismatch after downgrade"¶

    Solutions: - Verify compaction completed - Check for in-flight transactions during migration - Re-run with fresh source state

    Issue: "Applications failing after downgrade"¶

    Solutions: - Applications may use UPDATE/DELETE - Modify applications or keep as ACID - Consider Iceberg for modern ACID needs

    Best Practices¶

    1. Compact before migration: Cleaner source files
    2. Freeze writes: Prevent changes during migration
    3. Test applications: Verify with downgraded tables
    4. Document changes: Track which tables were downgraded
    5. Consider Iceberg: For modern ACID needs
