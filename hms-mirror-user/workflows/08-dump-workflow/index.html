    DUMP Workflow¶

    Overview¶

    The DUMP strategy extracts schema definitions (DDL) from a cluster and generates SQL scripts without requiring a target cluster connection. This is useful for documentation, offline planning, backup, and scenarios where you need to capture metadata for later use.

    No data is moved and no changes are made to any cluster.

    When to Use¶

    • Documentation: Capture current schema state for documentation
    • Offline planning: Generate scripts for review before migration
    • Backup: Create schema backups for disaster recovery
    • Audit: Record schema definitions for compliance
    • Script generation: Create DDL scripts for manual execution
    • No target available: When target cluster isn't ready yet

    Prerequisites¶

    1. Source connection only: Only LEFT connection needed
    2. Read permissions: Access to source metastore
    3. No target required: RIGHT connection not used

    Step-by-Step Workflow¶

    Step 1: Configure Source Connection¶

    1. Create LEFT Connection:
    2. HiveServer2 URI of source cluster
    3. HCFS Namespace of source storage
    4. Platform Type for correct DDL syntax

    Step 2: Define Dataset¶

    1. Create Dataset with databases/tables to dump
    2. Configure filters as needed
    3. All matching tables will be included in dump

    Step 3: Create Configuration¶

    1. Basic Settings:

    2. createIfNotExists: Enable (adds IF NOT EXISTS to DDL)

    3. Optional Settings:

    4. View migration: Enable to include views
    5. ACID settings: Configure for ACID table handling

    Step 4: Create Job¶

    1. Data Strategy: Select DUMP
    2. References:
    3. Dataset: Your dataset
    4. Configuration: Your configuration
    5. LEFT Connection: Source cluster
    6. RIGHT Connection: Not required (will be ignored)

    Step 5: Execute¶

    1. Execute the job
    2. HMS-Mirror generates scripts without connecting to target

    Step 6: Review Output¶

    Output files generated:

    reports/
└── job-id/
    ├── db1_execute.sql        # CREATE statements
    ├── db1_clean.sql          # DROP statements
    ├── db2_execute.sql
    ├── db2_clean.sql
    ├── runStatus.yaml         # Execution summary
    └── distcp_workbook.md     # Data location info

    Example execute.sql content: 

    -- Database: analytics
CREATE DATABASE IF NOT EXISTS analytics
COMMENT 'Analytics data warehouse'
LOCATION 'hdfs://cluster/warehouse/analytics.db';

-- Table: analytics.fact_sales
CREATE EXTERNAL TABLE IF NOT EXISTS analytics.fact_sales (
    sale_id BIGINT COMMENT 'Unique sale identifier',
    product_id INT,
    quantity INT,
    amount DECIMAL(10,2),
    sale_date DATE
)
PARTITIONED BY (year INT, month INT)
STORED AS PARQUET
LOCATION 'hdfs://cluster/warehouse/analytics.db/fact_sales'
TBLPROPERTIES (
    'transient_lastDdlTime'='1234567890'
);

-- Add partitions
ALTER TABLE analytics.fact_sales ADD IF NOT EXISTS
    PARTITION (year=2024, month=1)
    LOCATION 'hdfs://cluster/warehouse/analytics.db/fact_sales/year=2024/month=1';

    Use Cases¶

    Use Case 1: Pre-Migration Planning¶

    Generate scripts to review before actual migration:

    1. Run DUMP on source cluster
2. Review generated DDL
3. Identify potential issues:
   - Unsupported table types
   - Complex views
   - Special characters in names
4. Modify scripts if needed
5. Plan migration strategy

    Use Case 2: Schema Backup¶

    Create regular schema backups:

    # Scheduled job to dump schemas weekly
hms-mirror \
  --strategy DUMP \
  --output-dir /backups/schema/$(date +%Y%m%d)

# Store in version control
git add /backups/schema/*
git commit -m "Weekly schema backup $(date)"

    Use Case 3: Cross-Environment Script Generation¶

    Generate scripts for different environments:

    1. DUMP from production
2. Modify scripts:
   - Change database names (add _dev prefix)
   - Update locations for dev storage
   - Remove production-specific settings
3. Execute on development cluster

    Use Case 4: Compliance Documentation¶

    Generate schema documentation for auditors:

    1. Run DUMP on all databases
2. Include output in compliance package
3. Store with timestamp for audit trail
4. Use for data lineage documentation

    Configuration Options¶

    Include Everything¶

    # Dump all table types and views
migrateNonNative: true
migrateVIEW:
  on: true
migrateACID:
  on: true

    Filter Specific Tables¶

    # Dataset configuration
databases:
  - databaseName: analytics
    filter:
      includeRegEx: "fact_.*|dim_.*"
      excludeRegEx: ".*_temp|.*_bak"

    Output File Details¶

    File Contents Use
    *_execute.sql CREATE statements Schema creation
    *_clean.sql DROP statements Schema removal/cleanup
    runStatus.yaml Execution metadata Audit/tracking
    distcp_workbook.md Location information Data planning

    Validation Checklist¶

    • All databases included in output
    • All tables have DDL generated
    • Partition definitions included
    • Table properties preserved
    • Views included (if enabled)
    • Output files properly formatted

    Troubleshooting¶

    Issue: "Missing tables in output"¶

    Solutions: - Check dataset filters aren't excluding tables - Verify connection permissions - Check table types (non-native may need explicit enable)

    Issue: "Views failing to dump"¶

    Solutions: - Enable migrateVIEW.on in configuration - Views with missing base tables may fail

    Issue: "Partition information missing"¶

    Solutions: - Very large partition counts may be truncated - Check partition limit settings

    Issue: "Special characters causing issues"¶

    Solutions: - Table/column names with special characters may need escaping - Review and manually fix generated scripts

    Best Practices¶

    1. Regular backups: Schedule DUMP jobs for schema backup
    2. Version control: Store generated scripts in git
    3. Review before use: Always review scripts before execution
    4. Environment adaptation: Modify scripts for different environments
    5. Include metadata: Use runStatus.yaml for audit trails

    Using DUMP Scripts¶

    Execute on Target¶

    # Connect to target cluster and run scripts
beeline -u "jdbc:hive2://target:10000" \
  -f /path/to/db_execute.sql

    Modify for Different Environment¶

    # Change locations for different cluster
sed 's|hdfs://source-cluster|hdfs://target-cluster|g' \
  db_execute.sql > db_execute_target.sql

# Add prefix to database names
sed 's|CREATE DATABASE|CREATE DATABASE dev_|g' \
  db_execute.sql > db_execute_dev.sql
