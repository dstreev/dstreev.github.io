    On-Premises to Cloud Migration Workflow¶

    Overview¶

    This workflow covers migrating Hive data from on-premises Hadoop clusters to cloud-based data platforms. This includes migrations to AWS (S3/EMR), Azure (ADLS/HDInsight/Synapse), and Google Cloud (GCS/Dataproc).

    Migration Scenarios¶

    Scenario Source Target Strategy
    Connected On-prem HDFS Cloud Storage SQL/HYBRID via VPN
    Disconnected On-prem HDFS Cloud Storage SCHEMA_ONLY + distcp
    Intermediate On-prem HDFS Cloud Storage Multi-hop via staging

    Cloud Target Options¶

    AWS¶

    Target Storage: s3a://bucket-name/warehouse/
Target Compute: EMR, CDP on AWS
Connection: Direct Connect or VPN

    Azure¶

    Target Storage: abfs://container@account.dfs.core.windows.net/warehouse/
Target Compute: HDInsight, CDP on Azure, Synapse
Connection: ExpressRoute or VPN

    Google Cloud¶

    Target Storage: gs://bucket-name/warehouse/
Target Compute: Dataproc, CDP on GCP
Connection: Cloud Interconnect or VPN

    Step-by-Step Workflow¶

    Phase 1: Cloud Environment Setup¶

    Step 1.1: Provision Cloud Resources¶

    1. Storage:
    2. Create storage bucket/container
    3. Configure appropriate IAM/ACLs

    4. Set up encryption (at-rest and in-transit)

    5. Compute:

    6. Provision Hive cluster (EMR/HDInsight/Dataproc/CDP)
    7. Configure networking

    8. Set up HiveServer2 with appropriate authentication

    9. Networking:

    10. Establish connectivity (VPN/Direct Connect/ExpressRoute)
    11. Configure security groups/firewall rules
    12. Test connectivity

    Step 1.2: Configure Cloud Credentials¶

    On-prem cluster needs cloud storage access:

    For AWS S3: 

    <!-- core-site.xml -->
<property>
  <name>fs.s3a.access.key</name>
  <value>YOUR_ACCESS_KEY</value>
</property>
<property>
  <name>fs.s3a.secret.key</name>
  <value>YOUR_SECRET_KEY</value>
</property>

    Or use IAM roles with instance profiles.

    For Azure ADLS Gen2: 

    <!-- core-site.xml -->
<property>
  <name>fs.azure.account.auth.type</name>
  <value>OAuth</value>
</property>
<property>
  <name>fs.azure.account.oauth.provider.type</name>
  <value>org.apache.hadoop.fs.azurebfs.oauth2.ClientCredsTokenProvider</value>
</property>

    Phase 2: Connection Configuration¶

    Step 2.1: On-Premises (LEFT) Connection¶

    name: "on-prem-production"
hcfsNamespace: "hdfs://nameservice1"
hs2Uri: "jdbc:hive2://hiveserver:10000/default;principal=hive/_HOST@REALM"
platformType: "HDP3"  # or CDH6, CDP7_x

    Step 2.2: Cloud (RIGHT) Connection¶

    AWS Example: 

    name: "aws-datalake"
hcfsNamespace: "s3a://company-datalake"
hs2Uri: "jdbc:hive2://emr-master:10000/default"
platformType: "EMR6"

    Azure Example: 

    name: "azure-datalake"
hcfsNamespace: "abfs://datalake@storageaccount.dfs.core.windows.net"
hs2Uri: "jdbc:hive2://hdinsight.azurehdinsight.net:443/default;transportMode=http;httpPath=/hive2"
platformType: "CDP7_2"

    Phase 3: Choose Migration Strategy¶

    Strategy A: Direct Connected (VPN/Direct Connect)¶

    When clusters can communicate directly:

    1. Strategy: SQL, HYBRID, or EXPORT_IMPORT
    2. Data Flow: On-prem Hive → Cloud Storage
    3. Best For: Moderate data volumes with good bandwidth
    strategy: HYBRID
hybrid:
  exportImportPartitionLimit: 100
  sqlPartitionLimit: 500
  sqlSizeLimit: 1073741824

    Strategy B: Intermediate Storage¶

    When direct connectivity is limited:

    1. Set up staging bucket accessible from both environments
    2. Phase 1: On-prem to staging
    3. Phase 2: Staging to cloud destination
    transferConfig:
  intermediateStorage: "s3a://migration-staging-bucket/"

    Strategy C: Offline Migration¶

    When no direct connectivity:

    1. SCHEMA_ONLY: Create tables on cloud
    2. Export data: Use distcp to staging or ship
    3. Import data: Copy to cloud storage
    4. Update locations: Or let SCHEMA_ONLY set correct locations

    Phase 4: Configuration¶

    Step 4.1: Create Migration Configuration¶

    # On-Prem to Cloud Configuration

createIfNotExists: true

# ACID handling - typically downgrade for cloud
migrateACID:
  on: true
  downgrade: true

# Cloud warehouse directories
transferConfig:
  warehouse:
    externalDirectory: "s3a://bucket/warehouse/external"
    managedDirectory: "s3a://bucket/warehouse/managed"

# Optimization for long-distance transfer
optimization:
  autoTune: true
  compressTextOutput: true

    Phase 5: Data Transfer¶

    For Connected Migration (SQL/HYBRID)¶

    1. Create and validate job
    2. Execute migration
    3. Data transfers directly to cloud storage

    For Disconnected Migration (SCHEMA_ONLY + distcp)¶

    Step 5.1: Run SCHEMA_ONLY migration

    # Creates tables on cloud pointing to cloud locations
hms-mirror --strategy SCHEMA_ONLY ...

    Step 5.2: Copy data via distcp

    # From on-prem cluster with cloud credentials
hadoop distcp \
  -Dmapreduce.map.memory.mb=4096 \
  -Dmapreduce.job.queuename=migration \
  -update -delete \
  hdfs://nameservice1/warehouse/db.db/ \
  s3a://bucket/warehouse/external/db.db/

    Step 5.3: Verify and repair partitions

    -- On cloud cluster
MSCK REPAIR TABLE db.table;

    Phase 6: Cloud-Specific Optimization¶

    AWS S3 Optimization¶

    optimization:
  overrides:
    properties:
      fs.s3a.fast.upload:
        RIGHT: "true"
      fs.s3a.fast.upload.buffer:
        RIGHT: "bytebuffer"
      fs.s3a.multipart.size:
        RIGHT: "104857600"  # 100MB parts

    Azure ADLS Optimization¶

    optimization:
  overrides:
    properties:
      fs.azure.write.request.size:
        RIGHT: "67108864"  # 64MB

    Phase 7: Validation¶

    Step 7.1: Schema Validation¶

    -- On cloud cluster
SHOW DATABASES;
USE migrated_db;
SHOW TABLES;
DESCRIBE FORMATTED table_name;

    Step 7.2: Data Validation¶

    -- Compare row counts
SELECT COUNT(*) FROM table_name;

-- Sample data verification
SELECT * FROM table_name LIMIT 100;

    Step 7.3: Performance Testing¶

    1. Run representative queries
    2. Compare execution times
    3. Tune cloud configurations

    Phase 8: Cutover¶

    1. Freeze source: Stop writes to on-prem
    2. Final sync: Migrate any remaining changes
    3. Validate: Confirm all data current
    4. Switch applications: Update connection strings
    5. Monitor: Watch for issues
    6. Decommission: Remove on-prem when ready

    Network Considerations¶

    Bandwidth Planning¶

    Data Volume 100 Mbps 1 Gbps 10 Gbps
    1 TB ~24 hours ~2.5 hours ~15 min
    10 TB ~10 days ~24 hours ~2.5 hours
    100 TB ~100 days ~10 days ~24 hours

    Optimizing Transfer¶

    1. Parallel distcp: Use multiple mappers
    2. Compression: Compress if not already
    3. Off-peak hours: Schedule during low usage
    4. Staging approach: For very large datasets

    Cost Considerations¶

    Cost Type Consideration
    Egress Cloud charges for data out
    Storage Cloud storage costs
    Compute Query processing costs
    Network VPN/Direct Connect fees

    Troubleshooting¶

    Issue: "Slow transfer speeds"¶

    Solutions: - Increase distcp parallelism - Check network bandwidth - Use staging approach for large datasets

    Issue: "Authentication failures to cloud"¶

    Solutions: - Verify cloud credentials configuration - Check IAM roles/policies - Validate service principals

    Issue: "Permission denied on cloud storage"¶

    Solutions: - Check bucket/container policies - Verify IAM permissions - Validate cross-account access if applicable

    Issue: "Table data not visible after migration"¶

    Solutions: - Run MSCK REPAIR TABLE - Verify file permissions on cloud storage - Check partition locations

    Best Practices¶

    1. Start small: Migrate test databases first
    2. Monitor costs: Track cloud spending during migration
    3. Plan bandwidth: Size network appropriately
    4. Use staging: For large datasets or unreliable connections
    5. Validate incrementally: Check data as you migrate
    6. Keep source: Don't delete on-prem until validated

    Validation Checklist¶

    • Cloud resources provisioned
    • Connectivity established
    • Credentials configured
    • Test migration successful
    • Full migration completed
    • Data validated
    • Applications tested
    • Performance acceptable
    • Cutover completed
