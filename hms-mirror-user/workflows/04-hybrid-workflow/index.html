    HYBRID Migration Workflow¶

    Overview¶

    The HYBRID strategy intelligently combines EXPORT_IMPORT and SQL approaches based on each table's characteristics. HMS-Mirror analyzes partition counts and table sizes, then automatically selects the most efficient migration method for each table.

    This provides the best of both strategies: fast EXPORT_IMPORT for smaller tables and scalable SQL for larger ones.

    When to Use¶

    • Mixed table sizes: Databases with both small and large tables
    • Automated optimization: Don't want to manually categorize tables
    • Performance optimization: Best throughput without manual tuning
    • Partition variety: Some tables have few partitions, others have many

    Decision Logic¶

    HMS-Mirror applies these rules for each table:

    IF partition_count <= exportImportPartitionLimit (default: 100)
   AND table_size <= sqlSizeLimit (default: 1GB)
THEN use EXPORT_IMPORT
ELSE IF partition_count <= sqlPartitionLimit (default: 500)
THEN use SQL
ELSE SKIP (table exceeds all limits)

    Prerequisites¶

    1. Linked clusters: Network connectivity between source and target
    2. Shared storage access: For EXPORT_IMPORT portion
    3. Compute resources: For SQL portion
    4. Export staging: Accessible directory for EXPORT operations

    Step-by-Step Workflow¶

    Step 1: Configure Connections¶

    Same as SQL/EXPORT_IMPORT workflows - ensure both clusters are accessible.

    Step 2: Define Dataset¶

    1. Create Dataset with target databases/tables
    2. Consider partition filters to pre-filter very large tables
    3. Tables exceeding all limits will be flagged for manual handling

    Step 3: Create Configuration¶

    1. Basic Settings:
    2. createIfNotExists: Enable

    3. Standard ACID and optimization settings

    4. Transfer Settings:

    5. exportBaseDirPrefix: Location for EXPORT operations
    6. warehouse: Target warehouse directories

    Step 4: Create Job¶

    1. Data Strategy: Select HYBRID

    2. Hybrid Settings (critical for this strategy):

    Setting Default Description
    exportImportPartitionLimit 100 Max partitions for EXPORT_IMPORT
    sqlPartitionLimit 500 Max partitions for SQL
    sqlSizeLimit 1GB Max size for SQL strategy
    1. Adjust thresholds based on your environment:
    2. Larger clusters: Increase limits
    3. Resource-constrained: Decrease limits
    4. Partition-heavy workload: Increase exportImportPartitionLimit

    Step 5: Validate and Review Strategy Selection¶

    1. Run validation

    2. Review the Strategy Assignment Report: 

      Database: analytics
├── fact_sales (500 partitions, 50GB) → SQL
├── dim_product (0 partitions, 100MB) → EXPORT_IMPORT
├── dim_customer (0 partitions, 500MB) → EXPORT_IMPORT
├── fact_inventory (1000 partitions, 100GB) → SKIPPED
└── summary_daily (365 partitions, 5GB) → SQL

    3. Adjust limits if too many tables are skipped

    Step 6: Execute Migration¶

    1. Execute in Execute mode
    2. HMS-Mirror processes tables in optimized order:
    3. EXPORT_IMPORT tables grouped for efficiency
    4. SQL tables processed with appropriate parallelism
    5. Monitor per-table progress and strategy used

    Step 7: Validate Results¶

    1. Verify all tables migrated
    2. Check tables that used each strategy
    3. Validate data integrity for both approaches

    Tuning Hybrid Thresholds¶

    For Partition-Heavy Environments¶

    # Increase EXPORT_IMPORT threshold for environments with many partitions
hybrid:
  exportImportPartitionLimit: 200  # Up from 100
  sqlPartitionLimit: 1000          # Up from 500

    For Large Table Environments¶

    # Increase size limits for clusters with ample resources
hybrid:
  sqlSizeLimit: 5368709120  # 5GB instead of 1GB

    For Resource-Constrained Environments¶

    # Lower limits to prevent resource exhaustion
hybrid:
  exportImportPartitionLimit: 50
  sqlPartitionLimit: 200
  sqlSizeLimit: 536870912  # 512MB

    Strategy Selection Examples¶

    Table Profile Partitions Size Strategy Selected
    Small dimension 0 10 MB EXPORT_IMPORT
    Medium dimension 0 500 MB EXPORT_IMPORT
    Large dimension 0 2 GB SQL
    Partitioned fact (daily) 365 100 GB SQL
    Partitioned fact (hourly) 8760 500 GB SKIPPED
    Small partitioned 50 200 MB EXPORT_IMPORT

    Handling Skipped Tables¶

    Tables exceeding all limits need manual handling:

    1. Split by partition range: Migrate partition ranges separately
    2. Use distcp: Use SCHEMA_ONLY + external data copy
    3. Increase limits: If cluster can handle larger workloads
    4. Dedicated migration: Create separate job with higher limits
    -- Example: Migrate partition ranges
-- Job 1: Historical partitions
-- Dataset filter: partition_date < '2023-01-01'

-- Job 2: Recent partitions
-- Dataset filter: partition_date >= '2023-01-01'

    Validation Checklist¶

    • Strategy assignment reviewed
    • EXPORT_IMPORT tables migrated
    • SQL tables migrated
    • Skipped tables identified and planned
    • Data integrity verified for all tables
    • Export staging cleaned up

    Troubleshooting¶

    Issue: Too many tables using SQL (slow)¶

    Solution: Increase exportImportPartitionLimit to shift more tables to faster EXPORT_IMPORT

    Issue: Too many tables skipped¶

    Solution: - Increase sqlPartitionLimit - Handle skipped tables with dedicated jobs or SCHEMA_ONLY + distcp

    Issue: EXPORT_IMPORT tables failing¶

    Solution: Check export directory accessibility and permissions

    Issue: SQL tables timing out¶

    Solution: - Decrease sqlSizeLimit to shift large tables to EXPORT_IMPORT - Increase query timeouts in optimization overrides

    Issue: Inconsistent results between strategies¶

    Solution: Both strategies should produce identical data - verify source consistency

    Best Practices¶

    1. Start with defaults: Default thresholds work for most environments
    2. Review validation: Check strategy assignments before execution
    3. Monitor resources: Watch YARN queue utilization during execution
    4. Handle skipped tables: Plan for tables exceeding limits
    5. Test in dev first: Validate threshold settings in non-production
