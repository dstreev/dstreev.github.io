    HDP to CDP Migration Workflow¶

    Overview¶

    This workflow covers the complete migration from Hortonworks Data Platform (HDP 2.x or 3.x) to Cloudera Data Platform (CDP). It addresses the key differences between platforms including Hive version changes, ACID table handling, and warehouse directory conventions.

    Migration Challenges¶

    Challenge HDP 2.x/3.x CDP 7.x HMS-Mirror Solution
    Hive Version Hive 1.x/2.x/3.x Hive 3.x/4.x Automatic DDL translation
    ACID Tables Optional/Limited Full support ACID migration/downgrade
    Managed Tables Legacy behavior Strict ACID Automatic conversion
    Warehouse Layout Single warehouse Separate managed/external Directory translation
    Table Properties HDP-specific CDP conventions Property conversion

    Prerequisites¶

    1. Source cluster (HDP):
    2. HDP 2.6+, 3.0, or 3.1
    3. HiveServer2 accessible

    4. HDFS accessible

    5. Target cluster (CDP):

    6. CDP 7.1+ (7.2+ recommended for Iceberg)
    7. HiveServer2 accessible

    8. Target storage configured (HDFS, S3, ADLS)

    9. Network:

    10. Connectivity between clusters (for SQL/HYBRID/EXPORT_IMPORT)
    11. Or shared storage access (for COMMON)

    Step-by-Step Workflow¶

    Phase 1: Assessment¶

    Step 1.1: Inventory Source Environment¶

    Connect to HDP cluster and gather information:

    -- Count databases and tables
SHOW DATABASES;

-- For each database
USE database_name;
SHOW TABLES;

-- Identify ACID tables
SELECT TBL_NAME, TBL_TYPE
FROM metastore.TBLS t
JOIN metastore.DBS d ON t.DB_ID = d.DB_ID
WHERE t.TBL_TYPE = 'MANAGED_TABLE';

-- Check for transactional tables
DESCRIBE FORMATTED table_name;
-- Look for: transactional=true

    Step 1.2: Identify Migration Scope¶

    Document: - Total databases and tables - ACID vs non-ACID table counts - Large tables (>100GB) - Partition-heavy tables (>500 partitions) - Views and their dependencies - Non-native tables (HBase, etc.)

    Phase 2: Connection Setup¶

    Step 2.1: Configure HDP (LEFT) Connection¶

    1. HiveServer2 URI: 

      jdbc:hive2://hdp-hiveserver2:10000/default;principal=hive/_HOST@REALM

    2. HCFS Namespace: 

      hdfs://hdp-nameservice

    3. Platform Type: Select HDP2 or HDP3

    4. Test connection: Verify READ access

    Step 2.2: Configure CDP (RIGHT) Connection¶

    1. HiveServer2 URI: 

      jdbc:hive2://cdp-hiveserver2:10000/default;ssl=true;principal=hive/_HOST@REALM

    2. HCFS Namespace (choose based on storage):

    3. HDFS: hdfs://cdp-nameservice
    4. S3: s3a://cdp-datalake-bucket

    5. ADLS: abfs://container@account.dfs.core.windows.net

    6. Platform Type: Select CDP7_1, CDP7_2, or CDP7_3

    7. Test connection: Verify WRITE access

    Phase 3: Configuration¶

    Step 3.1: Create Migration Configuration¶

    # HDP to CDP Migration Configuration

# Basic Settings
createIfNotExists: true
migrateNonNative: false  # Enable if HBase tables needed

# ACID Table Handling (Critical for HDP->CDP)
migrateACID:
  on: true
  only: false
  downgrade: true  # Convert to EXTERNAL (recommended)
  artificialBucketThreshold: 2  # Remove fake buckets from HDP
  partitionLimit: 500

# Views
migrateVIEW:
  on: true

# Transfer Settings
transferConfig:
  warehouse:
    externalDirectory: "/warehouse/tablespace/external/hive"
    managedDirectory: "/warehouse/tablespace/managed/hive"

# Optimization
optimization:
  autoTune: true
  skipStatsCollection: false

    Step 3.2: Handle HDP-Specific Issues¶

    Issue: Artificial Buckets HDP required bucket clauses for ACID. Set artificialBucketThreshold to remove:

    migrateACID:
  artificialBucketThreshold: 2  # Remove CLUSTERED BY with <=2 buckets

    Issue: Legacy Managed Tables Pre-Hive 3 "managed" tables without ACID need conversion:

    # These become EXTERNAL with purge flag in CDP
migrateACID:
  downgrade: true

    Phase 4: Dataset Definition¶

    Step 4.1: Create Dataset¶

    1. Add databases to migrate
    2. Configure filters:
    3. Exclude temporary tables: excludeRegEx: ".*_temp|.*_staging"
    4. Focus on specific schemas if phasing

    Step 4.2: Consider Phased Approach¶

    For large environments, phase the migration:

    Phase A: Non-ACID tables (simpler) 

    # Dataset A: Non-ACID only
filter:
  tableTypes: ["EXTERNAL_TABLE"]

    Phase B: ACID tables (requires downgrade) 

    # Dataset B: ACID tables
migrateACID:
  on: true
  only: true  # Only ACID tables
  downgrade: true

    Phase 5: Job Creation and Execution¶

    Step 5.1: Select Strategy¶

    Scenario Recommended Strategy
    Clusters connected, format OK HYBRID
    Clusters connected, format conversion SQL
    Clusters not connected SCHEMA_ONLY + distcp
    Testing first LINKED
    Shared storage COMMON

    Step 5.2: Create Job¶

    1. Data Strategy: Based on scenario
    2. References: Select your Dataset, Configuration, Connections
    3. Options:
    4. Enable Disaster Recovery if setting up DR
    5. Configure hybrid thresholds for mixed tables

    Step 5.3: Validate¶

    1. Run validation
    2. Review:
    3. Table compatibility
    4. ACID table handling
    5. Location translations
    6. Partition counts

    Step 5.4: Dry Run¶

    1. Execute dry run
    2. Review generated scripts
    3. Verify DDL translations

    Step 5.5: Execute Migration¶

    1. Execute job
    2. Monitor progress
    3. Handle errors as they occur

    Phase 6: Validation¶

    Step 6.1: Schema Validation¶

    -- On CDP cluster
SHOW DATABASES;

-- Compare table counts
USE migrated_db;
SHOW TABLES;

-- Verify table structures
DESCRIBE FORMATTED table_name;

    Step 6.2: Data Validation¶

    -- Row count comparison
SELECT COUNT(*) FROM table_name;

-- Sample data check
SELECT * FROM table_name LIMIT 100;

-- For partitioned tables
SHOW PARTITIONS table_name;

    Step 6.3: Application Testing¶

    1. Connect applications to CDP
    2. Run test workloads
    3. Verify query results match HDP

    Phase 7: Cutover¶

    Step 7.1: Final Sync (if needed)¶

    For tables that changed during migration: 1. Identify changed tables 2. Re-run migration for deltas 3. Or use sync mode for DR setup

    Step 7.2: Switch Applications¶

    1. Update connection strings to CDP
    2. Verify applications work
    3. Monitor for errors

    Step 7.3: Decommission HDP¶

    After validation period: 1. Stop HDP workloads 2. Archive HDP data (optional) 3. Decommission HDP cluster

    ACID Migration Considerations¶

    Option A: Downgrade to EXTERNAL (Recommended)¶

    migrateACID:
  downgrade: true

    Benefits: - Simpler migration - Works with all strategies - No transactional overhead on target

    Option B: Preserve ACID¶

    migrateACID:
  downgrade: false

    Requirements: - CDP supports full ACID - Use SQL strategy - More complex, longer migration

    Common Issues¶

    Issue: "Bucket clause mismatch"¶

    HDP ACID required buckets, CDP doesn't: 

    migrateACID:
  artificialBucketThreshold: 2

    Issue: "Managed table location outside warehouse"¶

    CDP enforces warehouse paths for managed tables: 

    migrateACID:
  downgrade: true  # Convert to EXTERNAL

    Issue: "Statistics missing after migration"¶

    optimization:
  skipStatsCollection: false
enableAutoTableStats: true

    Issue: "View dependencies failing"¶

    Migrate base tables before views. Views are processed last.

    Validation Checklist¶

    • All databases created on CDP
    • All tables created with correct schemas
    • ACID tables properly converted
    • Data accessible and queryable
    • Views functional
    • Statistics collected
    • Applications tested
    • Performance acceptable
