    Large Table Handling Workflow¶

    Overview¶

    Tables with very large data volumes (100GB+) or extreme partition counts (1000+) require special handling during migration. This workflow covers strategies for efficiently migrating large tables without overwhelming cluster resources.

    Identifying Large Tables¶

    By Data Volume¶

    -- Find tables by estimated size
SELECT
  d.NAME as db_name,
  t.TBL_NAME,
  p.PARAM_VALUE as total_size
FROM metastore.TBLS t
JOIN metastore.DBS d ON t.DB_ID = d.DB_ID
JOIN metastore.TABLE_PARAMS p ON t.TBL_ID = p.TBL_ID
WHERE p.PARAM_KEY = 'totalSize'
  AND CAST(p.PARAM_VALUE AS BIGINT) > 107374182400  -- 100GB
ORDER BY CAST(p.PARAM_VALUE AS BIGINT) DESC;

    By Partition Count¶

    -- Find tables by partition count
SELECT
  d.NAME as db_name,
  t.TBL_NAME,
  COUNT(p.PART_ID) as partition_count
FROM metastore.TBLS t
JOIN metastore.DBS d ON t.DB_ID = d.DB_ID
LEFT JOIN metastore.PARTITIONS p ON t.TBL_ID = p.TBL_ID
GROUP BY d.NAME, t.TBL_NAME
HAVING COUNT(p.PART_ID) > 500
ORDER BY partition_count DESC;

    Size Categories¶

    Category Data Size Partition Count Recommended Approach
    Small < 10 GB < 100 Standard migration
    Medium 10-100 GB 100-500 HYBRID or SQL
    Large 100 GB - 1 TB 500-2000 Dedicated handling
    Very Large > 1 TB > 2000 Partition ranges or distcp

    Strategy Options¶

    Option 1: Dedicated Large Table Jobs¶

    Create separate jobs for large tables:

    # Large table dataset
name: "large-tables-only"
databases:
  - databaseName: warehouse
    tables:
      - "fact_transactions"
      - "event_logs"

    With tuned configuration: 

    optimization:
  overrides:
    properties:
      hive.tez.container.size:
        RIGHT: "16384"  # Larger containers
      hive.exec.parallel:
        RIGHT: "true"
      hive.exec.parallel.thread.number:
        RIGHT: "16"

hybrid:
  sqlPartitionLimit: 2000  # Increased limit
  sqlSizeLimit: 10737418240  # 10GB limit

    Option 2: Partition Range Migration¶

    Migrate partitions in ranges:

    Dataset 1: Historical (2020-2022) 

    databases:
  - databaseName: warehouse
    tables:
      - "fact_transactions"
    filter:
      # Add partition filter in table selection
      partitionFilter: "year <= 2022"

    Dataset 2: Recent (2023-present) 

    databases:
  - databaseName: warehouse
    tables:
      - "fact_transactions"
    filter:
      partitionFilter: "year >= 2023"

    Option 3: SCHEMA_ONLY + DistCp¶

    For very large tables:

    1. Create schema: SCHEMA_ONLY migration
    2. Copy data: External distcp
    3. Repair partitions: MSCK REPAIR TABLE
    # Step 1: Schema only migration
hms-mirror --strategy SCHEMA_ONLY ...

# Step 2: Parallel distcp
hadoop distcp \
  -Dmapreduce.job.maps=100 \
  -strategy dynamic \
  hdfs://source/warehouse/db.db/large_table \
  hdfs://target/warehouse/db.db/large_table

# Step 3: Repair partitions
beeline -e "MSCK REPAIR TABLE db.large_table"

    Option 4: Incremental Migration¶

    For continuously growing tables:

    Day 1: Migrate historical data (bulk)
Day 2-N: Migrate daily increments
Final: Last sync before cutover

    Tuning for Large Tables¶

    Memory Configuration¶

    optimization:
  overrides:
    properties:
      # Increase container memory
      hive.tez.container.size:
        BOTH: "16384"  # 16GB
      hive.tez.java.opts:
        BOTH: "-Xmx13107m"

      # Increase AM memory for large DAGs
      tez.am.resource.memory.mb:
        BOTH: "8192"

    Parallelism¶

    optimization:
  overrides:
    properties:
      # Enable parallel execution
      hive.exec.parallel:
        BOTH: "true"
      hive.exec.parallel.thread.number:
        BOTH: "8"

      # Dynamic partition handling
      hive.exec.dynamic.partition.mode:
        RIGHT: "nonstrict"
      hive.exec.max.dynamic.partitions:
        RIGHT: "10000"
      hive.exec.max.dynamic.partitions.pernode:
        RIGHT: "2000"

    Queue Isolation¶

    optimization:
  overrides:
    properties:
      # Use dedicated queue
      tez.queue.name:
        BOTH: "migration"
      mapreduce.job.queuename:
        BOTH: "migration"

    DistCp Best Practices¶

    Basic DistCp¶

    hadoop distcp \
  -update \
  -delete \
  -skipcrccheck \
  hdfs://source/path \
  hdfs://target/path

    Optimized DistCp¶

    hadoop distcp \
  -Dmapreduce.job.maps=200 \
  -Dmapreduce.job.reduces=50 \
  -Dmapreduce.map.memory.mb=4096 \
  -strategy dynamic \
  -bandwidth 500 \
  -update \
  -skipcrccheck \
  -p \
  hdfs://source/warehouse/db.db/large_table \
  hdfs://target/warehouse/db.db/large_table

    Parallel DistCp (by partition)¶

    #!/bin/bash
# Parallel distcp for partitions
for year in 2020 2021 2022 2023; do
  hadoop distcp \
    -Dmapreduce.job.maps=50 \
    -strategy dynamic \
    hdfs://source/warehouse/db.db/table/year=$year \
    hdfs://target/warehouse/db.db/table/year=$year &
done
wait

    Monitoring Large Migrations¶

    Progress Tracking¶

    -- Monitor partition counts during migration
SELECT COUNT(*) as migrated_partitions
FROM db.table
WHERE 1=1;  -- Target cluster

-- Compare with source
-- hdfs dfs -count -h hdfs://source/warehouse/db.db/table

    Resource Monitoring¶

    • YARN Resource Manager: Queue utilization
    • Application logs: Query progress
    • HDFS: Storage utilization

    Validation for Large Tables¶

    Row Count Validation¶

    -- For partitioned tables, compare by partition
SELECT year, month, COUNT(*) as row_count
FROM db.large_table
GROUP BY year, month
ORDER BY year, month;

    File Count Validation¶

    # Source file count
hdfs dfs -count -h hdfs://source/warehouse/db.db/table

# Target file count
hdfs dfs -count -h hdfs://target/warehouse/db.db/table

    Size Validation¶

    # Source size
hdfs dfs -du -s -h hdfs://source/warehouse/db.db/table

# Target size (may differ slightly due to replication/compression)
hdfs dfs -du -s -h hdfs://target/warehouse/db.db/table

    Troubleshooting¶

    Issue: "Query timeout on large table"¶

    Solutions: - Increase query timeout settings - Migrate in partition ranges - Use SCHEMA_ONLY + distcp

    Issue: "Out of memory"¶

    Solutions: - Increase container memory - Reduce parallelism - Process fewer tables concurrently

    Issue: "Too many mappers/reducers"¶

    Solutions: - Limit parallelism in configuration - Use queue isolation - Spread across multiple days

    Issue: "Slow progress"¶

    Solutions: - Check for data skew in partitions - Monitor network bandwidth - Consider off-peak execution

    Issue: "Partition repair taking too long"¶

    Solutions: - For tables with many partitions, MSCK REPAIR is slow - Consider using RECOVER PARTITIONS - Or create partitions explicitly

    Best Practices¶

    1. Isolate large tables: Separate from regular migration
    2. Test with subset: Validate approach on sample partitions
    3. Schedule strategically: Run during low-activity periods
    4. Monitor continuously: Watch resource utilization
    5. Validate incrementally: Check as partitions complete
    6. Have fallback plan: Be ready to switch strategies

    Example: 5TB Table Migration¶

    Table: warehouse.fact_events (5TB, 10,000 partitions)

Day 1: Schema migration
- Run SCHEMA_ONLY to create table structure
- Verify table exists on target

Day 2-4: Historical data (distcp)
- Copy partitions year <= 2022 (~4TB)
- Run overnight with high parallelism
- Validate each day's progress

Day 5: Recent data (distcp)
- Copy partitions year = 2023 (~1TB)
- Faster due to less volume

Day 6: Partition repair
- Run MSCK REPAIR TABLE
- Validate partition counts

Day 7: Final validation
- Row count comparison
- Sample query testing
- Performance validation

Day 8: Cutover
- Switch applications
- Monitor for issues
