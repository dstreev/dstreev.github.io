    Disaster Recovery Setup Workflow¶

    Overview¶

    This workflow covers setting up a Disaster Recovery (DR) replica of your Hive data warehouse using HMS-Mirror. DR mode creates tables with safe defaults (no data deletion on drop) and enables incremental sync for keeping the replica current.

    DR Architecture Options¶

    Option 1: Shared Storage DR¶

    Primary Cluster                     DR Cluster
+---------------+                   +---------------+
| Active        |                   | Standby       |
| Metastore     |                   | Metastore     |
+---------------+                   +---------------+
       |                                   |
       v                                   v
+------------------------------------------+
|     Shared Storage (S3/ADLS/GCS)         |
|     with cross-region replication        |
+------------------------------------------+

    Best for: Cloud environments with storage replication

    Option 2: Separate Storage DR¶

    Primary Cluster                     DR Cluster
+---------------+                   +---------------+
| Active        |                   | Standby       |
| Metastore     |                   | Metastore     |
+---------------+                   +---------------+
       |                                   |
       v                                   v
+---------------+                   +---------------+
| Primary       | ====distcp====>  | DR Storage    |
| Storage       |                   |               |
+---------------+                   +---------------+

    Best for: On-premises or isolated environments

    Key DR Features¶

    Feature Purpose
    disasterRecovery: true Enables DR mode
    external.table.purge: false Prevents accidental data deletion
    sync: true Enables incremental sync
    Read-only tables DR tables are read-only by default

    Step-by-Step Workflow¶

    Phase 1: Initial DR Setup¶

    Step 1.1: Configure Connections¶

    1. LEFT (Primary) Connection:
    2. Production cluster HiveServer2

    3. Production storage namespace

    4. RIGHT (DR) Connection:

    5. DR cluster HiveServer2
    6. DR storage namespace

    Step 1.2: Create DR Configuration¶

    # Disaster Recovery Configuration
createIfNotExists: true

# DR-specific settings
# (Note: These are set via Job options, not Configuration)

# Transfer settings
transferConfig:
  warehouse:
    externalDirectory: "/dr/warehouse/external"
    managedDirectory: "/dr/warehouse/managed"

# ACID handling
migrateACID:
  on: true
  downgrade: true  # Recommended for DR

# Statistics
enableAutoTableStats: true

    Step 1.3: Create Dataset¶

    1. Include all databases for DR
    2. Consider excluding temporary/staging tables: 
      filter:
  excludeRegEx: ".*_temp|.*_staging|.*_backup"

    Step 1.4: Create DR Job¶

    1. Data Strategy: Choose based on connectivity
    2. Connected: SQL, HYBRID, or EXPORT_IMPORT
    3. Shared storage: COMMON

    4. Separate storage: SCHEMA_ONLY + distcp

    5. Enable Disaster Recovery:

    6. Check "Disaster Recovery Job" option

    7. Sync Mode: Leave disabled for initial load

    Step 1.5: Execute Initial DR Load¶

    1. Execute the job
    2. Monitor progress
    3. This creates DR tables with:
    4. external.table.purge = false
    5. Read-only semantics
    6. DR-appropriate locations

    Phase 2: Incremental Sync¶

    Step 2.1: Enable Sync Mode¶

    After initial load, create a sync job:

    1. Create new job (or modify existing)
    2. Enable Disaster Recovery option
    3. Enable Sync option

    Step 2.2: Configure Sync Schedule¶

    Run sync jobs on a schedule:

    Option A: Scheduled via HMS-Mirror API 

    # Cron job to trigger sync
0 */4 * * * curl -X POST http://hms-mirror:8080/api/v1/jobs/{job-key}/run

    Option B: External Scheduler 

    # Airflow, Control-M, or similar
hms-mirror-cli --job dr-sync-job --execute

    Step 2.3: Sync Behavior¶

    When sync is enabled: - Only new/modified tables are processed - Partitions are compared and updated - Deleted source objects are NOT auto-removed (safety)

    Phase 3: DR Validation¶

    Step 3.1: Regular Validation¶

    Schedule periodic validation:

    -- Compare table counts
-- Run on both clusters
SELECT database_name, COUNT(*) as table_count
FROM metastore.TBLS t
JOIN metastore.DBS d ON t.DB_ID = d.DB_ID
GROUP BY database_name;

-- Compare row counts (sample)
SELECT COUNT(*) FROM critical_table;

    Step 3.2: DR Drill¶

    Periodically test DR readiness:

    1. Connect test applications to DR cluster
    2. Run read-only workloads
    3. Verify query results match production
    4. Document any discrepancies

    Phase 4: Failover Procedure¶

    Step 4.1: Decision to Failover¶

    Criteria for failover: - Primary cluster unavailable - Storage corruption detected - Extended outage expected

    Step 4.2: Execute Failover¶

    1. Stop sync jobs: Prevent partial updates
    2. Verify DR state: Run validation queries
    3. Update table properties (if write needed): 
      -- Enable purge for specific tables if needed
ALTER TABLE db.table SET TBLPROPERTIES ('external.table.purge'='true');
    4. Switch applications: Update connection strings
    5. Verify application function: Test critical workflows

    Step 4.3: Post-Failover¶

    1. Investigate primary failure
    2. Plan recovery/failback
    3. Document incident

    Phase 5: Failback Procedure¶

    After primary is restored:

    Step 5.1: Sync Changes to Primary¶

    If writes occurred on DR:

    1. Create reverse sync job (DR as source)
    2. Sync changed tables back to primary
    3. Validate primary state

    Step 5.2: Resume Normal Operations¶

    1. Switch applications back to primary
    2. Resume DR sync jobs
    3. Validate DR is current

    DR Job Configuration¶

    Initial Load Job¶

    # Job settings
name: "dr-initial-load"
description: "Initial DR replica creation"
strategy: HYBRID  # or SQL, EXPORT_IMPORT
disasterRecovery: true
sync: false  # Initial load

    Sync Job¶

    # Job settings
name: "dr-sync"
description: "Incremental DR sync"
strategy: HYBRID
disasterRecovery: true
sync: true  # Incremental sync

    Table Properties in DR Mode¶

    Property Value Purpose
    external.table.purge false Prevent data deletion on DROP
    EXTERNAL true All tables become EXTERNAL
    hms_mirror_dr_table true Marks as DR-managed

    Monitoring DR¶

    Key Metrics¶

    • Time since last sync
    • Table count comparison
    • Failed sync tables
    • Storage lag (for replicated storage)

    Alerts¶

    Configure alerts for: - Sync job failures - Large table count discrepancies - Extended sync duration - Storage replication lag

    Troubleshooting¶

    Issue: "Sync taking too long"¶

    Solutions: - Review changed table volume - Consider more frequent syncs - Optimize large table handling

    Issue: "Tables missing in DR"¶

    Solutions: - Check sync job logs - Verify dataset includes all databases - Run full sync to catch up

    Issue: "Stale data in DR"¶

    Solutions: - Verify sync schedule running - Check for sync failures - Run manual sync if needed

    Issue: "Cannot write to DR tables"¶

    Solutions: - DR tables are read-only by design - Modify table properties if failover required - Consider if write is actually needed

    Best Practices¶

    1. Test failover regularly: Don't wait for disaster
    2. Monitor sync lag: Know your RPO
    3. Document procedures: Clear runbooks for failover
    4. Automate validation: Regular automated checks
    5. Separate credentials: DR cluster should have independent auth
    6. Consider RTO: Test how long failover takes

    Validation Checklist¶

    • DR cluster accessible
    • All tables replicated
    • Table counts match
    • Sample data verified
    • Sync job scheduled
    • Failover procedure documented
    • DR drill completed
