    EXPORT_IMPORT Migration Workflow¶

    Overview¶

    The EXPORT_IMPORT strategy uses Hive's native EXPORT and IMPORT commands to migrate tables between clusters. Data is exported to a staging location on the source cluster (accessible by target), then imported to create the table on the target cluster.

    This approach preserves exact file formats and layouts while providing atomic table migration with metadata integrity.

    When to Use¶

    • Exact format preservation: When you need bit-for-bit identical files
    • Atomic migrations: Need complete table transfer or rollback
    • Moderate partition counts: Tables with fewer than ~100 partitions
    • Linked clusters: Shared or accessible storage between clusters
    • Metadata preservation: EXPORT/IMPORT preserves all table properties

    When NOT to Use¶

    • Tables with >100 partitions (use SQL or HYBRID instead)
    • When format conversion is required
    • When clusters cannot share storage access
    • Very large tables (EXPORT creates additional copies)

    Prerequisites¶

    1. Shared Storage: Target cluster must be able to read from export location
    2. Export Directory: Accessible path for staging exported data
    3. Connections: Both clusters configured and tested
    4. Permissions:
    5. EXPORT permission on source tables
    6. WRITE to export staging directory
    7. IMPORT permission on target
    8. CREATE DATABASE/TABLE on target

    Step-by-Step Workflow¶

    Step 1: Configure Export Location¶

    The export location must be accessible from BOTH clusters. Options:

    Option A: Source HDFS (if target can read) 

    /apps/hive/warehouse/export_staging/

    Option B: Shared Cloud Storage 

    s3a://shared-bucket/hms-mirror-export/
abfs://container@account/hms-mirror-export/

    Option C: Intermediate Storage 

    # Configure in transferConfig.intermediateStorage
s3a://migration-staging/export/

    Step 2: Configure Connections¶

    1. LEFT (Source) Connection:
    2. HiveServer2 with EXPORT permissions

    3. HCFS Namespace of source storage

    4. RIGHT (Target) Connection:

    5. HiveServer2 with IMPORT/CREATE permissions
    6. HCFS Namespace for target storage
    7. Must be able to read from export location

    Step 3: Define Dataset¶

    1. Create new Dataset
    2. Add databases and tables
    3. Important Filters:
    4. Set maxPartitions filter to avoid large partition tables
    5. Consider using HYBRID strategy for mixed workloads
    6. Save the Dataset

    Step 4: Create Configuration¶

    1. Create new Configuration

    2. Transfer Settings: 

      transferConfig:
  exportBaseDirPrefix: "/apps/hive/warehouse/export_"
  # Or for cloud:
  # exportBaseDirPrefix: "s3a://bucket/hms-mirror-export/"

    3. Basic Settings:

    4. createIfNotExists: Enable

    5. saveWorkingTables: Disable (to clean up exports)

    6. ACID Handling (if applicable):

    7. migrateACID.on: Enable

    8. Note: ACID tables require special handling with EXPORT_IMPORT

    9. Warehouse Configuration:

    10. Set target warehouse directories if different from defaults

    Step 5: Create Job¶

    1. Navigate to Jobs > Create New Job

    2. Job Details:

    3. Name: prod-export-import-migration

    4. Description: Document scope

    5. Data Strategy: Select EXPORT_IMPORT

    6. References:

    7. Select Dataset, Configuration, Connections

    8. Hybrid Settings:

    9. exportImportPartitionLimit: Maximum partitions (default: 100)

    Step 6: Validate Job¶

    1. Run validation
    2. Check for:
    3. Export directory accessibility
    4. Table partition counts within limits
    5. Storage capacity for export staging
    6. Review partition counts - tables exceeding limit will be skipped

    Step 7: Dry Run¶

    Review generated commands:

    -- Source cluster (LEFT)
EXPORT TABLE source_db.source_table
TO '/apps/hive/warehouse/export_source_db/source_table';

-- Target cluster (RIGHT)
IMPORT TABLE target_db.target_table
FROM '/apps/hive/warehouse/export_source_db/source_table'
LOCATION 'hdfs://target/warehouse/target_db.db/target_table';

    Step 8: Execute Migration¶

    1. Execute in Execute mode
    2. Monitor stages:
    3. EXPORT phase: Data copied to staging directory
    4. IMPORT phase: Table created and data registered on target
    5. Handle failures and retry as needed

    Step 9: Validate Results¶

    1. Verify tables on target: 

      SHOW TABLES IN target_db;
DESCRIBE FORMATTED target_table;

    2. Verify data integrity: 

      SELECT COUNT(*) FROM target_table;

    3. Compare file counts and sizes if needed

    Step 10: Cleanup¶

    After successful validation:

    1. Remove export staging data: 

      hdfs dfs -rm -r /apps/hive/warehouse/export_*/

    2. Or use the cleanup scripts generated by HMS-Mirror

    Configuration Example¶

    # Configuration for EXPORT_IMPORT
transferConfig:
  exportBaseDirPrefix: "s3a://migration-bucket/export/"
  remoteWorkingDirectory: "hms_mirror_working"

migrateACID:
  on: true
  downgrade: true

createIfNotExists: true
saveWorkingTables: false

    Validation Checklist¶

    • Export staging location accessible from both clusters
    • All tables exported successfully
    • All tables imported on target
    • Row counts match source
    • Table properties preserved
    • Partition structures intact
    • Export staging cleaned up

    Troubleshooting¶

    Issue: "Export path not accessible from target"¶

    Solution: - Use shared cloud storage for export location - Configure intermediate storage - Ensure cross-cluster filesystem access is configured

    Issue: "Table too large to export"¶

    Solution: - EXPORT creates a copy, doubling storage temporarily - Use SQL strategy for very large tables - Consider HYBRID to auto-select

    Issue: "Too many partitions"¶

    Solution: - Reduce exportImportPartitionLimit to fail faster - Use SQL strategy for partition-heavy tables - Use HYBRID for automatic selection

    Issue: "ACID table export failed"¶

    Solution: - ACID tables require active transactions to complete - Run ACID compaction before export - Consider SQL strategy with ACID downgrade

    Issue: "Permission denied on export path"¶

    Solution: - Ensure Hive user can write to export directory - Check HDFS/S3 permissions - Verify service accounts have cross-cluster access

    Storage Considerations¶

    EXPORT_IMPORT temporarily requires extra storage:

    Source Data:     100 GB
Export Staging:  100 GB  (temporary copy)
Target Data:     100 GB
------------------------
Peak Usage:      300 GB
Final Usage:     200 GB  (after cleanup)

    Plan for 3x storage during migration, 2x after cleanup.

    Generated Artifacts¶

    File Purpose
    *_execute.sql EXPORT and IMPORT statements
    *_clean.sql Cleanup for export staging
    runStatus.yaml Execution statistics
