    Iceberg Conversion Workflow¶

    Overview¶

    Apache Iceberg is an open table format designed for large analytic datasets. This workflow covers converting existing Hive tables to Iceberg format during migration, enabling features like hidden partitioning, schema evolution, time travel, and ACID semantics.

    Iceberg Benefits¶

    Feature Hive Tables Iceberg Tables
    Schema Evolution Limited Full support
    Partition Evolution Requires rewrite Dynamic
    Time Travel Not available Built-in
    ACID ORC-only Format-agnostic
    Hidden Partitioning No Yes
    Snapshot Isolation Limited Full

    Prerequisites¶

    1. Target platform support: CDP 7.2+, Spark 3.x, or compatible platform
    2. Source table format: ORC, Parquet, or Avro (or will be converted)
    3. SQL strategy: Iceberg conversion requires SQL-based migration
    4. Compute resources: Sufficient resources for data rewriting

    Iceberg Configuration Options¶

    File Format Translation¶

    Option Description
    SAME Preserve source format if compatible
    PARQUET Convert all to Parquet
    ORC Convert all to ORC
    AVRO Convert all to Avro
    STANDARD Convert to Parquet (recommended)

    Iceberg Version¶

    Version Features
    1 Original format, wider compatibility
    2 Row-level deletes, improved metadata (recommended)

    Step-by-Step Workflow¶

    Step 1: Verify Target Platform¶

    Confirm Iceberg support:

    -- On target cluster
CREATE TABLE test_iceberg (id INT)
STORED BY ICEBERG;

-- If this works, Iceberg is available
DROP TABLE test_iceberg;

    Step 2: Configure Connections¶

    Standard connection setup with Iceberg-capable target.

    Step 3: Create Configuration¶

    # Iceberg Conversion Configuration

createIfNotExists: true

# Iceberg Conversion Settings
icebergConversion:
  enable: true
  fileTypeTranslation: PARQUET  # or SAME, ORC, AVRO, STANDARD
  version: 2  # Iceberg format version
  inplace: false  # true for same-cluster conversion
  tableProperties:
    write.format.default: parquet
    write.parquet.compression-codec: zstd
    write.metadata.compression-codec: gzip
    commit.retry.num-retries: "4"

# ACID tables should be downgraded first
migrateACID:
  on: true
  downgrade: true

# Required: SQL strategy for conversion
# (Set via Job, not Configuration)

    Step 4: Create Dataset¶

    1. Include tables to convert
    2. Consider excluding:
    3. Very large tables (migrate separately)
    4. Tables not benefiting from Iceberg
    5. Temporary/staging tables

    Step 5: Create Job¶

    1. Data Strategy: Must be SQL for Iceberg conversion
    2. References: Select configuration with Iceberg enabled
    3. Note: EXPORT_IMPORT and HYBRID do not support Iceberg conversion

    Step 6: Validate¶

    1. Run validation
    2. Review:
    3. Table format compatibility
    4. Partition structure translation
    5. Schema compatibility

    Step 7: Execute Conversion¶

    1. Execute the job
    2. For each table:
    3. Iceberg table created on target
    4. Data written in specified format
    5. Iceberg metadata generated

    Generated DDL Example: 

    -- Original Hive table becomes:
CREATE TABLE target_db.converted_table (
  id BIGINT,
  name STRING,
  created_date DATE
)
PARTITIONED BY (year INT, month INT)
STORED BY ICEBERG
STORED AS PARQUET
LOCATION 's3a://bucket/warehouse/target_db.db/converted_table'
TBLPROPERTIES (
  'format-version'='2',
  'write.format.default'='parquet'
);

    Step 8: Verify Iceberg Tables¶

    -- Verify table is Iceberg
DESCRIBE FORMATTED converted_table;
-- Look for: STORED BY ICEBERG

-- Check Iceberg metadata
SELECT * FROM converted_table.snapshots;

-- Time travel query
SELECT * FROM converted_table FOR SYSTEM_VERSION AS OF 1;

    Conversion Scenarios¶

    Scenario 1: Simple Table Conversion¶

    Source: 

    CREATE EXTERNAL TABLE events (
  event_id BIGINT,
  event_type STRING,
  event_time TIMESTAMP
)
STORED AS PARQUET;

    Target Iceberg: 

    CREATE TABLE events (
  event_id BIGINT,
  event_type STRING,
  event_time TIMESTAMP
)
STORED BY ICEBERG
STORED AS PARQUET;

    Scenario 2: Partitioned Table Conversion¶

    Source: 

    CREATE EXTERNAL TABLE sales (
  sale_id BIGINT,
  amount DECIMAL(10,2),
  sale_date DATE
)
PARTITIONED BY (year INT, month INT)
STORED AS PARQUET;

    Target Iceberg: 

    CREATE TABLE sales (
  sale_id BIGINT,
  amount DECIMAL(10,2),
  sale_date DATE,
  year INT,
  month INT
)
PARTITIONED BY (year, month)
STORED BY ICEBERG
STORED AS PARQUET;

    Note: Iceberg uses partition transforms, partitions become regular columns.

    Scenario 3: ACID Table to Iceberg¶

    Source (ACID): 

    CREATE TABLE orders (
  order_id BIGINT,
  customer_id INT,
  total DECIMAL(10,2)
)
STORED AS ORC
TBLPROPERTIES ('transactional'='true');

    Conversion process: 1. ACID table read via SQL (materializes transactions) 2. Data written as Iceberg Parquet 3. Full Iceberg ACID capabilities available

    Scenario 4: Format Conversion¶

    Source (Text): 

    CREATE EXTERNAL TABLE legacy_data (
  col1 STRING,
  col2 STRING
)
ROW FORMAT DELIMITED FIELDS TERMINATED BY ','
STORED AS TEXTFILE;

    Target Iceberg (with format conversion): 

    -- Configured with fileTypeTranslation: PARQUET
CREATE TABLE legacy_data (
  col1 STRING,
  col2 STRING
)
STORED BY ICEBERG
STORED AS PARQUET;

    In-Place Iceberg Conversion¶

    For converting tables within the same cluster:

    icebergConversion:
  enable: true
  inplace: true

    In-Place Process: 1. Source files must already be Iceberg-compatible (Parquet/ORC/Avro) 2. Uses Iceberg's native in-place migration procedure 3. No data copying - only metadata transformation 4. Much faster than SQL-based conversion

    Requirements: - Source cluster supports Iceberg - Files already in compatible format - Table structure supports Iceberg

    Table Properties¶

    Common Iceberg table properties:

    icebergConversion:
  tableProperties:
    # File format
    write.format.default: parquet
    write.parquet.compression-codec: zstd  # snappy, gzip, lz4

    # Metadata
    write.metadata.compression-codec: gzip
    write.metadata.delete-after-commit.enabled: "true"

    # Compaction
    write.target-file-size-bytes: "536870912"  # 512MB

    # History
    history.expire.max-snapshot-age-ms: "432000000"  # 5 days

    Partition Evolution¶

    One of Iceberg's key features - partitioning can change without rewriting data:

    -- After migration, evolve partitions
ALTER TABLE converted_table ADD PARTITION FIELD bucket(16, customer_id);

-- Future writes use new partitioning
-- Old data remains readable

    Troubleshooting¶

    Issue: "Iceberg not available on target"¶

    Solutions: - Verify Iceberg is installed on target cluster - Check Hive/Spark Iceberg integration - Verify platform version (CDP 7.2+ required)

    Issue: "Partition structure incompatible"¶

    Solutions: - Iceberg handles most partition types - Complex partition expressions may need adjustment - Consider partition evolution after migration

    Issue: "Conversion very slow"¶

    Solutions: - Iceberg conversion rewrites all data - Consider smaller batches - Increase compute resources - Use in-place conversion if possible

    Issue: "ACID tables not converting"¶

    Solutions: - Enable ACID migration with downgrade - SQL strategy is required - Ensure transactions are complete

    Issue: "Schema mismatch after conversion"¶

    Solutions: - Iceberg may handle types slightly differently - Review column type mappings - Adjust source schema if needed

    Best Practices¶

    1. Start with non-critical tables: Test conversion process
    2. Use version 2: Unless compatibility requires version 1
    3. Choose Parquet: Best balance of features and compatibility
    4. Configure compression: Use zstd for good compression/speed
    5. Set table properties: Configure for your workload
    6. Test time travel: Verify snapshot capabilities work
    7. Plan for large tables: May need separate handling

    Validation Checklist¶

    • Target supports Iceberg
    • Tables created as Iceberg format
    • Data accessible via queries
    • Partition structure correct
    • Time travel queries work
    • Schema matches expectations
    • Performance acceptable
