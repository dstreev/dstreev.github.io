    Workflows¶

    Overview¶

    This guide provides step-by-step workflows for common HMS-Mirror migration scenarios. Each workflow document covers a specific use case or data strategy, including prerequisites, configuration steps, execution, and validation.

    Workflow Categories¶

    Strategy-Based Workflows¶

    These workflows focus on specific data migration strategies and when to use them:

    Workflow Description Use When
    SCHEMA_ONLY Transfer schema definitions only Data moved externally via distcp
    SQL Migrate data using Hive SQL Linked clusters, format conversion needed
    EXPORT_IMPORT Use Hive's native export/import Linked clusters, moderate data sizes
    HYBRID Auto-select SQL or EXPORT_IMPORT Mixed table sizes, optimal performance
    LINKED Create linked tables to source data Testing/validation before full migration
    STORAGE_MIGRATION Relocate data within a cluster Cloud migration, namespace changes
    COMMON Shared storage between clusters Clusters sharing same HDFS/S3
    DUMP Extract schema DDL scripts Documentation, offline planning
    CONVERT_LINKED Convert linked tables to regular After data copy to target

    Use-Case Workflows¶

    These workflows address common migration scenarios:

    Workflow Description
    HDP to CDP Migration Complete HDP 2.x/3.x to CDP 7.x migration
    On-Premises to Cloud Migrate from on-prem to cloud storage
    ACID Table Migration Special handling for transactional tables
    ACID Downgrade Convert ACID tables to EXTERNAL
    Iceberg Conversion Convert Hive tables to Iceberg format
    Disaster Recovery Setup Set up DR replica with incremental sync
    Database Rename/Prefix Migrate with database naming changes
    Phased Migration Multi-phase migration for large environments
    Testing with LINKED Test target cluster before full migration

    Advanced Workflows¶

    Workflow Description
    Multi-Hop Migration Migrations via intermediate storage
    Cross-Environment Dev/Test/Prod environment handling
    Large Table Handling Strategies for very large tables
    Partition-Heavy Tables Handling tables with many partitions

    Quick Start Decision Tree¶

    Use this decision tree to select the right workflow:

    START
  |
  v
Are clusters network-connected (linked)?
  |
  +-- NO --> Can you set up intermediate storage?
  |            |
  |            +-- YES --> Multi-Hop Migration (19)
  |            +-- NO  --> SCHEMA_ONLY (01) + external distcp
  |
  +-- YES --> Do you need to migrate data?
               |
               +-- NO  --> Just schema? --> SCHEMA_ONLY (01)
               |           Need testing? --> LINKED (05)
               |
               +-- YES --> Is this within same cluster?
                            |
                            +-- YES --> STORAGE_MIGRATION (06)
                            +-- NO  --> Do tables share storage?
                                         |
                                         +-- YES --> COMMON (07)
                                         +-- NO  --> Need format conversion?
                                                      |
                                                      +-- YES --> SQL (02)
                                                      +-- NO  --> Mixed table sizes?
                                                                   |
                                                                   +-- YES --> HYBRID (04)
                                                                   +-- NO  --> EXPORT_IMPORT (03)

    Common Prerequisites¶

    Before starting any workflow:

    1. Connections Configured: Both source (LEFT) and target (RIGHT) connections must be tested and working
    2. Dataset Defined: Tables to migrate must be identified in a Dataset
    3. Configuration Created: Migration settings must be defined in a Configuration
    4. Permissions Verified: User must have appropriate Hive and HDFS/S3 permissions on both clusters
    5. Network Connectivity: For linked strategies, clusters must be able to communicate
    6. Driver Availability: Required JDBC drivers must be installed

    Workflow Document Structure¶

    Each workflow document follows this structure:

    1. Overview: What this workflow accomplishes
    2. When to Use: Scenarios where this approach is appropriate
    3. Prerequisites: What must be in place before starting
    4. Step-by-Step Instructions: Detailed configuration and execution steps
    5. Validation: How to verify successful migration
    6. Troubleshooting: Common issues and solutions
    7. Related Workflows: Links to complementary approaches

    • Configuration Tooltips: Hover over info icons for field descriptions
