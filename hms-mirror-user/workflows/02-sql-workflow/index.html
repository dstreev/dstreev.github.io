    SQL Migration Workflow¶

    Overview¶

    The SQL strategy uses Hive SQL INSERT statements to migrate data from the source cluster to the target cluster. Data is read from source tables via HiveServer2 and written to newly created target tables using distributed SQL execution.

    This approach leverages Hive's query engine for data transfer, enabling format conversion, schema transformation, and optimization during migration.

    When to Use¶

    • Format conversion needed: Converting between file formats (e.g., Text to Parquet, Avro to ORC)
    • Schema transformation: Adding/removing columns, changing data types
    • Data filtering: Migrating subset of data based on predicates
    • Compression changes: Applying different compression during migration
    • Linked clusters: Source and target clusters can communicate via network
    • Moderate data sizes: Tables that can be processed within Hive query timeouts

    Prerequisites¶

    1. Linked Clusters: Source and target must have network connectivity
    2. Connections: Both LEFT and RIGHT connections tested and working
    3. HCFS Access: Target cluster must be able to write to its storage namespace
    4. Compute Resources: Sufficient YARN/Tez resources for distributed queries
    5. Permissions:
    6. READ on source tables
    7. CREATE DATABASE/TABLE on target
    8. WRITE to target storage locations

    Step-by-Step Workflow¶

    Step 1: Configure Connections¶

    1. LEFT (Source) Connection:
    2. HiveServer2 URI with appropriate authentication
    3. HCFS Namespace of source storage
    4. Platform Type matching source cluster version

    5. Test connection to verify READ access

    6. RIGHT (Target) Connection:

    7. HiveServer2 URI with WRITE permissions
    8. HCFS Namespace for target storage
    9. Platform Type for target cluster
    10. Test connection to verify CREATE/WRITE access

    Step 2: Define Dataset¶

    1. Create a new Dataset
    2. Add databases to include
    3. Configure table filters:
    4. Use regex patterns for include/exclude
    5. Set partition limits to prevent resource exhaustion
    6. Consider table size limits for very large tables
    7. Save the Dataset

    Step 3: Create Configuration¶

    1. Create a new Configuration
    2. Basic Settings:
    3. createIfNotExists: Enable for idempotent runs

    4. migrateNonNative: Enable if HBase tables exist

    5. ACID Migration (if source has ACID tables):

    6. migrateACID.on: Enable
    7. migrateACID.downgrade: Enable to convert to EXTERNAL

    8. migrateACID.partitionLimit: Set appropriate limit (default: 500)

    9. Optimization Settings:

    10. autoTune: Enable for automatic parameter tuning
    11. compressTextOutput: Enable if using text format

    12. Configure custom overrides if needed

    13. Statistics (optional):

    14. enableAutoTableStats: Enable for query optimizer

    15. enableAutoColumnStats: Enable for detailed statistics

    16. Transfer Settings:

    17. Configure warehouse directories if different from defaults
    18. Set intermediate storage if needed

    Step 4: Create Job¶

    1. Navigate to Jobs > Create New Job
    2. Job Details:
    3. Name: Descriptive name (e.g., analytics-sql-migration)

    4. Description: Document scope and purpose

    5. Data Strategy: Select SQL

    6. References:

    7. Dataset: Your defined dataset
    8. Configuration: Your configuration
    9. LEFT Connection: Source cluster

    10. RIGHT Connection: Target cluster

    11. Hybrid Settings (affects SQL partition limits):

    12. sqlPartitionLimit: Maximum partitions per table (default: 500)

    13. sqlSizeLimit: Maximum table size in bytes (default: 1GB)

    14. Review and create the job

    Step 5: Validate Job¶

    1. Click Validate on the job
    2. Review validation results:
    3. Table compatibility check
    4. Location translation preview
    5. Partition count verification
    6. Size estimation (if available)
    7. Address any validation errors

    Step 6: Dry Run¶

    1. Execute job in Dry Run mode
    2. Review generated SQL scripts: 
      -- Example generated SQL
CREATE DATABASE IF NOT EXISTS target_db;
USE target_db;

CREATE TABLE target_table (
  col1 STRING,
  col2 INT
) STORED AS PARQUET
LOCATION 'hdfs://target/warehouse/target_db.db/target_table';

INSERT INTO target_table
SELECT * FROM source_db.source_table;
    3. Verify:
    4. Table DDL is correct
    5. INSERT statements have proper column mappings
    6. Partition handling is appropriate

    Step 7: Execute Migration¶

    1. Execute job in Execute mode
    2. Monitor progress:
    3. Per-table execution status
    4. Error messages if any

    5. Duration tracking

    6. Handle any failures:

    7. Review error logs
    8. Fix configuration issues
    9. Re-run failed tables only

    Step 8: Validate Results¶

    1. Connect to target cluster

    2. Verify table creation: 

      SHOW DATABASES;
USE your_database;
SHOW TABLES;

    3. Verify data: 

      -- Row count comparison
SELECT COUNT(*) FROM target_table;

-- Sample data check
SELECT * FROM target_table LIMIT 10;

-- For partitioned tables
SHOW PARTITIONS target_table;

    4. Verify statistics (if enabled): 

      DESCRIBE FORMATTED target_table;
-- Check numRows, rawDataSize statistics

    Configuration Tuning¶

    For Large Tables¶

    optimization:
  autoTune: true
  overrides:
    properties:
      hive.exec.parallel:
        RIGHT: "true"
      hive.exec.dynamic.partition.mode:
        RIGHT: "nonstrict"
      hive.exec.max.dynamic.partitions:
        RIGHT: "10000"
      hive.tez.container.size:
        RIGHT: "8192"

    For ACID Tables¶

    migrateACID:
  on: true
  downgrade: true  # Convert to EXTERNAL
  partitionLimit: 500
  artificialBucketThreshold: 2  # Remove fake buckets

    For Format Conversion¶

    Set target table properties in the Configuration or use Iceberg conversion for modern formats.

    Validation Checklist¶

    • All databases created on target
    • All tables created with correct schemas
    • Row counts match between source and target
    • Partition structures preserved
    • Data accessible via queries
    • Statistics collected (if enabled)
    • Performance acceptable for queries

    Troubleshooting¶

    Issue: Query timeout on large tables¶

    Solutions: - Increase Hive query timeout settings - Use HYBRID strategy to switch large tables to EXPORT_IMPORT - Reduce table scope and migrate in batches

    Issue: Out of memory during INSERT¶

    Solutions: - Increase Tez container size in optimization overrides - Reduce concurrent operations - Use DISTRIBUTE BY for better partition handling

    Issue: Partition limit exceeded¶

    Solutions: - Increase sqlPartitionLimit in hybrid settings - Use EXPORT_IMPORT for partition-heavy tables - Split migration by partition ranges

    Issue: ACID table migration failing¶

    Solutions: - Ensure migrateACID.on is enabled - Enable downgrade to convert to EXTERNAL - Check source ACID table transaction status

    Issue: Slow performance¶

    Solutions: - Enable autoTune for automatic optimization - Set custom overrides for parallel execution - Consider time-of-day for reduced cluster load

    File Purpose

    *_clean.sql Cleanup/rollback scripts
    runStatus.yaml Execution statistics
