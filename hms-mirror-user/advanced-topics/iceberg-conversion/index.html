    Iceberg Conversion¶

    HMS-Mirror can convert traditional Hive tables to Apache Iceberg format during migration. This guide covers Iceberg concepts and conversion options.

    What is Apache Iceberg?¶

    Apache Iceberg is a high-performance table format for huge analytic datasets.

    Key Benefits¶

    Feature Description
    Time Travel Query data as of a specific point in time
    Schema Evolution Add, rename, drop columns without rewriting data
    Partition Evolution Change partitioning without rewriting data
    Hidden Partitioning Users don't need to know partition columns
    ACID Transactions Full transactional support
    Multi-Engine Works with Spark, Flink, Trino, Hive, and more

    Iceberg vs Traditional Hive¶

    Aspect Traditional Hive Iceberg
    Metadata Metastore only Manifest files + Metastore
    Transactions Limited (ACID tables) Full support
    Schema changes Requires ALTER Transparent
    Partition changes Requires rewrite Transparent
    Query engines Hive, Spark Many engines
    Time travel Not supported Built-in

    When to Convert to Iceberg¶

    Good Candidates¶

    • Tables that need schema evolution
    • Multi-engine access (Spark + Hive + Trino)
    • Large tables benefiting from Iceberg optimizations
    • Tables requiring time travel capabilities
    • New data lake implementations

    Poor Candidates¶

    • Small dimension tables (overhead not worth it)
    • Tables with complex Hive UDFs
    • Tables used only by legacy Hive applications
    • Temporary or staging tables

    Conversion Options¶

    Conversion Modes¶

    In-Place Conversion¶

    Converts the existing table to Iceberg format.

    iceberg:
  enabled: true
  conversionMode: IN_PLACE

    Process 

    ┌─────────────────┐                ┌─────────────────┐
│   Hive Table    │  ──Convert──►  │  Iceberg Table  │
│   (Original)    │                │  (Same name)    │
└─────────────────┘                └─────────────────┘

    Characteristics - Original table is converted - Same table name retained - Data files may be rewritten - Minimal storage overhead

    Shadow Conversion¶

    Creates a new Iceberg table alongside the original.

    iceberg:
  enabled: true
  conversionMode: SHADOW

    Process 

    ┌─────────────────┐
│   Hive Table    │  (Retained)
│   (Original)    │
└─────────────────┘
        │
        │ Copy
        ▼
┌─────────────────┐
│  Iceberg Table  │  (New)
│  (table_iceberg)│
└─────────────────┘

    Characteristics - Original table preserved - New table with suffix (e.g., _iceberg) - Safe rollback option - Requires additional storage

    File Format Translation¶

    Choose the output file format for Iceberg tables.

    iceberg:
  fileFormat: PARQUET  # PARQUET, ORC, or AVRO
    Format Best For
    PARQUET General purpose, wide compatibility
    ORC Heavy Hive usage, compression
    AVRO Schema evolution, row-level access

    Iceberg Version¶

    Select the Iceberg format version.

    iceberg:
  version: 2  # 1 or 2
    Version Features
    V1 Basic features, wider compatibility
    V2 Row-level deletes, position deletes

    Recommendation: Use V2 for new implementations.

    Configuration¶

    Basic Iceberg Conversion¶

    dataStrategy: ICEBERG_CONVERSION

iceberg:
  enabled: true
  conversionMode: SHADOW
  version: 2
  fileFormat: PARQUET

    Full Configuration¶

    iceberg:
  enabled: true
  conversionMode: SHADOW
  version: 2
  fileFormat: PARQUET

  # Table properties to set on Iceberg tables
  tableProperties:
    write.format.default: parquet
    write.parquet.compression-codec: zstd
    history.expire.max-snapshot-age-ms: 432000000  # 5 days

    With Storage Migration¶

    Combine Iceberg conversion with storage migration:

    dataStrategy: ICEBERG_CONVERSION

transfer:
  warehouse:
    external: 's3a://data-lake/iceberg/warehouse'

iceberg:
  enabled: true
  conversionMode: IN_PLACE
  version: 2
  fileFormat: PARQUET

    Migration Process¶

    Pre-Migration¶

    1. Verify Iceberg support
    2. Target cluster has Iceberg catalog
    3. Required Iceberg libraries installed

    4. Compatible Hive/Spark versions

    5. Evaluate tables

    6. Identify conversion candidates
    7. Estimate storage requirements

    8. Plan table naming

    9. Test conversion

    10. Convert small table first
    11. Verify queries work
    12. Check performance

    Migration Steps¶

    1. Configure HMS-Mirror 

      iceberg:
  enabled: true
  conversionMode: SHADOW
  version: 2

    2. Run dry run

    3. Review generated DDL

    4. Verify Iceberg properties

    5. Execute conversion

    6. Monitor progress

    7. Watch for errors

    8. Validate results

    9. Query converted tables
    10. Verify row counts
    11. Test time travel

    Post-Migration¶

    1. Update applications
    2. Point to Iceberg tables

    3. Update query syntax if needed

    4. Configure maintenance

    5. Set up compaction

    6. Configure snapshot expiration

    7. Remove shadow originals (if using SHADOW)

    8. After validation complete
    9. Keep for rollback period

    Generated DDL¶

    Iceberg Table Creation¶

    HMS-Mirror generates Iceberg-specific DDL:

    -- Create Iceberg table
CREATE TABLE database.table_iceberg (
  id BIGINT,
  name STRING,
  created_date DATE
)
USING ICEBERG
PARTITIONED BY (date_trunc('month', created_date))
TBLPROPERTIES (
  'format-version' = '2',
  'write.format.default' = 'parquet'
);

    Data Migration¶

    -- Insert data from source
INSERT INTO database.table_iceberg
SELECT * FROM database.table_source;

    Table Properties¶

    Common Iceberg Properties¶

    Property Description Example
    format-version Iceberg format version 2
    write.format.default Default file format parquet
    write.parquet.compression-codec Compression zstd, snappy
    history.expire.max-snapshot-age-ms Snapshot retention 432000000
    write.target-file-size-bytes Target file size 536870912

    Setting Properties¶

    iceberg:
  tableProperties:
    format-version: "2"
    write.format.default: "parquet"
    write.parquet.compression-codec: "zstd"
    write.target-file-size-bytes: "536870912"

    Partition Handling¶

    Hive Partitioning¶

    Traditional Hive partitioning: 

    -- Hive table
PARTITIONED BY (year INT, month INT)

    Iceberg Partitioning¶

    Iceberg supports hidden partitioning: 

    -- Iceberg table
PARTITIONED BY (year(event_date), month(event_date))

    Partition Translation¶

    HMS-Mirror translates partitions:

    Hive Partition Iceberg Equivalent
    year INT year(date_col)
    month INT month(date_col)
    day INT day(date_col)
    bucket INT bucket(N, col)

    Querying Iceberg Tables¶

    Basic Queries¶

    Iceberg tables work with standard SQL:

    SELECT * FROM database.iceberg_table
WHERE created_date > '2024-01-01';

    Time Travel¶

    Query historical data:

    -- As of specific timestamp
SELECT * FROM database.iceberg_table
TIMESTAMP AS OF '2024-01-15 10:00:00';

-- As of specific snapshot
SELECT * FROM database.iceberg_table
VERSION AS OF 1234567890;

    Metadata Queries¶

    -- View snapshots
SELECT * FROM database.iceberg_table.snapshots;

-- View history
SELECT * FROM database.iceberg_table.history;

-- View files
SELECT * FROM database.iceberg_table.files;

    Maintenance¶

    Compaction¶

    Combine small files:

    -- Rewrite data files
CALL system.rewrite_data_files('database.iceberg_table');

    Snapshot Expiration¶

    Remove old snapshots:

    -- Expire snapshots older than timestamp
CALL system.expire_snapshots('database.iceberg_table',
  TIMESTAMP '2024-01-01 00:00:00');

    Orphan File Cleanup¶

    Remove unreferenced files:

    CALL system.remove_orphan_files('database.iceberg_table');

    Troubleshooting¶

    Common Issues¶

    "Iceberg not supported" 

    Error: Iceberg format not available on cluster
    Solution: Install Iceberg libraries, configure catalog

    "Partition conversion failed" 

    Error: Cannot translate partition specification
    Solution: Manual partition mapping may be needed

    "Type not supported" 

    Error: Data type X not supported in Iceberg
    Solution: Transform data types before conversion

    Verification¶

    Check table is Iceberg 

    DESCRIBE FORMATTED database.table;
-- Look for: Provider: iceberg

    Verify row counts 

    SELECT COUNT(*) FROM database.hive_table;
SELECT COUNT(*) FROM database.iceberg_table;

    Test time travel 

    SELECT * FROM database.iceberg_table
VERSION AS OF 1;

    Best Practices¶

    Planning¶

    1. Start with non-critical tables
    2. Test thoroughly before production
    3. Plan storage for shadow mode
    4. Document rollback procedure

    Conversion¶

    1. Use SHADOW mode initially - Easy rollback
    2. Verify data accuracy - Compare counts
    3. Test queries - Ensure compatibility
    4. Monitor performance - May change

    Operations¶

    1. Set up compaction schedule
    2. Configure snapshot expiration
    3. Monitor storage usage
    4. Plan for schema evolution

    Example Workflows¶

    Simple Conversion¶

    # Convert sales tables to Iceberg
dataStrategy: ICEBERG_CONVERSION

iceberg:
  enabled: true
  conversionMode: SHADOW
  version: 2
  fileFormat: PARQUET

# Dataset: sales database
databases:
  - name: sales_db
    includePattern: "fact_.*|dim_.*"

    Full Migration with Iceberg¶

    # Migrate from HDP to CDP with Iceberg
dataStrategy: ICEBERG_CONVERSION

transfer:
  warehouse:
    external: 's3a://data-lake/iceberg'

iceberg:
  enabled: true
  conversionMode: IN_PLACE
  version: 2
  fileFormat: PARQUET
  tableProperties:
    write.parquet.compression-codec: "zstd"
