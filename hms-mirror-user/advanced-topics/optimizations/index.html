    Query Optimizations¶

    HMS-Mirror provides several optimization settings to improve migration performance and tune Hive queries. This guide covers available optimizations and when to use them.

    Overview¶

    Optimizations control: - Query execution behavior - Data compression - Statistics collection - Session property overrides

    Optimization Settings¶

    Sort Dynamic Partition Inserts¶

    Orders data before inserting into partitioned tables.

    optimizations:
  sortDynamicPartitionInserts: true

    How It Works

    Without sorting: 

    Data arrives in random order
  ↓
Multiple files per partition
  ↓
Many small files created

    With sorting: 

    Data sorted by partition columns
  ↓
Sequential writes per partition
  ↓
Fewer, larger files

    Benefits - Fewer output files - Better write performance - Improved read performance later - Reduced small file problem

    When to Use - Large partitioned tables - Many partitions - When small files are a concern

    Trade-offs - Adds sorting overhead - May increase memory usage - Longer individual query time

    Auto Tune¶

    Automatically applies optimal settings.

    optimizations:
  autoTune: true

    What It Does - Analyzes table characteristics - Adjusts parallelism settings - Tunes memory allocation - Optimizes join strategies

    When to Use - Default for most migrations - When unsure of optimal settings - Mixed table sizes

    Text Compression¶

    Compression for text-based formats.

    optimizations:
  textCompression: true
  compressionCodec: GZIP  # or SNAPPY, LZ4, ZSTD

    Available Codecs

    Codec Ratio Speed CPU
    GZIP High Slow High
    SNAPPY Medium Fast Low
    LZ4 Medium Very Fast Low
    ZSTD High Fast Medium

    Recommendations - SNAPPY - Default, good balance - ZSTD - Best overall for most cases - GZIP - Maximum compression, slow - LZ4 - Maximum speed, lower compression

    Statistics¶

    Control statistics gathering during migration.

    optimizations:
  buildShadowStatistics: true

    Purpose - Enables query optimizer - Improves execution plans - Better performance post-migration

    Options

    Option Description
    buildShadowStatistics Build stats during migration
    Skip statistics Faster migration, build later

    Property Overrides¶

    Set custom Hive session properties for LEFT and/or RIGHT clusters.

    Configuration Structure¶

    optimizations:
  overrides:
    - key: "property.name"
      leftValue: "value-for-left"
      rightValue: "value-for-right"

    Common Override Properties¶

    Dynamic Partitioning¶

    overrides:
  - key: "hive.exec.dynamic.partition"
    rightValue: "true"
  - key: "hive.exec.dynamic.partition.mode"
    rightValue: "nonstrict"
  - key: "hive.exec.max.dynamic.partitions"
    rightValue: "10000"
  - key: "hive.exec.max.dynamic.partitions.pernode"
    rightValue: "1000"

    Memory Settings¶

    overrides:
  - key: "hive.tez.container.size"
    rightValue: "8192"
  - key: "hive.tez.java.opts"
    rightValue: "-Xmx6144m"

    Parallelism¶

    overrides:
  - key: "hive.exec.parallel"
    rightValue: "true"
  - key: "hive.exec.parallel.thread.number"
    rightValue: "16"

    Compression¶

    overrides:
  - key: "hive.exec.compress.output"
    rightValue: "true"
  - key: "mapreduce.output.fileoutputformat.compress.codec"
    rightValue: "org.apache.hadoop.io.compress.SnappyCodec"

    Vectorization¶

    overrides:
  - key: "hive.vectorized.execution.enabled"
    rightValue: "true"
  - key: "hive.vectorized.execution.reduce.enabled"
    rightValue: "true"

    LEFT vs RIGHT Values¶

    Scope Use Case
    leftValue Settings for source cluster queries
    rightValue Settings for target cluster queries
    Both When same setting needed on both

    Example 

    overrides:
  # Different values per cluster
  - key: "hive.exec.parallel.thread.number"
    leftValue: "8"    # Smaller source cluster
    rightValue: "16"  # Larger target cluster

  # Same value for both
  - key: "hive.exec.dynamic.partition.mode"
    leftValue: "nonstrict"
    rightValue: "nonstrict"

    Optimization Scenarios¶

    High-Volume Migration¶

    For migrating large amounts of data:

    optimizations:
  sortDynamicPartitionInserts: true
  autoTune: true

  overrides:
    - key: "hive.exec.parallel"
      rightValue: "true"
    - key: "hive.exec.parallel.thread.number"
      rightValue: "16"
    - key: "hive.tez.container.size"
      rightValue: "8192"
    - key: "hive.exec.max.dynamic.partitions"
      rightValue: "50000"

    Many Small Tables¶

    For datasets with numerous small tables:

    optimizations:
  sortDynamicPartitionInserts: false  # Not needed for small tables
  autoTune: true

  overrides:
    - key: "hive.exec.parallel"
      rightValue: "true"
    - key: "hive.exec.parallel.thread.number"
      rightValue: "32"  # Higher parallelism

    Large Partitioned Tables¶

    For tables with many partitions:

    optimizations:
  sortDynamicPartitionInserts: true

  overrides:
    - key: "hive.exec.dynamic.partition"
      rightValue: "true"
    - key: "hive.exec.dynamic.partition.mode"
      rightValue: "nonstrict"
    - key: "hive.exec.max.dynamic.partitions"
      rightValue: "100000"
    - key: "hive.exec.max.dynamic.partitions.pernode"
      rightValue: "10000"
    - key: "hive.optimize.sort.dynamic.partition"
      rightValue: "true"

    Memory-Constrained Environment¶

    When cluster resources are limited:

    optimizations:
  sortDynamicPartitionInserts: false  # Reduces memory
  autoTune: false

  overrides:
    - key: "hive.exec.parallel.thread.number"
      rightValue: "4"  # Lower parallelism
    - key: "hive.tez.container.size"
      rightValue: "2048"  # Smaller containers
    - key: "hive.auto.convert.join"
      rightValue: "false"  # Avoid large hash joins

    Compression-Focused¶

    Minimize storage with maximum compression:

    optimizations:
  textCompression: true
  compressionCodec: ZSTD

  overrides:
    - key: "hive.exec.compress.output"
      rightValue: "true"
    - key: "parquet.compression"
      rightValue: "ZSTD"
    - key: "orc.compress"
      rightValue: "ZSTD"

    Property Reference¶

    Execution Properties¶

    Property Description Default
    hive.exec.parallel Enable parallel execution false
    hive.exec.parallel.thread.number Parallel threads 8
    hive.vectorized.execution.enabled Enable vectorization true

    Dynamic Partition Properties¶

    Property Description Default
    hive.exec.dynamic.partition Enable dynamic partitions true
    hive.exec.dynamic.partition.mode Strict or nonstrict strict
    hive.exec.max.dynamic.partitions Max total partitions 1000
    hive.exec.max.dynamic.partitions.pernode Max per node 100

    Memory Properties¶

    Property Description Default
    hive.tez.container.size Container memory (MB) 4096
    hive.tez.java.opts JVM options -Xmx3276m
    hive.auto.convert.join.noconditionaltask.size Map join size 10MB

    Compression Properties¶

    Property Description Values
    hive.exec.compress.output Compress output true/false
    mapreduce.output.fileoutputformat.compress.codec Codec class Various
    parquet.compression Parquet codec SNAPPY, GZIP, ZSTD
    orc.compress ORC codec SNAPPY, ZLIB, ZSTD

    Tuning Tips¶

    General Guidelines¶

    1. Start with autoTune - Good baseline
    2. Enable parallel execution - Faster migrations
    3. Use sortDynamicPartitionInserts - For partitioned tables
    4. Monitor and adjust - Based on actual performance

    When to Increase Settings¶

    • More partitions → Higher partition limits
    • Larger tables → More memory
    • More tables → Higher parallelism
    • Slow writes → Enable sort dynamic partition

    When to Decrease Settings¶

    • Memory errors → Reduce container size
    • Cluster pressure → Lower parallelism
    • Small tables → Disable sorting overhead

    Monitoring Impact¶

    Watch for: - Query execution time - Memory usage - Output file sizes - Error rates

    Troubleshooting¶

    Memory Errors¶

    Error: Container killed by YARN for exceeding memory limits

    Solution 

    overrides:
  - key: "hive.tez.container.size"
    rightValue: "8192"  # Increase
  - key: "hive.tez.java.opts"
    rightValue: "-Xmx6144m"  # 75% of container

    Partition Errors¶

    Error: Maximum number of dynamic partitions exceeded

    Solution 

    overrides:
  - key: "hive.exec.max.dynamic.partitions"
    rightValue: "50000"
  - key: "hive.exec.max.dynamic.partitions.pernode"
    rightValue: "5000"

    Slow Performance¶

    Diagnose - Check query plans - Monitor resource usage - Review output file sizes

    Solutions 

    optimizations:
  sortDynamicPartitionInserts: true
  overrides:
    - key: "hive.exec.parallel"
      rightValue: "true"
    - key: "hive.vectorized.execution.enabled"
      rightValue: "true"

    Best Practices¶

    1. Test optimizations on small datasets first
    2. Document your override choices
    3. Monitor cluster resources during migration
    4. Adjust based on actual performance
    5. Consider cluster capacity
    6. Use autoTune for initial runs
