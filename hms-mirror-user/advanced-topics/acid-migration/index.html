    ACID Table Migration¶

    ACID (Atomicity, Consistency, Isolation, Durability) tables in Hive require special handling during migration. This guide covers ACID table concepts and migration options.

    Understanding ACID Tables¶

    What are ACID Tables?¶

    ACID tables are Hive managed tables that support: - Atomicity - Transactions complete fully or not at all - Consistency - Database remains consistent - Isolation - Concurrent transactions don't interfere - Durability - Committed changes persist

    ACID Table Characteristics¶

    Property Description
    Transaction support INSERT, UPDATE, DELETE operations
    Table type Always managed (not external)
    Storage format ORC with ACID metadata
    Bucketing Usually bucketed (Hive 2.x required it)
    Compaction Requires periodic compaction

    ACID Version Differences¶

    Hive 2.x (Legacy) - ACID v1 format - Bucketing required - Limited update/delete support - Delta files for changes

    Hive 3.x (CDP) - ACID v2 format - Bucketing optional - Full CRUD support - Improved compaction - Better performance

    Migration Challenges¶

    Why ACID Migration is Complex¶

    ┌─────────────────┐                    ┌─────────────────┐
│  Legacy ACID    │                    │   CDP ACID      │
│  (Hive 2.x)     │   Challenge ──►    │   (Hive 3.x)    │
├─────────────────┤                    ├─────────────────┤
│ • ACID v1       │                    │ • ACID v2       │
│ • Required      │                    │ • Optional      │
│   bucketing     │                    │   bucketing     │
│ • Delta files   │                    │ • Improved      │
│                 │                    │   storage       │
└─────────────────┘                    └─────────────────┘

    Common Issues¶

    1. Format incompatibility - V1 and V2 formats differ
    2. SQL limitations - Can't INSERT directly from V1 to V2
    3. Export restrictions - Some ACID tables can't be exported
    4. Bucket mismatches - Different bucketing schemes
    5. In-flight transactions - Active transactions block export

    Migration Options¶

    HMS-Mirror provides several options for ACID tables:

    Option 1: Include ACID Tables¶

    Migrate ACID tables as-is, preserving transactional properties.

    acid:
  includeAcidTables: true
  downgradeAcid: false

    Requirements - Use EXPORT_IMPORT strategy - Compatible Hive versions - Both clusters support ACID

    Best For - Hive 3.x to Hive 3.x migrations - When transactional properties are required - Tables actively using UPDATE/DELETE

    Option 2: Downgrade to External¶

    Convert ACID tables to non-transactional external tables.

    acid:
  includeAcidTables: true
  downgradeAcid: true

    Process 1. Read data from ACID table 2. Create external table on target 3. Insert data using SQL 4. Result is non-transactional table

    Best For - Legacy to CDP migrations - When transaction support not needed - Simplifying table management

    Option 3: ACID Only Mode¶

    Process only ACID tables, skip all others.

    acid:
  includeAcidTables: true
  acidOnly: true

    Use Case - Phased migration (ACID tables separately) - Focused ACID troubleshooting - When ACID tables need special handling

    Option 4: In-Place Migration¶

    Migrate ACID tables within the same cluster.

    acid:
  inPlace: true

    Use Case - Storage migration (HDFS to S3) - Same cluster, different location - Namespace changes

    Configuration Settings¶

    Basic ACID Settings¶

    Setting Description Default
    includeAcidTables Process ACID tables false
    acidOnly Only process ACID tables false
    downgradeAcid Convert to external false
    inPlace Same cluster migration false

    Advanced Settings¶

    Setting Description Default
    partitionLimit Max partitions for ACID 0 (unlimited)
    artificialBucketThreshold Min buckets for SQL 0
    migrateAcid Enable ACID migration false

    Configuration Examples¶

    Include ACID, Keep Transactional 

    acid:
  includeAcidTables: true
  migrateAcid: true
  downgradeAcid: false

    Downgrade All ACID Tables 

    acid:
  includeAcidTables: true
  downgradeAcid: true

    ACID Tables Only with Partition Limit 

    acid:
  includeAcidTables: true
  acidOnly: true
  partitionLimit: 500

    Strategy Selection for ACID¶

    Which Strategy to Use¶

    Scenario Recommended Strategy
    Keep ACID properties EXPORT_IMPORT
    Downgrade to external SQL or HYBRID
    Large ACID tables EXPORT_IMPORT
    Small ACID tables (downgrade) SQL
    Mixed tables HYBRID with downgrade

    HYBRID Behavior with ACID¶

    When using HYBRID strategy:

    ┌─────────────────┐
│  ACID Table     │
│  Encountered    │
└────────┬────────┘
         │
         ▼
┌─────────────────────────┐
│ downgradeAcid enabled?  │
└────────┬────────────────┘
         │
    ┌────┴────┐
    │         │
    ▼         ▼
  Yes        No
    │         │
    ▼         ▼

│  SQL   │  │EXPORT_IMPORT│

    Partition Considerations¶

    Partition Limits¶

    ACID tables with many partitions require special handling.

    acid:
  partitionLimit: 500  # Max partitions for SQL strategy

    When Limit is Exceeded - SQL strategy → Switch to EXPORT_IMPORT - EXPORT_IMPORT → May take longer - Job fails if no alternative

    Recommendations by Partition Count¶

    Partitions Recommendation
    < 100 SQL works well
    100 - 500 SQL or EXPORT_IMPORT
    500 - 1000 EXPORT_IMPORT preferred
    > 1000 EXPORT_IMPORT only

    Bucket Handling¶

    Bucket Threshold¶

    Control when SQL strategy can be used based on bucket count.

    acid:
  artificialBucketThreshold: 2

    Purpose - Tables with buckets may perform poorly with SQL - Threshold triggers switch to EXPORT_IMPORT - Higher threshold = more tables use SQL

    Bucket Considerations¶

    Bucket Count SQL Performance Recommendation
    0 (none) Good SQL is fine
    1-4 Acceptable SQL usually works
    5+ Poor Use EXPORT_IMPORT

    Migration Workflow¶

    Pre-Migration Steps¶

    1. Identify ACID tables 

      SELECT * FROM information_schema.tables
WHERE table_type = 'MANAGED_TABLE';

    2. Check for active transactions 

      SHOW TRANSACTIONS;

    3. Run compaction (recommended) 

      ALTER TABLE table_name COMPACT 'major';

    4. Verify compaction complete 

      SHOW COMPACTIONS;

    Migration Steps¶

    1. Configure ACID settings
    2. Choose include/exclude/downgrade options

    3. Set appropriate limits

    4. Select strategy

    5. EXPORT_IMPORT for keeping ACID

    6. SQL or HYBRID for downgrade

    7. Run dry run

    8. Verify ACID tables are handled correctly

    9. Check for partition/bucket issues

    10. Execute migration

    11. Monitor for ACID-specific errors
    12. Check transaction states

    Post-Migration Steps¶

    1. Verify data
    2. Row counts match

    3. Sample data comparison

    4. Check table properties

    5. Transactional property (if kept)

    6. Storage location

    7. Run compaction on target (if ACID) 

      ALTER TABLE table_name COMPACT 'major';

    Troubleshooting ACID Migration¶

    Common Errors¶

    "Cannot export transactional table" 

    Error: Cannot export ACID table using SQL strategy
    Solution: Use EXPORT_IMPORT strategy or enable downgrade

    "Active transaction on table" 

    Error: Table has active transactions
    Solution: Wait for transactions to complete or abort them

    "Incompatible ACID version" 

    Error: Source ACID v1 incompatible with target
    Solution: Enable downgrade to convert to external

    "Bucket count mismatch" 

    Error: Target bucket specification differs
    Solution: Enable downgrade or adjust bucket threshold

    Verification Queries¶

    Check if table is ACID 

    DESCRIBE FORMATTED table_name;
-- Look for: transactional=true

    Check ACID version 

    SHOW TBLPROPERTIES table_name ('transactional_properties');

    Compare row counts 

    -- Source
SELECT COUNT(*) FROM source_db.table_name;
-- Target
SELECT COUNT(*) FROM target_db.table_name;

    Best Practices¶

    Before Migration¶

    1. Inventory ACID tables - Know what you're migrating
    2. Run major compaction - Consolidate delta files
    3. Quiesce activity - Pause writes during migration
    4. Document bucket configurations - For reference

    During Migration¶

    1. Use EXPORT_IMPORT - Most reliable for ACID
    2. Set appropriate limits - Avoid timeout/memory issues
    3. Monitor progress - ACID tables often take longer
    4. Handle failures individually - Don't restart entire job

    After Migration¶

    1. Verify transactional property - If kept
    2. Test INSERT/UPDATE/DELETE - If ACID preserved
    3. Run compaction - On target if ACID
    4. Update applications - Point to new tables

    Decision Matrix¶

    Requirement Keep ACID Downgrade
    Need UPDATE/DELETE ✓
    Simpler management ✓
    Legacy to CDP ✓ (recommended)
    CDP to CDP ✓ Optional
    Performance priority ✓
    Compliance (transactions) ✓

    Example Configurations¶

    HDP to CDP (Downgrade)¶

    # Downgrade all ACID tables to external
acid:
  includeAcidTables: true
  downgradeAcid: true
  partitionLimit: 500

# Use HYBRID to automatically select strategy
dataStrategy: HYBRID

    CDP to CDP (Preserve)¶

    # Keep ACID properties
acid:
  includeAcidTables: true
  migrateAcid: true
  downgradeAcid: false

# Use EXPORT_IMPORT for ACID tables
dataStrategy: EXPORT_IMPORT

    Phased ACID Migration¶

    # First job: Non-ACID tables
acid:
  includeAcidTables: false

# Second job: ACID tables only
acid:
  includeAcidTables: true
  acidOnly: true
  downgradeAcid: true
