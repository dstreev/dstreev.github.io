    Storage Migration¶

    Storage migration moves table data to a new storage location within the same cluster. This guide covers the STORAGE_MIGRATION strategy and related concepts.

    What is Storage Migration?¶

    Storage migration relocates table data while keeping the same Hive metastore:

    Same Cluster
┌─────────────────────────────────────────────────────────────────┐
│                                                                  │
│  ┌─────────────────┐                ┌─────────────────┐         │
│  │   Old Storage   │  ───Move───►   │   New Storage   │         │
│  │                 │                │                 │         │
│  │  hdfs://old/    │                │  s3a://new/     │         │
│  │  warehouse/     │                │  warehouse/     │         │
│  └─────────────────┘                └─────────────────┘         │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                    Hive Metastore                        │   │
│  │   (Table locations updated to new storage)               │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘

    Use Cases¶

    HDFS to Cloud Storage¶

    Migrate from on-premises HDFS to cloud object stores:

    Source Target
    hdfs://namenode/warehouse s3a://bucket/warehouse
    hdfs://namenode/warehouse abfs://container@account/warehouse
    hdfs://namenode/warehouse gs://bucket/warehouse

    Namespace Changes¶

    Move data between HDFS namespaces:

    Source Target
    hdfs://old-ns/data hdfs://new-ns/data
    hdfs://cluster1/warehouse hdfs://cluster2/warehouse

    Ozone Migration¶

    Move from HDFS to Apache Ozone:

    Source Target
    hdfs://cluster/warehouse o3fs://bucket.volume/warehouse
    hdfs://cluster/warehouse ofs://ozone/volume/bucket/warehouse

    Warehouse Reorganization¶

    Restructure warehouse directory layout:

    Source Target
    /apps/hive/warehouse /warehouse/managed
    /data/external /warehouse/external

    Configuration¶

    Basic Settings¶

    dataStrategy: STORAGE_MIGRATION

transfer:
  warehouse:
    external: 's3a://new-bucket/warehouse/external'
    managed: 's3a://new-bucket/warehouse/managed'

    Translation Types¶

    How paths are transformed from source to target.

    RELATIVE Translation¶

    Preserves the relative path structure from the source.

    Source: hdfs://old-cluster/warehouse/db/table
Target: s3a://bucket/warehouse/db/table
         └──────────────────┬───────────┘
                    Preserved structure

    Configuration: 

    transfer:
  storageMigration:
    translationType: RELATIVE

    Use When - Simple namespace swap - Keeping existing directory structure - Path hierarchy should be preserved

    ALIGNED Translation¶

    Maps full source paths to new target paths with complete control.

    Source: hdfs://old-cluster/apps/hive/warehouse/db/table
Target: s3a://bucket/warehouse/db/table
         └─────────┬─────────┘
           New base path

    Configuration: 

    transfer:
  storageMigration:
    translationType: ALIGNED

    Use When - Changing directory structure - Complex path transformations - Multiple source paths to single target

    Data Movement Strategies¶

    How data is actually moved.

    SQL¶

    Uses INSERT...SELECT to move data through Hive.

    transfer:
  storageMigration:
    dataMovementStrategy: SQL

    Characteristics - Works through HiveServer2 - Data is re-serialized - Can apply transformations - Good for small-medium tables

    DISTCP¶

    Uses Hadoop DistCp for direct file copy.

    transfer:
  storageMigration:
    dataMovementStrategy: DISTCP

    Characteristics - Direct file copy - Preserves file format exactly - Better for large datasets - Requires DistCp access

    EXPORT_IMPORT¶

    Uses Hive's native export/import.

    transfer:
  storageMigration:
    dataMovementStrategy: EXPORT_IMPORT

    Characteristics - Native Hive mechanism - Handles ACID tables - Creates intermediate files

    MANUAL¶

    Generates scripts but doesn't execute data movement.

    transfer:
  storageMigration:
    dataMovementStrategy: MANUAL

    Characteristics - User handles data movement - Maximum control - Use with external tools

    Data Flow Direction¶

    Controls whether target cluster pulls or source pushes.

    PULL (Default)¶

    Target cluster pulls data from source.

    transfer:
  storageMigration:
    dataFlow: PULL

    Source                      Target
┌────────┐    Read from    ┌────────┐
│        │ ◄─────────────  │        │
│  Data  │                 │ Process│
│        │  Write to       │        │
│        │ ◄─────────────  │        │
└────────┘                 └────────┘

    PUSH¶

    Source cluster pushes data to target.

    transfer:
  storageMigration:
    dataFlow: PUSH

    Source                      Target
┌────────┐   Write to      ┌────────┐
│        │ ───────────►    │        │
│ Process│                 │  Data  │
│        │                 │        │
└────────┘                 └────────┘

    Namespace Translation¶

    Global Location Map¶

    Define mappings from source to target namespaces.

    transfer:
  globalLocationMap:
    "hdfs://old-nameservice": "s3a://data-lake"
    "hdfs://archive-ns": "s3a://archive-bucket"

    How It Works 

    Table Location: hdfs://old-nameservice/warehouse/db/table
                └────────┬───────────┘
                    Matched

New Location:   s3a://data-lake/warehouse/db/table
                └──────┬──────┘
                   Replaced

    Multiple Namespace Mapping¶

    transfer:
  globalLocationMap:
    "hdfs://prod-ns/warehouse/managed": "s3a://prod-bucket/managed"
    "hdfs://prod-ns/warehouse/external": "s3a://prod-bucket/external"
    "hdfs://archive-ns": "s3a://archive-bucket"

    Warehouse Directories¶

    External Warehouse¶

    Location for external tables on target.

    transfer:
  warehouse:
    external: 's3a://bucket/warehouse/external'

    Managed Warehouse¶

    Location for managed tables on target.

    transfer:
  warehouse:
    managed: 's3a://bucket/warehouse/managed'

    Complete Example¶

    transfer:
  warehouse:
    external: 's3a://data-lake/warehouse/external'
    managed: 's3a://data-lake/warehouse/managed'
  globalLocationMap:
    "hdfs://hdp-cluster": "s3a://data-lake"
  storageMigration:
    translationType: ALIGNED
    dataMovementStrategy: DISTCP
    dataFlow: PULL

    Migration Workflow¶

    Phase 1: Planning¶

    1. Inventory tables and locations 

      SELECT t.TBL_NAME, s.LOCATION
FROM TBLS t JOIN SDS s ON t.SD_ID = s.SD_ID;

    2. Identify unique namespaces

    3. List all source storage locations

    4. Plan target locations

    5. Calculate storage requirements

    6. Total data size

    7. Target storage capacity

    8. Plan network/credentials

    9. Access to both storage systems
    10. Appropriate permissions

    Phase 2: Configuration¶

    1. Create HMS-Mirror configuration 

      dataStrategy: STORAGE_MIGRATION

transfer:
  warehouse:
    external: 's3a://target/external'
    managed: 's3a://target/managed'
  storageMigration:
    translationType: RELATIVE
    dataMovementStrategy: DISTCP

    2. Define dataset

    3. Select databases to migrate

    4. Filter tables if needed

    5. Configure connection

    6. Only LEFT cluster needed
    7. Ensure access to both storage systems

    Phase 3: Dry Run¶

    1. Run dry run
    2. Generate migration scripts

    3. Review planned changes

    4. Verify translations

    5. Check old → new location mappings

    6. Confirm no unexpected paths

    7. Estimate duration

    8. Based on data sizes
    9. Network bandwidth

    Phase 4: Execution¶

    1. Execute migration
    2. Start during low-activity period

    3. Monitor progress

    4. Monitor data movement

    5. Track DistCp or SQL progress

    6. Watch for errors

    7. Verify completion

    8. All tables processed
    9. Data accessible at new locations

    Phase 5: Validation¶

    1. Verify table access 

      SELECT COUNT(*) FROM database.table;

    2. Check locations updated 

      DESCRIBE FORMATTED database.table;

    3. Compare row counts

    4. Source vs target

    5. Test queries

    6. Run representative queries
    7. Verify performance

    Special Considerations¶

    Partitioned Tables¶

    Each partition location is updated:

    Before:
  Table: hdfs://old/warehouse/db/sales
  Partitions:
    year=2023: hdfs://old/warehouse/db/sales/year=2023
    year=2024: hdfs://old/warehouse/db/sales/year=2024

After:
  Table: s3a://new/warehouse/db/sales
  Partitions:
    year=2023: s3a://new/warehouse/db/sales/year=2023
    year=2024: s3a://new/warehouse/db/sales/year=2024

    ACID Tables¶

    Special handling for transactional tables:

    • Use EXPORT_IMPORT for best results
    • Consider compaction before migration
    • Verify ACID properties after migration

    External Tables with Custom Locations¶

    Tables with non-standard locations:

    # Table at: hdfs://cluster/custom/path/my_data
# Not under warehouse directory

transfer:
  globalLocationMap:
    "hdfs://cluster/custom": "s3a://bucket/custom"

    Large Tables¶

    For very large tables:

    1. Use DISTCP - More efficient than SQL
    2. Consider parallel jobs - Split by database
    3. Off-peak execution - Minimize impact
    4. Incremental if possible - Migrate in batches

    Troubleshooting¶

    Permission Errors¶

    Error: Access denied to s3a://bucket/path

    Solutions - Verify IAM roles/credentials - Check bucket policies - Ensure HMS-Mirror user has access

    Path Not Found¶

    Error: Source path does not exist

    Solutions - Verify source table locations - Check namespace spelling - Ensure storage is accessible

    Slow Migration¶

    Causes - Large data volumes - Network bandwidth - Cluster resources

    Solutions - Use DISTCP over SQL - Increase parallelism - Run during off-peak - Split into multiple jobs

    Location Not Updated¶

    Causes - Translation type mismatch - Missing namespace mapping

    Solutions - Check globalLocationMap - Verify translationType setting - Review dry run output

    Best Practices¶

    Planning¶

    1. Test with small dataset first
    2. Verify storage access before migration
    3. Calculate and provision storage
    4. Plan rollback procedure

    Execution¶

    1. Run dry run first
    2. Migrate during low-activity periods
    3. Monitor progress actively
    4. Keep source data until verified

    Validation¶

    1. Verify row counts
    2. Test query performance
    3. Check all partitions
    4. Validate ACID tables specifically

    Cleanup¶

    1. Remove source data after verification
    2. Update any hard-coded paths
    3. Document new locations
    4. Update backup procedures
