    Kerberos Authentication¶

    This guide covers configuring HMS-Mirror for Kerberos-secured DataLake environments.

    Overview¶

    Kerberos is the standard authentication mechanism for enterprise DataLake environments powered by Hive and Hive Metastore. HMS-Mirror supports Kerberos authentication for HiveServer2 connections. Kerberos credentials are managed centrally through the Preferences > Kerberos page in the web UI.

    Prerequisites¶

    On the HMS-Mirror Server¶

    1. Kerberos client installed 

      # RHEL/CentOS
yum install krb5-workstation

# Ubuntu/Debian
apt-get install krb5-user

    2. krb5.conf configured

    3. Located at /etc/krb5.conf

    4. Contains realm and KDC information

    5. Network access to KDC

    6. Port 88 (UDP/TCP) to Key Distribution Center
    7. Port 749 (TCP) for kadmin (if needed)

    Kerberos Configuration File¶

    Example /etc/krb5.conf:

    [libdefaults]
    default_realm = EXAMPLE.COM
    dns_lookup_realm = false
    dns_lookup_kdc = false
    ticket_lifetime = 24h
    renew_lifetime = 7d
    forwardable = true

[realms]
    EXAMPLE.COM = {
        kdc = kdc.example.com
        admin_server = kdc.example.com
    }

[domain_realm]
    .example.com = EXAMPLE.COM
    example.com = EXAMPLE.COM

    Managing Kerberos in HMS-Mirror¶

    HMS-Mirror provides a centralized interface for managing Kerberos authentication through the web UI at Utilities > Preferences > Kerberos.

    Configuring Kerberos Credentials¶

    1. Navigate to Utilities > Preferences in the web UI
    2. Select the Kerberos tab
    3. Configure your authentication method:

    Using a Keytab (Recommended for Production)¶

    1. Principal: Enter your Kerberos principal (e.g., hms-mirror@EXAMPLE.COM)
    2. Keytab File: Upload your keytab file through the interface
    3. Refresh Schedule (optional): Set a cron expression for automatic ticket renewal (e.g., 0 */4 * * * for every 4 hours)
    4. Auto Recovery: Enable to automatically re-initialize tickets if they expire

    Using Password¶

    1. Principal: Enter your Kerberos principal
    2. Password: Enter your Kerberos password
    3. Configure refresh schedule as needed

    Monitoring Ticket Status¶

    The Kerberos tab displays: - Current ticket validity status - Ticket expiration time - Next scheduled refresh time - Error messages if authentication fails

    Manual Ticket Refresh¶

    Click the Refresh button to manually refresh the Kerberos ticket at any time.

    HiveServer2 Connection Configuration¶

    When connecting to a Kerberized HiveServer2, the service principal must be specified in the JDBC URL, not in the connection properties.

    Important: Principal in URL¶

    The Kerberos service principal for HiveServer2 must be included in the JDBC URI:

    jdbc:hive2://hs2.example.com:10000/default;principal=hive/_HOST@EXAMPLE.COM

    Correct - Principal in URI: 

    hiveServer2:
  uri: jdbc:hive2://hs2.example.com:10000/default;principal=hive/_HOST@EXAMPLE.COM

    Incorrect - Principal in connection properties (will not work): 

    # DO NOT DO THIS
hiveServer2:
  uri: jdbc:hive2://hs2.example.com:10000/default
  connectionProperties:
    principal: hive/_HOST@EXAMPLE.COM  # This won't work!

    Connection Examples¶

    Binary Transport 

    hiveServer2:
  uri: jdbc:hive2://hs2.example.com:10000/default;principal=hive/_HOST@EXAMPLE.COM
  connectionProperties:
    transportMode: binary

    HTTP Transport 

    hiveServer2:
  uri: jdbc:hive2://hs2.example.com:10001/default;transportMode=http;httpPath=cliservice;principal=HTTP/_HOST@EXAMPLE.COM

    With SSL 

    hiveServer2:
  uri: jdbc:hive2://hs2.example.com:10000/default;ssl=true;principal=hive/_HOST@EXAMPLE.COM
  connectionProperties:
    sslTrustStore: /path/to/truststore.jks
    sslTrustStorePassword: changeit

    With Zookeeper Discovery 

    hiveServer2:
  uri: jdbc:hive2://zk1:2181,zk2:2181,zk3:2181/;serviceDiscoveryMode=zooKeeper;zooKeeperNamespace=hiveserver2;principal=hive/_HOST@EXAMPLE.COM

    Metastore Direct Connections¶

    Direct Metastore connections use database authentication (username/password), not Kerberos. The Kerberos ticket managed in Preferences is used for HiveServer2 connections.

    metastoreDirect:
  type: MYSQL
  uri: jdbc:mysql://metastore.example.com:3306/hive
  username: hive_user
  password: password

    Principal Format¶

    Understanding Principals¶

    service/hostname@REALM
  │       │       │
  │       │       └── Kerberos realm (uppercase)
  │       └── Fully qualified hostname
  └── Service name (hive, HTTP, etc.)

    _HOST Placeholder¶

    The _HOST placeholder is automatically replaced with the actual hostname:

    hive/_HOST@EXAMPLE.COM
  ↓
hive/hs2.example.com@EXAMPLE.COM

    Common Service Principals¶

    Service Principal Format
    HiveServer2 (Binary) hive/_HOST@REALM
    HiveServer2 (HTTP) HTTP/_HOST@REALM
    Knox HTTP/_HOST@REALM

    Troubleshooting¶

    Common Errors¶

    "GSS initiate failed" 

    javax.security.sasl.SaslException: GSS initiate failed

    Causes - No valid ticket - Check the Kerberos tab in Preferences for ticket status - Wrong principal in URL - Clock skew between client and server

    Solutions 1. Check Preferences > Kerberos for ticket status 2. Click Refresh to re-initialize the ticket 3. Verify the principal in your connection URL is correct 4. Check time synchronization

    "Server not found in Kerberos database" 

    KrbException: Server not found in Kerberos database

    Causes - Wrong principal name in the connection URL - Service not registered in KDC

    Solutions - Verify principal exists: kadmin -q "listprincs *hive*" - Check hostname resolution - Verify _HOST replacement is working

    "Clock skew too great" 

    KrbException: Clock skew too great

    Solution 

    # Sync time with NTP
sudo ntpdate -s time.nist.gov
# or
sudo systemctl restart chronyd

    "Cannot find KDC" 

    Cannot locate KDC for realm

    Solutions - Verify /etc/krb5.conf configuration - Check DNS resolution for KDC - Verify network connectivity to port 88

    Diagnostic Commands¶

    # Check Kerberos configuration
cat /etc/krb5.conf

# Test KDC connectivity
telnet kdc.example.com 88

# Verbose kinit (for testing outside application)
KRB5_TRACE=/dev/stdout kinit user@REALM

# Verify DNS
nslookup hs2.example.com
host -t SRV _kerberos._udp.EXAMPLE.COM

    Debug Logging¶

    Enable Kerberos debugging:

    Java System Property 

    java -Dsun.security.krb5.debug=true -jar hms-mirror-server.jar

    In application.properties 

    logging.level.org.apache.hadoop.security=DEBUG
logging.level.sun.security.krb5=DEBUG

    Multi-Realm Setup¶

    Cross-Realm Authentication¶

    When source and target clusters are in different realms:

    # /etc/krb5.conf
[realms]
    SOURCE.COM = {
        kdc = kdc.source.com
    }
    TARGET.COM = {
        kdc = kdc.target.com
    }

[capaths]
    SOURCE.COM = {
        TARGET.COM = .
    }
    TARGET.COM = {
        SOURCE.COM = .
    }

    Configuration for Multiple Realms¶

    When working with multiple realms, ensure your Kerberos principal (configured in Preferences > Kerberos) has cross-realm trust established, and specify the appropriate service principal in each connection URL:

    # LEFT connection (SOURCE.COM)
leftConnection:
  hiveServer2:
    uri: jdbc:hive2://hs2.source.com:10000/default;principal=hive/_HOST@SOURCE.COM

# RIGHT connection (TARGET.COM)
rightConnection:
  hiveServer2:
    uri: jdbc:hive2://hs2.target.com:10000/default;principal=hive/_HOST@TARGET.COM

    Best Practices¶

    Security¶

    1. Use keytabs for production - Configure via Preferences > Kerberos
    2. Restrict keytab access - Secure keytab files before uploading
    3. Use dedicated service accounts - Not personal accounts
    4. Enable auto-recovery - Ensures continuous operation

    Reliability¶

    1. Configure refresh schedule - Set automatic ticket renewal in Preferences
    2. Monitor ticket status - Check the Kerberos tab regularly
    3. Enable auto-recovery - Automatically re-initialize expired tickets

    Operations¶

    1. Document principal names - For troubleshooting
    2. Coordinate with security team - For keytab creation
    3. Test connections - Use the connection test feature after configuring Kerberos

    Complete Configuration Example¶

    Step 1: Configure Kerberos Credentials¶

    1. Navigate to Utilities > Preferences > Kerberos
    2. Enter principal: hms-mirror@EXAMPLE.COM
    3. Upload keytab file
    4. Set refresh schedule: 0 */4 * * * (every 4 hours)
    5. Enable auto-recovery
    6. Save configuration

    Step 2: Create Kerberized Connection¶

    In Connections, create a new connection:

    name: prod-kerberized-cluster
legacyHive: false

hiveServer2:
  uri: jdbc:hive2://hs2.example.com:10000/default;principal=hive/_HOST@EXAMPLE.COM;ssl=true
  connectionProperties:
    transportMode: binary
    sslTrustStore: /opt/hms-mirror/security/truststore.jks
    sslTrustStorePassword: changeit

sessionProperties:
  hive.exec.dynamic.partition: "true"
  hive.exec.dynamic.partition.mode: "nonstrict"

    Step 3: Test Connection¶

    Click Test Connection to verify Kerberos authentication is working correctly.
