    Security¶

    This page covers JWT session management and the encryption architecture used to protect sensitive data in HMS-Mirror.

    JWT Secret Management¶

    HMS-Mirror uses JWT (JSON Web Tokens) for session authentication. Configuring a persistent JWT secret ensures that user sessions survive application restarts.

    Default Behavior¶

    When no JWT secret is configured (or the configured value is shorter than 32 bytes), HMS-Mirror automatically generates a random secret at startup and logs a warning. The application functions normally, but all sessions are lost on each restart.

    Setting a Persistent JWT Secret¶

    There are several ways to configure a persistent JWT secret, listed in order of priority (highest first):

    Method 1: Environment Variable (Highest Priority)¶

    Set the HMS_MIRROR_JWT_SECRET environment variable:

    export HMS_MIRROR_JWT_SECRET="<your-secret>"

    Generate a suitable secret:

    openssl rand -base64 48

    This method is recommended for Docker and Cloudera Manager deployments.

    Method 2: Admin UI¶

    Navigate to Preferences > Security in the HMS-Mirror web UI (requires Admin role):

    • Generate & Save: Click to auto-generate a cryptographically secure 48-byte secret. The secret is saved to web.properties and takes effect after restart.
    • Custom Secret: Enter your own secret (minimum 32 characters) and click "Save Secret".

    The UI shows whether a JWT secret is currently configured. The actual secret value is never displayed.

    Method 3: web.properties File¶

    Add the jwt.secret property to ~/.hms-mirror/settings/web.properties:

    jwt.secret=<your-base64-secret-at-least-32-bytes>

    Method 4: Docker¶

    # docker-compose.yml
environment:
  - HMS_MIRROR_JWT_SECRET=<your-secret>

    Configuration Priority¶

    When multiple methods are used, the highest priority wins:

    Priority Source Notes
    1 (highest) HMS_MIRROR_JWT_SECRET env var Overrides all other sources
    2 web.properties (jwt.secret=...) Set via Admin UI or manually
    3 Default (empty) Triggers random generation with warning

    What Happens When the Secret Changes¶

    When the JWT secret changes between restarts (or when switching from a random to a persistent secret), all existing user sessions become invalid. Users are automatically redirected to the login page and must log in again with their credentials.

    When the secret stays the same across restarts, user sessions continue seamlessly.

    Minimum Secret Length¶

    The JWT secret must be at least 32 bytes (characters) to meet the HS256 signing algorithm requirements. Secrets shorter than this are treated as unconfigured, and a random secret is generated instead.

    Token Lifetimes¶

    Token Default Lifetime Purpose
    Access token 15 minutes Short-lived token for API requests
    Refresh token 7 days Used to obtain new access tokens without re-login

    Security Notes¶

    • The JWT secret value is never returned in any API response
    • The web.properties file stores the secret in plaintext; ensure appropriate file permissions
    • The environment variable (HMS_MIRROR_JWT_SECRET) always takes the highest priority
    • Random secret generation uses java.security.SecureRandom with 48 bytes (384 bits)

    Encryption Architecture¶

    HMS-Mirror implements enterprise-grade security controls to protect sensitive data including database credentials, Kerberos keytabs, and configuration files. This section details the cryptographic implementation, compliance standards, and security best practices employed throughout the application.

    Compliance Standards¶

    The CryptoService is designed to meet or exceed the following industry standards:

    Standard Description Requirement Met
    PCI-DSS Payment Card Industry Data Security Standard AES-256 encryption, secure key management
    HIPAA Health Insurance Portability and Accountability Act Encryption at rest, access controls
    GDPR General Data Protection Regulation Data protection by design, encryption
    SOC 2 Service Organization Control Encryption, integrity verification
    NIST National Institute of Standards and Technology AES-GCM, PBKDF2 with 65,536 iterations

    Encryption Algorithm¶

    AES-256-GCM (Galois/Counter Mode)¶

    HMS-Mirror uses AES-256-GCM for all encryption operations. This authenticated encryption mode provides:

    • Confidentiality - Data is encrypted and unreadable without the key
    • Integrity - Any tampering with ciphertext is detected via authentication tag
    • Authenticity - Verification that data was encrypted with the expected key

    Algorithm Parameters¶

    Parameter Value Purpose
    Algorithm AES/GCM/NoPadding Authenticated encryption
    Key Size 256 bits Maximum AES strength
    GCM Tag Length 128 bits Authentication/integrity verification
    IV (Nonce) 96 bits (12 bytes) NIST-recommended for GCM
    PBKDF2 Iterations 65,536 Key derivation from passwords
    Salt Length 32 bytes Unique per encryption operation

    Why AES-GCM?¶

    AES-GCM is the recommended mode for authenticated encryption because:

    1. Single-pass encryption - Encryption and authentication in one operation
    2. Parallelizable - High performance on modern hardware
    3. NIST approved - Recommended by NIST SP 800-38D
    4. Wide adoption - Used by TLS 1.3, IPsec, and SSH

    Encrypted Data Types¶

    HMS-Mirror encrypts the following sensitive data types:

    1. Kerberos Keytabs¶

    Service: KeytabStorageService

    Kerberos keytab files contain cryptographic keys for service authentication. These are binary files that must be protected with the highest security.

    Aspect Implementation
    Storage RocksDB with dedicated column family
    Encryption AES-256-GCM with master key
    Key Management Auto-generated master key stored in preferences
    Integrity SHA-256 hash stored with metadata
    At Rest Always encrypted in RocksDB
    In Transit Decrypted only when needed for authentication

    Storage Format: 

    RocksDB Key: data:{keytabId}
RocksDB Value: IV (12 bytes) || Ciphertext || GCM Tag (16 bytes)

    2. Connection Passwords¶

    Service: ConnectionPasswordService

    Database connection passwords for HiveServer2, Metastore Direct, and other JDBC connections.

    Aspect Implementation
    Storage Encrypted in ConnectionDto before persistence
    Marker Encrypted passwords prefixed with ENC:
    Encryption AES-256-GCM with dedicated master key
    Decryption On-demand when establishing connections

    Storage Format: 

    ENC:{Base64(IV || Ciphertext || Tag)}

    3. Kerberos Credential Passwords¶

    Service: KerberosCredentialPasswordService

    Passwords for password-based Kerberos authentication (alternative to keytabs).

    Aspect Implementation
    Storage Encrypted in KerberosCredential objects
    Encryption AES-256-GCM with dedicated master key
    Use Case Password-based Kerberos (non-keytab) authentication

    4. Trino Connection Passwords¶

    Service: TrinoConnectionPasswordService

    Credentials for Trino/Presto SQL connections used for metadata discovery.

    Aspect Implementation
    Storage Encrypted in TrinoConnectionDto
    Encryption AES-256-GCM with dedicated master key

    5. HCFS Configuration Files¶

    Service: HcfsConfigFileStorageService

    Hadoop configuration files (core-site.xml, hdfs-site.xml, etc.) that may contain sensitive properties.

    Aspect Implementation
    Storage Encrypted ZIP archives in RocksDB
    Allowed Files core-site.xml, hdfs-site.xml, yarn-site.xml, hive-site.xml, etc.
    Encryption AES-256-GCM with dedicated master key
    Extraction To temp directory with restrictive permissions

    6. Kerberos Preferences¶

    Service: KerberosPreferencesService

    Legacy global Kerberos configuration including keytab paths and passwords.

    Aspect Implementation
    Storage Encrypted in RocksDB preferences
    Encryption AES-256-GCM with dedicated master key

    Key Management Architecture¶

    Key Separation (v4.2+)¶

    HMS-Mirror implements secure key separation by storing encryption keys in a PKCS12 keystore, completely separate from the encrypted data stored in RocksDB. This architecture complies with:

    • PCI-DSS 3.5.2: Cryptographic keys must be stored separately from encrypted data
    • NIST SP 800-57: Key management best practices
    • Defense in Depth: Compromising the database does not expose encryption keys

    Keystore Architecture¶

    +-----------------------------------------------------------------------+
|                    PKCS12 Keystore (hms-mirror-keys.p12)              |
|                    Location: ~/.hms-mirror/keystore/                   |
|                    Password: External (env var or system property)     |
+-----------------------------------------------------------------------+
|  connections_key           -> AES-256 Master Key                      |
|  keytabs_key               -> AES-256 Master Key                      |
|  kerberos_credentials_key  -> AES-256 Master Key                      |
|  trino_connections_key     -> AES-256 Master Key                      |
|  hcfs_config_files_key     -> AES-256 Master Key                      |
|  kerberos_preferences_key  -> AES-256 Master Key                      |
+-----------------------------------------------------------------------+
                                    |
                          Password Required to Access
                                    |
                +---------------------------------------------+
                |         Password Source Options              |
                |  1. HMS_MIRROR_KEYSTORE_PASSWORD (env)       |
                |  2. -Dhms-mirror.keystore.password            |
                |  3. Default dev password (application.yaml)  |
                +---------------------------------------------+

    Key Generation Process¶

    Keys are:

    • Generated using java.security.SecureRandom
    • 256 bits in length
    • Stored in PKCS12 keystore (password-protected)
    • Generated once per domain and reused

    Providing the Keystore Password¶

    For production deployments, you must provide the keystore password externally:

    Option 1: Environment Variable (Recommended) 

    export HMS_MIRROR_KEYSTORE_PASSWORD="your-secure-password-here"
java -jar hmsm-server-*.jar

    Option 2: System Property 

    java -Dhms-mirror.keystore.password="your-secure-password-here" -jar hmsm-server-*.jar

    Option 3: Default Development Password If no environment variable or system property is set, HMS-Mirror uses the default development password defined in application.yaml (hms-mirror-dev-keystore-password). This must be overridden for production deployments since it is a well-known value.

    Migration from Previous Versions¶

    HMS-Mirror v4.2+ automatically migrates encryption keys from RocksDB to the keystore on first startup. After migration:

    • Keys are removed from RocksDB
    • All future encryption operations use the keystore
    • Existing encrypted data remains unchanged (same keys, different storage location)

    Keystore File Permissions¶

    The keystore directory and file are created with restrictive POSIX permissions to prevent unauthorized access:

    Path Permission Octal Purpose
    ~/.hms-mirror/keystore/ rwx------ 700 Directory requires execute for file access
    hms-mirror-keys.p12 r-------- 400 Read-only prevents tampering with keys

    Why these permissions?

    • Directory (700): Owner needs read/write/execute. Execute permission on directories allows accessing files inside. Without it, even the owner cannot create or read files.
    • Keystore file (400): Read-only after creation prevents accidental modification or tampering. The file is temporarily made writable (600) only during key rotation operations.

    Troubleshooting Permission Issues

    If you encounter AccessDeniedException when starting HMS-Mirror:

    # Fix directory permissions
chmod 700 ~/.hms-mirror/keystore

# If keystore file is corrupted or has wrong permissions, remove it
# (keys will be regenerated, but existing encrypted data will be unrecoverable)
rm ~/.hms-mirror/keystore/hms-mirror-keys.p12

    Warning

    Deleting the keystore file means losing all encryption keys. Any data encrypted with those keys (passwords, keytabs, config files) will become permanently unrecoverable.

    Password-Based Key Derivation¶

    For password-encrypted data, keys are derived using PBKDF2 with:

    • Algorithm: PBKDF2WithHmacSHA256
    • Salt: 32 random bytes (unique per encryption operation)
    • Iterations: 65,536 (NIST-recommended)
    • Output key size: 256 bits

    Temporary File Security¶

    When keytabs or configuration files are extracted for use, strict security controls are applied:

    File Permissions¶

    Permission Value Description
    Owner Read Yes Only the process owner can read
    Owner Write No Immutable after creation
    Group None No group access
    Other None No world access

    Temporary File Lifecycle¶

    1. Decrypt — Data is decrypted from RocksDB into memory
    2. Write — Written to temp file with POSIX permissions 0400 (owner read-only)
    3. Use — File used for authentication (e.g., Kerberos kinit)
    4. Cleanup — File deleted when session closes or application stops

    Cleanup Mechanisms¶

    1. Session-based cleanup - Temp files deleted when session closes
    2. Service shutdown - All temp files cleaned on application stop
    3. Explicit cleanup - cleanupTempFiles() method available
    4. Tracking - All temp file paths tracked for guaranteed cleanup

    Data Integrity Verification¶

    SHA-256 Content Hash¶

    Keytabs and configuration files are hashed on upload. This enables:

    • Detection of storage corruption
    • Verification of data authenticity
    • Duplicate detection

    GCM Authentication Tag¶

    The AES-GCM mode includes a 128-bit authentication tag that:

    • Verifies ciphertext integrity
    • Detects any bit flips or tampering
    • Automatically validated during decryption
    • Throws exception if verification fails

    Encryption Marker System¶

    Encrypted passwords use a marker prefix to distinguish them from plaintext:

    Original:    MySecretPassword123
Encrypted:   ENC:dGhpcyBpcyBhIGJhc2U2NCBlbmNvZGVkIGVuY3J5cHRlZCB2YWx1ZQ==

    This enables:

    • Safe re-encryption detection (don't double-encrypt)
    • Clear identification of encrypted values in logs/debugging
    • Backward compatibility with unencrypted legacy data

    Security Best Practices¶

    What HMS-Mirror Does¶

    1. Never logs sensitive data - Passwords and keys are never written to logs
    2. Memory cleanup - Sensitive data cleared after use where possible
    3. Minimal exposure - Decryption only when needed, not preemptively
    4. Separate keys - Each data type has its own master key
    5. Random IVs - Fresh IV for every encryption operation
    6. Strong RNG - Uses java.security.SecureRandom

    Recommendations for Operators¶

    1. Secure RocksDB directory - Restrict filesystem permissions to application user
    2. Regular backups - Back up RocksDB directory (data remains encrypted)
    3. Key rotation - Consider periodic key rotation for long-lived deployments
    4. Network security - Use TLS for all network connections
    5. Audit logging - Enable audit logs to track sensitive operations
