    JVM Settings¶

    This guide covers configuring Java Virtual Machine (JVM) settings for HMS-Mirror through the web UI.

    Overview¶

    JVM settings control how HMS-Mirror uses system resources including memory allocation, garbage collection, and logging. These settings are managed through Utilities > Preferences > JVM in the web UI.

    Accessing JVM Settings¶

    1. Navigate to Utilities > Preferences in the web UI
    2. Select the JVM tab
    3. Configure settings as needed
    4. Click Save Settings
    5. Restart HMS-Mirror for changes to take effect

    Important

    JVM settings are applied at application startup. After saving changes, you must restart HMS-Mirror for them to take effect.

    Settings File Location¶

    JVM settings are persisted to a properties file displayed at the top of the JVM tab. The default location is:

    ~/.hms-mirror/settings/jvm.properties

    This file is read by the startup script when launching HMS-Mirror.

    Memory Settings¶

    Initial Heap Size (-Xms)¶

    The initial amount of memory allocated to the JVM when HMS-Mirror starts.

    Setting Description Example
    Format Use 'm' for MB or 'g' for GB 4096m or 4g
    Recommended 25-50% of max heap 4096m

    Maximum Heap Size (-Xmx)¶

    The maximum amount of memory the JVM can use.

    Setting Description Example
    Format Use 'm' for MB or 'g' for GB 8192m or 8g
    Minimum At least 4GB for production 4g
    Recommended 50-75% of available system RAM 8g

    Guidelines: - Small migrations (< 100 tables): 4GB minimum - Medium migrations (100-500 tables): 8GB recommended - Large migrations (500+ tables): 12-16GB or more

    Note on Embedded LLM

    When using the Embedded Ollama LLM provider, the LLM runs as a separate native process outside the JVM. LLM memory requirements are managed by the operating system, not the JVM heap. See LLM Integration for LLM memory planning.

    Garbage Collector¶

    Select the garbage collection algorithm based on your workload:

    G1 Garbage Collector (Recommended)¶

    Best for: Most HMS-Mirror workloads, heaps > 4GB

    -XX:+UseG1GC
    • Balanced throughput and latency
    • Automatic region sizing
    • Good default choice

    Z Garbage Collector (ZGC)¶

    Best for: Large heaps with low latency requirements (Java 15+)

    -XX:+UseZGC
    • Sub-millisecond pause times
    • Handles very large heaps (multi-terabyte)
    • Requires more CPU overhead

    Parallel Garbage Collector¶

    Best for: Batch processing, throughput-oriented

    -XX:+UseParallelGC
    • Maximum throughput
    • Longer pause times acceptable
    • Good for non-interactive workloads

    Serial Garbage Collector¶

    Best for: Small heaps, testing, single-core systems

    -XX:+UseSerialGC
    • Single-threaded
    • Minimal overhead
    • Not recommended for production

    Network Settings¶

    Prefer IPv4 Stack¶

    Enable this option if you experience network connectivity issues with IPv6.

    -Djava.net.preferIPv4Stack=true

    When to enable: - Network timeouts connecting to clusters - DNS resolution issues - Mixed IPv4/IPv6 environments - Legacy cluster configurations

    Logging Settings¶

    Logging Level¶

    Controls the verbosity of application logs.

    Level Description
    TRACE Most detailed, includes all events
    DEBUG Detailed debugging information
    INFO Normal operational messages (default)
    WARN Warning conditions
    ERROR Error conditions only

    Recommendations: - Production: INFO or WARN - Troubleshooting: DEBUG - Deep debugging: TRACE

    Log File Prefix¶

    Sets the prefix for log file names. Default: hms-mirror-service

    Log files are created at: 

    ~/.hms-mirror/logs/{prefix}.log

    Additional JVM Options¶

    Enter any additional JVM options as a space-separated list. Use with caution.

    Common Options¶

    Heap Dump on Out of Memory: 

    -XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/tmp

    GC Logging (Java 11+): 

    -Xlog:gc*:file=/var/log/hms-mirror/gc.log:time,uptime,level,tags

    Remote Debugging: 

    -agentlib:jdwp=transport=dt_socket,server=y,suspend=n,address=5005

    JMX Monitoring: 

    -Dcom.sun.management.jmxremote -Dcom.sun.management.jmxremote.port=9010 -Dcom.sun.management.jmxremote.authenticate=false -Dcom.sun.management.jmxremote.ssl=false

    Configuration Examples¶

    Development Environment¶

    initialHeapSize=2g
maxHeapSize=4g
garbageCollector=G1GC
loggingThreshold=DEBUG

    Production - Standard¶

    initialHeapSize=4g
maxHeapSize=8g
garbageCollector=G1GC
loggingThreshold=INFO
preferIPv4Stack=true

    Production - Large Scale¶

    initialHeapSize=8g
maxHeapSize=16g
garbageCollector=G1GC
loggingThreshold=INFO
preferIPv4Stack=true
additionalOptions=-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/var/log/hms-mirror

    Action Buttons¶

    Save Settings¶

    Saves the current configuration to the properties file. Requires restart to take effect.

    Reset to Defaults¶

    Resets all JVM settings to their default values. Useful if you've made problematic changes.

    Cancel¶

    Discards unsaved changes and reverts to the last saved configuration.

    Troubleshooting¶

    Out of Memory Errors¶

    Symptoms: 

    java.lang.OutOfMemoryError: Java heap space

    Solutions: 1. Increase maxHeapSize 2. Enable heap dump for analysis 3. Check for memory leaks in job configuration

    Slow Startup¶

    Causes: - Initial heap too large for system - GC initialization overhead

    Solutions: 1. Reduce initialHeapSize 2. Ensure sufficient system RAM 3. Consider ZGC for large heaps

    GC Pauses¶

    Symptoms: - Application freezes periodically - Slow response times

    Solutions: 1. Switch to G1GC or ZGC 2. Increase heap size 3. Enable GC logging to analyze pause patterns

    Network Timeouts¶

    Symptoms: - Connection timeouts to clusters - DNS resolution failures

    Solutions: 1. Enable preferIPv4Stack 2. Check network configuration 3. Verify cluster hostnames resolve correctly

    Best Practices¶

    1. Set initial and max heap to same value in production to avoid resize overhead
    2. Use G1GC as the default garbage collector
    3. Enable heap dumps in production for post-mortem analysis
    4. Start with INFO logging and only increase verbosity when needed
    5. Monitor memory usage during initial deployments
    6. Test configuration changes in a non-production environment first
