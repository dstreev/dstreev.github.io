    LLM Integration¶

    HMS-Mirror v4 includes optional AI-powered analysis using Large Language Models (LLMs). This guide covers setup, configuration, and usage of LLM features.

    Overview¶

    LLM integration provides: - Configuration Analysis - Review settings and get recommendations - Job Analysis - Understand what a job will do before running - Report Analysis - Get insights from migration results - Strategy Suggestions - Help choosing the right approach

    Accessing LLM Settings¶

    Navigate to Utilities > Preferences in the web UI and select the LLM tab.

    Supported Providers¶

    HMS-Mirror supports multiple LLM providers organized into three categories:

    Embedded Providers¶

    Managed by HMS-Mirror, no external setup required.

    Provider Description
    Embedded Ollama Bundled Ollama, runs on HMS-Mirror server

    Local Providers¶

    Connect to LLM servers running in your environment.

    Provider Description
    Local Ollama External Ollama instance
    LM Studio LM Studio local server
    LocalAI LocalAI server
    OpenAI Compatible Any OpenAI-compatible local endpoint

    Hosted Providers¶

    Cloud-based services requiring API keys.

    Provider Description
    OpenAI ChatGPT API
    Anthropic Claude API
    Google AI Gemini API
    Azure OpenAI Azure-hosted OpenAI
    AWS Bedrock AWS-hosted models
    Groq Fast inference API
    Together AI Open model hosting
    Fireworks Fast model inference
    Grok xAI's Grok models

    Quick Start¶

    Embedded Ollama (Simplest)¶

    No external setup required:

    1. Navigate to Utilities > Preferences > LLM
    2. Select the Embedded tab
    3. Click Download Ollama if not already installed
    4. Click Start Server to start the embedded Ollama
    5. Go to Available Models and click Download on your preferred model
    6. Click the star icon on an installed model to set it as active
    7. Enable LLM features in General Settings
    8. Click Save Settings

    Local Ollama¶

    Use an existing Ollama installation:

    1. Install Ollama: https://ollama.ai
    2. Pull a model: ollama pull llama3.1
    3. In HMS-Mirror, navigate to Utilities > Preferences > LLM
    4. Select the Local tab
    5. Choose Ollama (Local)
    6. Set endpoint: http://localhost:11434
    7. Click Test to verify connectivity
    8. Click the star icon on a model to set it as active
    9. Enable LLM features and save

    Cloud Provider (OpenAI Example)¶

    1. Get API key from OpenAI
    2. Navigate to Utilities > Preferences > LLM
    3. Select the Hosted tab
    4. Choose OpenAI
    5. Enter your API key
    6. Click Test to verify connectivity
    7. Select a model (e.g., GPT-4o)
    8. Enable LLM features and save

    Model Parameters¶

    Parameter Description Range Default
    Temperature Creativity level 0.0-1.0 0.7
    Max Tokens Response length 100-8192 4096

    Temperature Guidelines - 0.0-0.3: Focused, deterministic (good for analysis) - 0.4-0.7: Balanced (recommended) - 0.8-1.0: Creative, varied (may be inconsistent)

    Using LLM Features¶

    Configuration Analysis¶

    Get AI-powered review of your migration configuration.

    Access 1. Navigate to Configurations 2. Click ✨ (sparkles) on a configuration card 3. View analysis in dialog

    What It Analyzes - Configuration completeness - Potential issues - Best practices - Recommendations

    Example Output 

    Configuration Analysis: hdp-migration-config

Summary
───────
This configuration is set up for migrating from HDP to CDP with
ACID table downgrade enabled.

Observations
────────────
• ACID tables will be converted to external tables
• Sort dynamic partition inserts is disabled
• No warehouse directories configured

Recommendations
───────────────
1. Consider enabling sortDynamicPartitionInserts for better
   performance with partitioned tables.

2. Set explicit warehouse directories to control where
   migrated tables are stored.

3. Review the partition limit (currently 0) - unlimited
   partitions may cause memory issues.

Risk Assessment
───────────────
• Low risk: Configuration is valid
• Medium risk: Missing warehouse paths may use defaults

    Job Analysis¶

    Understand what a job will do before running.

    Access 1. Navigate to Jobs → Build 2. Click ✨ (sparkles) on a job card 3. View analysis in dialog

    What It Analyzes - Job configuration summary - Expected operations - Potential risks - Prerequisites

    Example Output 

    Job Analysis: hdp-to-cdp-migration

Overview
────────
This job will migrate 45 tables from the sales_db database
using the HYBRID strategy.

Expected Operations
──────────────────
• 35 tables will use SQL strategy (< 1GB)
• 8 tables will use EXPORT_IMPORT (> 1GB)
• 2 ACID tables will be downgraded to external

Timing Estimate
───────────────
Based on table sizes and strategy selection:
• Estimated duration: 30-45 minutes
• Parallel operations: Up to 8 tables

Potential Issues
────────────────
1. Table 'large_transactions' is 50GB - consider running
   during off-peak hours.

2. 2 ACID tables will lose transactional properties.

Recommendations
───────────────
1. Run a dry run first to verify DDL generation.
2. Ensure target warehouse has 100GB+ free space.
3. Consider scheduling during maintenance window.

    Report Analysis¶

    Get insights from completed migration reports.

    Access 1. Navigate to Reports 2. Open a report 3. Click ✨ AI Analysis

    What It Analyzes - Success/failure patterns - Error categorization - Root cause analysis - Remediation suggestions

    Example Output 

    Report Analysis: hdp-to-cdp-migration

Executive Summary
─────────────────
Migration completed with 95% success rate. 3 tables failed
due to partition limits, 2 were skipped per configuration.

Success Analysis
────────────────
• 42 of 45 tables migrated successfully
• Average migration time: 45 seconds per table
• SQL strategy performed well for small tables

Failure Analysis
────────────────
Failed Tables:
1. order_items: Exceeded partition limit (1500 > 500)
2. event_log: Query timeout after 1 hour
3. audit_trail: Permission denied on target

Root Causes:
• Partition limit too conservative for some tables
• Large table without EXPORT_IMPORT strategy
• Missing permissions on target cluster

Recommendations
───────────────
1. Increase partitionLimit to 2000 and re-run failed tables
2. Switch event_log to EXPORT_IMPORT strategy
3. Grant CREATE TABLE permission for audit_trail

Next Steps
──────────
1. Address the 3 failed tables with suggested changes
2. Verify data in successfully migrated tables
3. Update applications to point to new locations

    General Settings¶

    The General Settings section at the top of the LLM tab controls global LLM behavior:

    Setting Description Default
    Enable LLM Features Turns AI analysis on/off throughout the application Off
    Max Tokens Maximum length of LLM responses 2048
    Temperature Controls response creativity (0.0-2.0) 0.7

    Temperature Guidelines - 0.0-0.3: Focused, deterministic (good for analysis) - 0.4-0.7: Balanced (recommended) - 0.8-1.0: Creative, varied (may be inconsistent)

    Managing Models¶

    Embedded Ollama Models¶

    Download Ollama Binary¶

    If using the Embedded provider for the first time: 1. Go to Utilities > Preferences > LLM 2. Select the Embedded tab 3. Click Download Ollama to download the Ollama binary 4. Wait for the download to complete

    Start/Stop the Server¶

    • Click Start Server to start the embedded Ollama server
    • Click Stop Server to stop it
    • The server status is displayed showing whether it's running

    Download a Model¶

    1. Go to Utilities > Preferences > LLM > Embedded
    2. Click Available Models tab
    3. Find the model you want and click Download
    4. Wait for download (progress bar shows status)
    5. The model will appear in Installed Models when complete

    Set Active Model¶

    Click the star icon (☆) on any installed model to set it as the active model for LLM features.

    Available Models¶

    Model Size Best For
    llama3.1 4.7GB General purpose
    mistral 4.1GB Fast, good quality
    codellama 3.8GB Code-focused
    phi 1.6GB Lightweight

    Delete a Model¶

    1. Go to Settings → LLM
    2. Click Manage Models
    3. Select model to remove
    4. Click Delete

    Model Selection Tips¶

    Use Case Recommended
    Quick analysis mistral, phi
    Detailed analysis llama3.1, gpt-4
    Code-heavy configs codellama
    Production accuracy gpt-4, claude-3-opus

    Prompt Customization¶

    Built-in Prompts¶

    HMS-Mirror includes optimized prompts for: - Configuration analysis - Job analysis - Report analysis

    Prompts are stored in:

└── report/
    └── detail/
        └── reportAnalysis.prompt

    Performance Considerations¶

    Local Models (Ollama)¶

    System Requirements - CPU: Modern multi-core (ARM or x86) - RAM: 8GB minimum, 16GB recommended - Storage: 5-10GB per model

    Performance Tips - Use smaller models for quick analysis - Larger models give better results but slower - GPU acceleration improves speed significantly

    Cloud Providers¶

    Considerations - Network latency affects response time - API costs per token - Rate limits may apply

    Cost Management - Use smaller models for routine analysis - Reserve larger models for complex cases - Monitor token usage

    Troubleshooting¶

    Model Won't Load¶

    Error: Failed to load model

    Solutions - Verify model is pulled - Check available memory - Restart Ollama service

    API Connection Failed¶

    Error: Cannot connect to LLM provider

    Solutions - Verify API key is correct - Check network connectivity - Confirm provider status

    Slow Responses¶

    Causes - Large model on limited hardware - Network latency (cloud) - Complex analysis request

    Solutions - Use smaller model - Reduce max tokens - Check network connection

    Empty or Poor Results¶

    Causes - Temperature too high - Max tokens too low - Model limitations

    Solutions - Lower temperature (0.3-0.5) - Increase max tokens - Try different model

    Privacy and Security¶

    Local Models (Recommended for Sensitive Data)¶

    • All processing on HMS-Mirror server
    • No data sent externally
    • Full privacy control

    Cloud Providers¶

    • Data sent to provider API
    • Review provider's data policies
    • Consider for non-sensitive configs

    Best Practices¶

    1. Use local models for sensitive configurations
    2. Review provider policies before using cloud
    3. Don't include secrets in analyzed configs
    4. Consider compliance requirements

    Disabling LLM Features¶

    If you don't want to use LLM features:

    1. Go to Settings → LLM
    2. Uncheck Enable LLM Features
    3. Save settings

    The ✨ icons will be hidden throughout the UI.
