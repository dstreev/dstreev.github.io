    Certificate Management¶

    This guide covers managing SSL/TLS certificates for secure connections to DataLake environments and other services from HMS-Mirror.

    Note

    This is where you register certificates for systems that use self-signed certificates for TLS. When connecting to clusters or services with self-signed SSL certificates, you must import those certificates here for HMS-Mirror to trust them.

    Overview¶

    HMS-Mirror uses a centralized truststore to manage SSL certificates for all outbound connections. This includes connections to: - HiveServer2 (JDBC over SSL) - Metastore databases (MySQL, PostgreSQL over SSL) - Knox Gateway - Other HTTPS services

    When to Use Certificate Management¶

    You need to import certificates when: - Connecting to clusters with self-signed SSL certificates - Using internal Certificate Authorities (CAs) not in the Java default truststore - Connecting through Knox Gateway with custom certificates - SSL connections fail with certificate validation errors

    Accessing Certificate Management¶

    Navigate to Utilities > Preferences in the web UI and select the Certificates tab.

    Application Truststore¶

    HMS-Mirror maintains a central truststore for all SSL connections. The truststore information is displayed at the top of the Certificates tab:

    Field Description
    Path Location of the truststore file
    Certificates Number of certificates currently installed

    Default truststore location: 

    ~/.hms-mirror/security/truststore.jks

    All SSL connections in HMS-Mirror automatically use certificates from this truststore.

    Importing Certificates¶

    Method 1: Upload Individual Certificate¶

    For single certificate files (.crt, .cer, .pem format):

    1. Navigate to Utilities > Preferences > Certificates
    2. In the Upload Certificate section:
    3. Click Choose File and select your certificate file
    4. Enter a meaningful Certificate Alias (e.g., cdp-cluster-ca)
    5. Click Import Certificate
    6. The certificate appears in the Installed Certificates table

    Supported Formats: - .crt - X.509 certificate - .cer - X.509 certificate (alternative extension) - .pem - PEM-encoded certificate

    Method 2: Import from Existing Truststore¶

    To import certificates from an existing JKS truststore:

    1. Navigate to Utilities > Preferences > Certificates
    2. In the Import from Truststore section:
    3. Click Choose File and select the .jks truststore file
    4. Enter the Truststore Password
    5. Click List Certificates
    6. A dialog shows all certificates in the truststore
    7. Select the certificates you want to import (or click Select All)
    8. Click Import Selected

    This is useful when: - Migrating from another tool with an existing truststore - Your cluster administrator provides a truststore with multiple certificates - You have a corporate CA bundle in JKS format

    Managing Installed Certificates¶

    Viewing Certificates¶

    The Installed Certificates table shows:

    Column Description
    Alias Unique identifier for the certificate
    Type Certificate type (usually X.509)
    File Original filename (if available)
    Size Certificate file size
    Actions Delete button

    Deleting Certificates¶

    To remove a certificate:

    1. Find the certificate in the Installed Certificates table
    2. Click the Delete button
    3. Confirm the deletion

    Note

    Removing a certificate will cause SSL connections that rely on it to fail.

    Obtaining Certificates¶

    From HiveServer2 / DataLake Environment¶

    Use OpenSSL to retrieve the certificate:

    # Replace hostname and port with your HiveServer2 details
openssl s_client -connect hs2.example.com:10001 -showcerts </dev/null 2>/dev/null | \
  openssl x509 -outform PEM > hs2-cert.pem

    From Knox Gateway¶

    openssl s_client -connect knox.example.com:8443 -showcerts </dev/null 2>/dev/null | \
  openssl x509 -outform PEM > knox-cert.pem

    From Database Server¶

    # MySQL
openssl s_client -connect mysql.example.com:3306 -starttls mysql -showcerts </dev/null 2>/dev/null | \
  openssl x509 -outform PEM > mysql-cert.pem

# PostgreSQL
openssl s_client -connect postgres.example.com:5432 -starttls postgres -showcerts </dev/null 2>/dev/null | \
  openssl x509 -outform PEM > postgres-cert.pem

    From Cluster Administrator¶

    Request the following from your cluster administrator: - The CA certificate used to sign cluster certificates - Any intermediate certificates in the chain - The certificate file in PEM or DER format

    Certificate Chain Handling¶

    If your cluster uses a certificate chain (server cert → intermediate CA → root CA), you may need to import: 1. The root CA certificate 2. Any intermediate CA certificates

    Import each certificate separately with descriptive aliases: - company-root-ca - company-intermediate-ca - cluster-server-cert

    Common Use Cases¶

    CDP Private Cloud¶

    CDP typically uses Knox Gateway with SSL. Import the Knox certificate:

    # Get Knox certificate
openssl s_client -connect knox.cdp.example.com:8443 </dev/null 2>/dev/null | \
  openssl x509 -outform PEM > cdp-knox.pem

    Then upload cdp-knox.pem with alias cdp-knox.

    HDP with SSL¶

    For HDP clusters with SSL enabled:

    # Get HiveServer2 certificate (binary transport)
openssl s_client -connect hs2.hdp.example.com:10000 </dev/null 2>/dev/null | \
  openssl x509 -outform PEM > hdp-hs2.pem

# Get HiveServer2 certificate (HTTP transport)
openssl s_client -connect hs2.hdp.example.com:10001 </dev/null 2>/dev/null | \
  openssl x509 -outform PEM > hdp-hs2-http.pem

    Self-Signed Certificates¶

    For development or test clusters with self-signed certificates:

    1. Obtain the self-signed certificate from the cluster
    2. Import it using the Upload Certificate feature
    3. Use a descriptive alias like dev-cluster-selfsigned

    Troubleshooting¶

    SSL Handshake Failure¶

    Error: 

    javax.net.ssl.SSLHandshakeException: sun.security.validator.ValidatorException:
PKIX path building failed

    Cause: The certificate or CA is not in the truststore.

    Solution: 1. Obtain the server's certificate using openssl 2. Import it via Certificates tab 3. Test the connection

    Certificate Expired¶

    Error: 

    javax.net.ssl.SSLHandshakeException: NotAfter: [date]

    Cause: The certificate has expired.

    Solution: 1. Obtain a new certificate from the server 2. Delete the old certificate 3. Import the new certificate

    Hostname Verification Failed¶

    Error: 

    javax.net.ssl.SSLPeerUnverifiedException: Hostname verification failed

    Cause: The hostname used doesn't match the certificate's CN or SANs.

    Solution: 1. Use the exact hostname in the certificate 2. Or add the hostname to DNS/hosts file 3. Some connections support disabling hostname verification (not recommended for production)

    Certificate Not Yet Valid¶

    Error: 

    javax.net.ssl.SSLHandshakeException: NotBefore: [date]

    Cause: The certificate's validity period hasn't started.

    Solution: 1. Verify system clock is correct 2. Wait until the certificate becomes valid 3. Or obtain a currently valid certificate

    Chain of Trust Issues¶

    Error: 

    unable to find valid certification path to requested target

    Cause: Missing intermediate or root CA certificates.

    Solution: 1. Import the entire certificate chain: - Root CA certificate - Intermediate CA certificates - Server certificate 2. Use aliases to identify each certificate in the chain

    Best Practices¶

    1. Use descriptive aliases that identify the source (e.g., prod-cdp-knox, dev-hdp-hs2)
    2. Document certificate purposes for team reference
    3. Track expiration dates and renew before expiration
    4. Import CA certificates rather than individual server certs when possible
    5. Test connections after importing certificates
    6. Backup the truststore before making changes
    7. Use PEM format for easier inspection and management
    8. Verify certificates before import using: 
      openssl x509 -in cert.pem -text -noout

    Certificate Information Commands¶

    View Certificate Details¶

    # View certificate information
openssl x509 -in certificate.pem -text -noout

# Check expiration date
openssl x509 -in certificate.pem -enddate -noout

# Verify certificate chain
openssl verify -CAfile ca-bundle.pem certificate.pem

    List Truststore Contents (JKS)¶

    keytool -list -v -keystore truststore.jks -storepass changeit

    Export Certificate from JKS¶

    keytool -exportcert -alias myalias -keystore truststore.jks -file cert.pem -rfc
