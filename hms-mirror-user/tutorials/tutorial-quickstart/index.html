    Quick Start Tutorial¶

    This tutorial walks you through your first HMS-Mirror migration in about 15 minutes. You'll perform a simple SCHEMA_ONLY migration to understand the workflow.

    Prerequisites¶

    Before starting, ensure you have:

    • HMS-Mirror installed and running (see Installation Guide)
    • Network access to at least one DataLake environment with HiveServer2
    • Valid credentials for the cluster (Kerberos ticket or username/password)

    What We'll Do¶

    1. Create a connection to a cluster
    2. Create a basic configuration
    3. Create a dataset with a few tables
    4. Build and run a job
    5. Review the results

    Step 1: Access the Web UI¶

    Open your browser and navigate to:

    http://localhost:8090/hms-mirror

    You'll see the HMS-Mirror dashboard with navigation options for Connections, Configurations, Datasets, and Jobs.

    Step 2: Create a Connection¶

    First, we need to tell HMS-Mirror how to connect to your cluster.

    2.1 Navigate to Connections¶

    Click Connections in the navigation menu.

    2.2 Create New Connection¶

    Click the New Connection button (or use the wizard).

    2.3 Fill in Basic Settings¶

    Enter the following information:

    Field Example Value Description
    Name my-cluster A friendly name for this connection
    Legacy No Check if this is HDP/CDH (pre-CDP)

    2.4 Configure HiveServer2¶

    In the HiveServer2 section:

    Field Example Value Description
    URI jdbc:hive2://your-host:10000/default Your HiveServer2 JDBC URL
    Driver Class org.apache.hive.jdbc.HiveDriver Usually auto-detected

    For Connection Properties, add any required settings: - transportMode → binary (or http for Knox) - ssl → true (if using SSL)

    2.5 Configure Authentication¶

    Select your authentication method:

    No Authentication (development clusters) - Leave default settings

    Kerberos (enterprise clusters) - Ensure you have a valid Kerberos ticket (klist to verify) - Add connection property: principal → hive/_HOST@REALM

    LDAP (username/password) - Add connection property: user → your-username - Add connection property: password → your-password

    2.6 Test the Connection¶

    Click Test Connection to verify everything works.

    Expected result: 

    ✓ Connection successful
  Connected to: Hive 3.1.3000
  Databases found: 15

    2.7 Save the Connection¶

    Click Save to store your connection.

    Step 3: Create a Configuration¶

    Configurations define how the migration behaves. For this tutorial, we'll use minimal settings.

    3.1 Navigate to Configurations¶

    Click Configurations in the navigation menu.

    3.2 Create New Configuration¶

    Click New Configuration to open the configuration wizard.

    3.3 Configuration Details¶

    On the Details page:

    Field Value
    Name quickstart-config
    Description Basic configuration for quick start tutorial

    Click Next.

    3.4 Migration Settings¶

    For this tutorial, accept the defaults: - Copy Views: No (leave unchecked) - Evaluate Partition Location: No

    Click Next.

    3.5 Optimizations¶

    Accept the defaults and click Next.

    3.6 Transfer Settings¶

    Accept the defaults and click Next.

    3.7 ACID Settings¶

    Accept the defaults and click Next.

    3.8 Conversions¶

    Accept the defaults (no Iceberg conversion).

    3.9 Save Configuration¶

    Click Save to create your configuration.

    Step 4: Create a Dataset¶

    Datasets define which databases and tables to migrate.

    4.1 Navigate to Datasets¶

    Click Datasets in the navigation menu.

    4.2 Create New Dataset¶

    Click New Dataset.

    4.3 Dataset Details¶

    Field Value
    Name quickstart-dataset
    Description Small dataset for tutorial

    4.4 Add a Database¶

    Click Add Database and enter:

    Field Value Description
    Database Name default Or any database you want to test
    Include Tables .* All tables (or a specific pattern)
    Exclude Tables (empty) Leave blank to include all

    Tip

    For your first test, choose a database with only a few small tables.

    4.5 Save Dataset¶

    Click Save to create your dataset.

    Step 5: Build a Job¶

    Now we'll combine everything into a job.

    5.1 Navigate to Jobs¶

    Click Jobs in the navigation menu, then click Build.

    5.2 Create New Job¶

    Click New Job or Build Job to start the job wizard.

    5.3 Job Details¶

    Field Value
    Name quickstart-job
    Description My first HMS-Mirror migration

    Click Next.

    5.4 Select Data Strategy¶

    Choose SCHEMA_ONLY for this tutorial.

    Why SCHEMA_ONLY?

    This strategy only creates table definitions on the target without moving data. It's safe, fast, and perfect for learning the workflow.

    Click Next.

    5.5 Select References¶

    Field Selection
    Dataset quickstart-dataset
    Configuration quickstart-config
    LEFT Connection my-cluster
    RIGHT Connection my-cluster (or another cluster if available)

    Note

    For SCHEMA_ONLY with the same cluster, you can use the same connection for both LEFT and RIGHT. The tables will be created in a different database (with a prefix or suffix based on configuration).

    Click Next.

    5.6 Options¶

    Accept the defaults for this tutorial.

    Click Next.

    5.7 Review Summary¶

    Review your job configuration:

    Job: quickstart-job
Strategy: SCHEMA_ONLY
Dataset: quickstart-dataset (1 database)
Configuration: quickstart-config
LEFT: my-cluster
RIGHT: my-cluster

    Step 6: Validate the Job¶

    Before running, validate the job to catch any issues.

    6.1 Click Validate¶

    On the job card or summary page, click Validate.

    6.2 Review Validation Results¶

    You should see:

    ✓ Job validation passed
  - Connections: Valid
  - Configuration: Valid
  - Dataset: Valid
  - Strategy compatibility: Valid

    If there are warnings or errors, address them before proceeding.

    Step 7: Run a Dry Run¶

    A dry run generates all the SQL scripts without actually executing them.

    7.1 Click Run (Dry Run)¶

    Click Run and ensure Dry Run is selected (or toggle it on).

    7.2 Monitor Progress¶

    Watch the progress indicator as HMS-Mirror: 1. Connects to clusters 2. Discovers tables 3. Analyzes each table 4. Generates DDL scripts

    7.3 Wait for Completion¶

    The job should complete in a few seconds to a couple of minutes, depending on the number of tables.

    Step 8: Review Results¶

    8.1 View the Report¶

    Once complete, click View Report on the job.

    8.2 Understand the Report¶

    The report shows:

    Summary Section 

    Total Tables: 5
Successful: 5
Failed: 0
Skipped: 0

    Table Details

    Each table shows: - Status (SUCCESS, FAILED, SKIPPED) - Strategy used - SQL statements generated - Any issues or warnings

    Generated SQL

    Expand a table to see the DDL:

    -- LEFT Cluster (Source)
USE default;
SHOW CREATE TABLE my_table;

-- RIGHT Cluster (Target)
USE default;
CREATE EXTERNAL TABLE IF NOT EXISTS my_table (
  id INT,
  name STRING,
  created_date DATE
)
STORED AS PARQUET
LOCATION 'hdfs://...'
TBLPROPERTIES (...);

    8.3 Download Scripts (Optional)¶

    Click Download to get the SQL scripts as files you can review or execute manually.

    Step 9: Execute (Optional)¶

    Once you're comfortable with what the dry run shows:

    9.1 Run with Execute¶

    Click Run again, but this time disable the Dry Run option.

    Warning

    This will actually create tables on your target cluster. Make sure you're ready.

    9.2 Monitor and Review¶

    The process is the same as dry run, but now the SQL is actually executed on the clusters.

    What You've Learned¶

    Congratulations! You've completed your first HMS-Mirror migration. You now know how to:

    • ✅ Create cluster connections
    • ✅ Configure migration settings
    • ✅ Define datasets with database/table selections
    • ✅ Build and validate jobs
    • ✅ Run dry runs to preview changes
    • ✅ Review migration reports

    Next Steps¶

    Now that you understand the basics:

    Try Different Strategies¶

    Experiment with other data strategies: - LINKED - Create views pointing to source data - SQL - Move data using INSERT statements - EXPORT_IMPORT - Use Hive's native export/import

    Learn More¶

    • Core Concepts - Understand LEFT/RIGHT, strategies, and components
    • Data Strategies - Deep dive into each migration strategy
    • Managing Connections - Advanced connection settings
    • Managing Configurations - Full configuration options

    Plan a Real Migration¶

    When you're ready for a real migration:

    1. Inventory your databases and tables
    2. Choose the appropriate strategy
    3. Test with a small subset first
    4. Run dry runs to validate
    5. Execute with monitoring

    Tip

    Use the AI Analysis feature (sparkles icon) on configurations and jobs to get recommendations and identify potential issues before running.
