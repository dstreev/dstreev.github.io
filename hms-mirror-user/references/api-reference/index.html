    API Reference¶

    HMS-Mirror provides a comprehensive REST API for programmatic access to all functionality. The API follows RESTful conventions and returns JSON responses.

    Interactive API Documentation (Swagger)¶

    HMS-Mirror includes built-in interactive API documentation powered by SpringDoc/OpenAPI 3.

    Resource URL
    Swagger UI http://<host>:8090/hms-mirror/swagger-ui.html
    OpenAPI JSON http://<host>:8090/hms-mirror/v3/api-docs

    The Swagger UI provides a complete, interactive reference for every endpoint, including request/response schemas, parameter details, and the ability to execute API calls directly from the browser. It is the authoritative source for request body schemas and response types.

    Tip

    Use the Swagger UI as your primary API exploration tool. This document provides an overview and quick reference, while Swagger provides full schema details, example payloads, and a "Try it out" feature for live testing.

    Base URL¶

    http://localhost:8090/hms-mirror/api/v1

    The context path /hms-mirror is configurable via server.servlet.context-path in application.yaml.

    Authentication¶

    HMS-Mirror uses JWT (JSON Web Token) authentication. All API endpoints except the public auth and health endpoints require a valid access token.

    Step 1 — Login¶

    POST /api/v1/auth/login
Content-Type: application/json

{
  "usernameOrEmail": "admin",
  "password": "your-password"
}

    Note

    The field is usernameOrEmail — you can provide either the account username or its registered email address.

    Response:

    {
  "accessToken": "eyJhbGciOiJIUzI1NiJ9...",
  "refreshToken": "eyJhbGciOiJIUzI1NiJ9...",
  "tokenType": "Bearer",
  "expiresIn": 900000,
  "user": {
    "id": "...",
    "username": "admin",
    "email": "admin@example.com",
    "roles": ["USER", "ADMIN"]
  }
}

    Step 2 — Use the access token (standard REST calls)¶

    Include the accessToken from the login response in the Authorization header for all subsequent requests:

    Authorization: Bearer <accessToken>

    curl -H "Authorization: Bearer $TOKEN" \
  http://localhost:8090/hms-mirror/api/v1/connections

    Step 3 — Use the access token for SSE streams¶

    Two LLM endpoints stream progress via Server-Sent Events (text/event-stream):

    • GET /api/v1/preferences/llm/embedded/download — Ollama binary download progress
    • GET /api/v1/preferences/llm/models/pull — Model pull progress

    The browser's native EventSource API cannot set custom HTTP headers, so pass the access token as a ?token= query parameter instead:

    GET /api/v1/preferences/llm/models/pull?model=llama3&token=<accessToken>

    # cURL equivalent (for testing)
curl -N -H "Accept: text/event-stream" \
  "http://localhost:8090/hms-mirror/api/v1/preferences/llm/models/pull?model=llama3&token=$TOKEN"

    Note

    Access tokens are short-lived (15 minutes), which limits the exposure window if a ?token= URL appears in server logs. Never use refresh tokens in query parameters.

    Step 4 — Refresh the access token¶

    Access tokens expire after 15 minutes. Use the refresh token to get a new access token without re-logging in:

    POST /api/v1/auth/refresh
Content-Type: application/json

{
  "refreshToken": "<refreshToken>"
}

    No Authorization header is required on this call. The response returns a new accessToken (and optionally a new refreshToken).

    Refresh tokens are valid for 7 days. After expiry the user must log in again.

    Token Lifecycle¶

    Token Lifetime Purpose
    accessToken 15 minutes Authenticate all API calls and SSE streams
    refreshToken 7 days Obtain a new accessToken without re-logging in

    Role-Based Access¶

    Endpoints are protected by role-based access control:

    Role Access
    USER Connections, jobs, datasets, configurations, reports, self-service profile
    ADMIN All USER access plus user management, preferences, master password, system configuration

    Admin-only endpoints require ROLE_ADMIN and return 403 Forbidden for standard users.

    Response Format¶

    Successful responses typically use this envelope:

    {
  "status": "SUCCESS",
  "data": { ... },
  "message": "Operation completed"
}

    Error responses:

    {
  "status": "ERROR",
  "message": "Error description"
}

    GET /compaction-status Check compaction status

    Conditional Availability¶

    Many API controllers are conditionally enabled based on feature flags and configuration properties:

    Condition Controllers Affected
    hms-mirror.rocksdb.enabled=true Connections, Configurations, Datasets, Jobs, Runtime Jobs, Runtime Reports, Migration Plans, HCFS Profiles, CM Connector, Data Browser, Discovery, Kerberos Credentials, Trino Connections, Preferences, Application Preferences, LLM, Report Import, CLI Import

    Example Usage¶

    cURL Examples¶

    Authenticate: 

    # Login and capture the access token
TOKEN=$(curl -s -X POST http://localhost:8090/hms-mirror/api/v1/auth/login \
  -H "Content-Type: application/json" \
  -d '{"usernameOrEmail":"admin","password":"your-password"}' | jq -r '.accessToken')

    List connections: 

    curl -H "Authorization: Bearer $TOKEN" \
  http://localhost:8090/hms-mirror/api/v1/connections

    Create a job: 

    curl -X POST -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  http://localhost:8090/hms-mirror/api/v1/jobs/my-job \
  -d '{
    "name": "my-job",
    "dataStrategy": "SCHEMA_ONLY",
    "dataset": "my-dataset",
    "configuration": "my-config",
    "leftConnection": "source-cluster",
    "rightConnection": "target-cluster"
  }'

    Start a job: 

    curl -X POST -H "Authorization: Bearer $TOKEN" \
  http://localhost:8090/hms-mirror/api/v1/runtime/jobs/start \
  -H "Content-Type: application/json" \
  -d '{"jobKey": "my-job", "dryRun": true}'

    Check system status: 

    curl -H "Authorization: Bearer $TOKEN" \
  http://localhost:8090/hms-mirror/api/v1/system/status
