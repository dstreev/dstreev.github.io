    CLI & Server Reference¶

    Legacy CLI Retired¶

    The standalone command-line interface from HMS-Mirror v3 has been retired in v4. The legacy CLI required complex YAML configuration files with deeply nested connection, migration, and optimization settings that were difficult to author and validate. Security credentials were stored in plain text configuration files, and the text-based reports lacked the interactive exploration capabilities needed for large-scale migrations.

    HMS-Mirror v4 replaces the CLI with a Web UI and a comprehensive REST API:

    • Web UI — The recommended way to interact with HMS-Mirror. Provides visual connection testing, guided job configuration, interactive report exploration, and integrated documentation. All configuration is stored securely in RocksDB with encrypted credentials.
    • REST API — Full programmatic access to all HMS-Mirror functionality. Ideal for automation, CI/CD pipelines, and scripted migration workflows. See the API Reference for the complete endpoint catalog, or explore interactively via the Swagger UI.

    Recommended Workflow

    Use the Web UI to create and configure connections, datasets, configurations, and jobs — taking advantage of visual validation, connection testing, and guided wizards. Then use the REST API to automate job execution, integrate with scheduling systems, or build custom migration pipelines. View reports through the Web UI for the best experience, though report details are also available via the API.

    REST API Workflow Examples¶

    The examples below demonstrate a complete migration workflow using the REST API — from creating connections and datasets to running jobs and retrieving reports. This approach is ideal for automation and repeatable migration pipelines.

    Prerequisites¶

    All API calls require a JWT token. Obtain one first:

    # Authenticate and extract the access token
TOKEN=$(curl -s -X POST http://localhost:8090/hms-mirror/api/v1/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"admin","password":"your-password"}' | jq -r '.data.accessToken')

# Verify the token works
curl -s -H "Authorization: Bearer $TOKEN" \
  http://localhost:8090/hms-mirror/api/v1/auth/me | jq .

    Step 1: Create Connections¶

    Connections define how HMS-Mirror connects to your Hive clusters. We recommend creating and testing connections through the Web UI first, since it provides immediate visual feedback on connectivity issues, driver availability, and Kerberos configuration.

    If you prefer the API:

    # Create a source (left) connection
curl -s -X POST http://localhost:8090/hms-mirror/api/v1/connections \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "hdp3-source",
    "description": "HDP 3.1 Source Cluster",
    "legacyHive": false,
    "hiveServer2": {
      "uri": "jdbc:hive2://hs2-source.example.com:10000/default",
      "connectionProperties": {
        "transportMode": "binary"
      }
    }
  }' | jq .

# Create a target (right) connection
curl -s -X POST http://localhost:8090/hms-mirror/api/v1/connections \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "cdp-target",
    "description": "CDP 7.1 Target Cluster",
    "legacyHive": false,
    "hiveServer2": {
      "uri": "jdbc:hive2://hs2-target.example.com:10000/default",
      "connectionProperties": {
        "transportMode": "binary"
      }
    }
  }' | jq .

# Test the connections
curl -s -X POST http://localhost:8090/hms-mirror/api/v1/connections/hdp3-source/test \
  -H "Authorization: Bearer $TOKEN" | jq .

curl -s -X POST http://localhost:8090/hms-mirror/api/v1/connections/cdp-target/test \
  -H "Authorization: Bearer $TOKEN" | jq .

    Step 2: Create a Dataset¶

    Datasets define which databases and tables to include in a migration. Each dataset specifies databases with optional include/exclude table patterns.

    # Create a dataset targeting the sales and inventory databases
curl -s -X POST http://localhost:8090/hms-mirror/api/v1/datasets \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "sales-migration",
    "description": "Sales and inventory databases for Q1 migration",
    "databases": [
      {
        "name": "sales_db",
        "tableIncludeFilterPattern": ".*",
        "tableExcludeFilterPattern": "tmp_.*"
      },
      {
        "name": "inventory_db",
        "tableIncludeFilterPattern": ".*"
      }
    ]
  }' | jq .

    Step 3: Create a Configuration¶

    Configurations define migration behavior — ACID handling, optimizations, and other settings.

    # Create a migration configuration
curl -s -X POST http://localhost:8090/hms-mirror/api/v1/config \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "standard-migration",
    "description": "Standard migration with ACID downgrade",
    "migrateACID": {
      "on": true,
      "downgrade": true
    },
    "optimization": {
      "sortDynamicPartitionInserts": true
    },
    "copyViews": true
  }' | jq .

    Step 4: Create a Job Definition¶

    Jobs bring together a dataset, configuration, and connections into an executable migration plan.

    # Create a job
curl -s -X POST http://localhost:8090/hms-mirror/api/v1/jobs/sales-q1-migration \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "sales-q1-migration",
    "description": "Migrate sales and inventory to CDP",
    "dataStrategy": "SCHEMA_ONLY",
    "dataset": "sales-migration",
    "configuration": "standard-migration",
    "leftConnection": "hdp3-source",
    "rightConnection": "cdp-target"
  }' | jq .

# Validate the job before running
curl -s -X POST http://localhost:8090/hms-mirror/api/v1/jobs/sales-q1-migration/validate \
  -H "Authorization: Bearer $TOKEN" | jq .

    Step 5: Run the Job¶

    Always start with a dry run to review the generated SQL before executing against your clusters.

    # Start a dry run
curl -s -X POST http://localhost:8090/hms-mirror/api/v1/runtime/jobs/start \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "jobKey": "sales-q1-migration",
    "dryRun": true
  }' | jq .

# Monitor job status
curl -s http://localhost:8090/hms-mirror/api/v1/runtime/jobs/current \
  -H "Authorization: Bearer $TOKEN" | jq .

# Once satisfied with the dry run, execute for real
curl -s -X POST http://localhost:8090/hms-mirror/api/v1/runtime/jobs/start \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "jobKey": "sales-q1-migration",
    "dryRun": false
  }' | jq .

    Step 6: Retrieve Reports¶

    Reports are best viewed through the Web UI, which provides interactive table-level detail, SQL previews, and downloadable artifacts. However, you can also retrieve report data through the API:

    # List reports for the job
curl -s http://localhost:8090/hms-mirror/api/v1/jobs/sales-q1-migration/reports \
  -H "Authorization: Bearer $TOKEN" | jq .

# List all runtime reports
curl -s http://localhost:8090/hms-mirror/api/v1/runtime/reports \
  -H "Authorization: Bearer $TOKEN" | jq .

# Get detailed report information (use the key from the reports list)
REPORT_KEY="sales-q1-migration_20260210_143000"
curl -s "http://localhost:8090/hms-mirror/api/v1/runtime/reports/details?key=$REPORT_KEY" \
  -H "Authorization: Bearer $TOKEN" | jq .

# Get database-level details with generated SQL
curl -s "http://localhost:8090/hms-mirror/api/v1/runtime/reports/database-details?key=$REPORT_KEY&database=sales_db" \
  -H "Authorization: Bearer $TOKEN" | jq .

# List report files (SQL scripts, markdown reports)
curl -s "http://localhost:8090/hms-mirror/api/v1/runtime/reports/files?key=$REPORT_KEY" \
  -H "Authorization: Bearer $TOKEN" | jq .

# Download a specific report file
curl -s "http://localhost:8090/hms-mirror/api/v1/runtime/reports/file?key=$REPORT_KEY&filename=RIGHT_execute.sql" \
  -H "Authorization: Bearer $TOKEN" -o right_execute.sql

# Download all report artifacts as a zip
curl -s "http://localhost:8090/hms-mirror/api/v1/reports/download-by-key?key=$REPORT_KEY" \
  -H "Authorization: Bearer $TOKEN" -o report.zip

    Putting It All Together: Automation Script¶

    Here's a complete example script that automates the workflow:

    #!/bin/bash
# HMS-Mirror REST API Migration Script
# Creates a dataset, builds a job, runs it, and downloads reports.

BASE_URL="http://localhost:8090/hms-mirror/api/v1"
USERNAME="admin"
PASSWORD="your-password"

# Authenticate
echo "Authenticating..."
TOKEN=$(curl -sf -X POST "$BASE_URL/auth/login" \
  -H "Content-Type: application/json" \
  -d "{\"username\":\"$USERNAME\",\"password\":\"$PASSWORD\"}" | jq -r '.data.accessToken')

if [ -z "$TOKEN" ] || [ "$TOKEN" = "null" ]; then
  echo "ERROR: Authentication failed"
  exit 1
fi

AUTH="-H \"Authorization: Bearer $TOKEN\""

# Create dataset
echo "Creating dataset..."
curl -sf -X POST "$BASE_URL/datasets" \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "nightly-migration",
    "databases": [{"name": "analytics_db", "tableIncludeFilterPattern": ".*"}]
  }'

# Create job
echo "Creating job..."
curl -sf -X POST "$BASE_URL/jobs/nightly-run" \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "nightly-run",
    "dataStrategy": "SCHEMA_ONLY",
    "dataset": "nightly-migration",
    "configuration": "standard-migration",
    "leftConnection": "hdp3-source",
    "rightConnection": "cdp-target"
  }'

# Validate
echo "Validating job..."
VALIDATION=$(curl -sf -X POST "$BASE_URL/jobs/nightly-run/validate" \
  -H "Authorization: Bearer $TOKEN")
echo "$VALIDATION" | jq .

# Run dry run
echo "Starting dry run..."
curl -sf -X POST "$BASE_URL/runtime/jobs/start" \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"jobKey": "nightly-run", "dryRun": true}'

# Poll for completion
echo "Waiting for completion..."
while true; do
  STATUS=$(curl -sf "$BASE_URL/runtime/jobs/current" \
    -H "Authorization: Bearer $TOKEN")
  RUNNING=$(echo "$STATUS" | jq -r '.data | length // 0')
  if [ "$RUNNING" = "0" ]; then
    echo "Job complete."
    break
  fi
  echo "  Still running..."
  sleep 5
done

# Download report
echo "Downloading reports..."
curl -sf "$BASE_URL/jobs/nightly-run/reports" \
  -H "Authorization: Bearer $TOKEN" | jq .

echo "Done. View detailed reports at: http://localhost:8090/hms-mirror"

    JVM Options¶

    Memory Settings¶

    # Increase heap size
java -Xmx8g -Xms4g -jar hms-mirror-server.jar

# With G1 garbage collector
java -Xmx8g -XX:+UseG1GC -jar hms-mirror-server.jar

    Recommended Settings by Scale¶

    Scale Heap Options
    Small (<100 tables) 2G -Xmx2g
    Medium (<1000 tables) 4G -Xmx4g -XX:+UseG1GC
    Large (>1000 tables) 8G+ -Xmx8g -XX:+UseG1GC

    Debug Settings¶

    # Enable remote debugging
java -Xdebug -Xrunjdwp:transport=dt_socket,server=y,suspend=n,address=5005 \
  -jar hms-mirror-server.jar

# Verbose GC logging
java -Xmx4g -verbose:gc -jar hms-mirror-server.jar

    Configuration Files¶

    External Configuration¶

    # Use external application.properties
java -jar hms-mirror-server.jar \
  --spring.config.location=/etc/hms-mirror/application.properties

# Use external YAML
java -jar hms-mirror-server.jar \
  --spring.config.location=/etc/hms-mirror/application.yml

    Profile-Based Configuration¶

    # Use production profile
java -jar hms-mirror-server.jar --spring.profiles.active=production

# Profile looks for application-production.yml

    Logging Options¶

    Log Level¶

    # Set log level
java -jar hms-mirror-server.jar \
  --logging.level.com.cloudera.utils.hms=DEBUG

# Multiple log levels
java -jar hms-mirror-server.jar \
  --logging.level.com.cloudera.utils.hms=DEBUG \
  --logging.level.org.apache.hive=WARN

    Log File¶

    # Custom log file
java -jar hms-mirror-server.jar \
  --logging.file.name=/var/log/hms-mirror/app.log

    Log Levels¶

    Level Description
    ERROR Only errors
    WARN Warnings and errors
    INFO Normal operation (default)
    DEBUG Detailed debugging
    TRACE Maximum detail

    Kerberos Options¶

    With Ticket Cache¶

    # Obtain ticket first
kinit user@REALM

# Then start server
java -jar hms-mirror-server.jar

    With Keytab¶

    java -Djava.security.auth.login.config=/path/to/jaas.conf \
  -jar hms-mirror-server.jar

    Example jaas.conf: 

    Client {
  com.sun.security.auth.module.Krb5LoginModule required
  useKeyTab=true
  keyTab="/path/to/keytab"
  principal="user@REALM";
};

    Kerberos Debug¶

    java -Dsun.security.krb5.debug=true \
  -jar hms-mirror-server.jar

    Environment Variables¶

    Set options via environment variables:

    export SERVER_PORT=8091
export HMS_MIRROR_ROCKSDB_PATH=/data/rocksdb
export HMS_MIRROR_KEYSTORE_PASSWORD=my-secure-password
export JAVA_OPTS="-Xmx8g -XX:+UseG1GC"

java $JAVA_OPTS -jar hms-mirror-server.jar

    Common Variables¶

    Variable Property Description
    SERVER_PORT server.port HTTP port
    HMS_MIRROR_ROCKSDB_PATH hms-mirror.rocksdb.path RocksDB location
    HMS_MIRROR_KEYSTORE_PASSWORD hms-mirror.config.keystore.password Keystore master password
    HMS_MIRROR_OUTPUT_DIR hms-mirror.config.output-dir Report output directory
    JAVA_OPTS — JVM options

    Using the Startup Script¶

    The distribution includes startup scripts:

    Linux/macOS¶

    # Make executable (first time)
chmod +x bin/hms-mirror.sh

# Start server
./bin/hms-mirror.sh

# With options
./bin/hms-mirror.sh --server.port=8091

    Script Features¶

    The startup script:

    • Sets appropriate JVM memory
    • Configures JAVA_HOME if needed
    • Adds aux_libs to classpath (for JDBC drivers)
    • Handles signals for clean shutdown

    Service Management¶

    Running as systemd Service¶

    Create /etc/systemd/system/hms-mirror.service:

    [Unit]
Description=HMS-Mirror Server
After=network.target

[Service]
Type=simple
User=hms-mirror
Group=hms-mirror
WorkingDirectory=/opt/hms-mirror
ExecStart=/usr/bin/java -Xmx4g -jar /opt/hms-mirror/lib/hms-mirror-server.jar
Restart=on-failure
RestartSec=10
Environment="JAVA_HOME=/usr/lib/jvm/java-17"
Environment="HMS_MIRROR_KEYSTORE_PASSWORD=your-production-password"

[Install]
WantedBy=multi-user.target

    Commands: 

    sudo systemctl daemon-reload
sudo systemctl enable hms-mirror
sudo systemctl start hms-mirror
sudo systemctl status hms-mirror
sudo systemctl stop hms-mirror

    View Service Logs¶

    sudo journalctl -u hms-mirror -f

    Docker Usage¶

    Running with Docker¶

    # Basic run
docker run -d -p 8090:8090 \
  --name hms-mirror \
  cloudera/hms-mirror:4.x.x

# With persistent storage
docker run -d -p 8090:8090 \
  -v /path/to/data:/opt/hms-mirror/data \
  --name hms-mirror \
  cloudera/hms-mirror:4.x.x

# With environment variables
docker run -d -p 8090:8090 \
  -e SERVER_PORT=8090 \
  -e JAVA_OPTS="-Xmx4g" \
  -e HMS_MIRROR_KEYSTORE_PASSWORD="your-production-password" \
  --name hms-mirror \
  cloudera/hms-mirror:4.x.x

    Docker Compose¶

    version: '3'
services:
  hms-mirror:
    image: cloudera/hms-mirror:4.x.x
    ports:
      - "8090:8090"
    volumes:
      - ./data:/opt/hms-mirror/data
      - ./config:/opt/hms-mirror/config
    environment:
      - JAVA_OPTS=-Xmx4g
      - HMS_MIRROR_KEYSTORE_PASSWORD=your-production-password
    restart: unless-stopped

    Common Startup Patterns¶

    Development¶

    # Quick start with defaults
java -jar hms-mirror-server.jar

# With debug logging
java -jar hms-mirror-server.jar \
  --logging.level.com.cloudera.utils.hms=DEBUG

    Testing¶

    # Different port to avoid conflicts
java -jar hms-mirror-server.jar --server.port=8091

# Separate RocksDB for testing
java -jar hms-mirror-server.jar \
  --hms-mirror.rocksdb.path=/tmp/hms-mirror-test/rocksdb

    Production¶

    java -Xmx8g -XX:+UseG1GC \
  -jar hms-mirror-server.jar \
  --spring.profiles.active=production \
  --server.port=8090 \
  --hms-mirror.rocksdb.path=/data/hms-mirror/rocksdb \
  --logging.file.name=/var/log/hms-mirror/app.log

    Health Check¶

    Verify Server Running¶

    # System health endpoint
curl http://localhost:8090/hms-mirror/api/v1/system/health

# Application version
curl http://localhost:8090/hms-mirror/api/v1/app/version

    Check Port¶

    # Check if port is in use
lsof -i :8090
netstat -tlnp | grep 8090

    Shutdown¶

    Graceful Shutdown¶

    # Send SIGTERM
kill $(pgrep -f hms-mirror-server)

# Or via systemd
sudo systemctl stop hms-mirror

    Force Shutdown¶

    # Send SIGKILL (last resort)
kill -9 $(pgrep -f hms-mirror-server)

    Exit Codes¶

    Code Meaning
    0 Normal exit
    1 General error
    143 Terminated by signal

    Troubleshooting¶

    Server Won't Start¶

    # Check if port is in use
lsof -i :8090

# Check Java version (17+ required)
java -version

# Check RocksDB permissions
ls -la ~/.hms-mirror

    Out of Memory¶

    # Increase heap
java -Xmx8g -jar hms-mirror-server.jar

    Permission Denied¶

    # Check file permissions
chmod -R 755 ~/.hms-mirror

# Check user
whoami
ls -la ~/.hms-mirror
