    Data Browser¶

    The Data Browser allows you to execute SQL queries directly against your Hive connections from within HMS-Mirror. This is useful for exploring databases, verifying table structures, and testing queries before running migrations.

    Accessing the Data Browser¶

    Navigate to Utilities > Data Browser from the main menu.

    Prerequisites¶

    Before using the Data Browser, you must have at least one connection configured with a successful HiveServer2 test. Only connections that have passed the HS2 connection test will appear in the connection selector.

    Opening a Connection Tab¶

    1. Select a connection from the dropdown menu at the top of the page
    2. Click New Tab to open a session to that connection
    3. Each tab represents an independent session to the selected cluster

    You can open multiple tabs to different connections simultaneously, allowing you to compare data across clusters.

    Writing SQL¶

    The SQL editor supports standard HiveQL syntax. Key features include:

    • Comments: Lines starting with -- are treated as comments and ignored during execution
    • Inline comments: Text after -- on any line is ignored (unless inside a quoted string)
    • Statement termination: Statements must end with a semicolon (;)
    • Multi-line statements: Statements can span multiple lines

    Example¶

    -- Show all databases
SHOW DATABASES;

-- List tables in a specific database
USE my_database;
SHOW TABLES;

-- Query with inline comment
SELECT * FROM my_table LIMIT 10; -- preview data

    Executing Queries¶

    Run Current Statement¶

    Executes the statement at the current cursor position.

    • Keyboard shortcut: ⌘+Enter (Mac) or Ctrl+Enter (Windows/Linux)
    • Button: Click Run Statement

    The statement is determined by the cursor position: - If the cursor is within a statement, that statement is executed - If the cursor is on the same line as a semicolon (even after it), the statement ending with that semicolon is executed

    Run Script¶

    Executes all statements in the editor sequentially.

    • Keyboard shortcut: ⌘+Shift+Enter (Mac) or Ctrl+Shift+Enter (Windows/Linux)
    • Button: Click Run Script

    Execution stops if any statement encounters an error.

    Viewing Results¶

    Query results appear in tabbed panels below the SQL editor:

    • Each executed query creates a new result tab
    • Result tabs show a success/error indicator
    • Query results display the first 100 rows in a scrollable table
    • Column names and data types are shown in the table header
    • NULL values are displayed as italicized "NULL"

    Result Types¶

    • Query Results: SELECT statements show data in a table format with row count
    • Update Results: DDL and DML statements show a success message with rows affected
    • Errors: Failed statements display the error message and SQL state

    Saving and Loading Scripts¶

    The Data Browser allows you to save scripts for later use and reload them when needed.

    Saving a Script¶

    1. Click the Save button in the toolbar
    2. Enter a name for your script
    3. Click Save

    If a script with that name already exists, you will be prompted to confirm overwriting. You can check "Always overwrite without asking" to skip this confirmation for future saves during your session.

    Loading a Script¶

    1. Click the Load button in the toolbar
    2. Select a script from the list of saved scripts
    3. The script content will replace the current editor content

    You can also delete saved scripts by hovering over a script name and clicking the trash icon.

    Scripts are stored persistently and will be available across sessions.

    Closing Sessions¶

    Click the X on any tab to close that session. This releases the connection back to the pool. Closing all tabs returns you to the initial state where you can select a new connection.

    Tips¶

    • Use SHOW DATABASES and SHOW TABLES to explore the cluster
    • Use DESCRIBE FORMATTED table_name to see detailed table metadata
    • Limit large queries with LIMIT to avoid long-running operations
    • The browser maintains separate sessions per tab, so transactions are isolated
