    Frequently Asked Questions¶

    Common questions about HMS-Mirror v4.

    General Questions¶

    What is HMS-Mirror?¶

    HMS-Mirror is a tool for migrating and synchronizing Apache Hive metadata and data between DataLake environments powered by Hive and Hive Metastore. It automates the complex process of moving tables, views, and data between environments like HDP to CDP, on-premises to cloud, or HDFS to object storage.

    What's new in version 4?¶

    Version 4 introduces: - Web-based UI - Visual configuration instead of config files - Wizard-driven workflows - Step-by-step guidance - AI-powered analysis - LLM integration for insights - RocksDB storage - Persistent configurations without external database - Improved job management - Save, validate, and rerun jobs - Iceberg conversion - Convert tables to Iceberg format

    Is HMS-Mirror open source?¶

    Yes, HMS-Mirror is open source software developed by Cloudera Labs.

    What clusters does HMS-Mirror support?¶

    Source clusters (LEFT): - HDP 2.x and 3.x - CDH 5.x and 6.x - CDP Private Cloud Base - CDP Public Cloud

    Target clusters (RIGHT): - CDP Private Cloud Base - CDP Public Cloud (AWS, Azure, GCP) - Open-source Hive deployments

    Does HMS-Mirror require downtime?¶

    No mandatory downtime, but consider: - For SCHEMA_ONLY: No impact on source - For data migration: Source tables are read (may affect performance) - For STORAGE_MIGRATION: Brief table unavailability possible - Best practice: Run during low-activity periods for large migrations

    Installation¶

    What are the system requirements?¶

    Requirement Minimum Recommended
    Java JDK 11 JDK 17
    Memory 2 GB 4 GB+
    Disk 500 MB 2 GB+

    Do I need to install anything on the clusters?¶

    No. HMS-Mirror connects to clusters via standard JDBC (HiveServer2) or direct Metastore connections. No agents or software installation on clusters is required.

    Can I run HMS-Mirror in Docker?¶

    Yes. Docker images are available: 

    docker run -d -p 8090:8090 cloudera/hms-mirror:4.x.x

    Where does HMS-Mirror store data?¶

    By default in ~/.hms-mirror/: - rocksdb/ - Configurations, jobs, datasets - reports/ - Migration reports - aux_libs/ - JDBC drivers - logs/ - Application logs

    Connections¶

    What authentication methods are supported?¶

    • No authentication
    • Kerberos
    • LDAP (username/password)
    • Knox Gateway

    Do I need both HiveServer2 and Metastore Direct?¶

    HiveServer2 is required for most operations (creating tables, moving data).

    Metastore Direct is optional but provides: - Faster metadata queries - Direct database access when HS2 unavailable

    How do I connect through Knox?¶

    Use the Knox gateway URI: 

    jdbc:hive2://knox.example.com:8443/;ssl=true;transportMode=http;httpPath=gateway/cdp-proxy-api/hive

    Can I connect to clusters in different Kerberos realms?¶

    Yes, with cross-realm trust configured. Configure both realms in /etc/krb5.conf with appropriate trust relationships.

    Data Strategies¶

    Which strategy should I use?¶

    Scenario Recommended
    Schema sync only SCHEMA_ONLY
    Quick test access LINKED
    Small tables (<1GB) SQL
    Large tables (>1GB) EXPORT_IMPORT
    Mixed table sizes HYBRID
    Same cluster, new storage STORAGE_MIGRATION
    Shared storage COMMON
    DDL documentation DUMP
    Table format upgrade ICEBERG_CONVERSION

    What's the difference between SQL and EXPORT_IMPORT?¶

    SQL Strategy - Uses INSERT...SELECT through HiveServer2 - Data is serialized/deserialized - Good for small-medium tables - Can apply transformations

    EXPORT_IMPORT Strategy - Uses Hive's native export/import - Preserves exact data formats - Better for large tables - Required for ACID tables (without downgrade)

    How does HYBRID work?¶

    HYBRID automatically selects SQL or EXPORT_IMPORT per table based on: - Table size - Partition count - Whether it's ACID - Bucket configuration

    Can I migrate ACID tables?¶

    Yes, with options: - Keep ACID: Use EXPORT_IMPORT strategy - Downgrade: Convert to external tables (easier, recommended for legacy migrations)

    Data Movement¶

    Does HMS-Mirror actually move data?¶

    Depends on strategy: - SCHEMA_ONLY, LINKED, DUMP, COMMON: No data movement - SQL, EXPORT_IMPORT, HYBRID: Yes, moves data - STORAGE_MIGRATION: Moves data within cluster

    How long does migration take?¶

    Varies greatly based on: - Data volume - Number of tables - Strategy used - Network bandwidth - Cluster resources

    Rough estimates (per table): - Schema only: Seconds - Small table (SQL): Minutes - Large table (EXPORT_IMPORT): Minutes to hours

    Can I run migrations in parallel?¶

    HMS-Mirror processes tables within a database sequentially but can process multiple operations concurrently. For maximum parallelism, split into multiple jobs by database.

    What if a migration fails midway?¶

    • Failed tables are noted in the report
    • Successful tables are not affected
    • Create a new job with only failed tables
    • Fix the issue and re-run

    ACID Tables¶

    Why are ACID tables special?¶

    ACID (transactional) tables have: - Different storage format - Version compatibility issues between Hive versions - Restrictions on SQL operations

    Should I downgrade ACID tables?¶

    Yes, if: - Migrating from HDP to CDP - Applications don't need UPDATE/DELETE - Simpler management preferred

    No, if: - Applications actively use transactions - UPDATE/DELETE operations required - Compliance requires transactional guarantees

    What happens when ACID tables are downgraded?¶

    • Converted to external tables
    • Lose transactional properties
    • Data is preserved
    • Can't use UPDATE/DELETE after

    Storage Migration¶

    Can I migrate from HDFS to S3?¶

    Yes, using STORAGE_MIGRATION strategy: 

    dataStrategy: STORAGE_MIGRATION
transfer:
  warehouse:
    external: s3a://bucket/warehouse/external
    managed: s3a://bucket/warehouse/managed

    Do I need to update table locations manually?¶

    No. HMS-Mirror updates all table and partition locations automatically based on your configuration.

    What about credentials for cloud storage?¶

    Configure in core-site.xml or pass through Hive session properties. HMS-Mirror uses the cluster's configured storage credentials.

    Iceberg Conversion¶

    What is Iceberg?¶

    Apache Iceberg is a high-performance table format providing: - Time travel queries - Schema evolution - Hidden partitioning - Multi-engine support

    Can I convert existing tables to Iceberg?¶

    Yes, using ICEBERG_CONVERSION strategy. Supports in-place or shadow conversion.

    Do I lose data during Iceberg conversion?¶

    No. Data is preserved. With SHADOW mode, the original table is also retained.

    Web UI¶

    Can I use HMS-Mirror without the web UI?¶

    The web UI is the primary interface in v4. The API is available for automation, but there's no standalone CLI mode like v3.

    How do I back up my configurations?¶

    1. Export individual items from the UI
    2. Back up the RocksDB directory: ~/.hms-mirror/rocksdb

    Can multiple users access HMS-Mirror simultaneously?¶

    Yes, but there's no user authentication or multi-tenancy. All users share the same configurations and jobs.

    LLM Integration¶

    Do I need LLM features?¶

    No. LLM features are optional and provide: - Configuration analysis - Job recommendations - Report insights

    All core functionality works without LLM.

    Is my data sent to cloud AI services?¶

    Depends on configuration: - Embedded/Local Ollama: All processing local, no data sent externally - Cloud providers (OpenAI, etc.): Data sent to provider APIs

    What models are recommended?¶

    Use Case Model
    Quick analysis mistral, phi
    Detailed analysis llama3.1, gpt-4
    Privacy-focused Embedded Ollama

    Troubleshooting¶

    Why can't I connect to my cluster?¶

    Common causes: 1. Network/firewall blocking connection 2. Wrong hostname or port 3. Service not running 4. Authentication failure (wrong credentials, no Kerberos ticket)

    Why is my migration slow?¶

    Possible causes: 1. Large data volumes 2. Using SQL for large tables (use EXPORT_IMPORT) 3. Network bandwidth limitations 4. Cluster resource constraints

    How do I enable debug logging?¶

    java -jar hms-mirror-server.jar \
  --logging.level.com.cloudera.utils.hms=DEBUG

    Where are the logs?¶

    Default: ~/.hms-mirror/logs/hms-mirror.log

    Migration Planning¶

    How should I plan a large migration?¶

    1. Inventory: Catalog all databases and tables
    2. Classify: Identify ACID, large tables, complex cases
    3. Test: Start with small, non-critical databases
    4. Validate: Verify results thoroughly
    5. Scale: Gradually migrate larger datasets
    6. Cutover: Plan application switchover

    Should I migrate everything at once?¶

    No. Recommended approach: 1. Phase 1: Schema only (validate structure) 2. Phase 2: Small tables 3. Phase 3: Large tables 4. Phase 4: Complex cases (ACID, many partitions)

    How do I validate a migration?¶

    • Compare row counts
    • Run sample queries on both clusters
    • Check table properties
    • Verify partition locations
    • Test application access
