    Understanding Connections¶

    Connections are the foundation of HMS-Mirror operations. This guide explains connection concepts, types, and authentication methods.

    What is a Connection?¶

    A connection defines how HMS-Mirror communicates with a DataLake environment powered by Hive and Hive Metastore. It encapsulates:

    • Endpoint information - Where to connect (host, port, URI)
    • Authentication - How to authenticate (Kerberos, LDAP, none)
    • Connection properties - Protocol settings (transport, SSL)
    • Session settings - Hive configuration overrides

    Connection Types¶

    HMS-Mirror supports multiple connection methods to interact with Hive.

    HiveServer2 (HS2)¶

    The primary and most common connection method.

    ┌─────────────────┐        JDBC         ┌─────────────────┐
│   HMS-Mirror    │─────────────────────│   HiveServer2   │
└─────────────────┘                     └─────────────────┘

    Characteristics - Uses JDBC protocol - Executes HiveQL statements - Supports all authentication methods - Required for data movement operations

    Use Cases - Creating tables - Inserting data - Running queries - Most migration operations

    Metastore Direct¶

    Direct connection to the Hive Metastore database.

    ┌─────────────────┐        JDBC         ┌─────────────────┐
│   HMS-Mirror    │─────────────────────│ Metastore DB    │
│                 │                     │ (MySQL/PG/Oracle)│
└─────────────────┘                     └─────────────────┘

    Characteristics - Connects to underlying database (MySQL, PostgreSQL, Oracle) - Read-only operations - Faster for metadata queries - Bypasses HiveServer2

    Use Cases - Fast metadata discovery - Partition information - Table statistics - When HS2 is unavailable

    When to Use Each¶

    Scenario HS2 Metastore Direct
    Create tables ✓
    Move data ✓
    Read metadata ✓ ✓ (faster)
    Partition discovery ✓ ✓ (faster)
    Statistics ✓ ✓
    No HS2 available ✓

    Connection URI Formats¶

    HiveServer2 URIs¶

    Basic Format 

    jdbc:hive2://<host>:<port>/<database>

    With Parameters 

    jdbc:hive2://<host>:<port>/<database>;param1=value1;param2=value2

    Examples

    Standard binary connection: 

    jdbc:hive2://hiveserver.example.com:10000/default

    HTTP transport: 

    jdbc:hive2://hiveserver.example.com:10001/default;transportMode=http;httpPath=cliservice

    SSL enabled: 

    jdbc:hive2://hiveserver.example.com:10000/default;ssl=true

    Zookeeper discovery: 

    jdbc:hive2://zk1.example.com:2181,zk2.example.com:2181,zk3.example.com:2181/;serviceDiscoveryMode=zooKeeper;zooKeeperNamespace=hiveserver2

    Knox gateway: 

    jdbc:hive2://knox.example.com:8443/;ssl=true;transportMode=http;httpPath=gateway/cdp-proxy-api/hive

    Metastore Database URIs¶

    MySQL/MariaDB 

    jdbc:mysql://metastore.example.com:3306/hive_metastore
jdbc:mysql://metastore.example.com:3306/hive_metastore?useSSL=true

    PostgreSQL 

    jdbc:postgresql://metastore.example.com:5432/hive_metastore
jdbc:postgresql://metastore.example.com:5432/hive_metastore?ssl=true

    Oracle 

    jdbc:oracle:thin:@metastore.example.com:1521:hive
jdbc:oracle:thin:@//metastore.example.com:1521/hive_service

    Authentication Methods¶

    No Authentication¶

    For development or test clusters without security.

    Configuration - No additional settings required - Simply provide the JDBC URI

    Example 

    hiveServer2:
  uri: jdbc:hive2://dev-cluster:10000/default

    Kerberos Authentication¶

    For enterprise clusters with Kerberos security.

    ┌─────────────────┐    1. kinit    ┌─────────────────┐
│     User        │───────────────►│       KDC       │
└─────────────────┘                └─────────────────┘
        │                                   │
        │                          2. TGT   │
        │◄──────────────────────────────────┘
        │
        │ 3. Request Service Ticket
        ▼
┌─────────────────┐    4. Ticket   ┌─────────────────┐
│   HMS-Mirror    │───────────────►│   HiveServer2   │
└─────────────────┘                └─────────────────┘

    Requirements - Valid Kerberos ticket (TGT) - Properly configured krb5.conf - Service principal in connection properties

    Configuration 

    hiveServer2:
  uri: jdbc:hive2://secure-cluster:10000/default
  connectionProperties:
    principal: hive/_HOST@EXAMPLE.COM

    Using a Keytab 

    hiveServer2:
  uri: jdbc:hive2://secure-cluster:10000/default
  connectionProperties:
    principal: hive/_HOST@EXAMPLE.COM
    keytab: /path/to/service.keytab

    LDAP Authentication¶

    Username and password authentication via LDAP.

    Configuration 

    hiveServer2:
  uri: jdbc:hive2://ldap-cluster:10000/default
  connectionProperties:
    user: myusername
    password: mypassword

    Knox Gateway Authentication¶

    Authentication through Apache Knox gateway.

    Configuration 

    hiveServer2:
  uri: jdbc:hive2://knox.example.com:8443/;ssl=true;transportMode=http;httpPath=gateway/cdp-proxy-api/hive
  connectionProperties:
    user: knoxuser
    password: knoxpassword

    Connection Properties¶

    Properties passed to the JDBC connection.

    Common Properties¶

    Property Description Values
    transportMode Transport protocol binary, http
    httpPath HTTP endpoint path cliservice, custom
    ssl Enable SSL/TLS true, false
    sslTrustStore Trust store path file path
    sslTrustStorePassword Trust store password string
    principal Kerberos principal service/_HOST@REALM
    user Username string
    password Password string

    SSL/TLS Properties¶

    Property Description
    ssl Enable SSL
    sslTrustStore Path to JKS trust store
    sslTrustStorePassword Trust store password
    sslTrustStoreType Trust store type (JKS, PKCS12)

    Kerberos Properties¶

    Property Description
    principal Service principal
    keytab Path to keytab file
    saslQop SASL quality of protection

    Performance Properties¶

    Property Description Default
    socketTimeout Socket timeout (ms) 0 (infinite)
    connectTimeout Connection timeout (ms) 10000
    fetchSize JDBC fetch size 1000

    Session Properties¶

    Hive session settings applied after connection.

    Common Session Properties¶

    Property Description Example
    hive.exec.dynamic.partition Enable dynamic partitions true
    hive.exec.dynamic.partition.mode Partition mode nonstrict
    hive.exec.max.dynamic.partitions Max partitions 10000
    hive.exec.parallel Enable parallel execution true
    mapreduce.job.reduces Number of reducers 10

    Setting Session Properties¶

    Session properties are set when the connection is established and apply to all operations using that connection.

    hiveServer2:
  uri: jdbc:hive2://cluster:10000/default
  sessionProperties:
    hive.exec.dynamic.partition: "true"
    hive.exec.dynamic.partition.mode: "nonstrict"
    hive.exec.parallel: "true"

    Cluster Environment Types¶

    HMS-Mirror recognizes different cluster environments that affect behavior.

    Legacy¶

    Pre-CDP clusters (HDP 2.x/3.x, CDH 5.x/6.x).

    Characteristics - Older Hive versions (1.x, 2.x) - Different ACID implementation - May have different default behaviors

    When to Select - HDP 2.x or 3.x clusters - CDH 5.x or 6.x clusters - Any non-CDP Hadoop distribution

    Non-Legacy (CDP)¶

    Modern CDP Private Cloud or Public Cloud.

    Characteristics - Hive 3.x - Modern ACID implementation - Iceberg support - Updated defaults

    When to Select - CDP Private Cloud Base - CDP Public Cloud - Modern Hive 3.x deployments

    Impact on Migration¶

    Feature Legacy Non-Legacy
    ACID tables V1 format V2 format
    External table purge Manual external.table.purge
    Managed table location warehouse/tablespace Separate managed warehouse
    Iceberg Not supported Supported

    Connection Pooling¶

    HMS-Mirror manages connection pools for efficiency.

    Pool Configuration¶

    Setting Description Default
    Initial Size Starting connections 2
    Max Total Maximum connections 10
    Max Idle Max idle connections 5
    Min Idle Min idle connections 1

    Pool Implementations¶

    DBCP2 - Apache Commons DBCP - Widely compatible - Default option

    HikariCP - High performance - Lower latency - Production recommended

    Hybrid - Combination approach - Adaptive behavior

    Testing Connections¶

    Always test connections before using them.

    What Gets Tested¶

    1. Network connectivity - Can reach the endpoint
    2. Authentication - Credentials are valid
    3. Authorization - User has permissions
    4. Basic operations - Can list databases

    Test Results¶

    Success 

    ✓ Connection successful
  Server: Hive 3.1.3000
  Databases: 25 found
  Response time: 234ms

    Failure 

    ✗ Connection failed
  Error: Authentication failed
  Details: GSS initiate failed
  Suggestion: Check Kerberos ticket (klist)

    Best Practices¶

    Security¶

    1. Use service accounts - Not personal accounts
    2. Minimal permissions - Only what's needed
    3. Rotate credentials - Regular password/keytab rotation
    4. Secure storage - Protect keytab files

    Reliability¶

    1. Test regularly - Catch issues early
    2. Use Zookeeper discovery - For HA setups
    3. Configure timeouts - Prevent hanging
    4. Monitor connections - Track pool usage

    Organization¶

    1. Clear naming - Include environment and purpose
    2. Document settings - Note special configurations
    3. Separate environments - Don't mix prod/dev
    4. Version control - Track connection changes

    Troubleshooting¶

    Common Issues¶

    Issue Cause Solution
    Connection refused Wrong host/port or service down Verify endpoint, check service
    Authentication failed Invalid credentials Check password/ticket
    Timeout Network issues Check firewall, increase timeout
    SSL errors Certificate mismatch Configure trust store
    Permission denied Insufficient privileges Grant required permissions

    See Troubleshooting Guide for detailed solutions.
