    Architecture Overview¶

    This document describes the technical architecture of HMS-Mirror v4, including its components, data flow, and integration points.

    Components¶

    Web UI (Frontend)¶

    The user interface is a modern React single-page application.

    Technology Stack - React 18 with TypeScript - Tailwind CSS for styling - React Router for navigation - Axios for API communication

    Key Features - Wizard-driven workflows - Real-time job monitoring - Interactive reports - AI analysis integration

    Directory Structure 

    frontend/
├── src/
│   ├── components/      # React components
│   │   ├── common/      # Shared components
│   │   ├── config/      # Configuration UI
│   │   ├── connections/ # Connection management
│   │   ├── datasets/    # Dataset management
│   │   ├── jobs/        # Job management
│   │   └── wizards/     # Wizard components
│   ├── api/             # API client code
│   ├── types/           # TypeScript definitions
│   └── utils/           # Utility functions

    Spring Boot Application (Backend)¶

    The server is a Spring Boot application providing REST APIs and business logic.

    Technology Stack - Java 17+ - Spring Boot 3.x - Spring Web for REST APIs - JDBC for database connections

    Package Structure 

    com.cloudera.utils.hms.mirror/
├── cli/                 # Command-line interface
├── domain/              # Domain models
│   ├── HmsMirrorConfig  # Main configuration
│   ├── TableMirror      # Table migration state
│   ├── DBMirror         # Database migration state
│   └── Cluster          # Cluster connection info
├── service/             # Business services
├── datastrategy/        # Migration strategies
├── connections/         # Connection pooling
├── web/                 # Web controllers
│   └── controller/
│       └── api/v1/      # REST API endpoints
└── repository/          # Data persistence

    Data Strategies¶

    Migration strategies are implemented as pluggable components.

    ┌─────────────────────────────────────────────────────────┐
│                  DataStrategy Interface                  │
└─────────────────────────────────────────────────────────┘
                           │
    ┌──────────────────────┼──────────────────────┐
    │                      │                      │
    ▼                      ▼                      ▼
┌─────────┐         ┌─────────────┐        ┌──────────┐
│ SCHEMA  │         │     SQL     │        │  EXPORT  │
│  ONLY   │         │             │        │  IMPORT  │
└─────────┘         └─────────────┘        └──────────┘
    │                      │                      │
    ▼                      ▼                      ▼
┌─────────┐         ┌─────────────┐        ┌──────────┐
│ LINKED  │         │   HYBRID    │        │ STORAGE  │
│         │         │             │        │MIGRATION │
└─────────┘         └─────────────┘        └──────────┘
    │                      │                      │
    ▼                      ▼                      ▼
┌─────────┐         ┌─────────────┐        ┌──────────┐
│ COMMON  │         │    DUMP     │        │ ICEBERG  │
│         │         │             │        │CONVERSION│
└─────────┘         └─────────────┘        └──────────┘

    Each strategy implements: - execute() - Perform the migration - validate() - Check prerequisites - generateDDL() - Create SQL statements

    Pool Implementations - DBCP2 - Apache Commons DBCP - HikariCP - High-performance pool - Hybrid - Combination approach

    RocksDB Storage¶

    Persistent storage for configurations, jobs, and state.

    ~/.hms-mirror/rocksdb/
├── connections/     # Saved connections
├── configurations/  # Migration configurations
├── datasets/        # Dataset definitions
├── jobs/            # Job configurations
└── reports/         # Migration reports

    Benefits - No external database required - Fast key-value access - Embedded and portable - Automatic compaction

    Firewall Requirements¶

    HMS-Mirror server needs outbound access to:

    1. HiveServer2 on both clusters
    2. Metastore database (if using direct access)
    3. Kerberos KDC (if using Kerberos auth)
    4. Knox Gateway (if accessing via Knox)
    5. LLM endpoints (if using cloud LLM providers)
