    Installation Guide¶

    This guide walks you through installing HMS-Mirror v4 and getting the server running. Choose the deployment method that fits your environment.

    Prerequisites¶

    System Requirements¶

    Requirement Minimum Recommended
    Java JDK 17 JDK 17+
    Memory 2 GB 4 GB+
    Disk Space 500 MB 2 GB+ (for RocksDB storage)
    OS Linux, macOS Linux

    Network Requirements¶

    HMS-Mirror needs network access to your DataLake environments:

    • HiveServer2 Port — Typically 10000 (or 10001 for HTTP transport)
    • Metastore Thrift Port — Typically 9083 (if using direct Metastore access)
    • Knox Gateway — If accessing through Knox (typically 8443)

    Ensure firewalls and security groups allow connections from the HMS-Mirror server to these ports on both source and target clusters.

    Kerberos Requirements (If Applicable)¶

    If your clusters use Kerberos authentication:

    • A valid /etc/krb5.conf file pointing to the correct KDC realm on the HMS-Mirror host
    • Network access to the KDC (Key Distribution Center)

    HMS-Mirror manages Kerberos authentication internally — keytabs and credentials are configured through the Web UI under Preferences > Kerberos. There is no need to kinit externally or manage ticket caches outside the application.

    Deployment Options¶

    Method Best For
    Cloudera Manager Production CDP clusters with CM-managed services
    Standalone Edge nodes, standalone servers, development
    Docker Containerized environments, quick evaluation
    Systemd Service Production Linux servers without CM

    Cloudera Manager Deployment¶

    Deploy HMS-Mirror as a managed service on a CDP cluster using a CSD (Custom Service Descriptor) and Parcel. This is the recommended deployment for CDP environments.

    Prerequisites¶

    • Cloudera Manager 7.0 or later
    • CDP cluster with at least one host available for the HMS-Mirror Server role
    • Admin access to Cloudera Manager
    • JDK 17+ installed on the target host

    Step 1 — Install the CSD¶

    The CSD tells Cloudera Manager how to configure and manage the HMS-Mirror service. It must be installed before the parcel.

    Obtain the CSD JAR (HMS_MIRROR-<version>.jar) from the release artifacts.

    1. Copy the CSD JAR to /opt/cloudera/csd/ on the Cloudera Manager host
    2. Restart the Cloudera Manager Server: 
      sudo systemctl restart cloudera-scm-server
    3. After restart, HMS Mirror will appear as an available service type

    Step 2 — Install and Activate the Parcel¶

    The parcel contains the HMS-Mirror application binaries that get distributed to cluster hosts.

    Obtain the parcel (HMS_MIRROR-<version>-<distro>.parcel) and its SHA checksum file from the release artifacts. Parcel builds are available for EL7, EL8, EL9, and Ubuntu.

    1. Copy the .parcel and .parcel.sha files to /opt/cloudera/parcel-repo/ on the Cloudera Manager host
    2. In Cloudera Manager, go to Parcels and click Check for New Parcels
    3. Find HMS_MIRROR in the list
    4. Click Distribute and wait for distribution to complete
    5. Click Activate to make the parcel available for use

    Step 3 — Add the Service¶

    1. In Cloudera Manager, go to your cluster and click Add Service
    2. Select HMS Mirror from the service list
    3. Assign the HMS Mirror Server role to a host
    4. Configure the service (see Configuration below)
    5. Click Start to launch the service

    CM Configuration¶

    All settings are available in the Cloudera Manager UI under the HMS Mirror service configuration.

    Core Settings¶

    Parameter Default Description
    Server Port 8090 HTTP/HTTPS port for Web UI and REST API
    Working Directory /var/lib/hms-mirror Root directory for all data (RocksDB, drivers, certificates, etc.)
    Log Directory /var/log/hms-mirror Application log files
    Maximum Concurrent Threads 10 Maximum migration threads (5–100)

    JVM Settings¶

    Parameter Default Description
    Initial Heap Size 4096 MB Initial JVM heap
    Maximum Heap Size 8192 MB Maximum JVM heap
    Garbage Collector G1GC GC algorithm (G1GC, ZGC, ParallelGC)
    Log Level INFO Application log verbosity (DEBUG, INFO, WARN, ERROR)

    Authentication¶

    Parameter Default Description
    Authentication Type local local, ldap, or oidc
    Keystore Password — Master password for encrypting credentials (required)

    LDAP and OIDC parameters are also configurable through the CM UI. See Authentication for details.

    Feature Flags¶

    Parameter Default Description
    Enable Migration Planner false Migration planning and tracking
    Enable Trino Connections false Trino JDBC connection support
    Enable LLM Features false AI-assisted analysis
    Create Sample Data false Create demo data on startup

    Auto-TLS¶

    When Auto-TLS is enabled on the cluster, HMS-Mirror automatically discovers and uses the cluster's keystores and truststores. The control script searches for:

    • cm-auto-host_keystore.jks — Server keystore (for HTTPS)
    • cm-auto-host_key.pw — Keystore password
    • cm-auto-in_cluster_truststore.jks — In-cluster truststore (preferred)
    • cm-auto-global_truststore.jks — Global truststore (fallback)

    No manual certificate configuration is needed when Auto-TLS is active.

    For custom certificates, configure the Client Truststore Location and Client Truststore Password in the SSL/TLS section of the service configuration.

    Web Links¶

    After the service starts, Cloudera Manager provides quick links:

    Link URL
    HMS Mirror Web UI https://<host>:<port>/hms-mirror
    API Documentation (Swagger) https://<host>:<port>/hms-mirror/swagger-ui.html
    Health Status https://<host>:<port>/hms-mirror/actuator/health

    Service Management¶

    Once deployed, manage HMS-Mirror through Cloudera Manager like any other service:

    • Start / Stop / Restart from the service Actions menu
    • View logs from the HMS Mirror Server role Logs tab
    • Monitor health via the built-in health check
    • Collect Diagnostics via the service command
    • Update configuration and restart when needed

    Upgrading via CM¶

    To upgrade to a new version:

    1. Obtain the new parcel and CSD JAR from the release artifacts
    2. Replace the CSD JAR in /opt/cloudera/csd/ and restart CM Server
    3. Upload the new parcel to /opt/cloudera/parcel-repo/
    4. In Cloudera Manager: Distribute and Activate the new parcel, Deactivate the old one
    5. Restart the HMS Mirror service

    CM Troubleshooting¶

    Issue Solution
    Service fails to start Check stderr log in CM role instance Logs tab
    Parcel not visible in CM Verify .parcel and .sha files are in /opt/cloudera/parcel-repo/ and click Check for New Parcels
    CSD not visible after install Restart the Cloudera Manager Server process
    SSL handshake failures Verify Auto-TLS is active or configure a custom truststore
    Java version error Ensure the host has JDK 17+ installed
    LDAP filter errors CM uses USERNAME_PLACEHOLDER instead of {0} — the control script handles the translation automatically

    Standalone Deployment¶

    Download¶

    Download the latest release (look for the hms-mirror-server-x.x.x.x-dist.tar.gz file):

    # Extract the archive
tar -xzf hms-mirror-server-x.x.x.x-dist.tar.gz

# Run the install
hms-mirror-server-install-x.x.x.x/setup.sh

    The release package contains: 

    hms-mirror-server-install-4.x.x.x/
├── bin/
│   ├── hmsm-server                    # Server startup script
│   └── hcfs-cli                       # HCFS CLI utility
├── lib/
│   └── hms-mirror-server-4.x.x.x-server.jar
└── setup.sh                           # Installation script

    After running setup.sh, HMS-Mirror is installed to $HOME/.hms-mirror/:

    $HOME/.hms-mirror/
├── bin/                        # Executable scripts
│   ├── hmsm-server             # Server startup script
│   └── hcfs-cli                # HCFS CLI utility
├── lib/                        # Application JARs
│   └── hms-mirror-server-4.x.x.x-server.jar
├── drivers/                    # Platform-specific Hive JDBC drivers
│   ├── cdp/                    # CDP drivers
│   ├── hdp/                    # HDP drivers
│   ├── cdh/                    # CDH drivers
│   └── apache/                 # Apache Hive drivers
├── data/                       # RocksDB persistent storage
├── certificates/               # SSL/TLS certificates and truststores
├── security/                   # PKCS12 keystore
├── settings/                   # Application settings
├── logs/                       # Application logs
├── reports/                    # Migration reports output
├── llms/                       # Local LLM models (if using Ollama)
└── hms-mirror.pid              # Server process ID file

    Running the Server¶

    After installation, use the hmsm-server script (added to your PATH by setup.sh):

    # Start the server
hmsm-server start

# Start with custom port
hmsm-server --server.port=8091

# Stop the server
hmsm-server stop

    Info

    Windows is not currently supported. HMS-Mirror runs on Linux and macOS.

    Docker Deployment¶

    Run HMS-Mirror in a Docker container. Persistent volume mounts are essential to preserve your data (connections, jobs, configurations, reports) across container restarts.

    Volumes¶

    HMS-Mirror stores all persistent data in its working directory. You must map a host directory to avoid data loss when the container is recreated.

    Container Path Purpose
    /opt/hms-mirror/data RocksDB database, drivers, certificates, keystore, settings, reports
    /var/log/hms-mirror Application logs

    Quick Start¶

    # Create host directories for persistent storage
mkdir -p ~/.hms-mirror/data ~/.hms-mirror/logs

# Run with persistent storage
docker run -d \
  --name hms-mirror \
  -p 8090:8090 \
  -v ~/.hms-mirror/data:/opt/hms-mirror/data \
  -v ~/.hms-mirror/logs:/var/log/hms-mirror \
  cloudera/hms-mirror:4.x.x

    Open http://localhost:8090/hms-mirror in your browser.

    Always Mount Persistent Volumes

    Without the -v volume mounts, all data (connections, jobs, configurations, reports, encryption keys) is stored inside the container and will be lost when the container is removed. Always mount host directories for /opt/hms-mirror/data and optionally /var/log/hms-mirror.

    Kerberos Support¶

    HMS-Mirror manages Kerberos authentication internally — keytabs are uploaded through the Web UI. The only external requirement is a valid krb5.conf pointing to the correct realm:

    mkdir -p ~/.hms-mirror/data ~/.hms-mirror/logs

docker run -d \
  --name hms-mirror \
  -p 8090:8090 \
  -v ~/.hms-mirror/data:/opt/hms-mirror/data \
  -v ~/.hms-mirror/logs:/var/log/hms-mirror \
  -v /etc/krb5.conf:/etc/krb5.conf:ro \
  cloudera/hms-mirror:4.x.x

    After starting, upload your keytab and configure the Kerberos principal through Preferences > Kerberos in the Web UI.

    Custom Truststore¶

    For clusters using self-signed certificates, mount a truststore:

    docker run -d \
  --name hms-mirror \
  -p 8090:8090 \
  -v ~/.hms-mirror/data:/opt/hms-mirror/data \
  -v ~/.hms-mirror/logs:/var/log/hms-mirror \
  -v /etc/krb5.conf:/etc/krb5.conf:ro \
  -v /path/to/truststore.jks:/etc/pki/truststore.jks:ro \
  -e JAVA_OPTS="-Djavax.net.ssl.trustStore=/etc/pki/truststore.jks -Djavax.net.ssl.trustStorePassword=changeit" \
  cloudera/hms-mirror:4.x.x

    Or upload certificates through the Web UI under Preferences > Certificates after starting.

    Configuration via Environment Variables¶

    docker run -d \
  --name hms-mirror \
  -p 8090:8090 \
  -v ~/.hms-mirror/data:/opt/hms-mirror/data \
  -v ~/.hms-mirror/logs:/var/log/hms-mirror \
  -e JAVA_OPTS="-Xmx8g -XX:+UseG1GC" \
  -e HMS_MIRROR_KEYSTORE_PASSWORD="your-production-password" \
  cloudera/hms-mirror:4.x.x
    Variable Default Description
    SERVER_PORT 8090 HTTP/HTTPS port
    JAVA_OPTS — JVM options (heap, GC, etc.)
    HMS_MIRROR_KEYSTORE_PASSWORD hms-mirror-dev-keystore-password Master password for encrypting credentials

    Docker Compose¶

    version: '3'
services:
  hms-mirror:
    image: cloudera/hms-mirror:4.x.x
    ports:
      - "8090:8090"
    volumes:
      - ./data:/opt/hms-mirror/data
      - ./logs:/var/log/hms-mirror
      - /etc/krb5.conf:/etc/krb5.conf:ro
    environment:
      - JAVA_OPTS=-Xmx8g -XX:+UseG1GC
      - HMS_MIRROR_KEYSTORE_PASSWORD=your-production-password
    restart: unless-stopped

    Managing the Container¶

    docker start hms-mirror       # Start
docker stop hms-mirror        # Stop
docker restart hms-mirror     # Restart
docker logs -f hms-mirror     # Follow logs
docker rm hms-mirror          # Remove (data preserved in mounted volumes)

    Docker Troubleshooting¶

    Issue Solution
    Container exits immediately Check logs: docker logs hms-mirror
    Port conflict Change the -p port mapping
    Data lost after restart Ensure volume mounts are configured (see Volumes)
    SSL connection failures Mount a truststore or upload certificates through the Web UI
    Out of memory Increase heap via JAVA_OPTS and Docker --memory limit

    Systemd Service¶

    For production Linux deployments without Cloudera Manager:

    # Create service file
sudo cat > /etc/systemd/system/hmsm-server.service << 'EOF'
[Unit]
Description=HMS-Mirror Server
After=network.target

[Service]
Type=forking
User=hms-mirror
Group=hms-mirror
WorkingDirectory=/home/hms-mirror/.hms-mirror
ExecStart=/home/hms-mirror/.hms-mirror/bin/hmsm-server start
ExecStop=/home/hms-mirror/.hms-mirror/bin/hmsm-server stop
PIDFile=/home/hms-mirror/.hms-mirror/hms-mirror.pid
Restart=on-failure
RestartSec=10
Environment="HMS_MIRROR_KEYSTORE_PASSWORD=your-production-password"

[Install]
WantedBy=multi-user.target
EOF

# Reload systemd
sudo systemctl daemon-reload

# Enable and start
sudo systemctl enable hmsm-server
sudo systemctl start hmsm-server

# Check status
sudo systemctl status hmsm-server

# View logs
sudo journalctl -u hmsm-server -f

    Building from Source¶

    Prerequisites¶

    Requirement Version
    Java JDK 17+
    Apache Maven 3.8+
    Node.js 18.x (LTS)
    npm 9.x+
    Git 2.x+

    Clone and Build¶

    git clone https://github.com/cloudera-labs/hms-mirror.git
cd hms-mirror

# Full build (includes frontend and documentation)
mvn clean package -DskipTests

# Build server module only
mvn clean package -pl server -am -DskipTests

# Build the CM parcel (after building server)
mvn clean package -pl parcel -am -DskipTests

    Project Structure¶

    hms-mirror/
├── pom.xml                 # Parent POM

│       ├── java/           # Java source code

    Build Artifacts¶

    After a successful build:

    server/target/
└── hms-mirror-server-4.x.x.x-server.jar    # Spring Boot executable JAR

csd/HMS_MIRROR/target/
└── HMS_MIRROR-4.x.x.x.jar                  # CSD JAR for Cloudera Manager

├── HMS_MIRROR-4.x.x.x-el9.parcel           # Parcel for EL9

    Running from Source¶

    # Using Maven (development)
cd server
mvn spring-boot:run

# Using the JAR directly
java -jar server/target/hms-mirror-server-4.x.x.x-server.jar

    Frontend Development¶

    For active frontend development with hot-reloading:

    1. Start the backend: 

      cd server && mvn spring-boot:run -Dspring-boot.run.arguments="--server.port=8080"

    2. In a separate terminal, start the Vite dev server: 

      cd server/src/main/frontend && npm run dev

    3. Access the application at http://localhost:3000/hms-mirror/

    First-Time Startup¶

    1. Start the server using any deployment method above

    2. Watch the startup logs: 

      INFO  - Starting HmsMirrorApplication v4.x.x
INFO  - RocksDB initialized at /home/user/.hms-mirror/data
INFO  - Tomcat started on port(s): 8090
INFO  - Started HmsMirrorApplication in 5.123 seconds

    3. Verify the server is running: 

      curl http://localhost:8090/hms-mirror/api/v1/system/health

    4. Open the Web UI at http://localhost:8090/hms-mirror

    5. Register your admin account — the first user registered is automatically granted the ADMIN role

    6. Set up connections — navigate to Connections and configure your source and target clusters

    Troubleshooting Installation¶

    Server Won't Start¶

    # Check if port is in use
lsof -i :8090

# Use a different port
java -jar hms-mirror-server.jar --server.port=8091

# Check Java version (17+ required)
java -version

    Cannot Connect to Clusters¶

    # Test HiveServer2 connectivity
telnet your-cluster.example.com 10000

# Test Metastore connectivity
telnet your-cluster.example.com 9083

    Kerberos Issues¶

    HMS-Mirror handles Kerberos authentication internally. Ensure:

    • /etc/krb5.conf exists and points to the correct KDC realm
    • Network connectivity to the KDC is available
    • The keytab is uploaded through the Web UI (Preferences > Kerberos)

    RocksDB Errors¶

    # Check directory permissions
ls -la ~/.hms-mirror/

# Fix permissions
chmod -R 755 ~/.hms-mirror/

# Backup and recreate (will lose saved configurations)
mv ~/.hms-mirror/data ~/.hms-mirror/data.backup
# Restart server to create fresh database
