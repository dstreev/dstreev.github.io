    Basic Triage¶

    Use this page to narrow down where the problem lives before collecting data or escalating.

    The Big Picture: Where Can the Problem Be?¶

    ┌────────────────────────────────────────────────────┐
│             Possible Problem Locations              │
├──────────────────┬─────────────────────────────────┤
│ HMS-Mirror       │ Server crash, config error, bug  │
│ application      │                                  │
├──────────────────┼─────────────────────────────────┤
│ Source cluster   │ Hive/HDFS unavailable, perms     │
├──────────────────┼─────────────────────────────────┤
│ Target cluster   │ Hive/HDFS unavailable, perms     │
├──────────────────┼─────────────────────────────────┤
│ Network          │ Firewall, DNS, latency           │
├──────────────────┼─────────────────────────────────┤
│ Customer config  │ Wrong credentials, wrong URI     │
└──────────────────┴─────────────────────────────────┘

    L1's job: Identify which bucket the problem is in. L2 diagnoses the specifics.

    Triage Question Script¶

    Ask the customer these questions in order:

    1. Is the server currently running?¶

    curl -s http://localhost:8090/hms-mirror/api/v1/system/health
    • UP → Server is fine, problem is elsewhere
    • DEGRADED / DOWN → Server has issues, escalate to L2
    • No response → Server not running — see Installation Verification

    2. What were they doing when the problem occurred?¶

    Answer Likely area
    Configuring a new connection Connection config issue
    Running a job Migration failure — collect job ID
    Just logging in Auth issue
    After upgrade Version/migration issue
    Randomly / always Server stability issue

    3. Is this the first time this has happened?¶

    • First time: New configuration or environmental change
    • Worked before, broke recently: Something changed — upgrade? cluster change? credentials expired?
    • Always fails: Likely a configuration error from the start

    4. Does the problem affect all jobs or just specific ones?¶

    • All jobs fail: Server-side or connection-level issue
    • One specific job: That job's configuration (dataset, connection, strategy choice)
    • Jobs with specific databases: Database-level permissions or access issue

    5. What error message does the customer see?¶

    Collect the exact error text. Common patterns:

    Error snippet Likely cause Next step
    Connection refused Service not running or firewall Check cluster service
    Authentication failed Wrong credentials or Kerberos Check credentials / kinit
    Permission denied User lacks Hive/HDFS permission Check ACLs on cluster
    Table already exists Target table exists Configuration setting
    OutOfMemoryError JVM heap too small L2 escalation
    RocksDB Database issue L2 escalation
    No suitable driver Missing JDBC driver JAR Driver installation
    SSL handshake TLS/certificate issue L2 escalation

    What L1 Can Resolve Directly¶

    Issue Resolution
    Server not running Guide through startup
    Account locked Admin unlocks via UI
    Wrong URL Correct to /hms-mirror context path
    Missing JDBC driver Guide to ~/.hms-mirror/aux_libs/ placement
    First-run setup Guide through First-Run Setup

    When to Stop Triaging and Escalate¶

    Escalate to L2 immediately (do not attempt further diagnosis) when:

    • Stack traces in the log that the customer can copy
    • RocksDB errors or corruption messages
    • Server keeps restarting (crash loop)
    • TLS/Kerberos/SSL errors
    • LDAP/OIDC authentication failures
    • All connections fail simultaneously after a working state

    Before escalating, complete the Data Collection Checklist.
