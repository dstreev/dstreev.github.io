    Data Collection Checklist¶

    Collect this information before every escalation. An incomplete data package will slow down L2 diagnosis and may require a second customer contact.

    Mandatory — Collect Before Every Escalation¶

    1. Environment Information¶

    Ask the customer or have them run:

    # HMS-Mirror version
cat ~/.hms-mirror/settings/web.properties | grep version
# OR check the JAR filename
ls ~/.hms-mirror/*.jar 2>/dev/null || ls /opt/hms-mirror/*.jar 2>/dev/null

# Java version
java -version 2>&1

# Operating system
uname -a

# Current working directory
echo $HMS_MIRROR_WORKING_DIR
# Default is: ~/.hms-mirror/

    Fill in the table:

    Field Value
    HMS-Mirror version
    Java version
    OS / distro
    Deployment mode (standalone / CM / Docker)
    Working directory path

    2. Problem Description¶

    Question Customer Answer
    What were you trying to do?
    When did the problem start?
    Has it ever worked?
    What changed recently?
    How many users are affected?

    3. Health Status¶

    curl -s http://localhost:8090/hms-mirror/api/v1/system/health

    Record the exact JSON response.

    4. Log Snippet¶

    # Last 200 lines of the main log
tail -200 ~/.hms-mirror/logs/hms-mirror.log

    If there are errors, also collect: 

    # Grep for all errors and warnings
grep -E "(ERROR|WARN)" ~/.hms-mirror/logs/hms-mirror.log | tail -50

    Job-Specific — Collect When a Migration Job Failed¶

    5. Job ID and Report¶

    1. Navigate to Jobs in the Web UI
    2. Find the failed job and note the Job ID (UUID)
    3. Download the full job report from the Reports tab

    Or from the filesystem: 

    ls ~/.hms-mirror/reports/
# Reports are organized by job ID
ls ~/.hms-mirror/reports/<job-id>/

    Reports contain: - conversion-result.yaml — full migration result with per-table status - *.html — human-readable report - SQL statements that were executed or generated

    6. Job Configuration¶

    From the Web UI: Jobs → [Job] → Export Configuration (download YAML)

    Or note: - Source connection name and platform type - Target connection name and platform type - Dataset: which databases and tables - Strategy used (SCHEMA_ONLY, SQL, EXPORT_IMPORT, etc.)

    Sanitize Credentials

    Before sending configuration, remove passwords from all JDBC URLs and credential fields. Replace with <REDACTED>.

    Connection-Specific — Collect When Connection Fails¶

    7. JDBC URL (Sanitized)¶

    Example: jdbc:hive2://hiveserver2.example.com:10000/default;principal=hive/hiveserver2.example.com@REALM
Remove: any ;password= parameters

    8. Driver Information¶

    ls ~/.hms-mirror/aux_libs/
ls ~/.hms-mirror/drivers/

    Security-Specific — Collect When Auth Fails¶

    9. Authentication Type¶

    grep "authentication" ~/.hms-mirror/settings/web.properties
    Auth Type Additional info needed
    local Username being used
    ldap LDAP server URL (no password), base DN
    oidc Provider URL, client ID (no secret)

    Escalation Package Summary¶

    Before submitting to L2, confirm you have:

    • HMS-Mirror version
    • Java version and OS
    • Exact error message or screenshot
    • tail -200 of the log file
    • Health endpoint response
    • Job ID (if applicable)
    • Job report files (if applicable)
    • Sanitized connection configuration
    • Authentication type in use
    • Description of what changed before the issue started
