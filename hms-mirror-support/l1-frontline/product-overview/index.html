    Product Overview¶

    Read this page before your first customer call. It gives you the vocabulary and context to ask the right questions.

    What is HMS-Mirror?¶

    HMS-Mirror (Hive Metastore Mirror) is a Hive table migration utility built by Cloudera. It copies Hive table definitions (metadata) and optionally their underlying data from one cluster to another.

    Common Use Cases¶

    Customer Scenario What They're Doing
    Upgrading HDP to CDP Moving tables from old cluster to new
    Moving to cloud (AWS, Azure, GCP) Migrating on-prem Hive to cloud object storage
    Cluster-to-cluster replication Copying a subset of tables to a second cluster
    Archiving / backup Dumping table definitions for safekeeping
    Testing environment creation Copying production tables to a dev cluster

    How the Product Works (Simplified)¶

    ┌──────────────────┐                    ┌──────────────────┐
│  SOURCE cluster  │                    │  TARGET cluster  │
│  (e.g., HDP)     │                    │  (e.g., CDP)     │
│                  │                    │                  │
│  Hive tables     │◄───HMS-Mirror────►│  Hive tables     │
│  (metadata +     │   reads source,    │  (created on     │
│   data files)    │   writes target    │   target)        │
└──────────────────┘                    └──────────────────┘
        │                                       ▲
        └──────── Data files (HDFS/S3) ─────────┘

    HMS-Mirror runs as a web server on a machine that can reach both clusters. Customers use the Web UI (browser) or REST API to:

    1. Configure cluster connections
    2. Define which databases/tables to migrate
    3. Choose a migration strategy
    4. Run the migration as a "Job"

    Key Concepts (Vocabulary)¶

    Term Meaning
    Connection How to reach a cluster — JDBC URL for HiveServer2, optional HCFS namespace for direct storage, platform type (HDP, CDP, cloud)
    Config (ConfigLite) Migration settings — ACID handling, Iceberg conversion, optimization flags
    Dataset Scope — which databases and tables to include/exclude
    Job Ties a Connection + Config + Dataset together with a chosen strategy. Running a job executes the migration.
    Strategy The migration approach: SCHEMA_ONLY (metadata only), SQL, EXPORT_IMPORT, HYBRID, STORAGE_MIGRATION, etc.
    Report The output of a job — a set of YAML/HTML files showing what succeeded, what failed, and the SQL that was executed
    RocksDB The embedded database that stores all configurations, jobs, users, and results

    What HMS-Mirror Is NOT¶

    • It is not a replication tool — changes made after a job runs are not continuously synced
    • It is not a Hadoop/Hive administrator — it cannot create clusters or fix cluster-side problems
    • It is not a data movement service — for large data moves it generates SQL or DistCP scripts to run on the cluster

    Deployment Modes¶

    Mode Description
    Standalone JAR java -jar hmsm-server-*.jar — developer/testing use
    Service script hmsm-server start/stop/restart — production on bare metal/VM
    Docker hmsm-server start-foreground — containerized
    Cloudera Manager CSD + Parcel — managed by CM lifecycle

    Accessing the Application¶

    Access Method URL
    Web UI http://host:8090/hms-mirror
    API http://host:8090/hms-mirror/api/v1/
    Health check http://host:8090/hms-mirror/api/v1/system/health
    Embedded docs http://host:8090/hms-mirror/docs/

    Context Path

    All URLs are prefixed with /hms-mirror. A request to the root / will return a 404. The correct base URL is /hms-mirror.

    Authentication Summary¶

    The product uses JWT (JSON Web Tokens) for authentication. Three modes are supported:

    Mode Description
    local Usernames/passwords stored in the embedded RocksDB database (default)
    ldap Delegated to an LDAP/Active Directory server
    oidc OAuth 2.0 / OpenID Connect (SSO)

    For most deployments you'll encounter, the mode is local.
