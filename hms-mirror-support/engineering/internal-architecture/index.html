    Internal Architecture¶

    Module Dependency Flow¶

    common  ──────────────────────────────────────────────────▶ server
  │                                                             │
  │ (shared domain models,                                      │
  │  DTOs, no Spring deps)                                      ▼
  │                                                            cli
  │                                                             │
  └─────────────────────────────────────────────────────▶ client
                                                    (REST client library)

    common has zero Spring dependencies — it's pure Java domain models. server is the Spring Boot application. cli wraps server for command-line use. client is a standalone REST client.

    Key Packages¶

    REST Layer (server/src/main/java/.../web/controller/api/v1/)¶

    Controller Handles
    AuthController Login, logout, refresh, change password
    UserController User CRUD, admin operations
    ConnectionController Connection CRUD, test
    ConfigController ConfigLite CRUD
    DatasetController Dataset CRUD
    JobController Job CRUD, run, history
    ReportController Report download
    StatusController Health, JVM, RocksDB status

    Service Layer (server/src/main/java/.../service/impl/)¶

    Services contain business logic. Each controller delegates to a service. Key interfaces:

    Interface Implementation Responsibility
    AuthService AuthServiceImpl Authentication, token issuance, lockout
    UserService UserServiceImpl User CRUD, password management
    ConnectionService ConnectionServiceImpl Connection CRUD, validation, testing
    JobService JobServiceImpl Job lifecycle, execution delegation
    HMSMirrorAppService HMSMirrorAppServiceImpl Migration engine orchestration

    Data Access Layer (server/src/main/java/.../repository/rocksDbImpl/)¶

    Each repository manages one type of entity in RocksDB. Key repositories:

    Repository Entity
    UserRepositoryRocksDbImpl User
    ConnectionRepositoryRocksDbImpl ConnectionDto
    ConfigRepositoryRocksDbImpl ConfigLiteDto
    DatasetRepositoryRocksDbImpl DatasetDto
    JobRepositoryRocksDbImpl JobDto

    Migration Strategies (server/src/main/java/.../datastrategy/)¶

    Each strategy is a Spring bean implementing a strategy interface:

    Strategy Class What It Does
    SchemaOnlyStrategy Copy DDL only
    SqlDataStrategy INSERT INTO SELECT *
    ExportImportStrategy Hive native EXPORT/IMPORT
    HybridStrategy Routes to SQL or EXPORT_IMPORT based on thresholds
    StorageMigrationStrategy Repoints tables to new storage location
    IcebergConversionStrategy Converts ORC/Parquet to Iceberg format
    LinkedStrategy Creates tables pointing at source storage
    CommonStrategy Tables sharing the same storage cluster
    DumpStrategy SQL script generation only (no execution)

    Domain Object Relationships¶

    JobDto
 ├── connectionRef: String  ────▶ ConnectionDto
 │                                  ├── hcfsNamespace
 │                                  ├── hs2Uri (JDBC)
 │                                  └── platform (HDP/CDP/CLOUD)
 ├── configRef: String      ────▶ ConfigLiteDto
 │                                  ├── acid settings
 │                                  ├── transfer settings
 │                                  └── optimization settings
 ├── datasetRef: String     ────▶ DatasetDto
 │                                  ├── databases[]
 │                                  ├── includeRegex
 │                                  └── excludeRegex
 └── strategy: DataStrategy  ──── (enum: SCHEMA_ONLY, SQL, ...)

ConversionResult (resolved at execution time)
 ├── Resolved ConnectionDto (source + target)
 ├── Resolved ConfigLiteDto
 ├── Resolved DatasetDto
 └── databases: Map<String, DBMirror>
      └── DBMirror
           └── tables: Map<String, TableMirror>
                └── TableMirror
                     ├── overall: MirrorStatus
                     └── actions: List<ActionResult>

    Security Filter Chain¶

    JwtAuthenticationFilter
    → Reads Authorization: Bearer or ?token= query param
    → Validates JWT, extracts user ID + roles
    → Sets Spring Security context

LoginRateLimitFilter
    → Rate-limits /api/v1/auth/login
    → 10 attempts per IP per minute (configurable)

Spring Security
    → @PreAuthorize annotations on controllers
    → Role checks: hasRole('ADMIN'), hasRole('USER'), etc.
    → Public endpoints: /api/v1/auth/login, /api/v1/system/health

    Jackson / YAML Serialization¶

    HMS-Mirror uses Jackson for both JSON (REST API) and YAML (configuration files). The YAML code point limit is set to 100MB (vs. SnakeYAML's 3MB default) to handle large migration reports:

    hms-mirror.config.yaml.codePointLimit=104857600

    Frontend Integration¶

    The React build is embedded in the JAR as static resources under: 

    server/src/main/resources/static/

    Spring Boot serves static files, and the ReactResourceConfig class has a custom PathResourceResolver that handles MkDocs-style use_directory_urls: true path patterns — requests to /hms-mirror/docs/some/path/ resolve to /hms-mirror/docs/some/path/index.html.

    Logback Configuration¶

    Log configuration is at server/src/main/resources/logback-hms-spring.xml. Early-init logging before Spring context loads uses JVM system properties:

    -DLOGGING_THRESHOLD=INFO
-DLOGGING_FILE_PREFIX=hms-mirror
-DLOGGING_DIR=~/.hms-mirror/logs

    These are set automatically by the hmsm-server startup script.
