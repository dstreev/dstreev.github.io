    Security Architecture¶

    Authentication Layers¶

    HMS-Mirror has three authentication modes:

    Mode Class When Used
    local LocalAuthService Default — credentials in RocksDB
    ldap LdapAuthService Enterprise AD/LDAP integration
    oidc OidcAuthService OAuth 2.0 / SSO

    Mode is selected by hms-mirror.authentication.type in web.properties. All modes produce JWT tokens after successful authentication — the rest of the system is token-agnostic.

    JWT Architecture¶

    Token Types¶

    Type Lifetime Purpose
    Access Token 15 minutes Authorize API requests
    Refresh Token 7 days Exchange for new access token

    Token Structure¶

    JWT tokens contain (in the payload): 

    {
  "sub": "<user-id-uuid>",
  "username": "alice",
  "roles": ["USER", "ADMIN"],
  "type": "access",
  "iat": 1740564000,
  "exp": 1740564900
}

    Token Processing — JwtAuthenticationFilter¶

    protected void doFilterInternal(...) {
    String jwt = extractJwtFromRequest(request);  // Bearer header or ?token= param

    if (jwt != null && jwtTokenProvider.validateToken(jwt)) {
        String tokenType = jwtTokenProvider.getTokenType(jwt);
        if ("access".equals(tokenType)) {
            // Build JwtUserPrincipal from token claims
            // Set into SecurityContextHolder
        }
    }
    filterChain.doFilter(request, response);
}

    The ?token= query parameter is supported for SSE endpoints (browser EventSource API cannot set custom headers). Short-lived 15-minute access tokens limit log-exposure risk.

    JWT Secret¶

    Configured via hms-mirror.jwt.secret. If empty, a random secret is generated at startup — all sessions are invalidated on restart.

    For production: use a long (64+ character), random, persistent secret: 

    hms-mirror.jwt.secret=<generate-with: openssl rand -base64 64>

    Encryption Keystore¶

    Sensitive values (connection passwords, etc.) are encrypted at rest using AES/GCM with keys from a PKCS12 keystore.

    ~/.hms-mirror/security/hms-mirror.p12

    Key hierarchy: 

    PKCS12 Keystore
    └── Master encryption key (AES-256)
            └── Used to encrypt/decrypt sensitive fields in RocksDB

    Configuration: 

    hms-mirror.keystore.path=~/.hms-mirror/security/hms-mirror.p12
hms-mirror.keystore.password=<keystore-password>

    Authorization Model¶

    Roles¶

    Role Description
    USER Standard access — can view/run jobs, manage own profile
    ADMIN Full access — manage users, reset passwords, all CRUD
    DATA_BROWSER Additional permission for data browsing features

    Endpoint Authorization¶

    Controllers use Spring Security's @PreAuthorize annotations:

    @PreAuthorize("hasRole('ADMIN')")  // Admin-only
@PreAuthorize("hasRole('USER')")   // Any authenticated user
@PreAuthorize("isAuthenticated()")  // Any authenticated user
// No annotation = public endpoint

    Public (no auth required) endpoints: - POST /api/v1/auth/login - GET /api/v1/system/health

    Rate Limiting¶

    LoginRateLimitFilter implements in-memory rate limiting on the login endpoint:

    • 10 attempts per IP per minute (configurable)
    • Returns 429 Too Many Requests when exceeded
    • Resets per window (not sliding window)

    Account lockout (separate from rate limiting): - 5 consecutive failed login attempts → account locked - Admin or system auto-unlock required - Lockout tracked per user, not per IP

    Security Event Logging¶

    The following events are logged (in addition to normal application logs):

    Event Level Logger
    Successful login INFO AuthServiceImpl
    Failed login (with attempt count) WARN AuthServiceImpl
    Account locked WARN AuthServiceImpl
    JWT validation failure WARN JwtAuthenticationFilter
    Rate limit exceeded WARN LoginRateLimitFilter
    Admin operations INFO Various

    Threat Model Summary¶

    The formal threat model is at docs/sdlc/threat-model.md. Key design decisions that impact security:

    Decision Rationale
    JWT access tokens are 15 minutes Limits window for stolen token abuse
    ?token= query param for SSE EventSource cannot set headers; short tokens limit log exposure
    AES/GCM for credential encryption Authenticated encryption prevents tampering
    PKCS12 keystore Standard Java keystore format; widely tooled
    Rate limiting on login Mitigates brute-force attacks
    Account lockout after 5 failures Defense in depth with rate limiting
    No plaintext passwords stored BCrypt hashing with salt

    Local Authentication: Password Policy¶

    Passwords stored via BCrypt (BCryptPasswordEncoder, strength 10).

    For createUser API (admin-created users): - If password supplied: encoded and stored - If no password: 16-character random password generated (alphanumeric + symbols), returned in response as generatedPassword (shown once, not stored in plaintext)

    Password reset flow: - Admin-initiated: PUT /api/v1/users/{id}/admin-reset-password — sets new password, clears lockout - Self-service: PUT /api/v1/auth/change-password — requires current password
