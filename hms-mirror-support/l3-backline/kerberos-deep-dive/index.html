    Kerberos Deep Dive¶

    Kerberos is a common source of obscure failures. This page goes beyond the basic kinit fix to cover realm issues, keytab management, and HMS-Mirror-specific Kerberos configuration.

    Kerberos Concepts Refresher¶

    User / Service          KDC (Key Distribution Center)
    │                           │
    ├──── AS-REQ (kinit) ───────►│
    │◄─── AS-REP (TGT) ──────────┤
    │                            │
    ├──── TGS-REQ (use TGT) ────►│
    │◄─── TGS-REP (service ticket)│
    │                            │
    ├──── AP-REQ (to Hive) ─────────────────►│ HiveServer2
    │◄─── AP-REP (authenticated) ────────────┤

    HMS-Mirror acts as a Kerberos client (equivalent to a user) connecting to HiveServer2 as a Kerberos service.

    Kerberos Configuration Locations¶

    # Kerberos client config
cat /etc/krb5.conf

# List current tickets
klist -e

# Check keytab contents
klist -kt /path/to/service.keytab

    HMS-Mirror Kerberos Configuration¶

    In the Connection JDBC URI: 

    jdbc:hive2://hiveserver2.example.com:10000/default;principal=hive/hiveserver2.example.com@REALM

    For Hadoop Kerberos config, the server needs access to: - krb5.conf (usually /etc/krb5.conf) - A valid ticket or keytab

    In web.properties or startup JVM args: 

    -Djava.security.krb5.conf=/etc/krb5.conf
-Djavax.security.auth.useSubjectCredsOnly=false

    Common Kerberos Errors and Root Causes¶

    "GSS initiate failed"¶

    javax.security.sasl.SaslException: GSS initiate failed [Caused by GSSException:
No valid credentials provided (Mechanism level: Failed to find any Kerberos tgt)]

    Root cause: No Kerberos TGT (Ticket Granting Ticket) available.

    Fix: 

    # Check if ticket exists and is valid
klist

# Get a new ticket
kinit user@REALM.EXAMPLE.COM
# or with keytab (non-interactive)
kinit -kt /path/to/service.keytab user@REALM.EXAMPLE.COM

    "Clock skew too great"¶

    KrbException: Clock skew too great (37) - PROCESS_TGS

    Root cause: Time difference between the HMS-Mirror host and the KDC is greater than 5 minutes (Kerberos default tolerance).

    Fix: 

    # Check current time on HMS-Mirror host
date -u

# Sync with NTP
sudo ntpdate -u <ntp-server>
# or on systemd systems
sudo timedatectl set-ntp true
sudo chronyc makestep

    "Server not found in Kerberos database"¶

    GSSException: No valid credentials provided
KrbException: Server not found in Kerberos database (7)

    Root cause: The service principal in the JDBC URI doesn't match what's registered in the KDC.

    Diagnosis: 

    # Check what principal HiveServer2 is using
# (on the HiveServer2 host, or ask the Hadoop admin)
grep "principal" /etc/hive/conf/hive-site.xml

# Verify the principal in the JDBC URI matches exactly
# jdbc:hive2://host:10000/default;principal=hive/host.example.com@REALM

    "KDC unreachable"¶

    KrbException: Cannot locate KDC for realm REALM.EXAMPLE.COM

    Root cause: Cannot reach port 88 (UDP/TCP) of the KDC.

    Diagnosis: 

    # Test KDC connectivity (UDP)
nc -zuv kdc.realm.example.com 88

# Check krb5.conf for KDC address
grep kdc /etc/krb5.conf

# Attempt kinit with verbose output
kinit -V user@REALM

    "Integrity check on decrypted field failed"¶

    KrbException: Integrity check on decrypted field failed (31)

    Root cause: Password/keytab mismatch, or the principal's password was changed after the keytab was generated.

    Fix: Regenerate the keytab from the KDC (requires Hadoop admin involvement).

    Keytab Management¶

    Keytabs are files containing pre-encrypted Kerberos credentials — they allow non-interactive kinit.

    Verify a Keytab¶

    # List principals in a keytab
klist -kt /path/to/service.keytab

# Test authentication with keytab
kinit -kt /path/to/service.keytab principal@REALM

# Confirm the resulting ticket
klist

    Configure HMS-Mirror to Use a Keytab¶

    For production deployments, configure a keytab so the server can authenticate automatically:

    In web.properties: 

    hms-mirror.kerberos.keytab=/path/to/hms-mirror.keytab
hms-mirror.kerberos.principal=hms-mirror/host.example.com@REALM

    Or pass as JVM args: 

    -Dhms-mirror.kerberos.keytab=/path/to/service.keytab
-Dhms-mirror.kerberos.principal=hms-mirror/host@REALM

    Multi-Realm Kerberos¶

    If source and target clusters are in different Kerberos realms, cross-realm trust must be configured at the KDC level. This is outside HMS-Mirror's control — escalate to the customer's Kerberos/security administrator if cross-realm trust is not set up.

    Symptom: Authentication works for source cluster but not target (or vice versa).

    Verification: 

    # Verify realm in target JDBC URI principal
# The principal realm must match the realm the KDC trusts

# List realm-specific configuration in krb5.conf
grep -A 10 "\[realms\]" /etc/krb5.conf
