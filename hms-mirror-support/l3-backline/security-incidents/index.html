    Security Incidents¶

    This page covers response procedures for suspected security incidents involving HMS-Mirror.

    Types of Security Incidents¶

    Incident Type Severity Response
    Unauthorized admin access Critical Immediate — change credentials, audit, investigate
    Account brute-force in progress High Rate limiter should be active; verify and escalate
    Suspected credential leak High Rotate credentials immediately
    Unauthorized job execution High Suspend the account, audit job history
    JWT token abuse Medium Revoke all sessions (rotate JWT secret)
    Privilege escalation (user → admin) Critical Audit user records, investigate

    Immediate Response: Unauthorized Access¶

    Step 1: Contain¶

    If unauthorized access is confirmed or strongly suspected:

    # Stop the server immediately to prevent further access
/path/to/hmsm-server stop

    Warning

    Stopping the server severs all active sessions. Notify the customer before doing this unless active data exfiltration is in progress.

    Step 2: Preserve Evidence¶

    Before making any changes, capture the current state:

    # Capture full log
cp ~/.hms-mirror/logs/hms-mirror.log /tmp/incident-$(date +%Y%m%d-%H%M)-audit.log

# Capture log with authentication events
grep -E "(login|auth|jwt|token|Admin|USER|ADMIN)" ~/.hms-mirror/logs/hms-mirror.log \
  > /tmp/incident-auth-events.log

    Step 3: Identify the Compromise Vector¶

    Look for:

    # Failed then successful logins (brute force succeeded?)
grep -E "Authentication failed|logged in successfully" ~/.hms-mirror/logs/hms-mirror.log | \
  grep -B 5 "logged in" | head -50

# Unusual IPs (if IP logging is enabled in a load balancer / proxy in front)
grep "logged in successfully" ~/.hms-mirror/logs/hms-mirror.log

# Admin operations performed
grep -E "ADMIN|createUser|deleteUser|setAdmin" ~/.hms-mirror/logs/hms-mirror.log | tail -50

# Jobs executed
grep "Starting job\|Job started" ~/.hms-mirror/logs/hms-mirror.log | tail -20

    Revoking All Active Sessions¶

    JWT tokens cannot be individually revoked (they are stateless). The only way to revoke ALL active sessions is to rotate the JWT secret:

    # Stop the server
/path/to/hmsm-server stop

# In web.properties, change or remove the JWT secret
# If removed, a new random secret is generated on next start
# All existing tokens become invalid

# Edit the secret
vi ~/.hms-mirror/settings/web.properties
# Change: hms-mirror.jwt.secret=<new-long-random-value>

# Restart
/path/to/hmsm-server start

    After this, all users (including legitimate ones) must log in again.

    Account Lockout Investigation¶

    If an account is reporting being "unexpectedly locked":

    Verify Current Lockout State via API¶

    # As admin, check user details
TOKEN=$(curl -s -X POST http://localhost:8090/hms-mirror/api/v1/auth/login \
  -H "Content-Type: application/json" \
  -d '{"usernameOrEmail":"admin","password":"adminpass"}' | \
  python3 -c "import sys,json; print(json.load(sys.stdin)['accessToken'])")

# Get user list with lockout status
curl -s http://localhost:8090/hms-mirror/api/v1/users \
  -H "Authorization: Bearer $TOKEN" | python3 -m json.tool

    Find Failed Login Events for a User¶

    grep "Authentication failed\|Failed login" ~/.hms-mirror/logs/hms-mirror.log | \
  grep "<username>" | tail -20

    Unlock via API¶

    # PUT /api/v1/users/{userId}/unlock
curl -X PUT http://localhost:8090/hms-mirror/api/v1/users/<userId>/unlock \
  -H "Authorization: Bearer $TOKEN"

    Privilege Escalation Audit¶

    If you suspect a user gained admin privileges they shouldn't have:

    # Check current admin users via API
curl -s http://localhost:8090/hms-mirror/api/v1/users \
  -H "Authorization: Bearer $TOKEN" | \
  python3 -c "import sys,json; users=json.load(sys.stdin); [print(u['username'], u.get('roles','')) for u in users if 'ADMIN' in str(u.get('roles',''))]"

# Check log for admin-granting events
grep -i "admin\|ROLE_ADMIN\|setAdmin" ~/.hms-mirror/logs/hms-mirror.log | tail -30

    Post-Incident¶

    After the incident is contained and investigated:

    1. Change all credentials — admin password, any configured service account passwords
    2. Rotate the JWT secret — invalidate all sessions
    3. Review user accounts — remove accounts that shouldn't exist
    4. Enable debug logging temporarily — to monitor for repeat attempts
    5. Document — what happened, timeline, remediation taken
    6. Escalate to Engineering if there is evidence of a software vulnerability (not just credential misuse)
