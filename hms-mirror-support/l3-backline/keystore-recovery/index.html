    Keystore Recovery¶

    HMS-Mirror uses two keystores:

    Keystore Purpose Location
    Encryption keystore Encrypts sensitive data at rest in RocksDB (passwords, credentials) ~/.hms-mirror/security/hms-mirror.p12
    TLS keystore HTTPS certificate and key for the web server ~/.hms-mirror/certificates/

    This page covers the encryption keystore. For TLS, see TLS/SSL Deep Dive.

    Understanding the Encryption Keystore¶

    The encryption keystore stores the master encryption keys used to encrypt: - Connection passwords stored in RocksDB - Other sensitive configuration values

    Configuration in web.properties: 

    hms-mirror.keystore.path=~/.hms-mirror/security/hms-mirror.p12
hms-mirror.keystore.password=changeit

    Keystore Loss = Encrypted Data Loss

    If the keystore is lost and no backup exists, any encrypted values in RocksDB (connection passwords, etc.) cannot be decrypted. The data structure is intact but encrypted fields are unrecoverable without the key.

    Keystore File Verification¶

    # Check the keystore file exists and is not empty
ls -la ~/.hms-mirror/security/

# Verify the keystore is readable with the configured password
keytool -list \
  -keystore ~/.hms-mirror/security/hms-mirror.p12 \
  -storetype PKCS12 \
  -storepass changeit

# Expected output shows the key entries

    Backup the Keystore¶

    # Create a backup with timestamp
cp ~/.hms-mirror/security/hms-mirror.p12 \
   ~/.hms-mirror/security/hms-mirror.p12.backup.$(date +%Y%m%d)

# Store this backup in a SECURE location off the server
# (encrypted storage, secrets vault, secure S3 bucket with server-side encryption)

    Backup schedule recommendation: After every configuration change that involves passwords, and at minimum weekly.

    Scenario: Keystore File Missing¶

    If the keystore file is missing:

    ls: cannot access ~/.hms-mirror/security/hms-mirror.p12: No such file or directory

    And you have a backup:

    cp /path/to/backup/hms-mirror.p12 ~/.hms-mirror/security/hms-mirror.p12
chmod 600 ~/.hms-mirror/security/hms-mirror.p12
/path/to/hmsm-server start

    And you do NOT have a backup:

    The encrypted fields in RocksDB are permanently unrecoverable. The procedure is:

    1. Stop the server
    2. Accept that connection passwords will need to be re-entered
    3. Start the server — HMS-Mirror will auto-generate a new keystore
    4. Re-configure all connection passwords in the UI
    5. Re-test all connections

    Scenario: Wrong Keystore Password¶

    If the password in web.properties doesn't match the keystore file:

    WARN  Keystore: Failed to load keystore — UnrecoverableKeyException

    Fix:

    1. If you know the correct password, update web.properties: 

      hms-mirror.keystore.password=correct-password

    2. If the password is unknown and no backup exists — same outcome as lost keystore above.

    Scenario: Keystore Corruption¶

    java.security.KeyStoreException: Failed to load keystore data

    Try restoring from backup. If no backup:

    1. Stop the server
    2. Move the corrupt keystore: 
      mv ~/.hms-mirror/security/hms-mirror.p12 \
   ~/.hms-mirror/security/hms-mirror.p12.corrupt.$(date +%Y%m%d)
    3. Start the server — a new keystore will be generated
    4. Re-enter all connection passwords

    Changing the Keystore Password¶

    # Change the keystore password
keytool -storepasswd \
  -keystore ~/.hms-mirror/security/hms-mirror.p12 \
  -storetype PKCS12 \
  -storepass oldpassword \
  -new newpassword

# Update web.properties to match
sed -i 's/hms-mirror.keystore.password=.*/hms-mirror.keystore.password=newpassword/' \
  ~/.hms-mirror/settings/web.properties

# Restart
/path/to/hmsm-server restart

    Best Practices for Key Management¶

    1. Back up the keystore immediately after first startup — before configuring any connections
    2. Store the keystore backup and password separately — in different secure systems
    3. Test restoration — periodically verify the backup can be restored
    4. Use a secrets manager in production (HashiCorp Vault, AWS Secrets Manager, etc.) to store the keystore password instead of plain text in web.properties
