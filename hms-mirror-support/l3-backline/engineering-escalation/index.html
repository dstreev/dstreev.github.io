    Engineering Escalation Package¶

    When L3 cannot resolve an issue and suspects a code defect, an incomplete escalation package will delay engineering investigation. This page defines the complete package.

    When to Escalate to Engineering¶

    Trigger Required Evidence
    Reproducible bug (not config/env issue) Exact steps to reproduce
    Unexpected API response (not 4xx from bad input) curl example with exact request/response
    RocksDB corruption with no environmental cause Data directory backup + full log
    Performance regression between versions Before/after comparison
    Security vulnerability suspected Private channel — do NOT file a public ticket
    Data loss or corruption in migration results Job ID, report files, SQL executed

    Complete Engineering Escalation Package¶

    1. Bug Report Title Format¶

    [BUG] [v<version>] <Concise one-line description>

Examples:
[BUG] [v4.1.5] EXPORT_IMPORT jobs fail silently for ACID tables with no error logged
[BUG] [v4.1.5] createUser API returns 500 when email contains a + character
[BUG] [v4.1.5] RocksDB corruption after ungraceful shutdown on Docker restart

    2. Environment¶

    HMS-Mirror version: 4.1.5.8  (exact — from JAR filename or API)
Java version: 17.0.9+9 (OpenJDK)
OS: RHEL 8.9
Deployment: Cloudera Manager CSD, parcel 4.1.5.8-el8
Working dir: /var/lib/hms-mirror/.hms-mirror

    3. Exact Reproduction Steps¶

    Number every step. Include exact commands, exact UI actions:

    1. Log in as admin (local auth)
2. Navigate to Connections → Create New
3. Enter: Name=TestConn, Platform=HDP, JDBC=jdbc:hive2://host:10000/default
4. Click "Test Connection"
5. Observe: 500 Internal Server Error
   Expected: "Connection successful" or a meaningful error

Frequency: 100% reproducible with this exact URI format

    4. Observed vs. Expected¶

    Description
    Observed HTTP 500 with no error message in response body
    Expected HTTP 400 with a human-readable error explaining why the URI is invalid, or HTTP 200 if the connection should succeed

    5. Relevant Log Excerpt¶

    Only the relevant lines — not the entire log. For exceptions, include the full stack trace:

    2026-02-26 10:15:33.421 ERROR c.c.u.h.m.service.ConnectionService - Unexpected error testing connection: TestConn
java.lang.NullPointerException: Cannot invoke "String.startsWith(String)" because "jdbcUrl" is null
    at com.cloudera.utils.hms.mirror.service.impl.ConnectionServiceImpl.validateUri(ConnectionServiceImpl.java:142)
    at com.cloudera.utils.hms.mirror.service.impl.ConnectionServiceImpl.testConnection(ConnectionServiceImpl.java:87)
    ... (truncated)

    6. API Request/Response (if applicable)¶

    # Request
curl -s -X POST \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{"name":"TestConn","platform":"HDP","jdbcUrl":"jdbc:hive2://host:10000/default"}' \
  http://localhost:8090/hms-mirror/api/v1/connections/test

# Response (500)
{"timestamp":"2026-02-26T10:15:33.421+00:00","status":500,"error":"Internal Server Error","path":"/hms-mirror/api/v1/connections/test"}

    7. Attached Files¶

    File Required? Why
    Last 500 lines of log Mandatory Context and stack trace
    Thread dump For hang/deadlock bugs Current thread state
    Heap dump For OOM bugs Memory analysis
    Job report files For migration bugs Execution details
    RocksDB backup For data/startup bugs State at time of failure
    Sanitized web.properties For config-dependent bugs Env reproduction

    Security Vulnerability Report¶

    For suspected security vulnerabilities, use a private channel — do NOT file a public GitHub issue.

    Include in the private report: 1. CVE category (if known): OWASP Top 10 category 2. Description: what can an attacker do? 3. Prerequisites: what access does an attacker need? 4. Proof of concept: minimal steps to demonstrate (not to exploit) 5. Severity assessment: CVSS score estimate if possible 6. Suggested remediation: your recommendation

    Patch Verification¶

    After engineering releases a fix, L3 verifies:

    1. The exact steps that previously reproduced the bug no longer reproduce it
    2. No regression in adjacent functionality
    3. The fix is confirmed on the customer's exact environment (version, OS, deployment mode)
    4. Document: "Verified fixed in version X.Y.Z.B on [date] by [L3 engineer]"
