    RocksDB Diagnostics¶

    RocksDB is the embedded key-value database that stores all HMS-Mirror state. Problems here directly prevent the application from functioning.

    RocksDB Location¶

    # Default location
~/.hms-mirror/data/

# If working directory overridden
${HMS_MIRROR_WORKING_DIR}/data/

    Signs of RocksDB Problems¶

    Log Pattern Severity
    RocksDB: Could not acquire lock Low — another instance running or stale lock
    RocksDB: Corruption detected High — data integrity issue
    RocksDB: IO error Medium — filesystem or disk issue
    RocksDB: No space left on device Medium — disk full
    Application starts but data is missing Medium — column family may be missing

    Quick Health Check¶

    # Check if RocksDB directory exists and has content
ls -la ~/.hms-mirror/data/
du -sh ~/.hms-mirror/data/

# Key files that should exist
ls ~/.hms-mirror/data/CURRENT    # RocksDB manifest pointer
ls ~/.hms-mirror/data/MANIFEST-* # Change log
ls ~/.hms-mirror/data/OPTIONS-*  # RocksDB options
ls ~/.hms-mirror/data/LOCK       # Exclusive lock file (exists only while running)

    Stale Lock File¶

    Symptom: Server reports "Could not acquire lock" even though no other instance is running.

    Cause: Previous instance crashed without releasing the lock file.

    Resolution:

    # 1. Confirm no HMS-Mirror processes are running
ps aux | grep hmsm-server

# 2. Only if NO process is running — remove the lock
rm -f ~/.hms-mirror/data/LOCK

# 3. Restart the server
/path/to/hmsm-server start

    Do Not Remove Lock While Server is Running

    Only remove the LOCK file if you have confirmed the server process is NOT running. Removing it while running will corrupt the database.

    Disk Space Recovery¶

    RocksDB writes compaction files that can accumulate. If disk is full:

    # Check total usage
du -sh ~/.hms-mirror/data/

# Check overall disk
df -h ~/.hms-mirror/

# Check for large SST files (RocksDB data files)
find ~/.hms-mirror/data/ -name "*.sst" -size +100M | sort -k5 -rn

# Log files in the data directory can also be deleted safely (after stopping server)
# These are RocksDB WAL (Write-Ahead Log) files ending in .log
find ~/.hms-mirror/data/ -name "*.log" -not -name "hms-mirror*"

    After clearing space, start the server and let RocksDB compact on its own.

    Backup Before Any Recovery¶

    Always backup before attempting recovery:

    # Stop the server first
/path/to/hmsm-server stop

# Create timestamped backup
cp -r ~/.hms-mirror/data/ ~/.hms-mirror/data.backup.$(date +%Y%m%d-%H%M)/

# Verify backup
ls -la ~/.hms-mirror/data.backup.$(date +%Y%m%d-%H%M)/

    Recovery Procedures¶

    Option 1: Restart (Fixes Many Transient Issues)¶

    /path/to/hmsm-server stop
sleep 5
/path/to/hmsm-server start

    RocksDB performs integrity checks on startup and can often self-repair minor issues.

    Option 2: Database Reset (Last Resort)¶

    DATA LOSS

    This procedure permanently deletes all user accounts, connections, configurations, jobs, and reports. Only use if all other options are exhausted and the customer accepts the consequences.

    1. Stop the server
    2. Export any recoverable data via the UI (connections, configs, datasets) if the server can still start
    3. Move the data directory:
    /path/to/hmsm-server stop
mv ~/.hms-mirror/data/ ~/.hms-mirror/data.broken.$(date +%Y%m%d-%H%M)/
    1. Start the server — a fresh RocksDB will be created:
    /path/to/hmsm-server start
    1. Re-create the initial admin account (see first-run-setup)
    2. Re-import any exported configurations

    Column Family Recovery¶

    HMS-Mirror uses RocksDB column families to separate data by type (users, connections, jobs, etc.). If one column family is corrupted but others are intact, partial data recovery may be possible. This requires Engineering involvement — escalate with:

    • Full RocksDB data directory backup
    • List of column families affected
    • Error messages from the log

    RocksDB Metrics via API¶

    If the server is running, you can query RocksDB status:

    TOKEN=$(curl -s -X POST http://localhost:8090/hms-mirror/api/v1/auth/login \
  -H "Content-Type: application/json" \
  -d '{"usernameOrEmail":"admin","password":"adminpass"}' | python3 -c "import sys,json; print(json.load(sys.stdin)['accessToken'])")

curl -s http://localhost:8090/hms-mirror/api/v1/status/rocksdb \
  -H "Authorization: Bearer $TOKEN" | python3 -m json.tool
