    Connection Troubleshooting¶

    A large fraction of HMS-Mirror support cases involve connection failures. This page covers each connection type systematically.

    Connection Types¶

    HMS-Mirror establishes two types of connections to each cluster:

    Connection Type Purpose Protocol
    HiveServer2 (HS2) Execute Hive DDL/DML, read metadata JDBC
    HCFS (Hadoop-Compatible File System) List directories, validate storage paths HDFS/WebHDFS/S3/ADLS

    Step 1: Test the Connection in the UI¶

    The Web UI has a built-in connection tester:

    1. Connections → [Connection] → Test Connection

    If the test fails, the error message will be displayed in the UI. Copy this message before diving into logs.

    HS2 / JDBC Connection Issues¶

    Symptom: "Connection refused"¶

    java.net.ConnectException: Connection refused (connection refused)

    Checklist:

    • Is HiveServer2 actually running on the source cluster?
    • Is the hostname correct? Test DNS resolution: nslookup <hostname>
    • Is the port correct? Default is 10000 (binary) or 10001 (HTTP mode)
    • Is a firewall blocking the connection? Test: telnet <hostname> 10000
    • Is the transportMode in the JDBC URI matching the server config?

    JDBC URI format check:

    # Binary transport (default)
jdbc:hive2://hiveserver2.example.com:10000/default

# HTTP transport (through Knox, load balancer, or HS2 in HTTP mode)
jdbc:hive2://hiveserver2.example.com:10001/default;transportMode=http;httpPath=/cliservice

# Knox gateway
jdbc:hive2://knox.example.com:8443/;ssl=true;transportMode=http;httpPath=gateway/cdp-proxy-api/hive

    Symptom: "Authentication failed" (non-Kerberos)¶

    javax.security.sasl.SaslException: Authentication failed
    • Verify the username and password in the connection configuration
    • Check if the Hive user has login permissions
    • For Knox: verify Knox credentials (separate from Hive credentials)

    Kerberos Authentication Failures¶

    Kerberos is common in CDP and secured HDP environments. It's also the most complex to debug.

    Quick Kerberos Check¶

    # On the HMS-Mirror host, check for a valid ticket
klist

# Expected output:
# Credentials cache: FILE:/tmp/krb5cc_1000
# Principal: hive-client@REALM.EXAMPLE.COM
# Valid starting     Expires            Service principal
# 02/26/2026 10:00  02/26/2026 20:00   krbtgt/REALM@REALM

    Common Kerberos Errors¶

    Error Cause Fix
    GSS initiate failed No valid Kerberos ticket Run kinit
    Clock skew too great Time difference > 5 minutes between hosts Sync NTP on HMS-Mirror host
    KDC unreachable Cannot reach KDC (port 88) Check network/firewall to KDC
    Client not found in Kerberos database Wrong principal name Verify principal format
    Server not found in Kerberos database Service principal doesn't exist in KDC Check the Hive service principal
    Pre-authentication data required Need password for kinit Use keytab: kinit -kt file.keytab principal@REALM

    Obtain a Kerberos Ticket for Testing¶

    # With password
kinit user@REALM.EXAMPLE.COM

# With keytab (non-interactive, for service accounts)
kinit -kt /path/to/service.keytab hive-client@REALM.EXAMPLE.COM

# Verify the ticket
klist

    Clock Skew Fix¶

    # Check time difference between HMS-Mirror host and KDC
# (ask customer to show output of both)
date

# Sync time
sudo ntpdate -u <ntp-server>
# or
sudo chronyc makestep

    HCFS / Storage Connection Issues¶

    HMS-Mirror optionally connects to HDFS or cloud storage to validate file paths and storage namespaces.

    HDFS Connection¶

    The HDFS namespace is configured as the "Default FS" in the connection:

    hdfs://nameservice/
# or
hdfs://namenode.example.com:8020/

    Common failures:

    Error Cause
    No such file or directory Wrong namespace path
    Permission denied HDFS user lacks read permission
    Name node is in standby mode NameNode HA failover — retry
    Network timeout Firewall blocking HDFS port 8020/50070

    Cloud Storage (S3/ADLS/GCS)¶

    For cloud storage, the "Default FS" looks like:

    s3a://bucket-name/
abfs://container@account.dfs.core.windows.net/
gs://bucket-name/

    Common failures:

    Error Cause
    Access Denied Wrong or expired cloud credentials
    NoSuchBucket Bucket name wrong or doesn't exist
    Endpoint not configured Cloud endpoint not set in Hadoop config

    Driver Availability¶

    Many connection failures at the JDBC level are actually missing driver failures:

    java.sql.SQLException: No suitable driver found for jdbc:hive2://...

    Solution: The appropriate JDBC driver JAR must be in ~/.hms-mirror/aux_libs/ or ~/.hms-mirror/drivers/.

    Platform Driver JAR
    HDP Hive 2/3 hive-jdbc-standalone-*.jar
    CDP Hive hive-jdbc-*.jar
    Hive Metastore (direct) mysql-connector-java-*.jar (if MySQL backend)
    Hive Metastore (direct) postgresql-*.jar (if PostgreSQL backend)

    After placing the JAR, restart HMS-Mirror — drivers are loaded at startup.

    Connection Test Checklist¶

    Before assuming a bug, verify:

    • Hostname resolves from the HMS-Mirror host (nslookup <hostname>)
    • Port is reachable (telnet <hostname> <port> or nc -zv <hostname> <port>)
    • Kerberos ticket is valid (klist) if using Kerberos
    • Correct driver JAR is in aux_libs/
    • transportMode in URI matches server (binary vs http)
    • SSL settings match (if TLS enabled on HiveServer2)
    • httpPath is correct (for Knox or HTTP mode)
