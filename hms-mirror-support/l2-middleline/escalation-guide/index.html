    L2 Escalation Guide¶

    When to Escalate to L3¶

    Escalate to L3 when:

    Situation Reason
    RocksDB corruption or "cannot open database" L3 has RocksDB recovery procedures
    Server crashes repeatedly with no clear log cause Stack trace analysis + potential code bug
    TLS certificate chain issues Requires certificate tooling knowledge
    Kerberos KDC configuration changes needed Customer's security team may need coordination
    Security incident (account compromise, unauthorized access) Security team must be looped in
    Performance issues persist after standard JVM tuning Deep profiling required
    Keystore / encryption key recovery needed High-risk procedure requires L3 oversight
    Bug suspected (reproducible, not a config error) Needs engineering review

    L3 Escalation Package¶

    Before escalating to L3, prepare this package:

    1. Full Log Bundle¶

    # Collect full log + rotated logs
tar -czf hms-mirror-logs-$(date +%Y%m%d).tar.gz ~/.hms-mirror/logs/

# OR just recent logs
tail -1000 ~/.hms-mirror/logs/hms-mirror.log > recent.log

    2. Configuration Snapshot (Sanitized)¶

    # Export web.properties with passwords redacted
sed 's/password=.*/password=<REDACTED>/g' \
    ~/.hms-mirror/settings/web.properties > web-sanitized.properties

    3. Java Thread Dump (For Hung/Slow Server)¶

    # Find the PID
PID=$(cat ~/.hms-mirror/hms-mirror.pid)

# Generate thread dump (non-destructive, can repeat)
kill -3 $PID
# Thread dump appears in the server log

# Or use jstack
jstack $PID > thread-dump-$(date +%H%M%S).txt

    4. JVM Heap Dump (For OOM Issues)¶

    # Requires jmap (same JDK as running the server)
PID=$(cat ~/.hms-mirror/hms-mirror.pid)
jmap -dump:format=b,file=heap-dump.hprof $PID

    Heap Dump Size

    Heap dumps can be multi-gigabyte files. Coordinate with L3 on how to transfer.

    5. System Resource Snapshot¶

    # CPU, memory, disk at time of issue
top -b -n 1 > system-top.txt
free -h >> system-top.txt
df -h ~/.hms-mirror >> system-top.txt

    6. RocksDB Status¶

    # Check if RocksDB directory is intact
ls -la ~/.hms-mirror/data/
du -sh ~/.hms-mirror/data/

[Customer name, production impact, SLA status]

    Bug Escalation to Engineering¶

    If you suspect a code bug (not a configuration or environment issue), gather:

    1. Exact reproduction steps — can another person reproduce this on a fresh install?
    2. HMS-Mirror version — exact version string
    3. The specific API call or UI action that triggers the bug
    4. Expected behavior vs. actual behavior
    5. Relevant log lines — only the lines that show the bug, not the full log
    6. Whether the bug is version-specific — did it work in a previous version?

    Submit to the engineering team with the label [BUG] in the ticket subject.
