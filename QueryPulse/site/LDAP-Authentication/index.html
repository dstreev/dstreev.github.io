    LDAP Authentication¶

    Scope: Global (Admin Only)

    Configure Query Pulse to authenticate users against an LDAP or Active Directory server instead of the local user database.

    Overview¶

    When LDAP authentication is enabled, non-admin users authenticate by verifying their credentials against your directory server. Query Pulse uses a two-phase bind: a service account searches for the user, then the user's own credentials are verified. The admin account always uses local authentication, ensuring access even if the LDAP server is unavailable.

    Supported Directory Servers:

    Server Search Filter Username Attribute
    Active Directory (sAMAccountName={0}) sAMAccountName
    FreeIPA (uid={0}) uid
    OpenLDAP (uid={0}) uid

    Enabling LDAP¶

    Set the authentication type to ldap and provide your directory server details:

    export QUERY_PULSE_AUTH_TYPE=ldap
export QUERY_PULSE_LDAP_URL=ldap://ldap.example.com:389
export QUERY_PULSE_LDAP_BIND_DN="cn=serviceaccount,dc=example,dc=com"
export QUERY_PULSE_LDAP_BIND_PASSWORD="service-password"
export QUERY_PULSE_LDAP_USER_SEARCH_BASE="ou=users,dc=example,dc=com"

    Restart the application after changing authentication settings.

    Configuration Reference¶

    Setting Environment Variable Default Description
    LDAP URL QUERY_PULSE_LDAP_URL (required) Server URL. Use ldap:// for plain or ldaps:// for SSL
    Bind DN QUERY_PULSE_LDAP_BIND_DN (required) Service account distinguished name
    Bind Password QUERY_PULSE_LDAP_BIND_PASSWORD (required) Service account password
    User Search Base QUERY_PULSE_LDAP_USER_SEARCH_BASE (required) Base DN where user searches begin
    User Search Filter QUERY_PULSE_LDAP_USER_SEARCH_FILTER (uid={0}) Filter to find users; {0} is replaced with the username
    Username Attribute QUERY_PULSE_LDAP_USERNAME_ATTR uid LDAP attribute containing the username
    Connect Timeout QUERY_PULSE_LDAP_CONNECT_TIMEOUT 5000 Connection timeout in milliseconds
    Read Timeout QUERY_PULSE_LDAP_READ_TIMEOUT 10000 Read timeout in milliseconds
    Use SSL QUERY_PULSE_LDAP_USE_SSL false Use LDAPS (SSL on port 636)
    Use StartTLS QUERY_PULSE_LDAP_USE_STARTTLS false Upgrade LDAP to TLS on port 389
    Skip SSL Verify QUERY_PULSE_LDAP_SKIP_SSL_VERIFY false Skip certificate verification (testing only)

    Connection Security¶

    Plain LDAP¶

    QUERY_PULSE_LDAP_URL=ldap://ldap.example.com:389
    Credentials are transmitted in clear text. Not recommended for production.

    LDAPS (SSL)¶

    QUERY_PULSE_LDAP_URL=ldaps://ldap.example.com:636
QUERY_PULSE_LDAP_USE_SSL=true
    Full SSL/TLS encryption from connection start.

    StartTLS¶

    QUERY_PULSE_LDAP_URL=ldap://ldap.example.com:389
QUERY_PULSE_LDAP_USE_STARTTLS=true
    Starts with a plain connection, then upgrades to TLS. Useful when your server supports TLS on the standard LDAP port.

    Certificate Verification¶

    By default, SSL connections verify the server certificate against the JVM truststore. If your LDAP server uses a self-signed or internal CA certificate, either:

    1. Import the CA certificate into the Query Pulse truststore (recommended)
    2. Set QUERY_PULSE_LDAP_SKIP_SSL_VERIFY=true (testing only)

    Directory Server Examples¶

    Active Directory¶

    export QUERY_PULSE_AUTH_TYPE=ldap
export QUERY_PULSE_LDAP_URL=ldap://ad.corp.example.com:389
export QUERY_PULSE_LDAP_BIND_DN="CN=svc-querypulse,CN=Users,DC=corp,DC=example,DC=com"
export QUERY_PULSE_LDAP_BIND_PASSWORD="service-password"
export QUERY_PULSE_LDAP_USER_SEARCH_BASE="CN=Users,DC=corp,DC=example,DC=com"
export QUERY_PULSE_LDAP_USER_SEARCH_FILTER="(sAMAccountName={0})"
export QUERY_PULSE_LDAP_USERNAME_ATTR=sAMAccountName

    FreeIPA¶

    export QUERY_PULSE_AUTH_TYPE=ldap
export QUERY_PULSE_LDAP_URL=ldaps://ipa.example.com:636
export QUERY_PULSE_LDAP_USE_SSL=true
export QUERY_PULSE_LDAP_BIND_DN="uid=svc-querypulse,cn=users,cn=accounts,dc=example,dc=com"
export QUERY_PULSE_LDAP_BIND_PASSWORD="service-password"
export QUERY_PULSE_LDAP_USER_SEARCH_BASE="cn=users,cn=accounts,dc=example,dc=com"
export QUERY_PULSE_LDAP_USER_SEARCH_FILTER="(uid={0})"
export QUERY_PULSE_LDAP_USERNAME_ATTR=uid

    OpenLDAP¶

    export QUERY_PULSE_AUTH_TYPE=ldap
export QUERY_PULSE_LDAP_URL=ldap://ldap.example.com:389
export QUERY_PULSE_LDAP_BIND_DN="cn=readonly,dc=example,dc=com"
export QUERY_PULSE_LDAP_BIND_PASSWORD="service-password"
export QUERY_PULSE_LDAP_USER_SEARCH_BASE="ou=people,dc=example,dc=com"
export QUERY_PULSE_LDAP_USER_SEARCH_FILTER="(uid={0})"
export QUERY_PULSE_LDAP_USERNAME_ATTR=uid

    User Management with LDAP¶

    Pre-Registration Required¶

    Users must be registered in Query Pulse before they can log in with LDAP credentials. The workflow is:

    1. An admin or the user registers an account in Query Pulse (username must match the LDAP username attribute)
    2. The user logs in using their LDAP directory password
    3. The local password (set during registration) is not used for login when LDAP is enabled

    Admin Account¶

    The admin account (default username: admin) always authenticates using its local password, regardless of LDAP configuration. This ensures:

    • Admin access is maintained when the LDAP server is unavailable
    • Emergency password reset via QUERY_PULSE_ADMIN_PASSWORD_RESET=true always works

    Account Lockout¶

    Failed login attempts are tracked even with LDAP authentication. After the configured number of failures (default: 5), the account is locked. An admin must re-enable the account.

    Docker Deployment¶

    # docker-compose.yml
environment:
  - QUERY_PULSE_AUTH_TYPE=ldap
  - QUERY_PULSE_LDAP_URL=ldap://ldap.example.com:389
  - QUERY_PULSE_LDAP_BIND_DN=cn=admin,dc=example,dc=com
  - QUERY_PULSE_LDAP_BIND_PASSWORD=password
  - QUERY_PULSE_LDAP_USER_SEARCH_BASE=ou=users,dc=example,dc=com
  - QUERY_PULSE_LDAP_USER_SEARCH_FILTER=(uid={0})
  - QUERY_PULSE_LDAP_USERNAME_ATTR=uid
  - QUERY_PULSE_LDAP_USE_SSL=false
  - QUERY_PULSE_LDAP_USE_STARTTLS=false
  - QUERY_PULSE_LDAP_SKIP_SSL_VERIFY=false

    Cloudera Manager Deployment¶

    In the Query Pulse service configuration:

    1. Set Authentication Type to ldap
    2. Fill in the LDAP URL, Bind DN, Bind Password, and User Search Base
    3. Adjust the User Search Filter and Username Attribute for your directory type
    4. Enable SSL or StartTLS if required
    5. Save and restart the service

    Note: In CSD configurations, use USERNAME_PLACEHOLDER instead of {0} in the search filter. Cloudera Manager will translate this automatically.

    Troubleshooting¶

    Issue Cause Solution
    "Invalid credentials" for LDAP users User not registered locally Register the user in Query Pulse first
    Connection timeout LDAP URL or port incorrect Verify URL and network connectivity
    SSL handshake failure Certificate not trusted Import CA cert into truststore or enable skip verification for testing
    "User not found in directory" Wrong search base or filter Verify search base DN and filter match your directory structure
    Admin can't log in via LDAP Expected behavior Admin always uses local authentication
