    Security & Certificates¶

    This section covers TLS configuration, certificate management, and encryption settings.

    TLS/mTLS Configuration¶

    Scope: Global (Admin Only)

    Configure application-level TLS encryption for secure communication.

    TLS Modes¶

    Mode Description
    One-Way TLS Server certificate only; clients verify server
    Mutual TLS (mTLS) Both server and client certificates required

    Server Certificate Configuration¶

    Distinguished Name (DN) Fields: - Common Name (CN): Server hostname - Organization (O): Company name - Organizational Unit (OU): Department - Country (C): Two-letter country code - State (ST): State or province - Locality (L): City

    Subject Alternative Names (SAN): - Additional hostnames or IP addresses - Critical for certificate validation

    Certificate Parameters: - Validity: 30-3650 days - Key Size: 2048 or 4096 bits - Protocol: TLSv1.2 or TLSv1.3

    Client Certificate Management (mTLS)¶

    When mTLS is enabled: 1. Generate client certificates for authorized clients 2. Download PKCS12 file for client configuration 3. Track certificate status and expiration 4. Revoke certificates when needed

    Configuration Workflow¶

    1. Enable TLS
    2. Generate or import server certificate
    3. Configure DN and SAN values
    4. Optionally enable mTLS
    5. Generate client certificates if needed
    6. Restart application to apply changes

    Certificates¶

    Scope: Hybrid (User-Scoped + Global for Admin)

    Manage X.509 certificates for SSL/TLS connections to external services.

    Certificate Sources¶

    JDK Default Certificates: - Automatically available - Read-only, cannot be modified - Include common CA certificates

    User Certificates: - Uploaded by individual users - User-scoped (not visible to other users) - Can be deleted by the owning user

    Global Certificates (Admin Only): - Created by admin users - Available to all users - Marked with "Global" badge

    Certificate Upload¶

    Individual Certificate Upload: - Supported formats: .crt, .cer, .pem - Specify alias for identification - Certificate details parsed automatically

    JKS Truststore Import: - Import multiple certificates at once - Provide truststore password - Select certificates to import

    Certificate Information Displayed¶

    • Alias (name)
    • Subject
    • Expiration date
    • File location
    • File size
    • Status badges (Global, Read-only)

    Truststore Management¶

    The application maintains a truststore at: 

    ~/.query-pulse/security/truststore.jks

    Operations available: - View truststore contents - Reset truststore (with backup) - Delete individual certificates

    Master Password & Encryption¶

    Scope: Global (Admin Only)

    The Master Password is the passphrase that protects the PKCS12 keystore containing Query Pulse's master encryption key. It is not the admin user login password. Every piece of sensitive data stored by Query Pulse -- connection passwords, Kerberos keytabs, Kerberos passwords, TLS passwords, and Git credentials -- is encrypted using keys derived from the master encryption key inside this keystore. If the master password is lost or changed without updating the configuration, all previously encrypted data becomes unrecoverable.

    Encryption Architecture¶

    The master password unlocks a PKCS12 keystore file that holds a single AES-256 master encryption key (alias: master-encryption-key). That master key is never used directly for encrypting application data. Instead, purpose-specific keys are derived from it using PBKDF2-HMAC-SHA256 with 100,000 iterations, providing cryptographic domain separation.

    All encryption uses AES-256-GCM (authenticated encryption) with a random 12-byte IV per operation and 128-bit GCM authentication tags. Additional Authenticated Data (AAD) ties each ciphertext to its owning entity (e.g., conn:<userId>:<connectionId>), preventing ciphertext from being moved between records.

                    |  QUERY_PULSE_KEYSTORE_   |
                    |  PASSWORD (env var)      |
                    +-----------+--------------+
                                |
                                v
                    +-----------+--------------+
                    |  PKCS12 Keystore         |
                    |  (query-pulse-keys.p12)  |
                    |                          |
                    |  [master-encryption-key]  |
                    |  (AES-256)               |
                    +-----------+--------------+
                                |
                    PBKDF2-HMAC-SHA256
                    (100,000 iterations)
                                |
          +----------+----------+----------+----------+
          |          |          |          |          |
          v          v          v          v          v
     connection  keytab    kerberos    tls      git

          |          |          |          |          |

     + random IV  + random IV
     + AAD        + AAD

    What's Encrypted¶

    Data Type Purpose Key Used For
    Connection passwords connection-passwords JDBC/database connection credentials
    Kerberos keytabs keytab-encryption Binary keytab file encryption
    Kerberos passwords kerberos-passwords Kerberos principal passwords
    TLS passwords tls-passwords SSL/TLS keystore/truststore passwords
    Git credentials git-credentials Repository access tokens
    General general Default/fallback encryption

    Keystore File Location¶

    Deployment Mode Default Path
    Standalone ~/.query-pulse/security/query-pulse-keys.p12
    Docker /data/query-pulse/security/query-pulse-keys.p12 (inside container)
    Cloudera Manager (CSD) /var/lib/query-pulse/security/query-pulse-security.p12

    The path is configurable via:

    • Spring property: query-pulse.security.keystore.path
    • Environment variable: QUERY_PULSE_KEYSTORE_PATH

    Configuration Properties¶

    Property Environment Variable Default Description
    query-pulse.security.keystore.path QUERY_PULSE_KEYSTORE_PATH ${query-pulse.working-directory-base}/security/query-pulse-keys.p12 Path to the PKCS12 keystore file
    query-pulse.security.keystore.password QUERY_PULSE_KEYSTORE_PASSWORD query-pulse-dev-keystore-password Password protecting the keystore

    WARNING: The default password query-pulse-dev-keystore-password is intended for development only. Always set a strong, unique password for production deployments.

    First Run Behavior¶

    On first startup, if no keystore file exists at the configured path:

    1. The parent directory is created if it doesn't exist.
    2. A new PKCS12 keystore is created.
    3. A random AES-256 master key is generated and stored in the keystore under the alias master-encryption-key.
    4. The keystore is saved and file permissions are restricted to owner read/write only.
    5. The master key is loaded into memory and the service reports initialized = true.

    On subsequent startups, the existing keystore is loaded and the master key is extracted using the configured password.

    Password Resolution Order¶

    When the application starts, the keystore password is resolved in this order:

    1. query-pulse.security.keystore.password (from environment variable QUERY_PULSE_KEYSTORE_PASSWORD or Spring config)
    2. Hardcoded development default: query-pulse-dev-keystore-password -- a warning is logged

    Passing the Master Password by Deployment Mode¶

    Standalone Mode (query-pulse.sh)¶

    Set the environment variable before starting, or configure it in query-pulse-env-local.sh:

    export QUERY_PULSE_KEYSTORE_PASSWORD="your-strong-password"
./query-pulse.sh start

    The startup script reads QUERY_PULSE_KEYSTORE_PASSWORD and passes it as a Spring Boot command-line override:

    --query-pulse.security.keystore.password=${KEYSTORE_PASSWORD}

    The environment template file (query-pulse-env.sh) documents this variable:

    # SECURITY/ENCRYPTION SETTINGS
# Keystore for encrypting sensitive data (passwords, credentials)
# export QUERY_PULSE_KEYSTORE_PATH=/var/lib/query-pulse/security/query-pulse-security.p12
# export QUERY_PULSE_KEYSTORE_PASSWORD=password

    Docker Mode (docker-compose.yml)¶

    Pass the password via environment variable before running docker-compose, or set it in a .env file:

    # Shell
export QUERY_PULSE_KEYSTORE_PASSWORD="your-strong-password"
docker-compose up -d

# Or in .env file
QUERY_PULSE_KEYSTORE_PASSWORD=your-strong-password

    The docker-compose.yml forwards it to the container:

    environment:
  - QUERY_PULSE_KEYSTORE_PASSWORD=${QUERY_PULSE_KEYSTORE_PASSWORD:-}

    Inside the container, the application's application.yaml resolves it via Spring's ${QUERY_PULSE_KEYSTORE_PASSWORD:...} placeholder.

    Cloudera Manager (CSD) Mode¶

    The CSD service.sdl defines two parameters on the Query Pulse Server role:

    CSD Parameter Label Type Default
    query_pulse_keystore_path Query Pulse Keystore Path path (localDataFile) /var/lib/query-pulse/security/query-pulse-security.p12
    query_pulse_keystore_password Query Pulse Keystore Password password (empty)

    These are configured in the Cloudera Manager UI under the Query Pulse Server role configuration. The SDL maps them to environment variables:

    "QUERY_PULSE_KEYSTORE_PATH": "${query_pulse_keystore_path}",
"QUERY_PULSE_KEYSTORE_PASSWORD": "${query_pulse_keystore_password}"

    The control.sh script picks them up and passes them as Spring Boot application arguments:

    APP_ARGS+=("--query-pulse.security.keystore.path=${KEYSTORE_PATH}")
if [ -n "$KEYSTORE_PASSWORD" ]; then
    APP_ARGS+=("--query-pulse.security.keystore.password=${KEYSTORE_PASSWORD}")
fi

    The password is sanitized in configuration logs:

    KEYSTORE_PASSWORD:     [SET - 24 chars]

    Changing the Master Password¶

    The master password can be changed through the Admin UI or the REST API. This changes the password that protects the keystore; it does not rotate the master encryption key inside it (that is a separate operation).

    Via the Admin UI¶

    1. Navigate to the Master Password panel in the Admin section.
    2. Enter the current password.
    3. Enter a new password (minimum 12 characters).
    4. Confirm the new password.
    5. Click Change Password.

    On success:

    • The old keystore is backed up with a datetime suffix (e.g., query-pulse-keys-2026-02-11T14-30-45-123456.p12).
    • A new keystore is created with the same keys but the new password.
    • The UI displays the backup path and a warning about updating configuration.

    Via the REST API¶

    POST /api/v1/admin/security/keystore/change-password

    Request body: 

    {
  "currentPassword": "old-password",
  "newPassword": "new-strong-password-at-least-12-chars"
}

    Success response: 

    {
  "status": "SUCCESS",
  "message": "Keystore password changed successfully",
  "backupPath": "/path/to/query-pulse-keys-2026-02-11T14-30-45-123456.p12",
  "entriesTransferred": 1,
  "warning": "You must update the keystore password in your configuration ..."
}

    This endpoint requires the ADMIN role.

    What Happens During Password Change¶

    1. The current password is validated by attempting to load the existing keystore.
    2. A backup of the old keystore is created with a timestamp suffix.
    3. A new keystore is created and all entries are transferred from the old keystore using the new password.
    4. The new keystore is written to a temporary file first, then atomically swapped.
    5. Restrictive file permissions (owner-only read/write) are set on the new keystore.
    6. The in-memory keystore reference is updated.

    Post-Change Requirements¶

    After changing the master password, you must:

    1. Update your deployment configuration with the new password:
      • Standalone: Update QUERY_PULSE_KEYSTORE_PASSWORD in your environment or query-pulse-env-local.sh.
      • Docker: Update QUERY_PULSE_KEYSTORE_PASSWORD in your .env file or docker-compose environment.
      • Cloudera Manager: Update the "Query Pulse Keystore Password" in the CM configuration UI.
    2. Restart the application for the new password to take effect on next startup.

    If the configuration is not updated, the application will fail to start on next restart with error KEYSTORE_004: Invalid keystore password.

    Master Key Rotation (Separate from Password Change)¶

    Key rotation replaces the master encryption key inside the keystore and re-encrypts all stored data. This is distinct from changing the keystore password. Key rotation is available via the REST API only.

    When to Rotate¶

    • Suspected key compromise
    • Compliance requirements (periodic rotation policy)
    • Personnel changes (key holders departing)

    Via the REST API¶

    POST /api/v1/admin/security/rotate-key?confirm=true

    Without confirm=true, the endpoint returns a confirmation prompt listing affected data types and a recommendation to create a database backup before proceeding.

    Rotation Process¶

    1. Collect: All encrypted data is decrypted with the current key (connections, Kerberos configs, keytabs).
    2. Rotate: A new AES-256 master key is generated. The old key is backed up in the keystore with a timestamped alias.
    3. Re-encrypt: All data is re-encrypted with the new key and saved back to the database.
    4. Clear cache: The derived key cache is cleared so new purpose-specific keys are derived from the new master key.

    Important: Key rotation should be performed during a maintenance window. If rotation fails mid-process, check logs and consider restoring from backup.

    Encryption Validation¶

    Validate that all encrypted data can still be decrypted via the REST API:

    GET /api/v1/admin/security/validate

    Returns success/failure counts for connections, Kerberos configs, and keytabs. Use this to verify encryption health after upgrades, key rotation, or suspected issues.

    Health Monitoring¶

    Monitor the overall encryption system health via the REST API:

    GET /api/v1/admin/security/health

    Key Management Status:

    • HEALTHY: Master key loaded, keystore accessible
    • NOT_INITIALIZED: Key management service failed to start

    Data Encryption Status:

    • HEALTHY: All stored data decrypts successfully
    • ISSUES_FOUND: One or more items failed to decrypt

    Additional Endpoints:

    Endpoint Method Description
    /api/v1/admin/security/status GET Keystore initialization status, path, derived key cache size
    /api/v1/admin/security/keystore GET Non-sensitive keystore info (path, algorithm, type)

    All security management endpoints require the ADMIN role.

    Error Codes¶

    The KeystoreException class provides structured error codes and recovery instructions:

    Error Code Description Recovery
    KEYSTORE_001 Keystore file not found Restart to create a new keystore; previously encrypted data is unrecoverable
    KEYSTORE_002 Permission denied Check file permissions on the keystore directory
    KEYSTORE_003 Keystore corrupted Delete corrupted keystore and restart; previously encrypted data is unrecoverable
    KEYSTORE_004 Wrong password Verify QUERY_PULSE_KEYSTORE_PASSWORD matches the password used when the keystore was created
    KEYSTORE_005 Key not found Keystore was likely recreated; affected data must be re-entered
    KEYSTORE_006 Key generation failed Check system entropy and Java security configuration
    KEYSTORE_007 Decryption failed Keys have changed; affected data must be re-entered
    KEYSTORE_008 Encryption failed Check system resources and Java security configuration
    KEYSTORE_009 Initialization failed Check logs and ensure the keystore directory is writable
