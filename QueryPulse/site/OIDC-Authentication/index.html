    OIDC / SSO Authentication¶

    Scope: Global (Admin Only)

    Configure Query Pulse to authenticate users via an OpenID Connect (OIDC) Identity Provider for Single Sign-On (SSO).

    Overview¶

    When OIDC authentication is enabled, the login page displays a "Login with SSO" button alongside the standard username/password form. Clicking the SSO button redirects users to your Identity Provider (IdP) for authentication. After successful login, the user is redirected back to Query Pulse with a valid session.

    Supported Identity Providers:

    Provider Issuer URI Pattern
    Keycloak https://keycloak.example.com/realms/{realm}
    Okta https://{domain}.okta.com/oauth2/default
    Azure AD https://login.microsoftonline.com/{tenant-id}/v2.0
    Google https://accounts.google.com
    Any OIDC-compliant provider Must support .well-known/openid-configuration

    Key Features:

    • Automatic endpoint discovery from issuer URI
    • Dynamic user provisioning on first login
    • Group-based access control
    • Group-based admin role assignment
    • Admin account can still use local password login

    Enabling OIDC¶

    Set the authentication type to oidc and provide your Identity Provider details:

    export QUERY_PULSE_AUTH_TYPE=oidc
export QUERY_PULSE_OIDC_ISSUER_URI=https://keycloak.example.com/realms/myrealm
export QUERY_PULSE_OIDC_CLIENT_ID=query-pulse
export QUERY_PULSE_OIDC_CLIENT_SECRET=your-client-secret

    Restart the application after changing authentication settings.

    Configuration Reference¶

    Setting Environment Variable Default Description
    Issuer URI QUERY_PULSE_OIDC_ISSUER_URI (required) OIDC provider base URL
    Client ID QUERY_PULSE_OIDC_CLIENT_ID (required) OAuth2 client ID
    Client Secret QUERY_PULSE_OIDC_CLIENT_SECRET (required) OAuth2 client secret
    Scopes QUERY_PULSE_OIDC_SCOPES openid profile email groups Space-separated OAuth2 scopes
    User Group QUERY_PULSE_OIDC_USER_GROUP (empty) Required group for access; empty allows all
    Admin Group QUERY_PULSE_OIDC_ADMIN_GROUP (empty) Group that grants ADMIN role; empty disables
    Dynamic User Create QUERY_PULSE_OIDC_DYNAMIC_USER_CREATE true Auto-create user on first OIDC login

    Authentication Flow¶

    1. User visits the login page and clicks Login with SSO
    2. Browser redirects to the Identity Provider's login page
    3. User authenticates with the IdP (password, MFA, etc.)
    4. IdP redirects back to Query Pulse with an authorization code
    5. Query Pulse exchanges the code for OIDC tokens
    6. User information is extracted from the ID token:
    7. Username from preferred_username claim (falls back to email prefix, then subject)
    8. Email from email claim
    9. Groups from groups claim (falls back to roles claim)
    10. Group access is validated (if configured)
    11. A local user account is created or updated
    12. Internal JWT tokens are issued and the user is logged in

    Group-Based Access Control¶

    Restricting Access to a Group¶

    Set QUERY_PULSE_OIDC_USER_GROUP to require users to be a member of a specific group:

    export QUERY_PULSE_OIDC_USER_GROUP=query-pulse-users

    Users not in this group will see an error: "You do not have permission to access this application."

    Leave this empty to allow all authenticated users.

    Granting Admin Role via Group¶

    Set QUERY_PULSE_OIDC_ADMIN_GROUP to automatically grant the ADMIN role to members of a specific group:

    export QUERY_PULSE_OIDC_ADMIN_GROUP=query-pulse-admins

    Admin status is synced on each login. If a user is removed from the admin group in the IdP, their ADMIN role is revoked at next login. Leave this empty to manage admin roles manually.

    User Provisioning¶

    Dynamic User Creation (Default)¶

    When QUERY_PULSE_OIDC_DYNAMIC_USER_CREATE=true (default), a local user account is automatically created the first time a user logs in via SSO. The account is created with:

    • Username from the OIDC preferred_username claim
    • Email from the OIDC email claim
    • USER role (and ADMIN role if in admin group)
    • A random password hash (OIDC users cannot log in with a password)

    Pre-Created Users¶

    When QUERY_PULSE_OIDC_DYNAMIC_USER_CREATE=false, users must be pre-created by an admin before they can log in via SSO. The local username must match the OIDC preferred_username claim.

    Identity Provider Setup¶

    Keycloak¶

    1. Create a new client in your realm:
    2. Client ID: query-pulse
    3. Client Protocol: openid-connect
    4. Access Type: confidential
    5. Set Valid Redirect URIs: https://query-pulse.example.com/login/oauth2/code/oidc
    6. Under Client Scopes, ensure groups scope is available
    7. Create a Group Membership mapper:
    8. Mapper Type: Group Membership
    9. Token Claim Name: groups
    10. Add to ID token: Yes
    11. Copy the client secret from the Credentials tab
    export QUERY_PULSE_OIDC_ISSUER_URI=https://keycloak.example.com/realms/myrealm
export QUERY_PULSE_OIDC_CLIENT_ID=query-pulse
export QUERY_PULSE_OIDC_CLIENT_SECRET=<from-credentials-tab>

    Okta¶

    1. Create a new Web Application in Okta
    2. Set Sign-in redirect URI: https://query-pulse.example.com/login/oauth2/code/oidc
    3. Set Sign-out redirect URI: https://query-pulse.example.com/login
    4. Under Sign On > OpenID Connect ID Token, add a groups claim:
    5. Claim name: groups
    6. Include in token type: ID Token
    7. Filter: Matches regex .*
    8. Note the Client ID and Client Secret
    export QUERY_PULSE_OIDC_ISSUER_URI=https://dev-12345.okta.com/oauth2/default
export QUERY_PULSE_OIDC_CLIENT_ID=<client-id>
export QUERY_PULSE_OIDC_CLIENT_SECRET=<client-secret>

    Azure AD (Microsoft Entra ID)¶

    1. Register a new application in Azure AD
    2. Under Authentication, add a Web platform redirect URI: https://query-pulse.example.com/login/oauth2/code/oidc
    3. Under Certificates & Secrets, create a new client secret
    4. Under Token Configuration, add a groups claim:
    5. Claim type: Groups
    6. Token type: ID
    7. Note the Application (client) ID and Directory (tenant) ID
    export QUERY_PULSE_OIDC_ISSUER_URI=https://login.microsoftonline.com/<tenant-id>/v2.0
export QUERY_PULSE_OIDC_CLIENT_ID=<application-id>
export QUERY_PULSE_OIDC_CLIENT_SECRET=<client-secret>

    Admin Account with OIDC¶

    The admin account (default username: admin) can always log in with local credentials, even when OIDC is the configured authentication type. Both the SSO button and the username/password form appear on the login page.

    This provides:

    • Emergency admin access when the IdP is unavailable
    • Password reset capability via QUERY_PULSE_ADMIN_PASSWORD_RESET=true

    Docker Deployment¶

    # docker-compose.yml
environment:
  - QUERY_PULSE_AUTH_TYPE=oidc
  - QUERY_PULSE_OIDC_ISSUER_URI=https://keycloak.example.com/realms/myrealm
  - QUERY_PULSE_OIDC_CLIENT_ID=query-pulse
  - QUERY_PULSE_OIDC_CLIENT_SECRET=your-client-secret
  - QUERY_PULSE_OIDC_SCOPES=openid profile email groups
  - QUERY_PULSE_OIDC_USER_GROUP=query-pulse-users
  - QUERY_PULSE_OIDC_ADMIN_GROUP=query-pulse-admins
  - QUERY_PULSE_OIDC_DYNAMIC_USER_CREATE=true

    Cloudera Manager Deployment¶

    In the Query Pulse service configuration:

    1. Set Authentication Type to oidc
    2. Fill in the OIDC Issuer URI, Client ID, and Client Secret
    3. Optionally configure User Group and Admin Group for access control
    4. Set Dynamic User Creation as desired
    5. Save and restart the service

    Troubleshooting¶

    Issue Cause Solution
    SSO button not visible OIDC not configured or config endpoint unreachable Verify issuer URI, client ID, and client secret are set
    "Access was denied by the identity provider" User denied access at IdP Check IdP user permissions and application assignment
    "You do not have permission" User not in required group Add user to the group specified by OIDC_USER_GROUP
    Redirect loop after login Callback URL mismatch Ensure IdP redirect URI matches https://<host>/login/oauth2/code/oidc
    User created without admin role User not in admin group Add user to the group specified by OIDC_ADMIN_GROUP
    "Authentication failed. No tokens received." Callback URL parameters missing Check IdP configuration and network connectivity
    Admin can't use SSO Expected behavior Admin uses local credentials; other users use SSO
    Provider discovery fails Issuer URI incorrect or unreachable Verify the URI returns a valid .well-known/openid-configuration
