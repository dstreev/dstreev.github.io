    Quick Start Guide¶

    Get Query Pulse up and running in minutes. Choose your installation method, configure authentication, and connect to your first data source.

    Prerequisites¶

    Requirement Details
    Java JDK 17 or later
    Memory 4 GB minimum, 8 GB recommended
    Disk 500 MB for the application, plus space for data and drivers
    Browser Chrome, Firefox, Edge, or Safari (modern versions)
    Network Access to your Hive, Trino, or Impala endpoints

    Installation Options¶

    Method Best For Guide
    Standalone Single-server deployments, edge nodes, development Standalone Installation
    Docker Containerized environments, quick evaluation Docker Installation
    Cloudera Manager CDP clusters with centralized management Cloudera Manager Installation

    First-Time Setup¶

    Creating the Admin Account¶

    On first launch, Query Pulse has no users. The first user to register automatically receives the ADMIN role.

    1. Open Query Pulse in your browser (http://<hostname>:8088)
    2. Click Register on the login page
    3. Enter a username, email, and password
    4. This first account becomes the admin

    The admin username defaults to admin, but you can choose any username during registration. Only the first registered user gets the ADMIN role automatically.

    Resetting the admin password (if locked out):

    # Set this environment variable and restart the application
export QUERY_PULSE_ADMIN_PASSWORD_RESET=true

    After restart, the admin password is cleared and you can log in and set a new one. Remember to unset the variable afterward.

    Creating Additional Users¶

    Local authentication (default): Users self-register through the login page, or an admin can create accounts from Settings > User Management.

    LDAP authentication: An admin must pre-create user accounts before users can log in. The local username must match the LDAP directory username. See LDAP Setup below.

    OIDC authentication: Users are automatically provisioned on first SSO login (when QUERY_PULSE_OIDC_DYNAMIC_USER_CREATE=true). See OIDC Setup below.

    Authentication Setup¶

    Query Pulse supports three authentication modes. Set the mode with:

    export QUERY_PULSE_AUTH_TYPE=local    # local | ldap | oidc

    Regardless of which mode you choose, the admin account always authenticates with local credentials. This ensures admin access even when LDAP or OIDC services are unavailable.

    Local Authentication (Default)¶

    No additional configuration needed. Users register and log in with a username and password stored in the application database.

    LDAP Authentication Setup¶

    Delegates password verification to an external LDAP or Active Directory server.

    Required parameters:

    export QUERY_PULSE_AUTH_TYPE=ldap
export QUERY_PULSE_LDAP_URL=ldap://ldap.example.com:389
export QUERY_PULSE_LDAP_BIND_DN="cn=serviceaccount,dc=example,dc=com"
export QUERY_PULSE_LDAP_BIND_PASSWORD="service-password"
export QUERY_PULSE_LDAP_USER_SEARCH_BASE="ou=users,dc=example,dc=com"

    Optional parameters:

    Variable Default Description
    QUERY_PULSE_LDAP_USER_SEARCH_FILTER (uid={0}) LDAP filter; {0} is replaced with the username
    QUERY_PULSE_LDAP_USERNAME_ATTR uid Attribute containing the username
    QUERY_PULSE_LDAP_USE_SSL false Use LDAPS (port 636)
    QUERY_PULSE_LDAP_USE_STARTTLS false Upgrade to TLS on port 389
    QUERY_PULSE_LDAP_SKIP_SSL_VERIFY false Skip certificate verification (testing only)

    Active Directory example:

    export QUERY_PULSE_AUTH_TYPE=ldap
export QUERY_PULSE_LDAP_URL=ldap://ad.corp.example.com:389
export QUERY_PULSE_LDAP_BIND_DN="CN=svc-querypulse,CN=Users,DC=corp,DC=example,DC=com"
export QUERY_PULSE_LDAP_BIND_PASSWORD="service-password"
export QUERY_PULSE_LDAP_USER_SEARCH_BASE="CN=Users,DC=corp,DC=example,DC=com"
export QUERY_PULSE_LDAP_USER_SEARCH_FILTER="(sAMAccountName={0})"
export QUERY_PULSE_LDAP_USERNAME_ATTR=sAMAccountName

    Important: Users must be pre-registered in Query Pulse before they can log in with LDAP. The local username must match the LDAP directory username. Register users via the admin UI or have them self-register first.

    For the full reference, see LDAP Authentication.

    OIDC / SSO Setup¶

    Enables Single Sign-On via an OpenID Connect identity provider (Keycloak, Okta, Azure AD, Google, or any OIDC-compliant provider).

    Required parameters:

    export QUERY_PULSE_AUTH_TYPE=oidc
export QUERY_PULSE_OIDC_ISSUER_URI=https://keycloak.example.com/realms/myrealm
export QUERY_PULSE_OIDC_CLIENT_ID=query-pulse
export QUERY_PULSE_OIDC_CLIENT_SECRET=your-client-secret

    Optional parameters:

    Variable Default Description
    QUERY_PULSE_OIDC_SCOPES openid profile email groups OAuth2 scopes to request
    QUERY_PULSE_OIDC_USER_GROUP (empty) Require membership in this group; empty allows all
    QUERY_PULSE_OIDC_ADMIN_GROUP (empty) Members of this group get the ADMIN role
    QUERY_PULSE_OIDC_DYNAMIC_USER_CREATE true Auto-create local accounts on first SSO login

    Identity provider setup:

    In your IdP, create a client (or application) with:

    • Redirect URI: https://<query-pulse-host>:8088/login/oauth2/code/oidc
    • Grant type: Authorization Code
    • Token claims: Ensure preferred_username, email, and groups (or roles) claims are included in the ID token

    Example with Keycloak:

    export QUERY_PULSE_AUTH_TYPE=oidc
export QUERY_PULSE_OIDC_ISSUER_URI=https://keycloak.example.com/realms/myrealm
export QUERY_PULSE_OIDC_CLIENT_ID=query-pulse
export QUERY_PULSE_OIDC_CLIENT_SECRET=<from-credentials-tab>
export QUERY_PULSE_OIDC_USER_GROUP=query-pulse-users
export QUERY_PULSE_OIDC_ADMIN_GROUP=query-pulse-admins

    When OIDC is enabled, the login page displays a Login with SSO button alongside the username/password form. The password form remains available for the admin account.

    For the full reference including Okta and Azure AD examples, see OIDC / SSO Authentication.

    Truststore Configuration¶

    When connecting to endpoints secured with self-signed or internal CA certificates (Hive over SSL, LDAPS, OIDC providers with private certificates), Query Pulse needs to trust those certificates.

    Uploading Certificates via the UI¶

    The simplest approach is to upload certificates through the application:

    1. Log in as any user (or admin for global certificates)
    2. Go to Settings > Certificates
    3. Click Upload Certificate
    4. Select a .crt, .cer, or .pem file and provide an alias
    5. The certificate is added to the application's truststore

    Importing from an existing JKS truststore:

    1. Go to Settings > Certificates
    2. Click Import from Truststore
    3. Select your .jks file and enter the truststore password
    4. Choose which certificates to import

    User-uploaded certificates are scoped to that user. Admin users can upload Global certificates that are available to all users.

    Configuring a System Truststore¶

    To point the entire JVM at a truststore (for LDAPS, OIDC provider connections, etc.):

    export QUERY_PULSE_TRUSTSTORE_LOCATION=/path/to/truststore.jks
export QUERY_PULSE_TRUSTSTORE_PASSWORD=changeit

    This sets the JVM system properties javax.net.ssl.trustStore and javax.net.ssl.trustStorePassword, so all outbound SSL connections (LDAP, OIDC discovery, JDBC) will trust the certificates in this truststore.

    For deployment-specific truststore examples, see the Standalone, Docker, or Cloudera Manager installation guides.

    Creating a Truststore¶

    If you have a CA certificate or server certificate in PEM format, create a JKS truststore with:

    keytool -importcert \
  -alias my-ca \
  -file ca-certificate.pem \
  -keystore truststore.jks \
  -storepass changeit \
  -noprompt

    To add multiple certificates:

    keytool -importcert -alias hive-ca -file hive-ca.pem -keystore truststore.jks -storepass changeit -noprompt
keytool -importcert -alias ldap-ca -file ldap-ca.pem -keystore truststore.jks -storepass changeit -noprompt
keytool -importcert -alias oidc-ca -file idp-ca.pem  -keystore truststore.jks -storepass changeit -noprompt

    Connecting to Your First Data Source¶

    Once Query Pulse is running and you have logged in:

    Step 1 — Install a JDBC Driver¶

    1. Go to Settings > JDBC Drivers
    2. Select the tab for your engine (Hive, Trino, Impala)
    3. Click Install next to the driver matching your platform (e.g., CDP 7.3.1 Hive)
    4. Wait for the download to complete

    Step 2 — Create a Connection¶

    1. Go to Settings > Connections
    2. Click Add Connection
    3. Provide:
    4. Connection Name — A descriptive label
    5. Platform Type — Must match the installed driver
    6. JDBC URL — Your endpoint URL

    Example JDBC URLs:

    # HiveServer2 with Kerberos
jdbc:hive2://hiveserver2.example.com:10000/default;principal=hive/_HOST@REALM.COM

# HiveServer2 with username/password
jdbc:hive2://hiveserver2.example.com:10000/default

# HiveServer2 via Knox Gateway (SSL)
jdbc:hive2://knox.example.com:8443/;ssl=true;transportMode=http;httpPath=gateway/cdp-proxy-api/hive

# Trino
jdbc:trino://trino.example.com:8443/hive/default?SSL=true

# HiveServer2 with ZooKeeper discovery (HA)
jdbc:hive2://zk1:2181,zk2:2181,zk3:2181/;serviceDiscoveryMode=zooKeeper;zooKeeperNamespace=hiveserver2
    1. Click Save & Test to verify connectivity

    Step 3 — Start Querying¶

    1. Click Workspaces in the navigation bar and create a workspace
    2. Open or create a .sql file
    3. Select your connection from the connection dropdown
    4. Write a query and click Run

    Common Environment Variables¶

    Variable Default Description
    QUERY_PULSE_SERVER_PORT 8088 HTTP/HTTPS port
    QUERY_PULSE_CONTEXT_PATH / URL context path
    QUERY_PULSE_HEAP_INIT 2048 Initial JVM heap (MB)
    QUERY_PULSE_HEAP_MAX 4096 Maximum JVM heap (MB)
    QUERY_PULSE_LOG_LEVEL INFO Log verbosity (DEBUG, INFO, WARN, ERROR)
    QUERY_PULSE_AUTH_TYPE local Authentication mode (local, ldap, oidc)
    QUERY_PULSE_ADMIN_PASSWORD_RESET false Reset admin password on startup
    QUERY_PULSE_SSL_ENABLED false Enable HTTPS for the application
    QUERY_PULSE_TLS_PROTOCOL TLSv1.2 TLS protocol version
    QUERY_PULSE_TRUSTSTORE_LOCATION (empty) Path to JKS truststore
    QUERY_PULSE_TRUSTSTORE_PASSWORD (empty) Truststore password
