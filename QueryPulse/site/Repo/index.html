    Repository First¶

    Query Pulse takes a "Repository First" approach to workspace management. Every workspace is backed by a full Git repository, enabling version control, collaboration, and remote synchronization for all your SQL development work.

    Core Philosophy¶

    Traditional SQL tools store queries as isolated files or database records. Query Pulse treats your SQL work as a proper software development project:

    • Version Control: Every change is tracked in Git history
    • Branching: Work on multiple features or experiments simultaneously
    • Collaboration: Push and pull from remote repositories
    • History: Full audit trail of all changes with commit messages
    • Recovery: Easily revert to any previous state

    Workspace Architecture¶

    When you create a workspace, Query Pulse automatically initializes a Git repository:

    ~/.query-pulse/workspaces/
├── {userId}/
│   └── {workspaceId}/
│       └── repo/           # Git repository root
│           ├── .git/       # Git metadata
│           └── files/      # Your SQL files

    Each workspace is isolated by user, ensuring privacy and security of your work.

    Creating Workspaces¶

    New Empty Workspace¶

    1. Click "Create Workspace" in the workspace selector
    2. Enter a workspace name
    3. Click "Create"

    A new Git repository is automatically initialized with a main branch.

    Clone from Remote Repository¶

    1. Click "Create Workspace" in the workspace selector
    2. Enter a workspace name
    3. Provide the remote repository URL (HTTPS or SSH)
    4. Select stored Git credentials (if required)
    5. Optionally specify a branch to checkout
    6. Click "Create"

    The clone operation runs asynchronously. The workspace status shows "Cloning" until complete.

    Git Integration¶

    Status Tracking¶

    The Git panel shows real-time repository status:

    Indicator Meaning
    Clean No uncommitted changes
    Modified Files changed but not staged
    Staged Files ready to commit
    Untracked New files not yet added to Git
    Conflicting Merge conflicts need resolution

    File Status in Tree¶

    The file tree displays Git status with visual indicators:

    • Green - New file (added)
    • Yellow - Modified file
    • Red - Deleted file
    • Gray - Untracked file

    Committing Changes¶

    1. Make changes to your SQL files
    2. Open the Git panel (Changes tab)
    3. Stage files individually or use "Stage All"
    4. Enter a commit message describing your changes
    5. Click "Commit"

    Branch Management¶

    Work on multiple development lines:

    • Create Branch: Start a new feature branch from any commit
    • Switch Branch: Checkout different branches to switch context
    • Merge Branch: Combine branches when work is complete
    • Delete Branch: Clean up merged branches

    History and Diffs¶

    • Commit History: View all commits with messages, authors, and timestamps
    • File Diffs: See exactly what changed between versions
    • Commit Details: Inspect individual commits with full diff information

    Remote Synchronization¶

    Connecting to Remotes¶

    After creating a workspace, add remote repositories:

    1. Open Git panel
    2. Go to Remotes section
    3. Click "Add Remote"
    4. Enter remote name (e.g., "origin") and URL
    5. Associate stored credentials for authentication

    Push and Pull¶

    Push: Upload your local commits to the remote repository - Select the remote and branch - Click "Push" - Uses stored credentials for authentication

    Pull: Download and merge remote changes - Select the remote - Click "Pull" - Automatically merges with your current branch

    Fetch: Download remote changes without merging - Updates remote tracking branches - Review changes before merging

    Credential Management¶

    Store credentials securely for remote operations:

    Type Use Case
    SSH Key GitHub/GitLab SSH URLs
    Personal Access Token HTTPS with token auth
    Username & Password Basic HTTPS auth

    Credentials are encrypted at rest and associated per-remote, allowing different credentials for different repositories.

    Advanced Features¶

    Stash Management¶

    Temporarily save uncommitted work:

    • Create Stash: Save current changes without committing
    • Apply Stash: Restore stashed changes (keeps stash)
    • Pop Stash: Restore and remove stash
    • Drop Stash: Delete a stash

    Tag Support¶

    Mark important versions:

    • Create Tag: Mark commits with version labels
    • Lightweight Tags: Simple pointers to commits
    • Annotated Tags: Tags with messages and metadata

    Conflict Resolution¶

    When merges result in conflicts:

    1. Conflicting files are marked in the file tree
    2. Open the file to see conflict markers
    3. Edit to resolve conflicts
    4. Stage the resolved file
    5. Commit the merge

    Workspace Status¶

    Workspaces track their operational state:

    Status Description
    PENDING Initial creation state
    INITIALIZING Git repository being set up
    CLONING Remote clone in progress
    READY Workspace ready for use
    ERROR Operation failed (see status message)

    Best Practices¶

    1. Commit Often: Make small, focused commits with clear messages
    2. Use Branches: Create branches for new features or experiments
    3. Write Good Messages: Describe why you made changes, not just what
    4. Push Regularly: Backup your work to remote repositories
    5. Review Before Commit: Use diffs to verify your changes
    6. Keep Main Clean: Merge completed, tested work to main branch

    Benefits of Repository First¶

    • Never Lose Work: Git history preserves every change
    • Experiment Safely: Create branches for risky changes
    • Collaborate: Share work through remote repositories
    • Audit Trail: Know who changed what and when
    • Portable: Clone workspaces to any machine
    • Industry Standard: Leverage familiar Git workflows
