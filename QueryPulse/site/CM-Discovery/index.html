    Cloudera Manager Discovery¶

    Cloudera Manager (CM) Discovery automates the creation of HiveServer2 connections by querying Cloudera Manager's API for cluster configurations. Instead of manually constructing JDBC URLs with the correct ports, transport modes, SSL settings, and Kerberos principals, you can discover and import them directly.

    How It Works¶

    Cloudera Manager API
       │
       │  /api/v41/clusters
       │  /api/v41/clusters/{name}/services
       │  /api/v41/clusters/{name}/services/{name}/config
       │  /api/v41/clusters/{name}/services/{name}/roles
       ▼
┌─────────────────────────────┐
│  Query Pulse Discovery      │
│                             │
│  1. Enumerate clusters      │
│  2. Find Hive services      │
│  3. Read HS2 configuration  │
│  4. Detect Kerberos/SSL     │
│  5. Build JDBC URLs         │
└──────────────┬──────────────┘
               │
               ▼
     User selects endpoints
     to import as connections

    Discovery reads the following from each cluster's Hive service configuration:

    Setting What It Determines
    hive_server2_transport_mode Binary (port 10000) or HTTP (port 10001) transport
    ssl_enabled Whether SSL/TLS is required
    hive_server2_authentication Kerberos or no authentication
    kerberos_princ_name Service principal for Kerberos connections
    dynamic_service_discovery ZooKeeper-based HA discovery vs direct host
    hive_zookeeper_quorum ZooKeeper connection string for HA
    hive_server2_zookeeper_namespace ZooKeeper namespace (default: hiveserver2)
    hs2_thrift_address_port Binary transport port
    hive_server2_thrift_http_port HTTP transport port
    hive_server2_thrift_http_path HTTP path (default: cliservice)

    CM Connections (Admin Setup)¶

    Scope: Global (Admin Only)

    Before users can discover endpoints, an administrator must configure one or more Cloudera Manager connections. These are global — all users share the same CM connections for discovery.

    Creating a CM Connection¶

    1. Navigate to Settings > Admin > Discovery
    2. Click Add CM Connection
    3. Fill in the connection details:
    Field Description Example
    Name Display name for this CM instance Production CM
    Description Optional notes CDP 7.1.9 Production Cluster
    CM URL Base URL of Cloudera Manager https://cm-host:7183
    Username CM API username admin
    Password CM API password (encrypted at rest)
    Use TLS Enable HTTPS for API calls Enabled for port 7183
    Skip TLS Verify Bypass certificate validation Use only for self-signed certs

    Skip TLS Verify

    Disabling TLS verification is a security risk. Use this only in development or when connecting to CM instances with self-signed certificates that cannot be added to the truststore.

    Testing Connectivity¶

    Click Test on any CM connection to verify that Query Pulse can reach the Cloudera Manager API. The connection status displays:

    • SUCCESS — API is reachable and credentials are valid
    • FAILED — Check the URL, credentials, or network connectivity

    URL Normalization¶

    The CM URL is automatically normalized. Trailing paths like /cmf or /cm are stripped, so all of these are equivalent:

    • https://cm-host:7183
    • https://cm-host:7183/cmf
    • https://cm-host:7183/cm

    Discovering Endpoints (User Workflow)¶

    Scope: User-Scoped (All Users)

    Any user can discover and import connections from the CM instances configured by an administrator.

    Step 1: Open the Import Modal¶

    1. Open the Connections panel
    2. Click Import from CM
    3. Select a CM connection from the list and click Discover

    Step 2: Review Discovered Endpoints¶

    Discovery queries each cluster and returns a list of HiveServer2 endpoints. Each entry shows:

    • Cluster name and version
    • Platform type (CDP 7.1, CDH 6, etc.)
    • Connection mode — ZooKeeper-based (HA) or direct host
    • Configuration badges:
      • Kerberos — Authentication requires a Kerberos principal
      • SSL — Connection uses SSL/TLS encryption
      • HTTP — Uses HTTP transport mode (vs binary)
      • ZooKeeper — Uses ZooKeeper service discovery for high availability

    Step 3: Select and Import¶

    1. Select the endpoints you want to import (or use Select All)
    2. If any selected endpoint is kerberized, choose a Kerberos Configuration to associate with the imported connections
    3. Click Import

    Import Results¶

    After import, you'll see a summary:

    • Created — New connections added to your account
    • Skipped — Connections that already exist (matched by name)
    • Warnings — Non-fatal issues (e.g., kerberized connection without a Kerberos config)
    • Errors — Connections that failed to import

    Imported connections start with an UNTESTED status. Use Test in the Connections panel to verify connectivity.

    Discovery Modes¶

    ZooKeeper-Based Discovery (HA)¶

    When HiveServer2 is configured for ZooKeeper service discovery, the imported JDBC URL uses the ZK quorum:

    jdbc:hive2://zk1:2181,zk2:2181,zk3:2181/;serviceDiscoveryMode=zooKeeper;zooKeeperNamespace=hiveserver2

    This provides automatic failover — the client discovers available HS2 instances through ZooKeeper at connect time.

    Direct Host Discovery¶

    When ZooKeeper discovery is not enabled, each HiveServer2 role is discovered as a separate endpoint with its hostname and port:

    # Binary transport
jdbc:hive2://hs2-host.example.com:10000/default

# HTTP transport
jdbc:hive2://hs2-host.example.com:10001/default;transportMode=http;httpPath=cliservice

    If a cluster has multiple HS2 roles, each one appears as a separate discoverable endpoint.

    Platform Detection¶

    Discovery automatically maps the Cloudera cluster version to the correct Query Pulse platform type, which determines the appropriate JDBC driver:

    Cluster Version Platform Type JDBC Driver
    CDH 5.x CDH 5 CDH 5 Hive
    CDH 6.x CDH 6 CDH 6 Hive
    CDP 7.0.x CDP 7.0 CDP 7.1.9 Hive
    CDP 7.1.x CDP 7.1 CDP 7.1.9 Hive
    CDP 7.1.9 SP1 CDP 7.1.9 SP1 CDP 7.1.9 Hive
    CDP 7.2.x CDP 7.2 CDP 7.3.1 Hive
    CDP 7.3.x CDP 7.3 CDP 7.3.1 Hive

    Note

    Ensure the corresponding JDBC driver is installed (via Settings > Admin > JDBC Drivers) before testing imported connections.

    Kerberos Handling¶

    When discovery detects that hive_server2_authentication is set to KERBEROS, it:

    1. Extracts the service principal from the kerberos_princ_name configuration
    2. Reads the Kerberos realm from Cloudera Manager's SECURITY_REALM setting
    3. Constructs the full principal (e.g., hive/_HOST@EXAMPLE.COM)
    4. Includes the principal in the generated JDBC URL

    At import time, you can associate a Kerberos Configuration (keytab + principal) with the imported connections. If no Kerberos configuration is available, the connection is still imported with a warning — you can configure Kerberos authentication later in the Connections panel.

    SSL/TLS Handling¶

    When ssl_enabled is detected in the Hive service configuration, imported connections include SSL parameters in the JDBC URL. The exact parameters depend on the transport mode:

    # Binary transport with SSL
jdbc:hive2://host:10000/default;ssl=true;principal=hive/_HOST@REALM

# HTTP transport with SSL
jdbc:hive2://host:10001/default;ssl=true;transportMode=http;httpPath=cliservice;principal=hive/_HOST@REALM

    If your HS2 instances use certificates from a private CA, you may need to configure a TLS truststore in the connection settings after import.

    Troubleshooting¶

    Discovery returns no endpoints¶

    • Verify the CM connection test passes
    • Confirm the cluster has a Hive or Hive on Tez service
    • Check that the CM API user has read access to cluster configurations

    "HTML login page" error¶

    The CM URL may be incorrect or the credentials are invalid. Cloudera Manager returns an HTML login page instead of a JSON API response when authentication fails. Verify:

    • The URL points to the CM API (typically port 7180 or 7183)
    • The username and password are correct
    • The user has API access permissions

    Imported connection fails to connect¶

    • Ensure the correct JDBC driver is installed for the detected platform
    • For Kerberos connections, verify a valid Kerberos configuration is associated and the keytab/principal are correct
    • For SSL connections, check that the client trusts the HS2 server certificate
    • Test network connectivity to the HS2 host and port
