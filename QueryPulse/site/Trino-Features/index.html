    Trino Features¶

    Query Pulse provides comprehensive support for Trino with real-time query statistics, configurable explain plans, SQL validation, and detailed operator-level analytics.

    Real-Time Query Statistics¶

    When executing queries against Trino connections, Query Pulse displays live query statistics from the Trino Query Stats API.

    Progress Display¶

    The progress display shows real-time updates during query execution:

    • Query State: Current execution state (QUEUED, PLANNING, RUNNING, FINISHING, FINISHED, FAILED)
    • Progress Percentage: 0-100% completion indicator
    • Elapsed Time: Total time since query started

    Query State Indicators¶

    States are color-coded for quick identification:

    State Color Description
    QUEUED Gray Query waiting in queue
    PLANNING Blue Query plan being generated
    RUNNING Blue Query actively executing
    FINISHING Blue Query completing final operations
    FINISHED Green Query completed successfully
    FAILED Red Query encountered an error

    Task Metrics¶

    Real-time task execution statistics:

    Metric Description
    Total Tasks Number of tasks created
    Running Tasks Currently executing tasks
    Completed Tasks Successfully finished tasks
    Failed Tasks Tasks that encountered errors

    Driver Metrics¶

    Detailed driver execution information:

    Metric Description
    Total Drivers Total drivers created
    Running Drivers Currently executing drivers
    Queued Drivers Drivers waiting to execute
    Blocked Drivers Drivers waiting on resources
    Completed Drivers Successfully finished drivers

    Data Processing Metrics¶

    Live data throughput statistics:

    Metric Description
    Physical Input Data Bytes read from storage
    Physical Input Positions Rows read from storage
    Processed Input Data Bytes processed after filtering
    Processed Input Positions Rows processed after filtering

    Timing Metrics¶

    Comprehensive timing breakdown:

    Metric Description
    Elapsed Time Total wall-clock time
    Execution Time Time spent executing
    Queued Time Time spent in queue
    Analysis Time Time for query analysis
    Planning Time Time for distributed planning
    CPU Time Total CPU time consumed
    Scheduled Time Total scheduled time
    Blocked Time Time waiting for resources

    Memory Metrics¶

    Memory consumption tracking:

    Metric Description
    Peak User Memory Maximum user memory used
    Peak Total Memory Maximum total memory used
    Current Total Memory Current memory reservation

    Explain Plan Features¶

    Query Pulse provides a configurable explain interface for Trino with multiple type and format options.

    Using Trino EXPLAIN¶

    How to Use: 1. Write or select a query in the SQL editor 2. Click the Trino EXPLAIN button (tree icon) 3. A configuration popup appears 4. Select TYPE and FORMAT options 5. Click "Execute"

    Explain Types¶

    Type Description
    LOGICAL Shows the logical execution plan
    DISTRIBUTED Shows the distributed execution plan across nodes

    LOGICAL Plan: Displays the query plan at the logical level, showing: - Operator tree structure - Predicate expressions - Column projections - Join conditions

    DISTRIBUTED Plan: Shows how the plan is distributed across Trino workers: - Stage boundaries - Data exchange operations - Partition strategies - Remote exchanges

    Output Formats¶

    Format Description File Extension
    TEXT Human-readable text format .txt
    GRAPHVIZ DOT graph format for visualization .dot
    JSON Structured JSON format .json

    TEXT Format: Best for quick review and sharing: 

    - Output[columnA, columnB]
    - RemoteExchange[GATHER]
        - TableScan[table, ...]

    GRAPHVIZ Format: Best for visual analysis: - Creates a DOT file that can be rendered as a graph - Shows operator relationships visually - Useful for complex query plans

    JSON Format: Best for programmatic analysis: - Structured data format - Contains all plan details - Can be processed by external tools

    File Output¶

    Explain operations create files in the workspace:

    Combination File Pattern
    LOGICAL + TEXT {filename}_trino_explain_logical_{n}.txt
    LOGICAL + GRAPHVIZ {filename}_trino_explain_logical_{n}.dot
    LOGICAL + JSON {filename}_trino_explain_logical_{n}.json
    DISTRIBUTED + TEXT {filename}_trino_explain_distributed_{n}.txt
    DISTRIBUTED + GRAPHVIZ {filename}_trino_explain_distributed_{n}.dot
    DISTRIBUTED + JSON {filename}_trino_explain_distributed_{n}.json

    Files are automatically opened in new editor tabs for viewing.

    SQL Validation¶

    Trino provides a fast validation feature to check SQL syntax and semantics without executing the query.

    Using Validate SQL¶

    How to Use: 1. Write or select a query in the SQL editor 2. Click the Validate SQL button (checkmark icon) 3. Results appear in the results panel

    Validation Results¶

    Valid Query: - Green panel displays "Valid" - Confirms query syntax is correct - Confirms all referenced objects exist - Confirms user has required permissions

    Invalid Query: - Red panel displays "Invalid" - Query has syntax errors, or - Referenced objects don't exist, or - User lacks required permissions

    Use Cases¶

    • Rapid Development: Check syntax while writing queries
    • Pre-Execution Check: Validate before running expensive queries
    • Permission Testing: Verify access without consuming resources
    • CI/CD Integration: Validate SQL files before deployment

    Command Executed: EXPLAIN (TYPE VALIDATE) <query>

    Execution Details¶

    After query completion, detailed execution information is available in the Execution Details panel.

    Query Statistics Grid¶

    Statistics are organized into logical groups:

    Task Progress: - Total, Completed, Running, Failed task counts

    Driver Progress: - Total, Completed, Running, Queued, Blocked driver counts

    Data Processing: - Physical and processed input sizes - Row counts at each stage

    Timing: - Complete timing breakdown by category

    Memory: - Peak and current memory usage

    Operator Summaries¶

    Detailed per-operator execution statistics table:

    Column Description
    Stage Stage.Pipeline.Operator ID
    Operator Operator type name
    Drivers Number of drivers for operator
    Input Input data size and rows
    Output Output data size and rows
    CPU Time CPU time consumed
    Peak Memory Maximum memory used

    Expandable Operator Details¶

    Click any operator row to see additional metrics:

    Input Metrics: - Physical input data size - Physical input positions (rows) - Processed input data size - Read time - Add input time

    Output Metrics: - Output data size - Output positions (rows) - Get output time

    Resource Usage: - Peak user memory - Peak total memory - Blocked time - Spilled data size (highlighted if > 0)

    Spill Detection¶

    When an operator spills data to disk, it's highlighted as a performance concern: - Spilled data size shown in warning color - Indicates memory pressure during execution - Suggests potential for query optimization

    Supported Query Types¶

    Explain and validation features are available for the following statement types:

    • SELECT
    • INSERT
    • UPDATE
    • DELETE
    • MERGE

    DDL statements (CREATE, ALTER, DROP) are not supported for explain operations.

    Variable Substitution¶

    All explain and validation operations support workspace and file variables:

    -- Using variables in queries
SELECT * FROM ${catalog}.${schema}.${table} WHERE date = '${run_date}'

    Variables are substituted before the command is executed. If undefined variables are detected, a dialog prompts for values.

    Best Practices¶

    1. Use LOGICAL EXPLAIN to understand the query structure and optimizer decisions
    2. Use DISTRIBUTED EXPLAIN when investigating parallelism and data distribution issues
    3. Use GRAPHVIZ format for complex queries to visualize the plan
    4. Use JSON format when you need to process the plan programmatically
    5. Use Validate SQL frequently during query development for rapid feedback
    6. Monitor Blocked Time in execution details - high values indicate resource contention
    7. Watch for Spilled Data - indicates memory pressure and potential optimization opportunities
    8. Review Operator Summaries to identify bottleneck operators in slow queries
